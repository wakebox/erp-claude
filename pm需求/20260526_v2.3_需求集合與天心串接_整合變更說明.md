### 需求 1：食材庫存類型管理與前端顯示補齊

#### 業務說明

食材依儲存特性分為兩類：
- **SHORT（短效期）**：如麵包、生鮮蔬菜，每次都需要根據銷售預測補貨
- **LONG（長效期）**：如冷凍肉品、醬料桶，一般門市會先叫貨保持庫存，只有快用完（當前庫存低於安全存量）時才需要補貨

此屬性是食材本身的固有特性，應由食材主檔統一管理。

#### 修改一：PDM 食材維護作業（主檔管理層）

| 修改項目 | 說明 |
|---------|------|
| 新增「庫存類型」欄位 | 下拉選單，選項：SHORT（短效期）/ LONG（長效期） |
| 欄位為必填 | 每筆食材主檔都必須設定庫存類型，不允許空值 |
| 列表頁新增顯示欄位 | 食材列表補顯示庫存類型，方便批量確認 |

#### 修改二：需求集合 BOM 試算明細頁（顯示層）

後端已依食材庫存類型做出不同預測邏輯，並已回傳對應旗標，但前端目前完全未取用。需補齊以下顯示：

| 修改項目 | 顯示條件 | 說明 |
|---------|---------|------|
| 庫存類型標籤 | 所有食材 | 顯示 `SHORT` / `LONG` 標籤 |
| 「判斷庫存」標示 | `storageType = LONG` 且庫存充足 | 在需求數量欄位顯示灰色「判斷庫存」文字，取代數字 |
| 安全存量警示圖示 | `storageType = LONG` 且庫存不足 | 顯示橘色警示圖示，並顯示當前庫存數值 |

### 需求 3：萬元用量拆分員工餐，員工餐欄位落 DB

#### 現行問題

歷史銷售資料混入損耗與員工餐消耗量，萬元用量偏高，無法正確反映純銷售行為所需的食材用量。

#### 修改方向

**新零售端（協調事項，非本文件修改範圍）**

員工餐用量來源為 POS 端收單時標記為員工餐的銷售記錄，由新零售端傳入時已拆分為一般銷售數量（用於萬元預測計算）與員工餐數量（獨立欄位）。

**後端修改（`crg_demand_forecast_detail` 新增欄位）**

| 新增欄位 | 型別 | 說明 |
|---------|------|------|
| `employee_meal_weekday` | DECIMAL(15,4) | 平日員工餐萬元用量 |
| `employee_meal_holiday` | DECIMAL(15,4) | 假日員工餐萬元用量 |

員工餐用量需落 DB，以便後續需求預估查詢與對帳使用。

**前端修改（BOM 試算明細頁）**

| 新增欄位 | 說明 | 編輯態 |
|---------|------|-------|
| 員工餐用量（平日） | 平日員工餐消耗萬元用量 | 唯讀 |
| 員工餐用量（假日） | 假日員工餐消耗萬元用量 | 唯讀 |

萬元預測的計算基礎改為只使用一般銷售數量，員工餐用量僅供顯示參考，不納入計算。

### 需求 4：多廠商對應說明（不修改系統）

**結論：不需修改系統，提供使用說明。**

**使用說明文字（供 BK 使用者參考）**：

> 當同一品號有多家廠商供貨時，請至「採購管理 → 廠商報價維護」，在各廠商的報價明細中設定「適用門市」欄位，指定該廠商供應的門市範圍。設定完成後，系統在進行食材需求預測試算時，會自動依每家門市對應到正確的廠商報價，確保建議訂貨量的箱規換算準確，門市也能正確判斷應向哪家廠商下單。

#### 延伸議題：多廠商均值成本計算（天心串接相關）

BK 反映同一物料若有不同單價的多家廠商（如麵包有兩個單價不同廠商），會因非門市因素造成成本差異，影響 PLV（損益）準確度。此問題與天心財務系統的成本計算架構相關：

- 力新端目前以「適用門市」設定對應單一廠商，單價取對應廠商的最新報價
- 若天心端需要以「均值成本」計算多廠商品項的成本，則需要在天心技術規格會議中確認均值成本的計算規則與實作方式

> 待確認 #T6-01：多廠商品項的均值成本計算規則——天心端是否支援同一品號多廠商的均值成本計算？力新端是否需要在傳送進貨單時附加額外的成本資訊？（PM + 天心技術規格會議）

### 需求 5：建議訂貨量計算公式

#### 公式

```
建議訂貨量 = (萬元需求量 × 預估營業額) - 盤點量 - 在途量
建議訂貨量最小值 = 0（不允許負數）
```

#### 在途量定義（依使用者角色）

| 使用者角色 | 在途量定義 | 資料來源 |
|-----------|-----------|---------|
| 門市 | 已向倉儲（MSS）下物流單，但貨物尚未到達門市的數量 | 物流模組，已有物流單但 `actual_arrival_date` 尚未填入的明細數量加總 |
| 總部 | 已向供應商下採購單，但貨物尚未送達（含尚未驗收入庫）的數量 | 採購模組，結轉驗收單明細的 `transit_qty` |

#### 門市微調上下限

門市人工微調訂貨量的上下限規則，以**系統參數**設定，不寫死於程式碼中，方便日後業務調整。

**新增系統參數**：

| 參數名稱 | 說明 | 建議預設值 |
|---------|------|---------|
| `ORDER_QTY_ADJUST_MAX_RATIO` | 最終訂貨量不得超過建議量的倍數 | 1.5（建議量的 150%） |
| `ORDER_QTY_ADJUST_MIN` | 最終訂貨量最小值 | 0 |

#### 後端修改

**`DemandForecastServiceImpl`**：SHORT 食材補齊庫存查詢，延伸現有 LONG 食材的查詢邏輯。

**`RawMaterialDemandHeadServiceImpl`**：品號合併後新增計算步驟：

```
依使用者角色取對應在途量
在途量 + 盤點量取得後：
建議量 = MAX(需求量 - 盤點量 - 在途量, 0)
```

#### 資料表新增欄位（`pdm_raw_material_demand_detail`）

| 欄位 | 型別 | 說明 |
|-----|------|------|
| `stock_qty` | DECIMAL(15,4) | 試算當下盤點量（snapshot） |
| `transit_qty` | DECIMAL(15,4) | 試算當下在途量（snapshot） |
| `suggested_order_qty` | DECIMAL(15,4) | 系統建議訂貨量 |
| `manual_order_qty` | DECIMAL(15,4) | 門市人工微調後最終訂貨量（null = 採用建議值） |

#### 前端新增欄位（非 BOM 食材明細表）

| 欄位 | 顯示名 | 編輯態 | 備註 |
|-----|-------|-------|------|
| `stockQty` | 盤點量 | 唯讀 | |
| `transitQty` | 在途量 | 唯讀 | |
| `suggestedOrderQty` | 建議訂貨量 | 唯讀 | |
| `manualOrderQty` | 最終訂貨量 | 可編輯 | 上下限由系統參數控制，超出時觸發防呆提示 |

### 需求 8：盤點/訂貨單位統一換算

#### 設計說明

系統已有廠商報價維護（#28）的以下欄位：
- `singlePackCount`：單一包裝量（即箱規，例如一箱 10 公斤）
- `pdm_unit_conv`：單位換算維護表，支援 unit → g / ml 的換算

因此不需要在食材主檔另外新增「門市操作單位」與「換算係數」欄位，直接使用廠商報價既有資料即可。

#### 後端修改

**整區模式補廠商查詢**：整區模式（未指定 storeId）時，以區內各店分別查廠商報價的 `singlePackCount`，計算各店的箱換算後回傳。

**門市操作量計算公式**：

```
基本換算（箱）：
門市操作量（箱）= suggestedOrderQty ÷ singlePackCount（四捨五入至整數）

細部換算（需走 pdm_unit_conv）：
門市操作量（其他單位）= suggestedOrderQty × (unit_conv_ratio) ÷ singlePackCount
```

#### 前端新增欄位（非 BOM 食材明細表）

| 欄位 | 顯示名 | 編輯態 | 說明 |
|-----|-------|-------|------|
| `operationQty` | 門市操作量 | 唯讀 | 顯示換算後數量 + 單位（如：5 箱） |

## 第二部分：天心財務系統串接

---

### 串接架構說明

天心財務系統負責成本計算與產出門市損益表。力新 ERP 需向天心輸出三類資料：

| 資料類型 | 觸發時機 | 力新來源 | 天心用途 |
|---------|---------|---------|---------|
| 基礎主資料（品號 + 門市） | 新增/停用時 | PDM 食材主檔、門市主檔 | 天心建立貨品與部門主檔 |
| 進貨單 / 調撥單 | 驗收確認歸檔後 | PMM 驗收確認 → WHS 入庫 | 天心計算進貨成本 |
| 銷貨單 + 組合單（BOM） | 每日批次 | POS 中繼系統 + PDM BOM 主檔 | 天心扣庫存並計算銷貨成本 |

**天心確認事項**（2026-05-14 會議 + 後續回覆）：
- 天心建議貨品代號與力新品號一致
- 即使代號一致，仍建議維護品號對應表，為未來擴充預留彈性
- 每張銷貨單品項需附對應組合單（BOM）
- 基礎資料需串接同步

### 模組 L0-1：物流管理行事曆收貨作業優化

#### 業務需求

BK 門市反映：收貨時只需要填寫「缺少的品項與數量」，其餘品項應視為自動完成（全數到貨），不需要逐列輸入實際到貨數量，以減少操作時間。

#### 修改內容

**現行設計**：門市收貨時需逐列填入每個品項的實際到貨數量，再按「儲存」建立入庫單。

**變更後設計**：

| 情況 | 操作方式 |
|-----|---------|
| 全數到貨的品項 | 系統自動帶入預計配送數量作為實際到貨數量，門市不需逐列輸入 |
| 有缺少或異常的品項 | 門市只需針對缺少或數量不符的品項修改實際到貨數量 |

**前端修改**：
- 點圓點載入明細時，「實際配送數量」欄位預設帶入「預計發送配送數量」
- 門市只需修改有差異的列，其餘維持預設值
- 按「儲存」時邏輯不變，仍以實際配送數量欄位的值建立入庫單

**修改原因**：減少門市每日收貨的操作負擔，符合 BK 門市實際作業習慣——貨品大多全數到貨，只有例外情況才需要修改數量。

### 模組 L0-2：當日進貨即時導入與入帳時效

#### 業務需求

BK 門市反映兩個相關問題：
1. 當日進貨貨單應可即時導入系統，不需事後補登
2. 進貨入帳應在進貨當天全數完成，否則遇到月/週盤時，若有當期進貨品項未入帳，盤點差異無法即時處理

#### 現行狀態

目前物流管理行事曆（#48）前端已支援點圓點後建立入庫單的操作路徑，但後端 PRD 尚未將「從物流管理行事曆觸發入庫」這條寫入路徑納入規格，驗收確認作業（#35）的入庫時效也沒有明確的時間要求。

#### 修改內容

**後端修改（物流管理行事曆 → 驗收確認作業）**

確認並補齊「從物流管理行事曆按儲存 → 觸發建立入庫單」的完整寫入路徑規格，包含：
- 入庫單建立時間點：門市在物流管理行事曆填入實際到貨數量並按「儲存」時，當下即建立入庫單並更新 WHS 庫存
- 入庫單的時間戳記：以門市實際按儲存的時間為準，確保入帳日期等於進貨日期

**業務規則補充**

| 規則 | 說明 |
|-----|------|

| 盤點前截止時間 | 月/週盤前，系統應提示門市確認當期所有進貨品項是否已完成入帳，避免盤點差異失準 |

### 模組 L0：門市在途量定義說明

#### 業務說明

門市視角的在途量需依配送來源拆分為兩種，判斷方式與最終天心進貨單的呈現方式如下：

#### 在途量定義拆分

| 配送來源 | 在途量定義 | 判斷依據 |
|---------|-----------|---------|
| MSS 倉配送（中央倉） | 已產出 CSV 上傳至 SFTP，但門市尚未確認收貨 | 物流串接記錄已存在，但物流管理行事曆尚未填入實際到貨數量（`actual_arrival_date` 為空） |
| 廠商直送 | 已下採購單，廠商尚未送達門市 | 採購模組結轉驗收單明細的 `transit_qty` |

#### 天心進貨單數量的最終依據

不論是 MSS 配送還是廠商直送，天心進貨單的數量都以**門市實際確認收貨的數量**為準，也就是門市在物流管理行事曆按「儲存」後建立的入庫單數量，而不是預計配送數量。

```
門市確認收貨（物流管理行事曆按「儲存」）
  → 建立 WHS 入庫單（實際到貨數量）
  → 非同步推送天心進貨單
      → MSS 配送：配送方 = 總公司（中央倉）
      → 廠商直送：配送方 = 外部廠商
      → 兩者金額與數量皆以入庫單為準
```

### 模組 T3：進貨單推送（PMM 驗收確認）

#### 進貨單配送方說明

對門市而言，不論貨物來源為何，在天心都是**進貨單**。各門市與總公司各有獨立統編，屬於不同法人，因此總倉配送到門市也是法人間的進貨行為，而非同法人內部的調撥。

兩種來源的差別只在於**配送方**：

| 進貨來源 | 天心單據類型 | 配送方 |
|---------|-----------|------|
| 供應商直送到門市 | 進貨單 | 外部供應商 |
| 從總倉（MSS）物流配送到門市 | 進貨單 | 總公司（中央倉，獨立統編） |

**判斷邏輯（已確認）**：廠商報價明細的 `useDeliveryType` 欄位對應 `pdm_logistics_type` 物流類型主檔（#23）。推送天心時依此欄位判斷配送方：
- 物流類型為「直送」→ 配送方為外部供應商
- 物流類型為「週配/月配」（中央倉配送）→ 配送方為總公司（中央倉）

#### 推送流程

```
驗收確認歸檔
  → processStockIn（現有流程）
  → 非同步觸發天心進貨單推送
      → 取驗收明細（品號、入庫數量、倉庫代號）
      → 查廠商報價明細的 useDeliveryType
        → 對應 pdm_logistics_type 判斷配送方
        → 直送 → 配送方 = 外部供應商（廠商代號）
        → 週配/月配 → 配送方 = 總公司（中央倉統編）
      → 查對應表（ext_prod_code_mapping）取天心貨品代號
      → 取廠商報價明細的 latestQuotePerKgL（最新報價 NTD/單位）組裝金額
      → 組裝天心進貨單（含配送方資訊與金額）
      → 呼叫天心 API 推送
      → 寫入天心串接記錄（模組 T5）
```

> **已確認**：廠商報價明細的 `latestQuotePerKgL`（最新報價 NTD/單位）欄位存在且可落 DB，計算公式為 `latestQuotePerPack ÷ singlePackCount`，由前端在兩欄皆有值時自動觸發後端計算並持久化，天心進貨單金額可直接取用此欄位。

### 模組 T4：銷貨單 + 組合單每日推送

#### 每日推送內容

每門市每日推送兩張銷貨單：
- **一般銷售銷貨單**：含各品項銷售數量 + 各品項組合單
- **員工餐銷貨單**：含員工餐品項數量 + 各品項組合單

**組合單（BOM）產出**：以單品代號查詢食材清單與標準用量，新增一支格式化輸出 API。

**套餐處理**：力新以單品展開後傳入，套餐僅作虛設代號（無金額、無 BOM），各子品項帶金額與各自的組合單。
