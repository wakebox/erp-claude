# PRD｜PDM — 單品維護作業

> 來源：逆向自 `kingmaker-module-pdm` 之食譜（recipe）相關後端程式碼（含 `controller/admin/recipe/PdmRecipeController.java`、`service/recipe/PdmRecipeServiceImpl.java`、`dal/.../recipe/PdmRecipeDO.java`、`service/recipe/listener/PdmRecipeStatusListener.java` 等）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

> **本 PRD 範圍**：本程式內部命名為「食譜（recipe）」，業務語意上對應「單品」——指漢堡王菜單上的一個販售品項（一個漢堡、一份雞塊、一杯飲料），背後關聯一份食材配方（單份用量配方）。本 PRD 以「單品」為主稱呼。

---

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 PDM 的**單品研發人員**，所屬部門通常是商品部或研發部。
公司每幾週要上一個新單品（例如「黑松露安格斯堡」「夏季芒果冰沙」），背後牽動：用了哪些食材、各用多少、烹煮怎麼做、料理訣竅是什麼、單份成本多少、營養成分是什麼。我把這些資料完整建檔到 ERP 後，採購才知道要備什麼料、需求預測才能算每店該進多少、店長才能在門市照配方做菜。

### 1.2 我要做什麼

- **新增單品**：填主檔（單品類別、單品代號、顯示名稱、品號、餐食類型、烹煮方式等）＋ 子表（單份用量配方：每樣食材用多少、什麼單位）＋ 營養成分子表。
- **修改單品**：在簽核完成「歸檔」前可以反覆編輯；歸檔後就鎖住。
- **追蹤審批進度**：在「待辦」頁看自己參與的單品單據目前的流程狀態。
- **刪除單品**：取消開發或測試資料可刪；連同單份用量配方子表一起清掉。
- **匯出**：依目前篩選匯出單品主檔 Excel。
- **查詢**：條件查詢含主檔多個欄位（單品類別、顯示名稱、品號、餐食類型等）。
- **取得全部單品（簡易清單）**：給需求預測、BOM、報價等下游模組做選項用。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一筆單品要能掛多筆「食材＋用量＋單位」 | BOM 體系核心：一份漢堡用幾克牛肉、幾克生菜、幾片麵包都要建檔 |
| 每筆食材用量要能帶「標準用量成本」 | 算單份標準成本必要 |
| 系統幫我抓「最新報價」「最近一次採購價」 | 成本變動隨時可見、不用人工抄 |
| 系統自動產出單據編號 | 不用人工編號 |
| 走簽核流程，建立時自動觸發 | 確保新單品經商品、研發、採購、財務核可 |
| 已歸檔的不能再隨便改 | 維護資料可信度 |
| 「料理步驟」「料理訣竅」要能存富文本 | 給門市員工照做的參考 |
| 顯示名稱、標籤名稱要分開存 | 行銷對外用語與內部分類用語可能不同 |

## 2. 功能目的

提供一套**主檔加兩張子表、走簽核流程歸檔生效、編號自動產出**的單品（菜單品項）管理機制。設計上：

1. **一筆主檔**統整一個販售品項的基本資料（顯示名稱、品號、餐食類型、單份標準成本、烹煮方式等）。
2. **單份用量配方子表（BOM 配方）**：每樣構成此單品的食材＋用量＋單位＋標準用量成本，是 BOM 體系最核心的關聯。
3. **營養成分子表**：每樣營養成分定義對應的每份含量，用於菜單卡片、法規揭示。
4. **單據編號**靠「選單管理」之取號機制產出。
5. **簽核流程**由「選單管理」綁定；歸檔後鎖住不可改。

下游模組（需求預測、報價、採購、店長備料）都以「已歸檔的單品＋其單份用量配方」作為計算基礎。

> **重要過渡背景**：本功能屬於 PDM 業務模組，與 OA 整合過渡關聯較低。建立人員、簽核者來源仰賴 ERP 使用者主檔；階段 5 後改以 OA 為準（推論）。

## 3. 業務邏輯背景

### 3.1 「主檔 + 兩張子表」的關聯結構

```
單品主檔（pdm_recipe）
  ├── 1:N 單份用量配方（pdm_single_serving_recipe）   ← 每樣構成食材＋用量＋單位＋成本
  └── 1:N 食譜營養成分（pdm_recipe_nutritional_contents） ← 每樣營養成分定義＋每份含量
```

兩張子表的關聯欄位都是「單品主檔 ID（recipeId）」。

### 3.2 編碼層級

- **原則代碼（structure）**：使用者選；對應「編碼原則維護」
- **大類（mainCategory）**：使用者選
- **單品類別（itemCategory）**：使用者選；對應「編碼項目維護」中的單品類別
- **單品（itemName）**：使用者選；對應特定單品命名

> **註**：本程式**未寫死**原則代碼與大類（不像食材維護作業寫死為 5）。使用者可自由選擇。

### 3.3 流程狀態

同食材維護作業：「待處理 / 待簽核 / 待歸檔 / 已歸檔」四階段。

### 3.4 「已歸檔保護」（同食材維護作業）

更新時的驗證：

- 該單品必須存在，否則拋「食譜不存在」
- 若該單品**未綁流程實例 ID** 且 流程狀態 = 「已歸檔」 → 拒絕，拋「食譜已歸檔，不能再進行更新」

意即：曾走過流程的單品（已綁 processInstanceId）即便狀態為「已歸檔」**仍可被修改**（驗證寬鬆，與食材維護作業相同議題，列入 §11）。

### 3.5 單據編號的產出

新增時呼叫 `menuService.generateSignCode("單品維護作業")`：

- 由「單品維護作業」選單讀取取號規則
- 寫到主檔的 `signCode` 欄位

### 3.6 BPM 流程綁定

新增時呼叫 `menuFlowProcessInstanceHelper.createProcessInstanceIfFlowOpen()`：

- 由「單品維護作業」選單讀取「啟用流程＋流程 Key」
- 若啟用：發起 Flowable 流程實例、寫回主檔 `processInstanceId`
- 業務鍵格式：`recipe:{單品主檔 ID}`

### 3.7 簽核結果監聽

`PdmRecipeStatusListener` 監聽 BPM 流程狀態事件：

- 動態從「選單管理」取得當前流程定義 Key 比對
- 業務鍵前綴必須為 `recipe:`
- 結果為 APPROVE → 更新該單品流程狀態為「已歸檔」
- **不處理 REJECT、CANCEL**（程式中無對應分支，連註解都沒有）

### 3.8 子表更新採「先全刪、再批量重建」

修改時的單份用量配方處理：

- 若清單非空 → 先依 `recipeId` 刪除所有既有記錄、再批量寫入新清單
- 若清單為 null 或空陣列 → 不刪不重建

寫入前對每筆強制 `setId(null)`、`setUpdater(null)`、`setUpdateTime(null)`，目的：

- 避免主鍵衝突（前端可能帶上既有 ID）
- 確保 updateTime 由框架自動產生

> **註**：營養成分子表的更新流程**未實作於 Service 內**（程式中只有查詢、未見「保存/更新」邏輯）。是否表示子表只能由其他端點維護？列入 §11。

### 3.9 刪除採「主檔先、子表後」

- 刪除主檔
- 刪除單份用量配方子表

> **註**：營養成分子表的刪除流程**未實作**。刪除主檔後該單品的營養成分子表仍會殘留（推論：列入 §11）。

### 3.10 兩支查詢端點

| 端點 | 用途 |
|---|---|
| `/page` 一般分頁 | 維運盤點，支援眾多條件 |
| `/todo-page` 待辦分頁 | 簽核者作業視角，先從 BPM 拉「我有任務」的流程實例 ID 清單再過濾 |

外加 `/allRecipe` 取得全部單品的簡易資訊清單（給下游模組做選項用）。

### 3.11 匯出 Excel

`/export-excel` 強制 `pageSize = PAGE_SIZE_NONE`，撈全部後輸出 Excel。同其他模組的 OOM 風險（列入 §11）。

### 3.12 子表獨立查詢端點

主檔的 `/get` 僅回主檔資料，**不**回子表內容。要取得子表須另外呼叫：

- `/single-serving-recipe/list-by-recipe-id`：查單份用量配方清單
- `/getSingleServingRecipe`：查單份用量配方 DTO 清單（含 JOIN 來的資訊）
- `/recipe-nutritional-contents/list-by-recipe-id`：查營養成分清單

### 3.13 「狀態」是布林（true/false）

不同於食材維護作業的「啟停狀態用 Integer 0/1」，本表的 `status` 是 `Boolean`，意義推論為「啟用/停用」但實際語意需確認（列入 §11）。

### 3.14 與漢堡王中繼系統的關聯

主檔的 `recipeProductId` 欄位指向漢堡王中繼系統中的單品 ID。對接邏輯由 PDM 模組與中繼 API 對接，本 PRD 不展開（列入 §11）。

## 4. 情境說明

### 4.1 正常流程 — 新增一個單品

商品開發人員阿凱要建立「夏季芒果冰沙」：

**主檔**：
- 原則代碼、大類、單品類別、單品（依編碼）
- 顯示名稱：`夏季芒果冰沙`
- 標籤名稱：`夏限・冰沙系列`
- 品號：`PROD-MNGSHK-001`
- 餐食類型：飲料
- 單份用量：500（毫升）
- 用量單位：毫升
- 單份標準成本：35（NTD）
- 烹煮方式：「冷凍芒果丁打成冰沙」
- 料理步驟：（富文本）
- 料理訣竅：（富文本）

**單份用量配方 4 筆**：
- 芒果丁 100g、單位克、最新報價 12 NTD/kg、標準用量成本 1.2 NTD、預設廠商「廠商 A」
- 牛奶 200ml、單位毫升、最新報價 80 NTD/L、標準用量成本 16 NTD
- 糖漿 30ml、單位毫升、最新報價 50 NTD/L、標準用量成本 1.5 NTD
- 冰塊 200g（適量）

送出後系統：

1. 取號：「單品維護作業」選單 → `RCP260520001`
2. 主檔流程狀態 = 「待處理」
3. 寫入主檔
4. 寫入單份用量配方 4 筆（每筆綁主檔 ID）
5. 呼叫 BPM 判斷是否啟用流程 → 若啟用發起流程、寫回 `processInstanceId`
6. 回傳新主檔 ID

### 4.2 正常流程 — 簽核通過後自動歸檔

阿凱送出後 BPM 流程跑：商品主管 → 採購主管 → 財務主管，全數通過。BPM 發出 APPROVE 事件，listener 更新主檔流程狀態為「已歸檔」。下游模組可開始引用。

### 4.3 異常情境 — 簽核被退回（REJECT）

某筆單品送簽被財務主管退回。BPM 事件結果 = REJECT。
listener **無 REJECT 分支**（連 if-else 都沒寫），事件就被略過。
結果：流程狀態保留在當前值（如「待簽核」），使用者無法重新編輯送單（除非 IT 手動改流程狀態）。**比食材維護作業更糟**（食材至少有 if-else 但被註解，本模組連 if-else 都沒有）。列入 §11。

### 4.4 異常情境 — 已歸檔後想修改

阿凱想把已歸檔的單品的料理訣竅補一段話。送出更新：

- 系統檢查存在 → 是
- 檢查「未綁流程 + 已歸檔」→ 該單品已綁流程實例 ID → **不擋**
- 主檔更新成功

意即已歸檔的單品在「曾走過流程」的情境下實際上**仍可被修改**。列入 §11。

### 4.5 業務規則分流情境 — 編輯時子表為空

阿凱想清空某單品的所有單份用量配方（要重建）。前端把子表清單清空送出：

- 系統判斷 `isNotEmpty(子表清單)` → **false**（空陣列）→ 跳過子表更新整段邏輯
- 結果：原本的子表記錄**仍在**

意即要清空配方目前要靠後端 API 操作或單獨刪除每一筆。列入 §11。

### 4.6 業務規則分流情境 — 編輯時子表帶上原有 ID

阿凱編輯一個單品，前端把原有的子表清單（含每筆 ID）一起帶上。系統處理：

1. 先依 `recipeId` 刪除所有既有子表記錄
2. 批量插入新清單；但每筆寫入前 `setId(null)` 把 ID 清空 → 系統會用序列產生新 ID

結果：子表的「ID」會在每次更新時**改變**（即便內容沒變）。如果有其他模組以 `singleServingRecipe.id` 作為外鍵引用，會被切斷（推論，列入 §11）。

### 4.7 待辦情境 — 簽核者視角

商品主管小華開啟「單品待辦」頁，篩選 `待簽核`：

1. 服務呼叫 BPM 取「我有任務」的流程實例 ID 清單
2. 用此清單到主檔分頁查
3. 回傳

### 4.8 多來源情境 — 下游模組要選單品

需求預測模組要列出所有單品讓使用者選。呼叫 `/allRecipe` 取得簡易清單（`PdmRecipeRespLimitVO`）：

- 包含基本欄位（ID、顯示名稱、品號、餐食類型等）
- 不包含子表
- 供下游下拉用

> **註**：本端點未過濾「已歸檔」狀態，下游可能撈到尚在簽核中的單品。列入 §11。

### 4.9 異常情境 — 刪除單品但營養成分子表殘留

阿凱刪除某單品。Service 流程：

1. 驗證存在 → 通過
2. 刪除主檔
3. 刪除單份用量配方子表（依 recipeId）
4. **不刪營養成分子表**

結果：營養成分子表記錄留下「指向已刪除主檔」的孤兒資料。列入 §11。

### 4.10 富文本情境 — 料理步驟／訣竅

阿凱在料理步驟欄位用富文本編輯器寫了「1. 取冰塊 200g 放入果汁機 ...」並插入照片網址、加粗等樣式。系統將整段 HTML 字串存入 `cookingSteps` 欄位。後續顯示時前端解析並渲染。

> **註**：後端**未限制長度**，富文本可能塞入大量內容（含 base64 圖片），影響資料表大小（推論，列入 §11）。

## 5. 操作流程

### 5.1 新增單品

```
[填寫主檔 + 單份用量配方子表]
  │
  ▼
[權限檢查：是否具備「PDM-單品-新增」]
  │
  ▼
[呼叫選單取號：「單品維護作業」→ signCode]
  │
  ▼
[組裝主檔]
  ├─ 流程狀態 = 「待處理」
  └─ 其他依表單值
  │
  ▼
[寫入主檔 → 取得主檔 ID]
  │
  ▼
[單份用量配方子表（若非空）]
  ├─ 每筆綁主檔 ID
  └─ 批量寫入
  │
  ▼
[呼叫 BPM：判斷選單流程是否啟用]
  ├─ 啟用 → 發起流程實例 → 寫回 processInstanceId
  └─ 未啟用 → 略過
  │
  ▼
[（注意：營養成分子表的寫入未在 Service 內實作）]
  │
  ▼
[回傳新主檔 ID]
```

### 5.2 修改單品

```
[編輯表單，送出]
  │
  ▼
[權限檢查：是否具備「PDM-單品-修改」]
  │
  ▼
[驗證存在]
  │
  ▼
[已歸檔保護：未綁流程 + 已歸檔 → 拒絕]
  │
  ▼
[更新主檔]
  │
  ▼
[單份用量配方子表（若非空）]
  ├─ 依 recipeId 全刪
  ├─ 每筆 setId(null)、setUpdater(null)、setUpdateTime(null)
```

## 7. 商業邏輯

### 7.1 新增的處理順序

1. 取號
2. 流程狀態預設「待處理」
3. 寫主檔
4. 寫單份用量配方子表（若有）
5. 發起 BPM 流程（若選單啟用）
6. 整段包在 `@Transactional` 中（失敗整批回滾）

### 7.2 修改的處理順序

1. 驗證存在＋已歸檔保護
2. 更新主檔
3. 子表「全刪、重建」（若清單非空）
4. 整段包在 `@Transactional` 中

> **註**：本模組的新增、修改、刪除都有 `@Transactional`，與食材維護作業不同（食材模組無此標註）。資料一致性較佳。

### 7.3 簽核退回與取消「完全不處理」

listener 中**只寫了 APPROVE 分支**，沒有 REJECT、CANCEL 的 if-else。事件被略過時不會留任何痕跡。

### 7.4 子表更新的 ID 重新分配

更新子表時每筆 `setId(null)` 清空原有 ID，由序列重新分配。意即：

- 「同一個食材搭配同一個單品」的 BOM 配方 ID 會在每次更新時變動
- 若有別處引用此 ID 會被切斷

### 7.5 「狀態」是布林型態

`status` 欄位為 `Boolean`，意義推論為「啟用/停用」但語意需確認。

### 7.6 營養成分子表的處理範圍

從程式看：

- 主檔 CRUD 流程中**沒有**寫入／更新／刪除營養成分子表的邏輯
- 只有查詢端點 `getRecipeNutritionalContentsListByRecipeId`

這意味著營養成分子表的維護必須**透過其他方式**（推論：可能由獨立的 CRUD 端點維護、或目前還沒實作）。列入 §11。

### 7.7 取得單一單品時不回傳子表

`getRecipe(id)` 只回主檔資料。前端編輯時要取子表必須另呼叫子表端點。

## 10. 功能範圍

### 10.1 包含的功能

- 單品主檔與單份用量配方子表的一次性新增、修改、刪除
- 自動取號
- 自動發起 BPM 簽核（若選單啟用）
- 簽核通過後自動歸檔
- 已歸檔保護（同食材維護作業之寬鬆條件）
- 一般分頁查詢（多欄位）
- 待辦分頁查詢
- 取得單一主檔
- 取得單份用量配方子表（兩支端點：DO 版、DTO 版）
- 取得營養成分子表
- 取得全部單品簡易清單
- 匯出 Excel
- 完整的 `@Transactional` 包覆

### 10.2 預留但尚未實作

- **簽核退回／取消處理**：listener 中無 REJECT、CANCEL 分支
- **營養成分子表 CRUD**：Service 內未實作新增、修改、刪除
- **子表更新的差異化處理**：目前是「先全刪再重建」，無 diff 比對
- **後端欄位驗證**：原則代碼、大類、單品類別、單品、餐食類型、食材 ID 等皆未驗證存在性
- **富文本長度限制**：料理步驟／訣竅未設上限

### 10.3 不包含

- 食材本身的維護（在「食材維護作業」）
- 編碼類別／項目／原則的維護（在各自的維護作業）
- 營養成分定義的維護（在「營養成分定義維護」）
- 餐食類型字典的維護（在「餐食類型維護」）
- 單位字典的維護（在「單位定義維護」）
- BPM 簽核流程定義本身
- 取號規則設定（在「選單管理」）
