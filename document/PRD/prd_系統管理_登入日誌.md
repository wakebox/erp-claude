## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的**資安／IT 主管／稽核**。
每天都有人從不同 IP、不同瀏覽器嘗試登入 ERP，有人成功、有人輸錯密碼、有人帳號被停用還在試。當有「帳號被盜」「異常 IP 登入」「重複嘗試錯誤密碼」這類資安事件時，我得能從一筆筆登入紀錄裡看出來。

### 1.2 我要做什麼

- **每日資安例行盤點**：看昨天有多少次登入嘗試、多少筆失敗、有沒有同一帳號短時間內多次失敗（疑似暴力破解）。
- **異常 IP 調查**：發現某個帳號從不熟悉的 IP 登入，要查那個 IP 在系統裡的歷史登入足跡。
- **協助使用者**：員工說「我帳號好像登不進去」，要查他是密碼錯、驗證碼錯、還是帳號被停用、還是根本沒有這個帳號。
- **稽核需要**：外稽要求提供「某段期間特定使用者的登入紀錄」、「失敗登入清單」。
- **匯出佐證**：把查到的結果輸出 Excel。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 系統要把每次登入嘗試（含成功與失敗）都記下來 | 沒有完整紀錄就無法事後調查 |
| 失敗要分得清楚「是什麼原因失敗」 | 暴力破解、員工忘記密碼、被停用，要應對方式不同 |
| 每筆要有「時間、帳號、來源 IP、結果」 | 資安最低要求 |
| 同一筆登入嘗試帶 TraceId | 可以跟同次請求的 Spring Security 流程串起來查 |
| 帳號要冗餘儲存（不能等 join 使用者表） | 帳號可能改名、使用者可能被刪，登入紀錄要能獨立存活 |
| 可以匯出 Excel | 給稽核者 |
| 「成功」「失敗」要能快速切換查 | 日常盤點時聚焦失敗為主 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 在認證流程中自動寫入登入日誌（不需業務 Service 介入） | 不要靠人為操作 |
| 涵蓋多種登入類型：帳號密碼、社交、手機、簡訊 | 各種登入入口都要被記錄 |
| 涵蓋兩種登出類型：使用者主動登出、強制退出 | 完整的會話軌跡 |
| 記錄失敗原因（密碼錯、帳號停用、驗證碼錯、帳號不存在） | 區分業務原因 |
| 條件查詢：IP（模糊）、帳號（模糊）、結果（成功／失敗）、時間區間 | 維運盤點 |
| 匯出 Excel | 外稽佐證 |
| 登入成功時順帶更新使用者「最後登入 IP、時間」 | 帳號維護用 |

## 2. 功能目的

提供一套**完整、可信、可匯出**的登入軌跡紀錄機制，涵蓋登入與登出兩種行為、四種登入入口、兩種登出方式、五種登入結果。設計上分兩塊：

1. **寫入面（被動、由認證流程觸發）**：認證服務在驗證帳密、驗證碼、檢查停用狀態的每個分支結束時，呼叫日誌寫入。寫入是**同步的**（與「操作日誌」異步不同），確保任何登入嘗試一定留下紀錄。
2. **查詢面（主動、顯性）**：在系統管理介面提供分頁查詢與匯出。

> **重要過渡背景**：經系統整合決策（見 `erp-claude` 記憶 `project_user_integration_decision.md`，2026-05-16），漢堡王整體 IT 走向以 Laravel OA 為使用者單一事實來源、**ERP 不再自簽 JWT**。本模組與「ERP 自管登入流程」綁定極深：若搬遷階段 5 後 ERP 不再執行帳密驗證、改由 OA 簽 token 後 ERP 透過 token 認領使用者，本模組的「寫入面」會被大幅改寫或廢棄；「查詢面」可能保留以提供歷史軌跡，但新登入動作的記錄責任會轉移至 OA。本 PRD 描述的是過渡期內、ERP 仍自管登入流程時的需求。

## 3. 業務邏輯背景

### 3.1 「登入日誌」同時記錄「登入」與「登出」

雖然名為「登入日誌」，本表實際涵蓋兩類事件：

| 大類 | 細項代號 | 場景 |
|---|---|---|
| **登入** | 帳號登入（100） | 員工輸入帳號密碼登入 |
| | 社交登入（101） | 透過第三方社交帳號登入（例如綁定的微信、LINE） |
| | 手機登入（103） | 以手機號搭配密碼登入 |
| | 簡訊登入（104） | 以手機號 + 一次性簡訊驗證碼登入 |
| **登出** | 自己主動登出（200） | 使用者點「登出」 |
| | 強制退出（202） | 管理員強制把某個 token 失效 |

對於漢堡王 ERP 目前的場景，主要使用「帳號登入」與「自己主動登出」；其他類型為框架預留。

### 3.2 「登入結果」有 5 種

| 代號 | 意義 | 觸發點 |
|---|---|---|
| 0 | 成功 | 通過所有驗證 |
| 10 | 帳號或密碼不正確 | 帳號不存在 或 密碼不對 |
| 20 | 使用者被禁用 | 帳號通過密碼驗證但狀態為停用 |
| 30 | 圖片驗證碼不存在 | 驗證碼未取或過期 |
| 31 | 圖片驗證碼不正確 | 驗證碼輸入錯誤 |

「帳號不存在」與「密碼錯誤」在程式上共用 `10` 代碼，**對外不揭示具體原因**（避免攻擊者用「帳號是否存在」做枚舉攻擊）。

### 3.3 寫入是同步的

不同於「操作日誌」採異步寫入策略，本模組的寫入是**同步**的：

- 在認證流程的每個結束點，認證 Service 直接呼叫 `loginLogService.createLoginLog(reqDTO)`
- 完成後才執行其他動作（拋例外、產 token、回應前端）
- 寫入耗時計入登入回應時間
- 寫入失敗會中斷登入流程（推論：未見特殊處理，列入 §11）

設計理由：登入是稽核重點，丟失任何一筆都是資安風險，因此選擇同步以確保可靠。

### 3.4 帳號欄位冗餘儲存

帳號（username）欄位**直接存字串**，不是只存使用者編號。理由有二：

- 帳號可以變更（使用者管理允許改 username），改了之後舊的登入日誌仍應顯示當時的帳號值
- 使用者可能被刪除，連使用者主檔都不在了，登入紀錄要能獨立存活
- 帳號不存在的失敗情境（外人亂試）→ 沒有使用者編號可記，但要記下他試的是什麼字串

### 3.5 使用者編號可為空

當以下情境發生時，`userId` 欄位為空：

- 帳號不存在（沒查到使用者）
- 驗證碼錯誤（連帳號都沒驗證到）
- 過期登出由排程觸發（推論：來源 `LoginLogCreateReqDTO.java:31` 該欄位未標 `@NotNull`）

帳號欄位這時候會記下使用者輸入的字串（可能是真的帳號、可能是亂打的）。

### 3.6 查詢的「狀態」是布林轉成複雜條件

查詢頁面的「狀態」是一個 `Boolean`（成功 / 失敗）：

- 傳 `true` → SQL 條件 `result = 0`（僅成功）
- 傳 `false` → SQL 條件 `result > 0`（任何失敗類型都算）
- 不傳 → 不加條件，全撈

這代表「失敗」這個篩選不分原因，要看細項只能從結果欄位再篩。

### 3.7 登入成功的副作用

當登入結果 = 成功時，本模組的寫入流程**順帶**呼叫「使用者管理」更新使用者主檔的：

- 最後登入時間
- 最後登入 IP

讓使用者列表能顯示「最近一次登入時間」、便於 IT 識別休眠帳號。

### 3.8 帳號欄位的長度與「為何允許空」

DTO 上 `username` **未標** `@NotEmpty`。原因是會員側的社交登入剛綁定時，可能連 username 都還沒有，是後續流程才補回來。本模組向下相容此情境（程式註解明說「Member 社交登入時，此時暫時沒有 username」）。

### 3.9 瀏覽器 UA 允許空

DTO 上 `userAgent` 也允許空。原因：當登入日誌是由「排程觸發過期登出」寫入時，沒有 HTTP Request 可取得 UA。

### 3.10 IP 欄位是必填

不論何種觸發情境，IP 都必填。即使是排程任務也得帶一個來源 IP（例如本機 IP 或排程節點 IP），這是為了確保稽核軌跡完整。

### 3.11 匯出本身會被存取日誌記錄

「匯出登入日誌」端點掛了 `@ApiAccessLog(operateType = EXPORT)`，匯出動作會被另一張「存取日誌」表記下。意即：誰、何時、用什麼條件匯出了多少筆登入日誌，自己也會留下紀錄。

### 3.12 不通過 BPM 簽核

純查詢與匯出功能，與審批流無關。

## 4. 情境說明

### 4.1 正常流程 — 使用者成功登入

員工小李 9:00 上班，瀏覽器打開 ERP 登入頁、輸入帳號 `lee123`、密碼 `Bk2026!`、圖片驗證碼。送出後：

1. 驗證碼通過
2. 系統查使用者主檔 → 找到 `lee123`
3. 密碼比對 → 正確
4. 檢查帳號狀態 → 啟用中
5. **同步寫入登入日誌**：類型=帳號登入(100)、帳號=`lee123`、使用者編號=小李的編號、結果=成功(0)、IP=`10.1.2.30`、UA=`Mozilla/5.0...`、TraceId=`xyz-001`
6. **更新使用者主檔**：最後登入時間=`2026-05-19 09:00:12`、最後登入 IP=`10.1.2.30`
7. 簽發 JWT token
8. 回應前端

### 4.2 異常情境 — 密碼錯誤

小李重新整理頁面後手滑密碼打錯 `Bk2025!`：

1. 驗證碼通過
2. 查使用者主檔 → 找到 `lee123`
3. 密碼比對 → 不正確
4. **同步寫入登入日誌**：類型=帳號登入(100)、帳號=`lee123`、使用者編號=小李的編號、結果=帳號或密碼不正確(10)、IP=`10.1.2.30`
5. 拋例外「帳號或密碼不正確」給前端

> **註**：密碼錯誤時 `userId` 仍會帶上（已經找到使用者）。下一個情境的「帳號不存在」則 `userId` 為空。

### 4.3 異常情境 — 帳號根本不存在（疑似嘗試攻擊）

外部攻擊者亂打帳號 `admin123`：

1. 驗證碼通過
2. 查使用者主檔 → **找不到** `admin123`
3. **同步寫入登入日誌**：類型=帳號登入(100)、帳號=`admin123`、使用者編號=**空**、結果=帳號或密碼不正確(10)、IP=`203.0.113.5`
4. 拋例外「帳號或密碼不正確」給前端（與密碼錯誤同樣的訊息，避免揭示帳號是否存在）

資安主管事後查詢時可篩出「結果=失敗 且 使用者編號為空」找出這類「亂試帳號」的紀錄。

### 4.4 異常情境 — 帳號被停用

離職員工小陳的帳號被 IT 停用，他想嘗試登入：

1. 驗證碼通過
2. 查使用者主檔 → 找到帳號
3. 密碼比對 → 正確（他知道自己舊密碼）
4. 檢查帳號狀態 → 停用
5. **同步寫入登入日誌**：類型=帳號登入(100)、帳號=`chen001`、使用者編號=小陳的編號、結果=使用者被禁用(20)、IP=`...`
6. 拋例外「帳號已被停用」

### 4.5 異常情境 — 驗證碼錯誤

員工小華輸了正確的帳密、但驗證碼看錯：

1. 驗證碼比對 → 失敗
2. **同步寫入登入日誌**：類型=帳號登入(100)、帳號=`hua005`、使用者編號=**空**（還沒走到查使用者那步）、結果=圖片驗證碼不正確(31)、IP=`...`
3. 拋例外「驗證碼錯誤」

> **註**：驗證碼錯誤時即便帳號是真實存在的，`userId` 仍為空。因為認證流程的順序是「先驗證碼、再查使用者」。

### 4.6 正常流程 — 使用者登出

小李工作完點「登出」：

1. 認證服務驗證 token、找出對應使用者
2. **同步寫入登入日誌**：類型=自己主動登出(200)、帳號=`lee123`、使用者編號=小李的編號、結果=成功(0)、IP=`...`
3. 註銷 token

### 4.7 正常流程 — 強制登出

資安主管小華發現小陳的帳號異常活動，到「線上使用者」頁面把該帳號強制登出：

1. 認證服務找出該 token 對應的使用者
2. **同步寫入登入日誌**：類型=強制退出(202)、帳號=`chen001`、使用者編號=小陳的編號、結果=成功(0)、IP=當下小陳所在 IP（推論）
3. 註銷 token，小陳下次操作 ERP 會被踢回登入頁

### 4.8 排程觸發情境 — token 過期登出

某個 token 因到期被排程清理：

1. 排程任務遍歷過期 token
2. 對每個 token 對應的使用者 **同步寫入登入日誌**：類型=強制退出(202)、結果=成功(0)、IP=**本機 IP**（排程節點 IP）、UA=**空**（無 Request）
3. 刪除 token

### 4.9 資安例行盤點情境 — 查昨日所有失敗

資安主管小華每天早上開啟登入日誌頁，輸入：

- 帳號：留空
- IP：留空
- 狀態：失敗
- 建立時間：`2026-05-18 00:00:00 ~ 2026-05-18 23:59:59`

送出。系統依條件分頁查詢（依日誌編號倒序），回傳 47 筆失敗紀錄。小華目視掃過去，注意到有 1 個 IP `203.0.113.5` 在 03:00–04:00 間出現 23 次失敗（同一 IP、不同帳號），判定為疑似暴力破解，立刻封 IP。

### 4.10 異常 IP 調查情境

小華接到通報，某帳號從不熟悉的 IP 登入。她在登入日誌查詢頁：

- IP：`203.0.113.99`（模糊比對，會抓到 `203.0.113.99` 開頭的所有 IP）
- 其他條件留空

回傳 8 筆紀錄：5 筆失敗（試了 5 個帳號）、3 筆成功（其中 1 筆帳號是 CFO）。小華立刻通知 CFO 改密碼並走帳號異動流程。

### 4.11 匯出佐證情境

小華要把上述調查結果存證，按「匯出」：

1. 系統檢查權限「登入日誌-匯出」
2. 強制不分頁、撈全部符合條件
3. 輸出 Excel：`登录日志.xls`
4. 本次匯出動作被存取日誌記下

## 5. 操作流程

### 5.1 登入時的日誌寫入（由認證服務觸發）

```
[使用者點「登入」]
  │
  ▼
[認證服務 AdminAuthServiceImpl.login]
  │
  ▼
[1. 驗證碼檢查]
  ├─ 失敗 → 同步寫日誌（驗證碼錯）+ 拋例外
  └─ 通過 ▼
  │
  ▼
[2. 查使用者主檔]
  ├─ 不存在 → 同步寫日誌（密碼不正確、userId 空）+ 拋例外
  └─ 存在 ▼
  │
  ▼
[3. 密碼比對]
  ├─ 不正確 → 同步寫日誌（密碼不正確、有 userId）+ 拋例外
  └─ 正確 ▼
  │
  ▼
[4. 帳號狀態檢查]
  ├─ 停用 → 同步寫日誌（使用者被禁用）+ 拋例外
  └─ 啟用 ▼
  │
  ▼
[5. 同步寫日誌（成功）]
  │
  ▼
[6. 更新使用者主檔：最後登入時間 + 最後登入 IP]
  │
  ▼
[7. 簽發 token、回應前端]
```

### 5.2 登出時的日誌寫入

```
[使用者點「登出」]
  │
  ▼
[認證服務找到對應使用者]
  │
  ▼
[同步寫日誌（自己主動登出 200，結果成功）]
  │
  ▼
[註銷 token]
```

```
[管理員強制登出某使用者] / [排程清理過期 token]
  │
  ▼
[找到對應使用者]
  │
  ▼
[同步寫日誌（強制退出 202，結果成功）]
  │   排程情境：IP 用本機 IP、UA 留空
  │
  ▼
[註銷 token]
```

### 5.3 分頁查詢

```
[輸入查詢條件]
  │
  ▼
[權限檢查：是否具備「登入日誌-查詢」]
  │
  ▼
[組成 SQL 條件]
  ├─ IP LIKE %?%
  ├─ 帳號 LIKE %?%
  ├─ 狀態 = true → result = 0
  ├─ 狀態 = false → result > 0
  └─ 建立時間 BETWEEN ? AND ?
  │
  ▼
[依日誌編號倒序分頁]
  │
  ▼
[BeanUtils 轉成回應 VO]
  │
  ▼
[依字典翻譯「日誌類型」「登入結果」]
  │
  ▼
[回傳分頁結果]
```

## 7. 商業邏輯

### 7.1 寫入採同步

寫入流程：`AdminAuthServiceImpl.createLoginLog()` → `loginLogService.createLoginLog()` → `mapper.insert()`，全部在登入主執行緒中執行。優點：

- 確保任何登入嘗試都留下紀錄（除非資料庫整個壞）
- 不會遺失（不像操作日誌異步可能丟）

代價：

- 登入回應時間 +1 次 DB insert
- 若資料庫慢，登入就跟著慢
- 寫入失敗會把例外拋回認證流程（推論：認證流程未額外 catch，列入 §11）

### 7.2 認證流程的每個失敗分支都寫日誌

不論驗證在哪一步失敗，都會先寫日誌再拋例外：

| 失敗點 | 結果代碼 | 是否有 userId |
|---|---|---|
| 驗證碼錯 | 31 | 無 |
| 帳號不存在 | 10 | 無 |
| 密碼不對 | 10 | 有 |
| 帳號被停用 | 20 | 有 |

寫日誌動作不被例外打斷，與後續拋例外是兩件事。

### 7.3 登入成功時更新使用者最後登入欄位

成功時呼叫 `userService.updateUserLogin(userId, ip)`：

- 更新「最後登入時間」（系統當下）
- 更新「最後登入 IP」（HTTP Request 的 client IP）

此動作在使用者管理頁可以看到「最後登入」欄位，便於 IT 識別休眠帳號。

### 7.4 帳號失敗時不揭示具體原因

對外回應一律「帳號或密碼不正確」（代碼 10），不論是帳號不存在或密碼錯。理由：

- 避免攻擊者用「帳號是否存在」做枚舉
- 內部日誌仍可區分（用 userId 是否為空）

### 7.5 查詢「狀態」採布林轉換

`LoginLogPageReqVO.status` 是 `Boolean`，到 Mapper 後：

- `true` → `result = 0`（成功）
- `false` → `result > 0`（任何失敗）
- `null` → 不加條件

意即「失敗」這個篩選把所有失敗類型一起篩出，不分原因。要看特定失敗類型只能事後在列表上目視篩。

### 7.6 多種登入入口共用同一份日誌結構

帳號登入、社交登入、手機登入、簡訊登入 4 種入口都呼叫同一個 `createLoginLog` 內部方法。差別只在於 `logType` 欄位的值。對於漢堡王 ERP 目前場景，主要使用「帳號登入」；其他類型為框架預留。

### 7.7 匯出強制不分頁

匯出端點把 `pageSize` 設為 `PAGE_SIZE_NONE`，撈全部符合條件的紀錄。資料量大時可能 OOM（同操作日誌的考量，列入 §11）。

### 7.8 不通過 BPM 簽核、無新增／修改／刪除 UI

本模組僅提供「分頁查詢」「匯出」兩支對外端點。寫入完全由認證服務內部觸發，UI 上看不到「新增登入日誌」「刪除登入日誌」按鈕。資料的不可竄改性靠：

- UI 不提供修改／刪除入口
- 主檔繼承 `BaseDO` 含軟刪除欄位但無對應 API
- 物理寫入後僅作為查詢與匯出之用

## 10. 功能範圍

### 10.1 包含的功能

- 認證流程自動寫入登入日誌（同步、不丟失）
- 涵蓋 4 種登入入口（帳號、社交、手機、簡訊）
- 涵蓋 2 種登出（主動登出、強制退出）
- 涵蓋 5 種登入結果（成功 + 4 種失敗）
- 帳號欄位冗餘儲存（不依賴使用者主檔）
- 登入成功順帶更新使用者最後登入時間／IP
- 分頁查詢（4 個條件：IP 模糊、帳號模糊、狀態、時間區間）
- Excel 匯出
- 匯出動作本身被存取日誌記錄
- 日誌類型與登入結果欄位的字典翻譯

### 10.2 預留但尚未實作

- **暴力破解偵測**：無「同一 IP / 同一帳號 N 分鐘內失敗 M 次自動封鎖」機制。需要靠人工或外部監控。
- **查詢條件擴充**：未支援「依使用者編號」「依日誌類型」「依結果代碼」「依 TraceId」精確篩選。
- **特定失敗類型篩選**：查詢的「狀態」是布林（成功／失敗），失敗時不分子原因。
- **使用者編號欄位的回填**：DTO 上 `userId` 未標 `@NotNull`，但實際應用上對於「帳號不存在」「驗證碼錯」情境一定為空。是否要把欄位設成清楚的「可為空但要明確標」設計，看視角。

### 10.3 不包含

- 操作日誌（屬另一份功能：操作日誌）
- 存取日誌 / API 訪問日誌（由 `@ApiAccessLog` 觸發，由匯出動作觸發本日誌外的另一張表）
- token 簽發、token 註銷、token 過期判斷（由 OAuth2 模組負責）
- 對日誌的修改／刪除（不提供，符合稽核原則）
- 日誌歸檔／清理策略
- 即時告警（連續失敗發 email / 簡訊給資安）
