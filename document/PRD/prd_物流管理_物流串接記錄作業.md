## 1. 功能概覽

我是漢堡王台灣 **IT 維運 / 物流規劃人員 / 系統管理員**。系統每日會自動產生 CSV 並 SFTP 上傳至 MSS 倉系統（#50 的 `/generateCsv`）。我需要追蹤：

## 2. 功能目的

物流串接記錄是「**對 MSS 外部系統同步的 audit log**」：

1. **同步歷史** — 每筆 SFTP 上傳的明細都記錄
2. **對帳依據** — 對 MSS 確認資料對齊
3. **失敗排查** — SFTP 失敗時的 troubleshooting

---

## 3. 業務邏輯背景

### 3.1 與 #51 共用程式碼

完全相同的 Controller (`/pdm/raw-material-logistics`)、Service、DO。差別僅在**使用者視角**：

| 視角 | #51 物流異動查詢 | #52 物流串接記錄 |
|---|---|---|
| 主要查什麼 | 實際到店 vs 需求 | SFTP 上傳成功 / 失敗 |
| 主要使用者 | 物流規劃 / 倉儲 | IT 維運 |
| 操作 | 看異動 + 修補數量 | 看串接成敗 + 重發 |

實務上前端可能用同一頁但加不同過濾預設值。

### 3.2 串接的 audit 不足

依 #50 §11 提到：

- SFTP 失敗只 log error 不拋例外
- DB 寫入與 SFTP 上傳沒在同一 transaction
- 可能造成「DB 有記錄但 SFTP 沒上傳」或反之

⚠️ 本功能查到的「串接記錄」只代表「**DB 已寫入**」，不能保證「**MSS 收到了**」（見 §11）。

### 3.3 沒有重發 API

程式碼中**無 `/resend` 或類似的端點** — 失敗的串接無法直接從 ERP 重新觸發 SFTP。需手動執行 `/generateCsv`（會建一筆新單而非更新舊單）。詳見 §11。

### 3.4 沒有狀態欄位

DO `RawMaterialLogisticsDO` 只有 4 個欄位：signCode / deliveryMode / deliveryMfrId / shippingDate。**沒有「SFTP 上傳狀態」「MSS 確認狀態」** — 完全靠 IT 看 SFTP 端的 log（見 §11）。

## 4. 情境說明

### 4.1 正常流程 — 對帳

IT 維運每月對帳：

1. GET /pdm/raw-material-logistics/page → 看 5 月所有串接單
2. 統計 31 天份，每天一筆 PR-...
3. 比對 MSS 端的 SFTP 接收紀錄

### 4.2 異常情境 — 失敗重發

某天 SFTP 失敗（#50 service catch IOException 後只 log）：

- ERP 的 DB **可能有 / 可能沒有** 寫入（看失敗發生在哪一步）
- 若 DB 沒寫入 → 本功能查不到該日記錄 → IT 知道失敗
- 若 DB 寫入但 SFTP 失敗 → 本功能查到記錄但 MSS 沒收到 → **誤報為成功** ⚠️
- 解法：手動重跑 `/generateCsv` → 但會建一筆新單，需手動清理舊單

### 4.3 規則分流 — 編輯重發

某 IT 想「重發」上週的失敗串接：

- 沒有對應的端點
- 解法 A：手動執行 `/generateCsv`（如果原 SQL 仍能撈到當日資料）
- 解法 B：人工複製 CSV 內容透過 FTP 客戶端上傳

詳見 §11。
