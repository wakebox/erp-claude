## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣總部的 **PDM／採購／物流規劃人員**。每樣食材／包材的供應節奏不同 — 牛肉餅每週兩次配送、長期庫存的辣椒粉每月補一次、區域直送商每天到貨。為了讓系統知道某料件的供應節奏，我需要先建立一份「物流類型」字典：每筆描述一種「配送模式 + 週期 + 週期內到貨天」組合。

### 1.2 我要做什麼

- 維護物流類型清單（例：直送週一三五、配送週二、月配送一次…）
- 為每種類型指定「模式」（直送 / 配送）、「週期類型」（週 / 月）、「物流週期天數」（如 1,3,5 表示週一三五）、「資料提供模式」、是否為預設
- 確保整個 ERP 只有**最多一筆**標為「預設」的物流類型，做為新建料件未指定時的 fallback
- 在食材／包材／單品的「物流屬性」上選用這份清單裡的某一筆

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 用統一的物流類型字典 | 牛肉「週一三五直送」、雞肉「週二四直送」… 沒有字典每筆食材都要重打 |
| 必須有「預設」，但只能一筆 | 新建料件忘了選物流類型時要有 fallback；多個預設會衝突 |
| 週期天數能用「1,3,5」這種字串表達 | 一週多天到貨用單一欄位記比較簡潔 |
| 模式 / 週期類型用下拉，不要自由文字 | 避免「直送」「Direct」「直接配送」三種寫法 |
| 變更有稽核紀錄 | 建立 / 修改人員與時間可追溯 |

| 功能 | 解決的問題 |

## 2. 功能目的

物流類型字典提供 ERP 內所有與「補貨節奏」相關欄位的下拉選項，主要消費者為：

1. **食材／包材／單品的物流屬性**：每樣料件指定一個物流類型，下游請購、採購、排程才知道何時下單、何時到貨
2. **需求預測與請購計劃**：把預測量按物流週期攤平成「某週要請購多少」
3. **物流管理模組（PRD #48–52）**：行事曆與配送排程引用此字典決定何時排車

設計理念：

- **預設唯一性**：透過 `defaultStr = '0'` 標記「預設」；保護是「只能有一筆 default_str = '0'」
- **平面結構**：無父子層級、無啟用 / 停用、無排序
- **週期天數用字串**：彈性高但不容易做有效性檢查（見 §11）

## 3. 業務邏輯背景

### 3.1 「預設」用 `defaultStr` 欄位

`defaultStr` 是字串型別（不是 Boolean），語意如下：

| 值 | 語意 |
|---|---|
| `0` | 預設（fallback） |
| `1` | 不是預設 |
| 其他 / null | 未設定（程式邏輯上視為「不是預設」） |

> ⚠️ **數字與業務語意反向**：直覺上「0 = false」「1 = true」，但這裡 0 是「是預設」、1 是「不是預設」。對前端與下游 service 都容易誤判（見 §11）。

### 3.2 「只能一筆預設」的保護

`validateLogisticsTypeExists()` 在更新時若 `defaultStr = '0'`，會檢查整張表是否已有其他筆 `default_str = '0'`，**有則整筆更新失敗**，拋 `LOGISTICS_TYPE_ALREADY_EXISTS`（訊息：「物流類型設定已存在預設的」）（來源：`LogisticsTypeServiceImpl.java:67-77`）。

**問題點**：

1. 此邏輯**只在更新時觸發**，**新增時不檢查** — 可以同時新增兩筆預設
2. count 條件沒有排除自己，因此把一筆已是預設的記錄重新存（連同 defaultStr=0）會被誤判為「已存在預設」，自己更新自己也失敗
3. 沒有「目前哪一筆是預設」的取得方法（如 `getDefault()`）— 下游想用 fallback 必須自己查 `default_str=0`

### 3.3 「物流週期」欄位（`logisticsCycle`）

文字欄位，描述「週期內哪幾天到貨」，使用「逗號分隔字串」表達：

| 範例 | 語意（推測） |
|---|---|
| `1,3,5` | 週一、週三、週五 |
| `2,4` | 週二、週四 |
| `1,15` | 每月 1 號、15 號（搭配 `cycleType='月'`） |

**問題點**：

- 純字串無格式驗證，可填任何文字（如「一三五」「Mon/Wed/Fri」）
- 與 `cycleType` 的搭配無交叉驗證（例如 cycleType=月 但 logisticsCycle 寫了 0–6 的週幾值）
- 下游模組必須自己 parse

### 3.4 「模式」「週期類型」「資料提供模式」

- `deliveryMode`：直送 / 配送
- `cycleType`：週 / 月
- `dataType`：資料提供模式 — 從程式碼推測是描述「補貨資料來源」（例：總部統一規劃 / 各店自報 / 中繼系統推送），無字典固化

三個欄位都是純字串，**後端沒有透過 `@DictFormat` 或字典轉換**。前端應提供下拉但無系統強制（見 §11）。

### 3.5 與其他基礎資料的對比

| 比較項目 | 物流類型 | 餐食類型 | 營養成分定義 |
|---|---|---|---|
| 狀態 (啟用 / 停用) | 無 | 有（String + Dict） | 無 |
| 預設旗標 | 有（defaultStr='0'，且只能一筆） | 無 | 有（defaultIngredient，多筆預設可共存） |
| 排序 | 無 | 無 | 有（sort Long） |
| 唯一性檢查 | 無 | 無 | 無 |
| 批次刪除 | 無 | 有 | 有 |
| editable 旗標 | 無 | 有（固定 true） | 有（固定 true） |

可見 PDM 內字典型基礎資料的設計**很不統一**（見 §11）。

### 3.6 沒有跨表保護

刪除物流類型時，**不檢查**是否有食材 / 包材 / 單品引用此筆。若刪除被引用的類型，會造成那些料件的物流屬性指向不存在 ID（孤兒）。

### 3.7 錯誤碼撞號

`ErrorCodeConstants.java`：

- `LOGISTICS_TYPE_NOT_EXISTS = 1_003_006_001`
- `LOGISTICS_TYPE_ALREADY_EXISTS = 1_003_006_001` ⚠️ **同一個錯誤碼**

兩個語意完全不同的錯誤共用同一個錯誤碼，前端無法分辨。屬於明確的 bug（見 §11）。

## 4. 情境說明

### 4.1 正常流程 — 新增「週一三五直送」

採購規劃人員小張要為新進的「牛肉餅 LB-04」設定物流節奏。她進入「PDM > 物流類型維護表」，點「新增」，填入：

- 名稱：週一三五直送
- 模式：直送
- 週期類型：週
- 物流週期：1,3,5
- 資料提供模式：總部統一
- 預設：1（不是預設）
- 備註：牛肉類專用

送出後系統直接 insert 成功。之後在牛肉餅的物流屬性下拉就能選到「週一三五直送」。

### 4.2 典型業務 — 設定「預設」物流類型

主管決定把「週一直送」設為新建料件的 fallback 預設。小張在清單頁找到「週一直送」，編輯，把預設改為 `0`。系統檢查整張表是否已有其他筆預設（不排除自己 ID）：

- 若沒有任何筆是預設（包含這筆自己） → 直接更新成功
- 若有任何筆是預設（包含這筆自己）→ 拒絕並回「物流類型設定已存在預設的」

⚠️ **發現的 bug**：因為 count 沒排除自己 ID，**這筆原本就是預設的情況下重存** （即使其他欄位都沒改，defaultStr 仍為 0），會把自己當「其他已存在預設」而拒絕。詳見 §11。

### 4.3 典型業務 — 取消預設

要把「週一直送」改為不預設。小張編輯，把預設改為 `1` 並送出。`validateLogisticsTypeExists` 看到 `!"0".equals(defaultStr)` 直接 return（跳過唯一性檢查），更新成功（來源：`LogisticsTypeServiceImpl.java:67-68`）。

### 4.4 使用者鍵入錯誤 — 沒有任何必填欄位

`LogisticsTypeSaveReqVO` **沒有 `@NotEmpty` / `@NotNull` 註解**（來源：`LogisticsTypeSaveReqVO.java:11-35`） — 整個 VO 完全沒驗證。可建出一筆「全部空白」的物流類型。**後端完全靠前端把關**（見 §11）。

### 4.5 異常情境 — 更新或刪除已不存在的 ID

更新或刪除時若 ID 不存在，拋 `LOGISTICS_TYPE_NOT_EXISTS`（錯誤碼 `1_003_006_001`，訊息：「物流類型設定不存在」）。

### 4.6 規則分流 — 新增兩筆預設

新增**不檢查預設唯一性**。可連續新增兩筆 `defaultStr='0'` 的記錄，系統不會擋。之後下游若以 `default_str='0'` 取 fallback，會撈到多筆（見 §11）。

### 4.7 匯出情境

採購主管要對照物流安排，點匯出 Excel。系統不分頁、全量匯出。檔名「物流類型設定.xls」，欄位包含 ID、名稱、模式、週期類型、物流週期、資料提供模式、預設、備註、建立時間（來源：`LogisticsTypeRespVO.java`）。

## 5. 操作流程

```
[PDM 人員進入「物流類型維護表」]
  │
  ├─ 分頁查詢 GET /pdm/logistics-type/page
  │    ├─ 權限檢查：pdm:logistics-type:query
  │    ├─ 過濾：logisticsName like、其他欄位 eq、createTime eq（注意是等於而非區間）
  │    ├─ 排序：ID desc
  │    └─ 回傳分頁
  │
  ├─ 取得單筆 GET /pdm/logistics-type/get?id=
  │    └─ 權限檢查：pdm:logistics-type:query
  │
  ├─ 新增 POST /pdm/logistics-type/create
  │    ├─ 權限檢查：pdm:logistics-type:create
  │    ├─ 無必填驗證、無預設唯一性檢查
  │    └─ insert，回新 ID
  │
  ├─ 更新 PUT /pdm/logistics-type/update
  │    ├─ 權限檢查：pdm:logistics-type:update
  │    ├─ 存在性檢查：ID 不存在 → 拋 LOGISTICS_TYPE_NOT_EXISTS
  │    ├─ 若 defaultStr='0' → 檢查整表是否已有 default_str='0'（不排除自己 ID）
  │    │    └─ 有 → 拋 LOGISTICS_TYPE_ALREADY_EXISTS
  │    └─ updateById
  │
  ├─ 單筆刪除 DELETE /pdm/logistics-type/delete?id=
  │    ├─ 權限檢查：pdm:logistics-type:delete
  │    ├─ 存在性檢查
  │    └─ deleteById（軟刪除，無跨表保護）
  │
  └─ 匯出 Excel GET /pdm/logistics-type/export-excel
       ├─ 權限檢查：pdm:logistics-type:export
       ├─ 套用同分頁查詢條件，全量
       └─ 寫出 Excel
```

## 7. 商業邏輯

### 7.1 新增

- 無任何前置檢查，直接 insert
- 即使 defaultStr='0' 也不檢查唯一性

### 7.2 更新

```
1. 取舊資料 → 不存在 → 拋 LOGISTICS_TYPE_NOT_EXISTS
2. 若新 defaultStr ≠ '0' → 直接 updateById
3. 若新 defaultStr == '0':
     count = SELECT count(*) WHERE default_str='0'    ← 不排除自己 ID
     if count > 0 → 拋 LOGISTICS_TYPE_ALREADY_EXISTS
4. updateById
```

**Bug**：步驟 3 沒排除自己 ID，導致「重存一筆原本就是預設的記錄」會被拒絕。

### 7.3 刪除

- 檢查存在 → 軟刪除
- 無跨表保護（被引用也照刪）

### 7.4 查詢

- 名稱模糊、其他欄位等值
- 強制條件：無（連 deleted=0 都沒明確寫，依賴 MyBatis-Plus 全域邏輯刪除攔截）
- 排序：ID 倒序

## 10. 功能範圍

### 10.1 包含的功能

- 物流類型的 CRUD（新增、查詢、編輯、單筆刪除）
- 分頁查詢含多欄位過濾
- Excel 匯出
- 更新時的「預設唯一性」保護（有 bug）

### 10.2 預留但尚未實作

- **批次刪除**（其他字典都有）
- **新增時的預設唯一性檢查**
- **必填驗證**（VO 完全沒驗證註解）

- **字典化**（模式 / 週期類型 / 資料提供模式都是裸字串）
- **欄位交叉驗證**（cycleType 與 logisticsCycle 的合理性）

### 10.3 不包含

- 料件本身的物流屬性設定（屬於食材維護 / 包材維護 / 單品維護）
- 物流配送排程（屬於物流管理模組 PRD #48–52）
- 物流週期天數的實際解析與套用（屬於下游使用模組）
