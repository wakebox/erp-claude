## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長 / 副店長 / 倉管員**（例：BK001 信義店）。每次有貨車到店，我都要：

> 「2026-05-19 上午 9:00 — 物流司機送來牛肉餅 50 公斤、麵包 200 個、生菜 8 公斤；我對照配送單清點實收量、確認效期、簽收。系統把實收量自動加進信義店當前庫存。」

### 1.2 我要做什麼

- 每天物流到貨：點收、確認數量／效期、簽收
- 異常時：標記少到／破損／效期不符，連動到物流異動查詢
- 跨店調撥收方：對應「我的店」接收到的調撥
- 盤點盈虧：盤點完成後系統自動寫入入庫單（推論：店長僅查看，不主動建立）

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看「今天我要收哪些貨」 | 預排點收人手與空間 |
| 對每品輸入「實收量」 | 計畫量未必等於實際量 |
| 標記異常並上傳照片 | 留下證據、便於申訴／補單 |
| 簽收後庫存即時更新 | 下游補貨判斷需即時 |
| 行動裝置可操作 | 在卸貨現場操作 |
| 只看自店單據 | 不要被全公司資料淹沒 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「待我點收」清單 | 一眼看今天待收 |
| 點收明細逐筆輸入實收量 | 對帳基礎 |
| 異常標記＋拍照上傳 | 留證據 |
| 簽收後自動寫入庫存 | 一次到位 |
| 自店歷史入庫單查詢 | 對帳、稽核 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 入庫作業管理（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理」 |
| 兄弟功能 | 倉儲查詢、安全存量設定、出庫管理、門市調撥 |
| 主要頁面 | 待點收清單、點收明細頁、自店歷史入庫單 |
| 簽核流程 | 多數自動入庫直接歸檔；差異情境**建議綁簽核**（待確認） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在貨車到店的當下，**自助完成「點收→輸入實收量→確認異常→簽收→寫入庫存」**的完整循環。在門市端建立「物流末端可視性與責任歸屬」：什麼時候、收了什麼、收了多少、有沒有異常，全程留軌跡。

設計理念：**「計畫量」與「實收量」分離、有差異即留痕、簽收即不可逆**。店長不憑空建立入庫單（入庫單由上游配送、調撥、盤點等業務行為自動產生），只負責「實收確認」與「異常標記」。

## 3. 業務邏輯背景

### 3.1 入庫單的來源（店長視角）

| 來源 | 觸發方式 | 店長要做什麼 |
|---|---|---|
| 物流配送（中央倉送到店） | 配送日當天系統自動建單 | **主要場景**：點收、簽收 |
| 採購驗收（總部對廠商驗收歸檔） | 總部歸檔即自動入庫 | 通常與店長無關（除非店為驗收地點） |
| 跨店調撥（他店撥入） | 對方店出庫後，自店收貨 | 同點收流程 |
| 盤點盈餘調整 | 盤點完成後系統自動入庫 | 通常店長僅查看 |
| 不良品迴轉入庫 | 不良品流程觸發 | 推論：店長不直接操作 |

### 3.2 入庫單的結構

- **單頭**：單據編號、流程狀態、來源單據編號、區域／倉別／倉名、出入庫類型（區分入／出）、入庫時間、主旨、入庫事由
- **明細**：每項食材的品號、申請數量、實收數量、單位、儲位、入庫事由、來源單據編號

### 3.3 「實收量」如何記錄

底層在明細層有「庫存異動數量」與「入庫計數」兩個欄位。**目前實作中，配送觸發的入庫單會把「計畫量」直接帶入「實收量」**（與物流串接記錄作業相同問題），不區分計畫 vs 實際。

⚠️ 對店長視角而言，這正是最重要的差異點：**店長要能在點收時填入「實際拿到多少」**，否則整套點收流程沒有意義。

### 3.4 庫存更新時機

簽收（單據狀態變為已歸檔）的當下，系統會：

1. 對每明細：依品號 + 倉庫找到當前庫存紀錄
2. 將實收量加總到當前庫存
3. 若是從調撥／盤點／不良品來的，回寫上游單據的「出入庫已完成」旗標

⚠️ 底層 PRD 已標示：

- 此批次更新**無交易保護**，中途失敗會留下半套資料
- 並行入庫**無樂觀鎖**，可能讀-改-寫競爭

### 3.5 異常與差異處理

當實收量 < 計畫量 時，應同步：

- 在明細上留下「差異原因」
- 觸發物流異動單（連動「物流異動查詢」）
- 通知區經理／配送廠商客服

> 推論；目前底層程式僅單純照「實收量」寫庫存，不會自動觸發異動單。

### 3.6 與總部視角的差異

| 對比項 | 總部／倉儲人員 | 店長 |
|---|---|---|
| 可看單據 | 全公司／自區 | **只能看自己門市** |
| 主要操作 | 全流程 CRUD | **點收確認**為主 |
| 來源 | 多元（採購驗收、調撥、退貨） | 主要為物流配送 |
| 行動裝置 | 桌機為主 | **手機／平板**（現場操作） |
| 數量編輯 | 開放 | 「實收量」可輸入，「計畫量」唯讀 |
| 簽核 | 視政策 | 差異情境建議綁簽核 |
| 入庫照片 | 不必要 | 建議拍貨況存證 |

## 4. 情境說明

### 4.1 正常流程 — 中央倉配送到店

5/19 上午 8:30，信義店店長小陳：

1. 打開「待點收」清單
2. 看到 1 張單：今日配送計畫，含牛肉餅 50 公斤、麵包 200 個、生菜 8 公斤、起司 5 公斤
3. 9:05 司機到店，小陳逐項清點：
   - 牛肉餅：實收 50 公斤、效期 2026-08-15 → OK
   - 麵包：實收 200 個、效期 2026-05-21 → OK
   - 生菜：實收 8 公斤、效期 2026-05-22 → OK
   - 起司：實收 5 公斤、效期 2026-09-30 → OK
4. 全數簽收
5. 系統把信義店當前庫存對應品項加總：牛肉餅 +50、麵包 +200…
6. 紙本簽收單由司機帶回

### 4.2 異常情境 — 少到

5/20 配送，計畫牛肉餅 50 公斤、實際到貨 40 公斤：

1. 小陳逐項清點，發現少 10 公斤
2. 牛肉餅輸入實收 40 公斤，差異原因「廠商備料不足」
3. 拍照（司機簽收單副本）
4. 提交 → 系統：
   - 庫存加 40 公斤（非 50）
   - **建議**：同步觸發物流異動單，通報廠商與總部
5. 簽收完成

⚠️ 第 4 步的「同步觸發異動單」目前底層程式未實作。

### 4.3 異常情境 — 破損

5/21 配送麵包 200 個，到貨時 20 個壓壞：

1. 小陳輸入實收 180 個、破損 20 個
2. 拍破損照片
3. 破損 20 個進入「不良品」流程（另開一張不良品單，店長視角的另一個功能）
4. 入庫單實收 180 個簽收

### 4.4 跨店調撥收方

板橋店撥 5 公斤雞塊到信義店：

1. 對方店出庫後，信義店收到一筆「待點收」單，來源單號 ST 開頭
2. 司機送達，小陳清點：實收 5 公斤
3. 簽收 → 庫存加 5 公斤 → 同步回寫調撥單狀態為「已完整收貨」

### 4.5 配送資料尚未到店但已歸檔

⚠️ 5/22 配送單在系統上顯示「已歸檔」，但司機還沒到店。原因：底層自動配送排程會在凌晨建單並直接歸檔，且「計畫量直接帶入實收量」。

對店長而言：此時「庫存帳上已加」、「實物還沒到」，與實況脫節。應在店長視角區分「已建單未到貨」與「已點收已到貨」兩個狀態。

> 待確認：店長視角是否要加入「待點收」狀態，與底層的「已歸檔」拆開？

| 點收人 | 點收時帶入 | 待新增 |
| 點收時間 | 簽收當下 | 待新增 |

### 6.2 入庫明細

| 欄位（業務名） | 說明 | 店長視角是否顯示 |
|---|---|---|

| 食材名稱 | 從食材主檔帶 | 是 |
| 計畫量 | 上游單據的應收數量 | 是（唯讀） |
| 實收量 | 點收時輸入 | 是（可輸入） |

- 差異原因字典（少到／破損／效期／品質）

## 7. 商業邏輯

### 7.1 鎖店規則

店長視角的所有讀寫，應在後端強制：

```
1. 解析登入者所屬門市
2. 列表只回該門市對應倉的入庫單
3. 點收時驗證：要簽收的單據必須屬於該店長
4. 不接受參數覆寫
```

> 推論；目前後端未實作。

### 7.2 點收 → 庫存更新

簽收時：

```
對每明細：
  1. 找出該店該食材的當前庫存紀錄
  2. 將「實收量」加總（不是「計畫量」）
  3. 若有差異 → 額外觸發物流異動單（待實作）
  4. 若是調撥／盤點／不良品來源 → 回寫上游單據狀態
```

### 7.3 差異容忍規則（推論）

| 差異幅度 | 處理 |
|---|---|
| 0% | 正常簽收 |
| ±5% 內 | 簽收 + log 差異 |
| ±5%–±20% | 簽收 + 通報區經理 |
| ±20% 以上 | 拒絕簽收，必須先聯絡廠商／配送 |

> 推論；實際門檻待 PM 定義。

### 7.4 簽收不可逆

一旦簽收，該入庫單不可在店長視角再次編輯。如需更正：

- 走「盤點調整」走 BPM 流程
- 或由區經理／總部走更高權限路徑

> 推論；目前底層仍允許編輯已歸檔單據（在某些條件下），對店長視角應收緊。

### 7.5 配送排程自動建單與「待點收」狀態

當前底層配送排程在凌晨建單時直接設為「已歸檔」（推論：因為計畫量已直接帶入實收量）。

對店長視角，建議：

- 自動建單時設為「待點收」
- 店長簽收時才轉為「已歸檔」
- 「實收量」欄位在「待點收」狀態下可編輯，歸檔後鎖定

| 「待點收」狀態與「已歸檔」拆分 | 目前自動建單直接歸檔 |
| 「實收量」與「計畫量」分離欄位 | 目前實收 = 計畫 |
| 差異自動觸發物流異動單 | 未實作 |
| 點收人／點收時間欄位 | 單頭無此欄位 |
| 拍照附件 | 未串檔案服務 |
| 批次更新庫存的交易保護 | 底層無 |
| 並行入庫的樂觀鎖 | 底層無 |
| 簽收後鎖定編輯 | 底層仍可在部分條件下編輯歸檔單據 |

### 10.3 不包含

- 出庫管理（屬「店長功能 — 出庫作業管理」）
- 調撥單本身的建立（屬「店長功能 — 門市調撥管理」）
- 不良品迴轉本身（屬不良品管理功能）
- 盤點本身（屬盤點管理功能）
