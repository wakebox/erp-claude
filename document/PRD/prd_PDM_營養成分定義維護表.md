## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣總部的 **PDM（產品資料管理）人員**，負責維護產品端與食材端共用的基礎資料字典。當營養師、研發或行銷部門要在系統中描述「一份華堡含 28g 蛋白質、550 大卡」時，他們可選用的「蛋白質」「熱量」這些營養成分項目，是由我事先在系統裡建好的。

### 1.2 我要做什麼

- 維護整個 ERP 內可被引用的營養成分清單（例：蛋白質 / Protein、脂肪 / Fat、碳水化合物、鈉、熱量…）
- 為每一個營養成分指定它的「呈現單位」（例：蛋白質用「公克」、熱量用「大卡」、鈉用「毫克」），單位來自單位定義維護表
- 標記哪些是「預設成分」（食材或食譜建檔時自動會帶出來的常見項目，例如蛋白質、脂肪、碳水化合物、熱量）
- 設定每項的「平均每餐建議攝取量」，給營養比對／健康標籤計算當參考
- 設定顯示排序，讓清單與報表上的營養成分以一致的順序呈現
- 既有的營養成分若已被食材／食譜引用，需被保護不能改動或刪除（規劃中，目前後端尚未完成此判斷）

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 用統一的營養成分項目清單 | 不同人在不同單品上各自寫「Protein」「蛋白質」「蛋白」，會導致統計與健康標籤無法彙整 |
| 確認每個項目的呈現單位是同一套 | 同一個「鈉」一邊用 mg、一邊用 g，營養標示會錯 |
| 把常見項目標為「預設」 | 每次建食材都要從幾十個項目挑很煩，預設項目可以自動帶出 |
| 知道每個項目的建議攝取量 | 行銷／健康標籤計算需要對照，例如標「一份提供每日蛋白質建議量的 30%」 |
| 已被使用的項目不能亂改 | 改了名稱或單位，過去的食材營養標示就會跟著錯 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 營養成分項目分頁查詢（依中文名、英文名、是否預設成分過濾） | 快速找到要編輯的項目 |
| 新增營養成分 | 擴充清單以支援新的營養標示需求 |
| 編輯營養成分 | 修改名稱、單位、建議攝取量、是否預設、排序 |
| 刪除（單筆／批次） | 移除錯誤或不再使用的項目 |
| 匯出 Excel | 提供給營養師、稽核或行銷部門對照、檢視 |
| 「是否可修改」旗標 | 預留：被食材／食譜引用過的項目應禁止修改（目前後端固定回傳 true，邏輯未實作） |

## 2. 功能目的

營養成分定義是 PDM 模組內的字典型基礎資料，作為「食材」與「食譜（單品）」中營養標示子表的下拉選項來源。其設計目的：

1. **統一營養標示語彙**：確保整個 ERP 對「蛋白質」「脂肪」「鈉」等項目使用同一個 ID，避免命名重複或拼字不一致導致統計失準
2. **解耦項目與單位**：項目名稱由本表維護、呈現單位指向「單位定義維護表」，往後要把鈉從 mg 改 g 只需改一處
3. **支援預設成分機制**：設計上「預設成分」項目會在食材／食譜建檔時自動帶入空白營養成分，減少使用者重複作業
4. **引用保護的預留設計**：回應 VO 上的 `editable` 旗標已預埋，待食材／食譜的營養成分子表完成後，會用它判斷哪些項目已被引用、禁止修改或刪除（來源：`NutritionalDefinitionsServiceImpl.java:84` TODO 註解）

## 3. 業務邏輯背景

### 3.1 「項目」與「單位」是雙層字典

營養成分定義表本身只描述「項目身份」（名稱、英文名、是否預設、排序、建議攝取量、預設單位），其中「單位」欄位不存字面值（如「g」「mg」），而是存「單位定義維護表」的 ID。換句話說：

- 蛋白質、脂肪、碳水化合物 → 共用單位「公克」
- 熱量 → 單位「大卡」
- 鈉、鈣 → 單位「毫克」

當單位定義表中「公克」改名為「g（公克）」時，所有指向它的營養成分顯示會一起更新，不需個別修改。

### 3.2 「預設成分」的意義

`defaultIngredient` 欄位為 boolean。標 true 的項目，預期在食材或食譜建檔時會被自動帶入該品項的「營養成分子表」，使用者只需填數值；標 false 的項目需要手動加入。本表只負責儲存這個旗標，自動帶入的實作位於食材與食譜模組（目前尚未實作，見 §10.2）。

### 3.3 「平均每餐建議攝取量」是字串

此欄位在後端定義為 `String`（來源：`NutritionalDefinitionsDO.java:49`、`NutritionalDefinitionsSaveReqVO.java:27`），非數值，因此可接受 `18`、`18g`、`18-25` 等不同寫法。**這代表系統不會用它做計算，只用於顯示／參考**。若行銷要做「達到每日建議量 30%」之類的計算，必須由前端／其他模組自行解析此字串，或政策上改為純數字（見 §11）。

### 3.4 編輯保護的預留邏輯（尚未啟用）

`NutritionalDefinitionsServiceImpl.getNutritionalDefinitionsPageWithEditable()` 在分頁查詢時會為每筆資料附加 `editable` 旗標，目前**固定設為 true**，等食材／食譜的營養成分子表建好後，預期會改為：

> 「若此營養成分已被任何食材或食譜引用，則 editable=false，前端禁用編輯／刪除按鈕」

來源證據：`NutritionalDefinitionsServiceImpl.java:84`「TODO 待[食材維護作業]和[食譜維護作業]的營養成分子表建立之後補上此處邏輯」。

## 4. 情境說明

### 4.1 正常流程 — 新增一個營養成分項目

營養師王小姐收到行銷需求：要在 BBQ 牛肉華堡上標示「膳食纖維」含量。她進入「PDM > 營養成分定義維護表」，看到目前清單中沒有「膳食纖維」項目。

她點「新增」，填入：
- 項目名稱：膳食纖維
- 英文項目名稱：Dietary Fiber
- 單位：選擇「公克」
- 平均每餐建議攝取量：8
- 預設成分：否（不是每個食材都會有膳食纖維資料）
- 顯示排序：60（排在常見三大營養素之後）

送出後系統回傳建立成功，王小姐回到食材維護作業，能在牛肉餅的營養成分子表加入「膳食纖維」並填入數值。

### 4.2 典型業務 — 查詢「預設成分」總覽

PDM 主管想確認目前系統有哪些項目會自動帶入新建食材。他進入清單頁，把「預設成分」勾選為「是」，按搜尋。系統回傳：蛋白質、脂肪、碳水化合物、熱量、鈉（共 5 筆，依顯示排序由小到大排列）。

### 4.3 使用者鍵入錯誤 — 漏填必填欄位

某次新進人員建檔「鈣」時忘了選單位就送出。系統回傳「單位不能为空」，並由前端把焦點移回單位下拉框（驗證來源：`NutritionalDefinitionsSaveReqVO.java:23`）。同樣情境也適用於「項目名稱」（`@NotEmpty`）與「顯示排序」（`@NotNull`）。

### 4.4 異常情境 — 更新或刪除不存在的 ID

若前端 cache 或重整未同步，使用者試圖編輯一筆已被別人刪除的營養成分，系統會以錯誤碼 `1_003_000_007 / NUTRITIONAL_DEFINITIONS_NOT_EXISTS`（訊息：「營養成分定義不存在」）拒絕（來源：`ErrorCodeConstants.java:43`、`NutritionalDefinitionsServiceImpl.java:62-66`）。

### 4.5 規則分流 — 批次刪除不做存在性檢查

單筆刪除前會檢查 ID 是否存在；**批次刪除則直接呼叫 `deleteByIds`，不檢查每筆是否存在**（來源：`NutritionalDefinitionsServiceImpl.java:57-60`）。前端送出的 ID 清單若混入不存在的 ID，會被靜默忽略，不會回報錯誤。

### 4.6 匯出情境

主管要做季度盤點，點「匯出 Excel」。系統不分頁、全量匯出所有營養成分，欄位包含：營養成分定義 ID、項目名稱、英文項目名稱、單位（注意：匯出的是單位 ID，非單位名稱）、平均每餐建議攝取量、預設成分、顯示排序、建立／修改人員與時間（來源：`NutritionalDefinitionsRespVO.java`、`NutritionalDefinitionsController.java:100-113`）。匯出檔名為「營養成分定義.xls」。

## 6. 欄位規格

| 欄位 | 中文業務語 | 型別 | 必填 | 說明 |
|---|---|---|---|---|
| id | 營養成分 ID | Long | 系統產生 | 主鍵（PostgreSQL sequence） |

## 7. 商業邏輯

### 7.1 新增 / 更新流程

1. 攔下 Request VO，由 Jakarta Validation 跑必填驗證
2. 更新時先查 ID 是否存在，不存在拋 `NUTRITIONAL_DEFINITIONS_NOT_EXISTS`
3. 用 BeanUtils 把 VO 複製成 DO
4. 直接 insert / updateById

**未實作的業務檢查**（依照其他模組常見模式，這裡是缺漏項）：

- 名稱唯一性檢查（同名項目可重複新增）
- 單位 ID 是否真存在於「單位定義維護表」（目前只檢查非 null，不檢查指向有效）
- 排序值衝突或重複

### 7.2 刪除流程

- 單筆：檢查存在 → 軟刪除（`deleteById`，由 BaseDO 軟刪除機制處理）
- 批次：直接呼叫 `deleteByIds`，**不檢查存在性、也不檢查是否被食材／食譜引用**
- 預期未來會加上「若已被引用則禁止刪除」的判斷（見 §10.2 預留）

### 7.3 分頁查詢的 editable 加工

`getNutritionalDefinitionsPageWithEditable()` 在資料庫查回 DO 後，對每筆建立對應的 RespVO，並把 `editable` 固定設為 `true`。控制器的 `/page` 端點呼叫的就是這個方法，因此前端目前永遠看到 editable=true。**待補的邏輯**：用該營養成分 ID 反查食材／食譜的營養成分子表，若有引用則設 false。

### 7.4 排序

清單預設依 `sort` 由小到大排序；DB 未在 sort 欄位建唯一約束，多筆同 sort 值的順序由資料庫實作決定，可能不穩定。

## 10. 功能範圍

### 10.1 包含的功能

- 營養成分項目的 CRUD（新增、查詢、編輯、單筆／批次刪除）

### 10.2 預留但尚未實作

- **editable 旗標的動態判斷**：目前固定 true，待食材／食譜營養成分子表完成後改為「被引用即鎖定」（來源：`NutritionalDefinitionsServiceImpl.java:84`）

- **單位有效性檢查**：目前只檢查 unitId 不為 null，不檢查指向「單位定義維護表」中存在且啟用的單位

### 10.3 不包含

- 「單位」本身的維護（屬於 [單位定義維護表]）
- 食材／食譜上實際的營養成分數值錄入（屬於食材維護作業、食譜／單品維護作業）
- 營養標示計算、健康標籤生成（屬於行銷／前台展示層）
