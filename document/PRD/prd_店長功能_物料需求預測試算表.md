## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）或門市行政助理。

食材預測（如牛肉餅、麵包）已經有專屬試算表（見兄弟 PRD）。但門市每天用掉的還不只食材：紙袋、外帶杯、餐巾紙、吸管、清潔劑、消毒液、抹布、辦公文具…這些「**包材／清潔用品／設備耗材**」不是用食譜算出來的，而是依「過去叫貨紀錄 + 預計銷量」推估的。

物料需求預測試算表（非 BOM）就是把「食材預測單」與「臨時需求單」彙整後，把同品號的需求加總、綁定廠商、算出預計配送日，產出一份「**這週每一天，我的店要從哪些廠商收到哪些品號、多少量**」的時間表。

> 「下週信義店預計：紙袋 5000 個、餐巾紙 12 包、清潔劑 8 瓶、吸管 3 箱…其中餐巾紙下週做行銷加碼到 18 包」

### 1.2 我要做什麼

- 一週訂貨節奏前：跑試算 → 看自店未來 7 天每天會收到什麼品號、多少量
- 對照當前庫存（紙袋還剩多少箱）：判斷哪些需要請總部多訂、哪些可以少訂
- 確認預計配送日：和門市點收人力排班對齊
- 看「沒對到廠商的品號」：有些是需要但廠商還沒談妥，要通報採購
- 試算結果儲存 → 進入後續採購流程（推論）

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看自店未來 7 天每天會收什麼 | 排點收人手、空間 |
| 把食材預測 + 臨時加單合在一起看 | 兩個來源分開看會漏 |
| 自動綁廠商、自動算配送日 | 我不要每筆都自己查 |
| 平日 vs 假日量自動切 | 週末用量明顯較大 |
| 沒對到廠商的品號要被標出來 | 否則會以為一切正常但其實漏單 |
| 只看自己門市 | 不要混入其他分店 |
| 試算結果要能查回去 | 對帳、追溯使用 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「我的門市」物料需求試算（一週區間） | 一鍵看下週每天每品號的配送計畫 |
| 食材預測 + 臨時需求自動合併 | 兩個來源在試算當下就合在一起 |
| 廠商自動綁定 + 預計配送日 | 看廠商分組、估點收時間 |
| 無廠商品號的警示（待補） | 避免靜默漏單 |
| 我的試算單一覽 | 找回過去跑過的單做對帳 |
| 需求日 drill-down | 點某天看具體品項清單 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 物料需求預測試算表（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合」 |
| 兄弟功能 | 食材需求預測試算表（店長視角）、臨時需求審核（店長視角） |
| 主要頁面 | 自店試算頁、我的試算單一覽、需求日 drill-down |
| 簽核流程 | **無**（底層 service 未綁 BPM，與兄弟功能不同） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在不依賴總部協助的前提下，**自助看到「未來一週每一天我的店會從哪些廠商收到哪些品號、多少量」**，並對照自店實況提早發現問題（廠商沒談、品號漏配、配送日不合理）。

設計理念：**彙整 + 可視化 + 鎖店**。本功能不重算食材，只把上游兩份單據（食材預測 / 臨時需求）依品號合併，補上廠商與配送日。

## 3. 業務邏輯背景

### 3.1 三層結構

| 層級 | 一張單代表什麼 |
|---|---|
| 單頭 | 一次試算的條件：自店、日期區間、可選的食材中類／小類 |
| 需求日 | 區間內每一天的一筆紀錄（如 5/25、5/26…），記載「這天參考了哪張食材預測單、哪張臨時需求單」 |
| 食材明細 | 每個需求日下的品號清單：品號、數量、廠商、預計配送日 |

### 3.2 兩個資料來源的合併規則

對日期區間內每一天、自店：

1. **找食材預測單**：撈該日該店在食材預測（兄弟 PRD #60）裡的食材清單
2. **找臨時需求單**：撈該日該店所有已核可的臨時需求單明細
3. **以品號為 key 合併**：同品號 → 數量相加；不同品號 → 各自加入清單
4. 兩個來源同一天都沒有 → 該日跳過、不留紀錄

### 3.3 廠商自動綁定

合併後逐筆對品號 + 自店查最新的「歸檔且未過期」廠商報價，補上：

- 廠商代號、廠商名稱
- 物流類型（週配／月配）
- 物流週期（如「週一三五」「每月 1 號 15 號」）

**沒對到廠商的品號會被靜默過濾掉、不寫入明細**。這是底層既有行為，店長視角**建議改為顯示警示而非靜默丟棄**（見 §11）。

### 3.4 預計配送日反推

依物流類型 + 週期反推需求日前最近一次的配送日：

- 週配（週一三五配）：需求日是週四 → 預計週三配送（推論）
- 月配（每月 1 號、15 號）：需求日是 5/25 → 預計 5/15 配送（推論）

實際邏輯涉及日期跳算，邊界情況多。

### 3.5 平日 vs 假日

每個品號的數量自上游食材預測單帶「平日總量」或「假日總量」：

- 需求日是週一至週四 → 取平日總量
- 需求日是週五至週日 → 取假日總量

### 3.6 沒有更新／刪除

底層 service 只提供「建立 + 查詢」。一旦試算建單，就**無法刪、無法改**。店長若跑錯（例如選錯週期、選錯類別），唯一辦法是再建一張新單覆蓋（但舊單仍在）。

### 3.7 無簽核流程

本功能在後端**未綁定 BPM 流程**。一旦試算完成、單據建立，下游採購可直接取用，沒有人工確認的緩衝。

> 推論：店長視角應視試算單為「**參考用報表**」而非「**已定案的訂單**」；真正的訂單仍會由採購端另行確認。

### 3.8 資料權限自動過濾

底層 service 已有「依登入者區域／門市自動過濾」邏輯：登入者若有門市權限且未指定門市，會強制套用為登入者門市。店長視角理論上會被自動鎖到自店，**但此鎖店行為未經驗證**（待確認）。

## 4. 情境說明

### 4.1 正常流程 — 週日晚上跑下週試算

5/24（週日）晚上 10 點，信義店店長小陳要看下週 5/25–5/31 的物料配送計畫：

1. 進「物料需求預測試算」頁
2. 系統自動鎖定信義店、預設下週日期區間
3. 點「試算」
4. 系統：
   - 對 7 天每天去找該店的食材預測單與臨時需求單
   - 兩個都沒有 → 跳過該天
   - 任一有 → 合併品號
   - 補廠商、算預計配送日
5. 結果：
   - 5/25（一）：紙袋 1000 個 / 包材商 / 預計 5/22（五）配送
   - 5/25（一）：餐巾紙 2 包 / 衛生用品商 / 預計 5/22 配送
   - 5/26（二）：跳過（無單）
   - 5/27（三）：清潔劑 8 瓶 / 清潔商 / 預計 5/25（一）配送
   - …
6. 小陳檢視 → 在「我的試算單一覽」找回這張單做後續對帳

### 4.2 典型業務 — 點開某天看細項

小陳關心 5/30（五）週末配送：

1. 從需求日列表點 5/30 → drill-down
2. 看到該天 12 個品號 / 4 個廠商
3. 對照當前庫存：紙袋還剩 3 箱（1500 個）夠用、餐巾紙剩 1 包不夠
4. 對「不夠」的品號標記、傳給總部採購（推論：目前無此功能，需手動）

### 4.3 異常情境 — 某品號沒對到廠商，靜默丟失

新進的「環保杯 X-99」剛在臨時需求單裡被加，但廠商報價還沒談妥：

- 合併後 jointVendor 查不到廠商
- 系統**靜默過濾**，明細不寫入
- 小陳完全不知道這品號被丟掉
- 結果：實際配送日該品號沒到、現場才發現

> 風險高，建議改為「顯示警示但仍寫入明細並標紅」（見 §11）。

### 4.4 異常情境 — 某日兩來源都沒有

5/26（二）信義店那天沒有食材預測單也沒有臨時需求單：

- 系統跳過該天
- 需求日列表不顯示這天
- 店長看到「5/25 後直接跳到 5/27」可能會困惑「為什麼缺一天？」（待確認 UX）

### 4.5 異常情境 — 跑錯範圍不能改

小陳不小心把日期區間填成 5/18–5/24（過去）：

1. 系統照樣建單、跑試算
2. 跑完才發現選錯
3. **無法刪、無法改**
4. 只能再建一張正確日期的單，但舊單仍在 DB

### 4.6 規則分流 — 平日 vs 假日量

同一品號（紙袋）：

- 5/25（一）→ 平日 → 取平日量 1000 個
- 5/30（六）→ 假日 → 取假日量 1500 個

平日／假日量是上游食材預測單裡就分好的兩組數值，本試算只負責「依日期選對應的那組」。

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「物料需求預測試算」]
  ├─ 1. 取登入者身份 → 自動帶入：自店（鎖店）
  ├─ 2. 選日期區間（預設下週一～週日）
  │    （食材中類／小類為選填，店長視角預設不填）
  ├─ 3. 點「試算」 → 後端：
  │    ├─ 建立試算單頭
  │    ├─ 對日期區間每一天：
  │    │    ├─ 查自店該日食材預測單
  │    │    ├─ 查自店該日臨時需求單
  │    │    ├─ 兩者皆空 → 跳過該日
  │    │    └─ 任一有 → 建一筆需求日記錄
  │    ├─ 對每筆需求日：
  │    │    ├─ 合併兩來源品號（同品號加總）
  │    │    ├─ 補廠商（無廠商者靜默丟棄）
  │    │    ├─ 算預計配送日
  │    │    └─ 依平日／假日套對應量
  │    └─ 批次寫入明細
  ├─ 4. 自動跳轉到該試算單明細頁
  ├─ 5. 在需求日列表點某天 → drill-down 看品項
  └─ 6. 在「我的試算單一覽」回查歷史
```

> **自店過濾**：底層 service 已有此邏輯，但店長視角是否真能被鎖到自店未經驗證（待確認）。

## 6. 欄位規格

### 6.1 試算頁（店長視角輸入）

| 欄位 | 店長視角預設 | 可否修改 |
|---|---|---|
| 區域 | 自店所屬區域 | 否（推論） |
| 門市 | 自店 | 否（推論） |
| 食材中類／小類 | 不填 | 是（進階） |
| 日期區間起／訖 | 下週週一～週日 | 是 |

### 6.2 需求日列表（每天一行）

| 欄位 | 說明 | 店長視角是否顯示 |
|---|---|---|
| 需求日 | 該天日期 | 是 |
| 食材預測單號 | 來源 1 的單號 | 是（可點進兄弟 PRD） |
| 臨時需求單號 | 來源 2 的單號（可多個） | 是 |
| 預計銷售量 | 該日預估萬元 | 否（總部視角為主） |

### 6.3 食材明細（點某天展開）

| 欄位 | 說明 | 店長視角是否顯示 |
|---|---|---|
| 品號 | 系統識別碼 | 否（次要） |
| 品名 | 由廠商報價／物料主檔帶 | 是 |
| 需求數量 | 該天該品要的量 | 是 |
| 廠商名稱 | 由廠商報價帶 | 是 |
| 預計配送日 | 系統反推 | 是 |
| 實際到店日／實際到店數量 | 入庫後回填 | 是（對帳用） |
| 入庫單號 | 入庫後回填 | 是 |

### 6.4 我的試算單一覽（單頭）

| 欄位 | 說明 |
|---|---|
| 試算建立時間 | 我跑試算的時間 |
| 日期區間 | 起～訖 |
| 食材中類／小類 | 若有篩選則顯示 |

## 7. 商業邏輯

### 7.1 鎖店過濾（**未驗證，建議補檢核**）

店長視角進入試算時：

```
1. 取登入者門市 ID
2. 強制：門市條件以登入者為準（覆寫前端傳入）
3. 試算過程：
   ├─ 查食材預測單時加門市條件
   ├─ 查臨時需求單時加門市條件
   └─ 廠商報價查詢以該門市為條件
4. 試算單一覽：自動加門市過濾
```

### 7.2 無廠商過濾的店長視角建議

底層既有行為是「無廠商品號靜默丟棄」。店長視角建議改為：

```
1. 仍寫入明細
2. 廠商名稱欄位顯示「（無廠商）」
3. 標紅或標警示圖示
4. 提示「請聯繫總部採購補建廠商報價」
```

### 7.3 唯讀範圍

店長不應有「廠商報價維護」「物料主檔維護」按鈕。試算單本身雖無法改、無法刪，但這是底層限制（不是店長視角的設計）。

### 7.4 預設值

- 日期區間：下週週一到週日
- 食材中類／小類：不填（全品號）

### 7.5 從試算到後續流程

試算結果寫入後：

- 明細裡的「實際到店日／數量／入庫單號」欄位**已預留**，但寫入邏輯屬於入庫作業（不在本 PRD 範圍）
- 試算結果作為採購單建立的依據（屬於採購管理模組，不在本 PRD 範圍）

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店所有試算單與明細 | 試算、查詢、drill-down |
| 區經理 | 區內所有店 | 同店長，多店切換 |
| 採購規劃／總部 | 全公司 | 試算、查詢（已實作於既有共用 API） |

> 底層權限字串：建立用一組、查詢需求日／明細用另一組（命名不一致；屬底層問題）。店長視角應一律具備「查詢」權限與「建立」（建立 = 跑試算）權限。

|---|---|
| 店長視角專屬 endpoint | 後端僅有 `controller/admin/rawmaterial1/*`，無 `/app/*` 對應路徑 |
| 自店過濾的店長視角驗證 | 共用 API 已有過濾邏輯，未針對店長路徑驗證 |
| 無廠商品號的警示 | 目前靜默丟棄 |
| 試算單更新／刪除 | 後端無對應端點 |
| 簽核流程 | 後端無 BPM 綁定 |
| 重複試算保護 | 同範圍可建多張單 |
| 與當前庫存的並排對照 | 試算結果未帶當前庫存資訊 |
| Excel 匯出 | 後端僅有給物流 MSS 的 CSV，無一般匯出 |
| 行動裝置 UI | 推論：店長使用情境包含現場操作，但目前無前端 |

### 10.3 不包含

- 食材預測（屬於兄弟 PRD：食材需求預測試算表）
- 臨時需求建立（屬於兄弟 PRD：臨時需求審核）
- 物料主檔／包材主檔維護（屬於 PDM 包材維護作業）
- 廠商報價維護（屬於採購管理）
- 採購單建立（屬於採購管理；本表是其輸入）
- 入庫管理（屬於庫存管理；明細表的「實際到店」欄位由入庫流程回填）
- 物流配送車輛追蹤（屬於物流配送查詢）

| 沒有簽核流程是否合理？店長視角是否要把試算結果當作「需經採購主管確認」？ | 業務政策 |
| 試算範圍若跨月（如 5/25–6/5），月配品的配送日邏輯是否正確？ | 邊界 |
| 行動裝置 / PWA 是否為必需？ | 開發範疇 |
| 是否要支援「Excel 匯出（限自店）」？ | UX |
| 與物流行事曆、物流配送查詢、物流串接記錄是否要在 UI 上串聯？ | 體驗一致性 |
