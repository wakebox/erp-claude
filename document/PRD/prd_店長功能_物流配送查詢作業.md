## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

每天我都在等貨。已經規劃好的配送，**派車後**會被打包進一張物流單，由中央倉的司機開車一站一站送。我關心的是：

> 「**這張單到我這裡了嗎？司機開到哪了？預計幾點能到？**」

「物流規劃查詢」告訴我「**該收什麼**」、「物流配送查詢」告訴我「**車到哪了**」。

### 1.2 我要做什麼

- 收貨前 30 分鐘：開啟看「最近哪張單正往我們店送」
- 該到沒到：查狀態（在途／延誤／已抵達）
- 同一天多張單：排點收順序
- 抵達時：確認單據資訊與司機核對
- 異常即時通報：看到延誤 → 聯絡司機或進「物流異動查詢」

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看「現在有沒有車正在送我們店」 | 即時排點收人手 |
| 看預計抵達時間 | 不被動等 |
| 看司機與車牌 | 抵達時對單核對 |
| 看物流單明細 | 收貨時逐項點 |
| 同一天多張單要分先後 | 安排同事輪流接 |
| 只看與自店相關的物流單 | 全公司資料無關 |
| 行動裝置可用 | 賣場現場操作 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 自店相關物流單列表 | 一眼看今日相關的所有派車 |
| 顯示派車狀態（已派／在途／抵達／已簽收） | 知道現在進行到哪一步 |
| 物流單明細展開 | 收貨時清點 |
| 司機／車牌／預計抵達顯示 | 對單核對（待確認資料來源） |
| 跳轉至「物流異動查詢」 | 異常時建異動單 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 物流配送查詢作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合（PDM）」 |
| 兄弟功能 | 物流管理行事曆、物流規劃查詢、物流異動查詢、物流串接記錄 |
| 主要頁面 | 自店物流單列表、物流單明細 |
| 簽核流程 | 無（純查詢） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在「**派車之後、抵達之前**」這段過程中，**即時掌握我的店預期收到的物流單**。

設計理念：
- **聚焦「進行中」狀態**（已派車、在途、待簽收），不是歷史對帳
- 與「物流規劃查詢」串接 — 規劃是派車前，配送是派車後
- 與「物流異動查詢」聯動 — 配送中發現異常 → 建異動單
- **唯讀＋鎖店**

## 3. 業務邏輯背景

### 3.1 物流單怎麼來

物流單由系統每日批次產出（中央倉配送的情況下），或由總部物流規劃人工建立：

```
需求預測明細 (規劃量)
       │
       ▼
   每日批次 → 產生「物流單」(單頭 + 多筆明細)
       │
       ▼
   中央倉收到清單 → 派車送各門市
```

物流單一頭多明細，**一張單可能同時送多家店**。店長視角看到的物流單，是「**至少含我的店其中一筆明細**」的單。

### 3.2 物流單包含什麼

- **單頭**：單據編號（PR + 時間戳）、配送模式、配送廠商代號、出貨日
- **明細**：品項、數量、廠商、需求門市、來源預測單號、實際到店數量、訂單類別

### 3.3 配送狀態（**目前後端未實作**）

底層物流單**沒有狀態欄位**（單頭只有：單據編號、配送模式、配送廠商代號、出貨日）。

「在途／已抵達／已簽收」這些對店長最重要的狀態 — **必須額外加欄位或外部系統串接**：

| 狀態 | 來源 |
|---|---|
| 已派車 | 物流單建立 = 已派 |
| 在途 | 待確認（GPS／物流公司 API／司機回報） |
| 已抵達 | 待確認 |
| 已簽收 | 與庫存入庫聯動（推論） |

> 待確認：漢堡王是否有合作物流公司 API、是否走中央倉的中繼系統。

### 3.4 與兄弟物流功能的差異（店長視角）

| 功能 | 看什麼 | 主要時機 |
|---|---|---|
| 物流管理行事曆 | 未來某天有沒有配送 | 排班 |
| 物流規劃查詢 | 派車前的規劃明細 | 細查、單號追蹤 |
| **物流配送查詢（本功能）** | **派車後車輛狀態、即將抵達** | **收貨當下、追車** |
| 物流異動查詢 | 實際到店 vs 預計差異 | 收貨後對帳 |
| 物流串接記錄 | ERP 給中央倉的批次紀錄 | IT 軌跡 |

> 本功能與「物流異動查詢」共用同一張底層物流單表，差別在「**配送中 vs 配送後**」 — 配送查詢看的是「進行中」、異動查詢看的是「結算」。

### 3.5 「我的店」是怎麼判定的

底層查詢無「自店過濾」邏輯。店長視角需在後端：
1. 從登入身份取得所屬門市
2. 篩出「明細含自店」的物流單
3. 由明細反查單頭

## 4. 情境說明

### 4.1 正常流程 — 收貨前等車

5/19（一）早上 8:40，BK001 信義店店長小陳：

1. 打開「物流配送查詢」
2. 列表顯示今日 1 張單：`PR20260519030000`
3. 看到狀態「在途」、預計抵達 09:15
4. 小陳通知同事 9:00 開始準備卸貨區
5. 9:18 司機抵達，狀態自動變「已抵達」

### 4.2 多單同日 — 排點收順序

某日同時有 2 張單（一張週配、一張月配）：

1. 列表顯示 2 張
2. 週配（食材）預計 09:00；月配（紙袋）預計 14:00
3. 安排早班一人點週配、午班一人點月配

### 4.3 異常情境 — 該到沒到

5/19 10:00，預計 09:15 抵達的車還沒到：

1. 進「物流配送查詢」，看到 `PR20260519030000` 狀態仍「在途」
2. 點開明細，看司機姓名／車牌（推論欄位）
3. 致電司機詢問
4. 若確認延誤 → 跳「物流異動查詢」建延誤紀錄

### 4.4 抵達後核對

司機抵達，遞送單據給店長：

1. 店長手機打開「物流配送查詢」
2. 找到 `PR20260519030000`
3. 點開明細與紙本核對：品項、數量、來源預測單號
4. 若一致 → 簽收（後續走「庫存入庫」）
5. 若有差異 → 跳「物流異動查詢」建短裝／破損紀錄

### 4.5 多店共配車（隱私情境）

同一張物流單送 3 家店，店長只能看到自己店那幾筆明細：

1. 列表顯示 `PR20260519030000`（含自店明細）
2. 點明細時，**只顯示自店該收的品項**
3. 其他店的明細不顯示

> 待確認：是否所有資訊（司機、車牌、路線）都會洩漏其他店訊息。

### 4.6 多店店長

待確認：是否需切換店別查看不同店的物流單。

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「物流配送查詢」]
  |
  +- 取得登入者身份 → 鎖定「自店」
  |
  +- 預設條件：今日／昨日的物流單（自店相關）
  |    +- 明細層先過濾自店 → 反查單頭
  |
  +- 列表呈現：
  |    每列：單據編號、出貨日、配送模式、廠商、預計抵達、狀態
  |
  +- 點某張單 → 明細頁
  |    +- 列：品項、數量、來源單號（限自店）
  |    +- 上方：司機、車牌、預計抵達、目前狀態
  |
  +- 互動：
  |    +- 點「跳異動查詢」→ 建異動單（帶單號）
  |    +- 點「已簽收」→ 走入庫流程（推論）
  |
  +- 從規劃查詢跳入：帶單號自動定位
```

| 司機姓名 | 是（**目前底層無此欄位**） |
| 車牌 | 是（**目前底層無此欄位**） |

### 6.2 明細

| 欄位（業務名） | 店長視角是否該顯示 |
|---|---|
| 品號 | 是（次要） |
| 食材名稱 | 是 |

| 單據編號 | 否 | 模糊／精確 |
| 門市 | 系統自填 | 鎖定為登入者自店 |

## 7. 商業邏輯

### 7.1 自動鎖店（明細層先過濾）

底層查詢需改為：
1. 從登入身份取得所屬門市
2. 在物流單明細表先以「自店」過濾
3. 由命中明細反查單頭
4. 回傳給店長：含自店明細的物流單 + 限自店明細

> 此邏輯在後端**尚未實作**。

### 7.2 預設時間範圍

- 列表預設「今日／昨日」（聚焦進行中）
- 可拉成「最近 7 日」做短期追溯

### 7.3 配送狀態的判定（**未實作**）

短期方案（無物流公司 API 整合時）：
- 物流單建立 = 「已派」
- 司機手動回報 = 「在途／抵達」
- 與庫存入庫聯動 = 「已簽收」

長期方案：串中央倉／物流公司 API 自動更新。

### 7.4 即時提醒（推論）

抵達前 15 分鐘推播提醒店長 — 後端排程未實作。

### 7.5 唯讀

店長視角不開放修改、刪除物流單。底層 API 提供 CRUD（總部專用），店長 UI 應隱藏這些按鈕。

### 7.6 與「物流異動查詢」的關係

兩功能共用同一張物流單表。配送查詢看「進行中」、異動查詢看「結算」 — 後端是同一支查詢端點，差別在前端切換的預設條件與顯示欄位。

## 10. 功能範圍

### 10.1 包含的功能

- 自店相關物流單列表查詢
- 物流單明細展開（限自店明細）
- 派車狀態顯示與切換（待補欄位）
- 跳轉至「物流異動查詢」
- 自店 Excel 匯出

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端無 `/app/*` 對應路徑 |
| 自店過濾（鎖店） | 既有 API 接受任意條件、無依登入者過濾 |
| 派車狀態欄位 | 物流單主表無此欄位 |
| 司機／車牌／預計抵達欄位 | 物流單主表無 |
| 物流公司 API 整合（GPS、即時位置） | 後端無此整合 |
| 抵達前推播提醒 | 後端排程未實作 |
| 自店 Excel 匯出限制 | 既有 Excel 匯出無門市過濾 |

### 10.3 不包含

- 建立物流單（屬中央倉系統與總部）
- 修改物流單（總部專用）
- 收貨點收實際入庫（屬「庫存入庫作業」）
- 異動單建立（屬「物流異動查詢」）
- ERP 與中央倉的批次紀錄（屬「物流串接記錄」）
