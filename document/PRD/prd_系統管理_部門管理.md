## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員（資訊／IT 角色），同時也是負責建立組織結構的人。
公司有總部、各事業處、各部門、各門市，每一個成員都歸屬一個「部門」。誰能看到哪些單據、報表、人員清單，背後都要靠這顆組織樹切資料權限。

### 1.2 我要做什麼

- **公司新成立部門時**：在組織樹的對應父節點下建一個新部門節點（例如「總部—資訊部」底下開「資訊部—開發課」）。
- **既有部門調整**：改名、改顯示順序、改部門負責人、更新部門電話／信箱。
- **既有部門裁撤**：先把該部門下的子部門處理掉、再把該部門停用或刪除。
- **設定部門負責人**：把該部門的「主管」對應到一位 ERP 使用者，這位使用者在「我管的部門」相關查詢可以看到自己負責的部門。
- **盤點組織樹**：用部門名模糊搜尋找節點、依「啟用／停用」狀態過濾查看。
- **回應其他模組**：採購、PDM、倉儲建單時要選「歸屬部門」，會打到我這裡來驗證「該部門是否啟用、是否存在」。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一個地方能看見整套組織樹的全貌 | 部門多、層級多，散在文件或 Excel 上難盤點 |
| 新建部門時要擋下亂選父節點、撞名 | 避免組織樹出現環路或同層撞名造成資料誤掛 |
| 改部門時要擋下「把自己的子部門設成自己的父部門」 | 一旦環路成立，遞迴查詢「子部門集合」會吃光記憶體或無窮迴圈 |
| 刪除部門時要擋下「還有子部門」的情況 | 避免子節點變孤兒、組織樹斷裂 |
| 停用一個部門時，其他模組在建單時要查不到它 | 過渡期解編、組織重整必備 |
| 一筆部門設定改完後，其他模組（如使用者管理）的「部門下展查詢」要立刻反映 | 維運成本：不應該要重啟服務 |
| 知道某位使用者是哪些部門的負責人 | 主管登入後想看自己管的所有部門 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 新增／修改／刪除部門 | 維護組織樹本體 |
| 條件查詢部門清單（含名稱模糊、狀態篩選） | 維運盤點 |
| 取得單一部門明細 | 編輯前回填 |
| 提供精簡部門清單（給下拉選單用） | 其他模組／頁面填表時用 |
| 提供「子部門編號集合」查詢（含快取） | 使用者管理之「部門下展查詢」、資料權限框架需要 |
| 提供「依負責人反查所負責部門」查詢 | 主管視角需求 |
| 提供「批次驗證部門編號＋啟用狀態」內部 API | 給其他模組建單時防呆 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 部門管理 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 使用者管理、角色管理、選單管理、崗位管理、字典管理 |
| 主要頁面 | 部門樹狀列表頁、部門新增／編輯彈窗 |
| 簽核流程 | 無（部門新增、修改、刪除皆即時生效，不綁簽核） |

## 2. 功能目的

提供一套**集中、樹狀、可驗證啟用狀態**的組織結構主檔。一筆「部門」紀錄同時承擔：

1. **組織歸屬定義**：每位 ERP 使用者掛在唯一一個部門下，影響其資料可見範圍。
2. **資料權限切分基礎**：使用者管理的「部門下展查詢」、資料權限框架的「本部門及以下」邏輯都倚賴此處的父子關係。
3. **跨模組防呆來源**：採購、PDM、倉儲等模組在「歸屬部門」欄位處選擇時，會回頭向本模組批次驗證部門編號是否存在、是否啟用。
4. **聯絡資訊載體**：負責人、電話、信箱讓跨部門協作有人找。

> **重要過渡背景**：經系統整合決策（見 `erp-claude` 記憶 `project_user_integration_decision.md`，2026-05-16），漢堡王整體 IT 走向以 Laravel OA 為使用者與組織單一事實來源。**部門資料**屬於組織主檔，長期規劃中將與 OA 對齊（推論：階段 5 後採同步或廢棄）。本 PRD 描述的是過渡期內、ERP 仍自管部門主檔時的需求；負責人欄位仍引用 ERP 使用者編號，未來會以 OA 員工編號為主。

## 3. 業務邏輯背景

### 3.1 樹狀結構的層級無上限（但有實作上限）

部門是一棵樹，每個節點有一個父節點，根層級的父節點以「根節點代號 0」表示。實作上以一個 `Short.MAX_VALUE`（32,767）的安全閥防止異常資料造成的死迴圈，正常業務遠遠用不到。

### 3.2 三道驗證守住組織樹的完整性

新增與修改一筆部門時，依序檢查三件事：

| 驗證 | 新增時 | 修改時 |
|---|---|---|
| 自己存在 | N/A | ✓ |
| 父節點合法 | ✓ | ✓ |
| 同層名稱唯一 | ✓ | ✓（排除自己） |

「父節點合法」的細項：

- 父節點為「根」（0）時直接放行
- 不能把自己設成自己的父節點
- 父節點必須存在
- **環路檢查**（僅修改時）：沿父節點往上一直回溯，若某一層追到了自己的編號，代表「把自己的子部門設成父部門」，整體拒絕

### 3.3 同層唯一性而非全系統唯一

部門名稱僅在「同一個父節點底下」唯一。不同父節點下可以重名（例如「總部—資訊部」與「台北分公司—資訊部」是允許的）。

### 3.4 刪除採物理刪除＋擋路保護

- 刪除時若該部門仍有子部門 → 整體拒絕
- 通過保護後直接物理刪除（藉由框架的軟刪除欄位機制，技術上仍可從歷史紀錄回查）
- **不**會自動處理：該部門下所掛載的使用者、所引用的單據。離開部門的使用者依然帶著已被刪除的部門編號（推論：需由「使用者管理」端另外處理，列入 §11）

### 3.5 子部門集合的快取設計

「給我這個部門以下所有子部門的編號清單」是高頻查詢（使用者分頁、資料權限框架都用）。本模組對此維護一份快取：

- **鍵**：部門編號
- **值**：所有後代部門編號集合
- **過期策略**：新增／修改／刪除任一筆部門 → 整批清空（因不知道牽動哪一支線）
- **建立時禁用資料權限**：避免「呼叫者剛好被資料權限限制了視野」造成快取裡缺資料，後面別的呼叫者撈到不完整集合

### 3.6 多租戶支援

部門表本身繼承多租戶基礎，每筆部門有所屬租戶，跨租戶不可見不可選。

### 3.7 部門狀態只有「啟用」「停用」兩態

沒有「準備中」「歷史」這類中間狀態。當組織重整或部門裁撤時，常見做法：

- 短期停辦 → 停用（保留記錄、人員仍可掛但其他模組無法新建）
- 永久裁撤且無關聯資料 → 刪除

## 4. 情境說明

### 4.1 正常流程 — 新增一個部門

IT 阿明收到通知：「新成立『門市營運部—北區課』」。阿明操作：

- 在組織樹中找到父節點「門市營運部」並按「新增子部門」
- 部門名稱：`北區課`
- 父部門：`門市營運部`（系統由樹中自動帶入）
- 顯示順序：`10`
- 負責人：選使用者「李大華」
- 聯絡電話：`02-1234-5678`
- 信箱：`bk-north@bk.com.tw`
- 狀態：啟用

送出後系統檢查：父部門存在 → 通過；同層下沒有別的子部門叫「北區課」 → 通過。建立成功，系統清空所有「子部門集合」快取（因為「門市營運部」與更上層節點的子部門集合都該包含這筆新資料）。

### 4.2 正常流程 — 修改部門負責人

「採購部」原負責人離職，要換成新主管小王。阿明點開「採購部」的編輯彈窗，把負責人改為小王，其他欄位不動。送出後系統依序檢查：自己存在 → 通過；父節點未變、合法 → 通過；同層名稱（採購部）唯一性檢查時抓到自己，但 ID 相同 → 通過。寫入成功，子部門集合快取整批清空。

### 4.3 異常情境 — 把子部門設成父部門

阿明手滑，要把「採購部」的父節點改成「採購部—國際採購組」（這是採購部的子部門）。送出後系統沿父節點往上回溯：採購部—國際採購組 → 採購部，發現追到了「採購部」這個自己。系統回應「不能设置自己的子部门为父部门」並拒絕。

### 4.4 異常情境 — 把自己設成自己的父部門

更直白的錯誤：阿明把「採購部」的父節點直接選成「採購部」自己。系統第一關就擋下，回應「不能设置自己为父部门」。

### 4.5 異常情境 — 同層名稱重複

阿明在「總部」底下要新增第二個「資訊部」（已有一個）。送出後系統回應「已经存在该名字的部门」並拒絕。同層下不可重名。

### 4.6 異常情境 — 父部門不存在

阿明從外部匯入工具直接呼叫 API，帶了一個不存在的父部門編號 `99999`。系統回應「父级部门不存在」並拒絕。

### 4.7 異常情境 — 刪除一個還有子部門的節點

阿明想把「PDM 部」整個刪掉，但底下還掛著「PDM—食材組」「PDM—包材組」。系統回應「存在子部门，无法删除」並拒絕。阿明必須先逐一處理子部門。

### 4.8 跨模組情境 — 採購建單選了一個已停用的部門

採購人員小李新增請購單，「歸屬部門」下拉裡看不到已停用的「採購部—臨時專案組」（精簡清單會過濾），但他不死心、直接用部門編號 `4012` 走 API 建單。採購模組在落地前呼叫「批次驗證部門編號」內部 API，系統回應「部门(採購部—臨時專案組)不处于开启状态，不允许选择」，拒絕建單。

### 4.9 跨模組情境 — 使用者管理之「部門下展查詢」

阿明在「使用者管理」頁查「採購部以下所有人」，使用者管理服務向部門服務取出「採購部」之後代部門編號集合（透過快取）。快取若存在 → 直接命中；若被前一筆部門新增操作清掉了 → 重新計算並寫回。

### 4.10 主管視角情境 — 主管查自己管的部門

主管小王登入 ERP，前端要他的「我負責的部門」清單。系統以小王的使用者編號到部門表反查 `負責人 = 小王` 的所有部門。回傳：「採購部、採購部—國際採購組」。

### 4.11 異常情境 — 修改一個不存在的部門

阿明拿到一個來路不明的部門編號 `99999` 並嘗試更新。系統回應「当前部门不存在」並拒絕。

## 6. 欄位規格

### 6.1 部門主檔欄位

| 欄位（業務名） | 型別／格式 | 必填 | 規則 | 備註 |
|---|---|---|---|---|
| 部門編號 | 數字 | 系統產生 | 全租戶內唯一 | 不可由使用者編修 |
| 部門名稱 | 文字 | 是 | 最長 30 字、同層唯一 | 顯示用 |
| 父部門編號 | 數字 | 否（未填預設為根 0） | 必須存在或為根、不可自指、不可形成環路 | — |
| 顯示順序 | 整數 | 是 | — | 同層內由小到大排序 |
| 部門負責人 | 使用者引用 | 否 | 引用 ERP 使用者編號（推論：未驗證該使用者是否存在、是否啟用） | 主管視角倚賴此欄位 |
| 聯絡電話 | 文字 | 否 | 最長 11 字（看似為手機格式約束，但市話亦可填） | — |
| 信箱 | 文字 | 否 | 合法 Email 格式、最長 50 字 | — |
| 部門狀態 | 列舉 | 是 | 啟用 / 停用（依共用狀態列舉） | — |
| 所屬租戶 | 系統寫入 | 系統寫入 | 由多租戶框架自動寫入 | 跨租戶不可見 |

## 7. 商業邏輯

### 7.1 新增的前置檢核（必過 2 關）

1. **父節點合法**：為根、或父節點存在
2. **同層名稱唯一**

未經檢核的欄位（如負責人是否存在、電話格式）不在後端強制驗證，前端責任。

### 7.2 修改的前置檢核（必過 3 關）

1. **自己存在**
2. **父節點合法（含環路檢查）**
3. **同層名稱唯一（排除自己）**

### 7.3 環路檢查的演算法

僅在「修改」時觸發。流程：

1. 取本次表單要設的父節點主檔
2. 沿著父節點的父節點一路往上抓
3. 任一層的父節點編號 = 自己編號 → 視為環路，拒絕
4. 抓到根節點或父節點失蹤 → 結束、視為合法
5. 最多迴圈 32,767 次（安全閥）

> **註**：此演算法的時間複雜度為 O(層深)，每往上一層觸發一次資料庫查詢。極端組織樹（深度 100+）會造成 100+ 次查詢。一般組織深度 5–10，無虞。

### 7.4 同層名稱唯一性的判斷

- 以「相同父節點 ＋ 相同名稱」查單筆紀錄
- 找不到 → 通過
- 找到、且新增情境 → 視為重複，拒絕
- 找到、且修改情境：若該筆編號 = 本次自己 → 通過；否則拒絕

### 7.5 刪除的擋路保護

- 該部門必須存在
- 該部門不可有子部門（以 `父部門編號 = 自己` 計數 > 0 判定）
- 通過後直接刪除主檔，**不**做任何後續處理：
  - 不會把該部門下的使用者「孤兒化處理」（推論：使用者管理需自行清理）
  - 不會處理引用該部門的歷史單據

### 7.6 子部門集合的快取設計細節

- **建立**：在計算當前部門的後代集合時，**臨時關閉資料權限**。理由：若某使用者只能看見「總部」以下，他建快取時撈到的集合會偏少，後續其他使用者（看得更廣的）拿到的快取就會缺資料。
- **失效**：所有新增、修改、刪除操作後，整批快取清空（`allEntries = true`）。理由：操作一筆部門會牽動其所有祖先節點的子部門集合，精準 evict 不划算。

> 此設計犧牲了一些快取效率以換取資料一致性。列入 §11 評估。

### 7.7 跨模組驗證的回應格式

`validateDeptList` 是給其他模組（採購、PDM 等）建單時批次驗證部門用的內部 API。它的回應採「異常驅動」：

- 任何一筆找不到 → 拋「当前部门不存在」例外（不會把其他失敗都收集後一次回報）
- 任何一筆狀態為停用 → 拋「部门(XX)不处于开启状态，不允许选择」例外

> 此設計遇到多筆異常時只會回報第一個，呼叫端拿不到完整失敗清單。列入 §11。

### 7.8 多租戶隔離

部門表繼承多租戶基礎類別，所有查詢自動加上「租戶 = 當前租戶」條件。跨租戶不可見不可選。多租戶過濾由框架自動處理，本服務無需顯式處理。

## 10. 功能範圍

### 10.1 包含的功能

- 部門之新增、修改、刪除
- 全部清單查詢（支援名稱模糊、狀態篩選；維運用）
- 精簡清單查詢（給其他模組下拉用）
- 單筆部門明細查詢
- 取得某部門以下所有子部門編號集合（內部 API、含快取）
- 依負責人反查所負責部門（內部 API）
- 批次驗證部門編號＋啟用狀態（內部 API）

### 10.2 預留但尚未實作

- **部門負責人合法性驗證**：欄位接受任意使用者編號，不檢查該使用者是否存在、是否啟用、是否屬於同租戶（推論：來源 `DeptSaveReqVO.java`、`DeptServiceImpl.java:54`）。
- **部門狀態變更的連動處理**：將一個部門從「啟用」改成「停用」時，**不會**自動處理其下的子部門。理論上若父節點停用、子節點應視為不可用（與選單管理之精簡清單的父鏈遮蔽邏輯一致），目前部門模組未實作此語意（列入 §11）。

### 10.3 不包含

- 部門下掛載的使用者（在「使用者管理」模組，使用者 DO 帶部門編號欄位）
- 部門所引用的單據（歷史單據之「歸屬部門」欄位由各業務模組自行設計）
- 崗位／角色（在「崗位管理」「角色管理」模組）
- 跨租戶共享部門（多租戶機制本身）
