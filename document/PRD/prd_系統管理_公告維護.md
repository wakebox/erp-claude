# PRD｜系統管理 — 公告維護

> 來源：逆向自 `kingmaker-module-system` 之公告相關後端程式碼（含 `controller/admin/notice/NoticeController.java`、`service/notice/NoticeServiceImpl.java`、`dal/.../notice/NoticeDO.java` 等）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

> **本 PRD 範圍**：本程式同時處理三種公告類型（通知、公告、行銷活動），共用同一張資料表與 CRUD 端點。本 PRD 聚焦於「通知」與「公告」兩種類型（即 type=1、2）的維護面；「行銷活動」（type=3）獨有的「接收人指派、我的公告」邏輯由另一份 PRD（行銷活動公告）描述。

---

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員或公司公關／HR／IT 中具有公告發佈權的人。
公司有訊息要全體員工看到（系統升級、政策更新、節慶通知、停機維護），就由我用這個功能把訊息掛到 ERP 首頁，所有人登入時都會看到。

### 1.2 我要做什麼

- **系統公告**：例如「本週六 22:00–24:00 系統維護，請提前離線」要發佈一則公告。
- **政策／規章變更**：HR 修改差勤辦法、財務調整報帳流程，要發布一則「公告」讓全員知悉。
- **節慶通知**：母親節活動、農曆春節辦公時程要通知全員。
- **既有公告的修改**：上線後發現錯字、時間有誤要回去改、改完仍然顯示。
- **下架／刪除**：活動結束、過時資訊要刪掉或停用。
- **手動推播**：發完一則重要訊息後，想立刻通知**目前在線**的同仁看到（而不是等他下次刷新頁面）。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一個地方能集中發布／管理全公司公告 | 散在 LINE 群組、Email 容易漏看 |
| 發完後可以隨時修改、停用、刪除 | 內容打錯、活動取消都是常態 |
| 有「狀態」可控制這則公告是否顯示給員工 | 草稿先打好、活動開始當天再啟用 |
| 重要公告能立刻推給目前在線的人 | 系統維護等緊急訊息不能等使用者主動刷新 |
| 列表頁可分頁、可搜尋標題、可依狀態篩選 | 公告數量會越積越多 |
| 公告類型可區分（系統通知、一般公告、行銷活動） | 不同類型有不同顯示位置與接收對象規則 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 新增／修改／刪除公告 | 公告生命週期 |
| 條件分頁查詢公告（標題模糊、狀態篩選） | 維運盤點 |
| 取得單一公告明細 | 編輯前回填 |
| 啟用／停用公告 | 草稿與下架的中間狀態 |
| 手動推播公告（給在線使用者） | 緊急訊息即時送達 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 公告維護 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 使用者管理、角色管理、部門管理、選單管理、操作日誌、登入日誌、行銷活動公告（共用本程式之 type=3 分支） |
| 主要頁面 | 公告列表頁、公告新增／編輯頁、公告預覽彈窗 |
| 簽核流程 | 無（新增、修改、刪除即時生效，不綁簽核） |

## 2. 功能目的

提供一個**集中、簡單、可即時推播**的公司內部訊息發佈中心。本功能將「通知（type=1）」「公告（type=2）」「行銷活動（type=3）」三種類型統一在同一張資料表與同一組 CRUD 端點，僅以「類型」欄位區分。設計上：

1. **通知／公告**：所有員工都能看到（不需要指派接收人），任何登入使用者皆可在自己的清單拉到這類訊息。
2. **行銷活動**：需要指派接收對象（依部門或依使用者），只有被指派者才看得到。本 PRD 不展開此分支邏輯。
3. **手動推播**：透過 WebSocket 把公告即時推給「**當下在線**的後台使用者」。離線者下次登入時仍能在公告列表看見。

> **重要過渡背景**：本模組屬於 ERP 系統管理底下的輔助功能，與「使用者單一事實來源由 Laravel OA 接管」的整合方向關聯較低。但「接收人」之指派與「我的公告」之查詢仰賴使用者／部門主檔；OA 整合後，這兩支查詢的人員／部門來源預期改以 OA 為準。本 PRD 描述的是過渡期內、ERP 仍自管使用者主檔時的需求。

## 3. 業務邏輯背景

### 3.1 公告分三種類型，行為不同

| 類型 | 程式代號 | 行為 |
|---|---|---|
| **通知** | 1 | 所有員工可見，無接收人指派 |
| **公告** | 2 | 所有員工可見，無接收人指派 |
| **行銷活動** | 3 | 僅指派的接收人可見；新增／修改時需同時維護接收人關聯 |

「通知」與「公告」在程式行為上**完全相同**，僅由「類型」欄位作標籤、給前端依類型分頁或分區顯示用（推論：實際 UI 區隔由前端決定）。

### 3.2 「狀態」就兩態

每則公告有「啟用／停用」狀態：

- 啟用：在公告列表頁正常顯示
- 停用：在維運列表仍看得到，但前端面向員工的列表會過濾掉（推論：篩選由前端或角色配置決定，後端「我的公告」端點目前**不**按狀態過濾，列入 §11）

### 3.3 寫入採完整覆寫策略

修改一則公告時，伺服端：

1. 先驗證該公告存在
2. 直接以表單內容整筆覆寫主檔
3. 把該公告現有的接收人關聯**全部刪除**
4. 若新類型為「行銷活動」，重新建立接收人關聯

這代表即便修改的是通知或公告（不需要接收人），系統仍會走「刪除舊接收人」這一步——只是因為這類公告本就沒有接收人關聯，所以效果為無。

### 3.4 手動推播只推「在線」的使用者

公告維護人按「推播」時：

1. 系統取得該公告完整內容
2. 透過 WebSocket 把完整公告物件廣播給所有目前**連線中的後台使用者**
3. 離線使用者拿不到此次推播，但下次登入時可在公告列表看到該公告

推播主題為 `notice-push`，使用者類型限定為「後台使用者」（即 admin 端）。

### 3.5 三支「無需權限」的端點

本控制器有 3 個 API 在程式上**未啟用權限驗證**：

| 端點 | 用途 | 風險 |
|---|---|---|
| `頁碼公告-無權限` | 為前端首頁提供公告列表，不要求權限 | 任何登入者可呼叫 |
| `取得公告-無權限` | 為前端首頁提供單筆明細 | 任何登入者可呼叫 |
| `我的公告` | 取得當前使用者可見的公告（含被指派的行銷類型） | 限定要登入（取登入者編號），但無權限字串 |

「無權限」端點是給前端公開呈現用，例如首頁不論角色都能看到全公司公告。但這意味著只要任一登入者就能拉到後台所有公告。是否要求「至少登入」由 Spring Security 預設策略決定，未額外限制（列入 §11）。

### 3.6 多租戶＋共用 BaseDO

公告主檔繼承一般 BaseDO（**未**繼承多租戶基礎），意即公告是**跨租戶共用**。所有租戶看到同一份公告主檔。

> **註**：與「部門」「使用者」這些跨租戶不可見的主檔不同。若日後改為「按租戶各自發佈公告」需要改 DO 與所有 Mapper（列入 §11）。

## 4. 情境說明

### 4.1 正常流程 — 發布一則系統維護公告

IT 主管阿明要發布「2026-05-22 (週五) 22:00–24:00 ERP 系統維護」的公告。他在「公告維護」頁點「新增」：

- 標題：`5/22 系統維護通知`
- 類型：通知（type=1）
- 內容：`本週五 22:00 起進行系統升級，預計 24:00 完成，期間將無法登入...`（富文本）
- 狀態：啟用

送出後系統建立公告紀錄，回到列表頁可看到。所有員工下次登入或刷新首頁時即可看到此公告。

### 4.2 正常流程 — 修改一則公告

阿明發完維護公告後發現時間打錯，把 22:00 改成 23:00。他點「編輯」、改內容、送出。系統檢查公告存在 → 通過，直接覆寫主檔內容。順便清空接收人關聯（本公告類型為「通知」，沒有接收人，所以無實際變動）。

### 4.3 正常流程 — 手動推播

阿明發完一則「系統 30 分鐘後緊急停機」的通知，怕大家沒看到。他在列表頁找到該公告、點「推播」。系統把該公告完整物件透過 WebSocket 推給目前在線的所有後台使用者，他們的瀏覽器立刻彈出通知。離線的同仁要等下次登入才看得到。

### 4.4 異常情境 — 推播一個不存在的公告編號

阿明手動呼叫推播 API 帶了一個不存在的編號 `99999`。系統回應「公告不能为空」（採用 hutool 的 `Assert.notNull`，回應為原生例外格式）並中止。

> **註**：此處錯誤訊息格式與專案常規的 `ServiceExceptionUtil.exception` 不一致。列入 §11。

### 4.5 業務規則分流情境 — 修改時連帶清接收人

阿明修改一則「行銷活動」公告（type=3），把活動延期改了標題與時間，**但接收人未變**。送出後系統流程：

1. 覆寫公告主檔
2. **刪除**該公告所有接收人關聯
3. 因類型仍為行銷活動 → 重新建立接收人關聯（依新表單的部門／使用者集合）

意即就算阿明沒動接收人欄位，他的前端表單也必須**把原本的接收人完整帶回來**送上去，否則接收人就被清空了（推論：列入 §11）。

### 4.6 異常情境 — 刪除一個不存在的公告

阿明拿到一個來路不明的公告編號 `99999` 並嘗試刪除。系統回應「公告不存在」並拒絕。

### 4.7 一般使用者情境 — 看自己的公告

員工小李登入後進首頁，前端呼叫「我的公告」端點：

1. 系統先查小李對應的「行銷活動」公告編號清單
2. 系統分頁查所有**非行銷類型**的公告（通知＋公告，所有人都看得到）
3. 若小李有被指派任何行銷活動 → 分頁查那些行銷類型公告
4. 兩段結果合併、依建立時間倒序排序、手動分頁後回傳

> **註**：「我的公告」採「先合併再手動分頁」策略，因此回傳的「分頁總筆數」會是合併集合總數，但分頁切片是程式內手動切，可能與其他分頁端點的行為不一致（列入 §11）。

### 4.8 異常情境 — 公告類型欄位帶了不在列舉內的值

阿明從外部工具呼叫 API，帶了 `type=99`。系統**不會檢查類型合法性**，照常建立。後續顯示時前端可能找不到對應的標籤而出錯（推論：列入 §11）。

## 6. 欄位規格

### 6.1 公告主檔欄位

| 欄位（業務名） | 型別／格式 | 必填 | 規則 | 備註 |
|---|---|---|---|---|
| 公告編號 | 數字 | 系統產生 | 全系統唯一 | 不可由使用者編修 |
| 公告標題 | 文字 | 是 | 最長 50 字 | — |
| 公告類型 | 整數 | 是 | 1=通知、2=公告、3=行銷活動 | 後端**未做列舉合法性檢查** |
| 公告內容 | 文字 | 否（推論） | 富文本 | 後端未限制長度 |
| 公告狀態 | 整數 | 否（推論） | 啟用 / 停用 | VO 上**未標 `@NotNull`**，但 Schema 上標 required；列入 §11 |
| 建立人 | 系統寫入 | 系統寫入 | 由框架自動帶入 | 唯讀 |
| 建立時間 | 日期時間 | 系統寫入 | — | 唯讀 |
| 更新人 | 系統寫入 | 系統寫入 | 由框架自動帶入 | 唯讀 |
| 更新時間 | 日期時間 | 系統寫入 | — | 唯讀 |

> 來源：`NoticeDO.java`、`NoticeSaveReqVO.java`、`NoticeRespVO.java`

### 6.2 接收人關聯欄位（僅行銷類型用）

| 欄位 | 型別 | 必填 | 備註 |
|---|---|---|---|
| 關聯編號 | 數字 | 系統產生 | — |
| 公告編號 | 數字 | 是 | 對應公告主檔 |
| 使用者編號 | 數字 | 是 | 對應使用者主檔 |
| 已讀狀態 | 布林 | 系統寫入 | 預設未讀 |

> 來源：`NoticeUserDO.java`

### 6.3 表單上的接收人欄位（行銷類型用）

| 欄位 | 規則 |
|---|---|
| 接收部門編號清單 | 多選；系統會自動展開到所有子部門 |
| 接收使用者編號清單 | 多選；自動去重；當前登入者自動加入 |

> 來源：`NoticeSaveReqVO.java`、`NoticeServiceImpl.java:136-177`

### 6.4 查詢輸入欄位

| 欄位 | 規則 |
|---|---|
| 公告標題 | 模糊比對 |
| 公告狀態 | 等於 |

> 來源：`NoticePageReqVO.java`、`NoticeMapper.java`

### 6.5 回傳補充欄位

`NoticeRespVO` 額外回傳：建立人、更新人、更新時間、接收部門清單、接收使用者清單。

> **註**：「接收部門清單」「接收使用者清單」雖在 VO 中定義，但 Service 端**未實作回填**。前端編輯既有公告時若想預先帶入接收人，目前拿不到（推論，列入 §11）。

## 7. 商業邏輯

### 7.1 新增無前置驗證（只驗欄位非空）

新增公告時，後端**不做**：

- 類型是否在列舉內
- 接收人指派的部門／使用者是否存在
- 狀態是否在列舉內
- 內容長度是否合理

僅依 VO 的 `@NotBlank`、`@Size`、`@NotNull` 做欄位層驗證。

### 7.2 修改採完整覆寫＋接收人重建

修改公告的處理順序：

1. 驗證公告存在
2. 以表單覆寫主檔
3. **無條件**刪除該公告所有現有接收人關聯
4. 若新類型為行銷活動 → 重新依表單建立接收人關聯

對「通知」「公告」類型而言，第 3 步刪除沒有實際效果（本就沒接收人）；對「行銷活動」類型而言，必須由前端把原本的接收人清單完整帶回來，否則會被清空。

### 7.3 推播僅作用於 WebSocket 在線使用者

- 不會發送 Email 或其他外部通知
- 離線使用者拿不到推播
- 推播目標限定為「後台類型」使用者（admin 端）
- 推播訊息主題：`notice-push`
- 推播內容：完整公告物件（含 ID、標題、類型、內容、狀態）

### 7.4 「我的公告」分頁策略

採「兩段查詢 + 程式內合併 + 程式內切分頁」：

1. 先取使用者被指派的行銷類型公告編號集合
2. 用同一份分頁條件分別查「非行銷類型」與「行銷類型（限定編號）」
3. 把兩段結果合併
4. 依「建立時間」倒序排序
5. 程式內依分頁參數切片

問題：兩段查詢各自帶 `pageNo/pageSize`，但只有合併後才切實際的分頁。對效能與分頁正確性都有疑慮（推論，列入 §11）。

### 7.5 行銷類型的接收人展開規則

- **指定部門**：系統取得「該部門 + 所有後代部門」之集合，再取該集合下所有使用者
- **指定使用者**：直接加入接收人集合
- **當前登入者**：發布人自己會自動成為接收人（理由推測：發布完想自己能在「我的公告」看到）
- **去重**：以 `Set` 收集，自動去重
- **已讀狀態**：每筆預設「未讀」

### 7.6 刪除採物理刪除＋連動清接收人

- 刪除公告主檔：直接 `deleteById`
- 同時刪除該公告所有接收人關聯（即便為通知／公告類型，本就無關聯，此步無實際效果）
- 整個刪除動作包在資料庫交易內，任一步失敗整批回滾

### 7.7 三支端點未啟用權限驗證

| 端點 | 是否需要登入 | 用途 |
|---|---|---|
| 公開分頁查詢（無權限版） | 取決於 Spring Security 預設策略 | 前端首頁公開呈現 |
| 公開單筆查詢（無權限版） | 同上 | 前端首頁點開單筆 |
| 我的公告 | 是（程式內取登入者編號） | 前端使用者中心 |

「無權限」端點若未在 Security 設定中明確允許未登入，仍需登入；但不會驗證任何業務權限。列入 §11。

### 7.8 跨租戶共用

公告主檔繼承一般 BaseDO（未繼承 TenantBaseDO），所有租戶共享同一份公告。實作上：

- A 租戶建的公告 B 租戶看得到
- 「我的公告」未過濾租戶，可能看到別租戶建的公告
- 對單一漢堡王台灣使用情境影響不大；多租戶情境需重新評估（列入 §11）

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| **公告管理員（具備「系統-公告-查詢」「-新增」「-修改」「-刪除」全部權限）** | 全部公告 | 新增、修改、刪除、推播 |
| **僅具備「系統-公告-查詢」者** | 全部公告 | 僅瀏覽公告維運列表 |
| **任何登入使用者** | 透過「我的公告」「公開分頁」端點看到所有非行銷類型公告，加上自己被指派的行銷類型公告 | 僅檢視 |
| **未配發本模組權限者** | 仍可透過三支「無權限」端點看到公告 | 僅檢視 |

> 設計上把「公告維運列表」與「員工面公告閱讀」拆成兩組端點：前者要求權限、後者不要求權限（任意登入者均可閱讀）。

## 10. 功能範圍

### 10.1 包含的功能

- 公告之新增、修改、刪除（涵蓋三種類型：通知、公告、行銷活動）
- 標題模糊、狀態篩選之分頁查詢
- 取得單一公告
- 啟用／停用（靠「狀態」欄位）
- 手動推播給在線後台使用者（WebSocket）
- 三支對員工面的公告查詢端點：公開分頁、公開單筆、我的公告

### 10.2 預留但尚未實作

- **接收人回填**：`NoticeRespVO` 有 `deptIds` / `userIds` 欄位，但 Service 未實作回填。前端編輯既有公告時拿不到原本的接收人清單。
- **公告類型列舉檢查**：後端不檢查 `type` 是否為 1/2/3，可能寫入未知值。
- **公告狀態列舉檢查**：VO 未對 `status` 標 `@NotNull` 與列舉檢查，可能寫入空值或未知值。
- **公告排程上下架**：目前僅靠「狀態」手動切換，未提供「自動於某時間上下架」機制（一般公告系統常見功能）。

### 10.3 不包含

- 行銷活動公告之「接收人指派、我的公告、合併分頁」深度規格（在「行銷活動公告」PRD）
- 推播至 Email、簡訊、LINE 等外部通道（僅 WebSocket）
- 已讀／未讀的更新與查詢介面（接收人關聯雖有 `readStatus` 欄位，但本模組未提供更新 API；列入 §11）
- 公告附件上傳（資料表無附件欄位）
- 公告與選單／角色的可見性綁定（除「無權限」端點外，皆走標準權限字串）
