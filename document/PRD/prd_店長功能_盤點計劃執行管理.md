## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長 / 副店長**（例：BK001 信義店）。

每月底（有時是季底、年底），總部會排一張「月度盤點計畫」並分派到我們店。我要在計畫排定的時段內，把計畫指定的食材清單**一項不漏**地清點完畢、把實盤值填入系統、必要時拍照與說明差異原因、送區經理覆核。覆核通過後，差異會透過盤盈虧單調整庫存帳。

> 「2026-05 月度計畫 CP202605-001：信義店、5/31 14:00～18:00 執行、含 87 項食材、容忍度 ±3%、差異需拍照、區經理覆核」

### 1.2 我要做什麼

- 看「分派到我店」的盤點計畫清單（含執行時段、進度）
- 進入計畫 → 取得指定的食材清單
- 對每食材清點、輸入實盤量
- 超出容忍度時必填差異原因（字典化）、附照片
- 提交區經理覆核
- 被退回時重盤
- 計畫完成後檢視差異報表

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一打開只看到「分派給我店」的計畫 | 不要被全公司計畫淹沒 |
| 看得到計畫時段與「還剩多少時間」 | 避免逾時 |
| 進度條（已盤幾項 / 總幾項） | 大計畫要分次做 |
| 中途可暫存、不會掉資料 | 廚房有客人會打斷 |
| 差異原因字典化、可一鍵選 | 不要每次打字 |
| 平板或手機操作、可拍照 | 在實際倉位填，不跑回辦公室 |
| 被退回時看得到區經理意見 | 知道哪裡要重盤 |
| 已盤項目鎖定避免誤改 | 防呆 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 自店「待執行計畫」清單 | 入口 |
| 計畫明細頁（含食材清單、進度、容忍度） | 主畫面 |
| 實盤輸入（每食材一列） | 主要操作 |
| 差異原因（字典）+ 拍照 | 留痕 |
| 暫存／續盤 | 操作彈性 |
| 一鍵提交送簽 | 結束流程 |
| 簽核退回後重盤 | 修正 |
| 計畫差異報表 | 事後對帳 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 盤點計劃執行管理（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理（WHS）」 |
| 兄弟功能 | 每日盤點作業（店長視角）、盤點計劃製定（總部）、盤點作業（盤盈虧結算） |

## 2. 功能目的

讓店長能在總部排定的時段內，**有節奏、可追蹤、留得下證據地** 完成一次「正式盤點」，產出三項輸出：

1. **每食材的實盤量** — 結帳對帳基礎
2. **差異說明與佐證** — 為盤盈虧的會計處理提供依據
3. **簽核軌跡** — 區經理覆核紀錄、必要時食安會簽

設計理念：**強鎖店、強時效、強留痕、強鎖項**。

與每日盤點的關鍵差異：本功能由「計畫」驅動（店長被動接收）、品項清單固定（不可加減）、必走簽核、差異會落到正式盤盈虧單。

## 3. 業務邏輯背景

### 3.1 三層架構（店長只負責中間）

| 階段 | 功能（後端對應） | 誰做 |
|---|---|---|
| 排計畫 | 盤點計畫製定 | 總部 / 區經理 |
| 執行 | **盤點計劃執行（本功能）** | **店長** |
| 結帳 | 盤點作業（盤盈虧、CheckTake） | 系統自動 + 區經理確認 |

### 3.2 計畫怎麼來

1. 總部建一張「盤點計畫」（單頭：主旨、區域、倉名、計畫起訖日）
2. 計畫含「品類清單」（指定要盤哪些食材）
3. 計畫送簽通過後，系統會拆成「計畫執行單列」（單頭，sign code 前綴 `CE`），分派到具體門市／倉
4. 店長在「我的待執行計畫」清單看到這張單，狀態為「待執行」

### 3.3 執行單與庫存連動

- 執行單上有「執行日期、容忍度、checkType（出庫 / 入庫 / 兩者 / 都不）」
- 店長提交後，盤盈虧處理會建一張盤盈虧單（CheckTake）
- 盤盈虧單觸發出入庫，最終回寫執行單的「stockType」旗標（0=已出庫、1=已入庫）

### 3.4 與每日盤點的差異

|  | 每日盤點 | 盤點計劃執行 |
|---|---|---|
| 發起 | 店長自發 | 總部排、分派到店 |
| 頻率 | 每天 | 月／季／年 |
| 範圍 | 店長自選重點品項 | 計畫指定的完整清單 |
| 容忍度 | 推論：較寬 | 較嚴 |
| 簽核 | 大額才需 | 必走 |
| 時效 | 當日完成即可 | 必須在計畫時段內 |
| 結果 | 直接調庫存（推論） | 透過盤盈虧單調整 |

### 3.5 「我的店」是怎麼判定的

**目前計畫執行單頭無「門市 ID」欄位**，只有「區域 ID／區域名稱／倉名／倉名名稱」。

> 待確認：與每日盤點相同問題 — 「倉名」能否唯一對應到一家店？多店共倉（中央倉）的計畫單，店長視角怎麼處理？此設計差異會影響鎖店實作。

### 3.6 跨模組依賴

| 依賴 | 用途 |
|---|---|
| WHS 盤點計畫（上游） | 計畫來源 |
| WHS 盤盈虧處理（下游） | 盤點結果結帳 |
| WHS 出入庫作業 | 盤盈虧落到庫存 |
| WHS 庫存主檔 | 系統值（賬面量） |
| BPM 簽核 | 區經理覆核 |
| 推播 / 通知中心 | 提醒店長執行、逾期警示 |
| 系統使用者主檔 | 登入者 → 所屬門市對應（推論：尚未建立） |

## 4. 情境說明

### 4.1 正常情境 — 月度盤點

5/31 14:00 BK001 信義店店長小李拿著平板進冷凍倉：

1. 系統推播提醒「2026-05 月度盤點 14:00 開始」
2. 小李打開 App，看到「我的待執行計畫」有 1 筆，狀態「待執行」，剩餘時間 03:59:42
3. 點進去看到 87 項食材清單（冷凍 32、冷藏 28、乾貨 27）
4. 從冷凍倉開始，每點完一項就填、暫存、繼續下一項
5. 進度條：32/87 → 60/87 → 87/87
6. 系統匯總顯示「3 項超容忍」
7. 對這 3 項補填差異原因（不明損耗 / 報廢 / 帳目轉錯倉），其中一項加附照片
8. 點「全部提交」→ 送區經理覆核
9. 隔天早上區經理覆核通過 → 系統自動建盤盈虧單 → 庫存被調整

### 4.2 異常情境 — 計畫逾時未盤完

某店店長因現場太忙，到 18:00 還剩 12 項未盤：

1. 系統把計畫狀態改為「逾期」
2. 店長無法再填（鎖定），或可以填但需另案說明（待確認）
3. 系統通知總部與區經理
4. 區經理決定：補時段 / 取消這次計畫 / 罰則

> 待確認：逾時處理規則。

### 4.3 異常情境 — 區經理打回去重盤

區經理覆核時發現某食材差異 30%，要求重盤：

1. 區經理在簽核頁標記「該項退回」+ 留意見
2. 系統把計畫狀態改回「執行中」
3. 店長收到通知，看到區經理意見
4. 重新清點該項、修正、再提交
5. 區經理覆核通過

> 待確認：打回的粒度（全張單退 / 單項退）。

### 4.4 異常情境 — 邊盤邊出入庫

盤點期間，櫃台還在賣餐點，廚房還在領食材：

1. 推論：理想做法是凍結出入庫，盤完才放
2. 實際：BK 是 24 小時店，無法凍結
3. 折衷：店長記下「正在盤點時段」的出入庫流水，事後系統補算回來

> 待確認：是否允許邊盤邊動？或必須挑離峰時段？

### 4.5 多店店長 / 區經理代盤

某區經理在店長休假時親自到店代盤：

1. 區經理登入後，可切換「以某店身分執行」
2. 完成盤點 → 仍走原計畫的簽核

> 待確認：是否允許代執行？或必須由店長本人操作？

## 7. 商業邏輯

### 7.1 鎖店過濾規則（**未實作，待補**）

```
1. 取登入者 → 所屬門市
2. 列計畫執行單 where 門市 ID = 自店 AND 狀態 IN (待執行, 執行中, 退回)
3. 進入計畫明細時，再次驗證該店有權執行
4. 已歸檔／逾期的計畫可唯讀查詢
```

### 7.2 時段控管

- 計畫時段內：可填、可暫存、可提交
- 時段前：唯讀（看清單但不能填）
- 時段後：依規則 — 鎖定 / 寬限期 / 自動歸為逾期

### 7.3 漏項防呆

- 必須所有品類都填實盤值才能提交（推論：除非有「故意跳過」機制 + 原因）
- 提交前系統做完整性檢查

### 7.4 容忍度與簽核

- 推論：容忍度可在計畫層設定（例 ±3%）
- 差異 ≤ 容忍度 → 該項免原因，但仍要簽核覆核
- 差異 > 容忍度 → 必填原因 + 可附照片，視差異絕對額大小決定會簽人員

### 7.5 與每日盤點的並行

- 同日同店有計畫盤時，每日盤點建議停用（避免雙計）
- 計畫盤完成後當日不再重做每日盤
- 待確認：是否強制互斥

### 7.6 盤盈虧落地

- 提交通過後系統自動產生盤盈虧單（CheckTake），明細含每食材的「盤盈／盤虧」
- 盤盈虧單可能再走一次簽核（會計或食安）
- 通過後觸發出入庫，回寫本單的 stockType

### 7.7 唯讀切換

- 待執行 → 可編輯計畫上下文，不可填實盤值
- 執行中 → 可填、可暫存
- 待簽 → 唯讀，等覆核
- 退回 → 可再編輯被退項
- 已歸檔／逾期 → 唯讀

│    實盤  [    14   ]   差 -6 ⚠
│    原因  [不明損耗 ▼]
│    照片  [+ 加]
├─────────────────────────────────

- 倉位顯眼（協助店長定位）

- 大字數字鍵盤
- 倉別 Tab 快速切換
- 提交前彈出完整性檢查
