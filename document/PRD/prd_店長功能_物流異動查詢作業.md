# PRD｜店長功能 — 物流異動查詢作業

> 來源：逆向掃描需求集合（PDM）的物流單後端程式碼。物流單 CRUD 與明細查詢底層已實作（見底層 PRD [prd_物流管理_物流異動查詢作業.md](prd_物流管理_物流異動查詢作業.md)）。
>
> ⚠️ 後端**目前無店長視角專屬 endpoint**；既有 API 為總部／物流規劃共用版本。「異動」在後端尚未有獨立欄位／表，僅以「實際到店數量」欄位呈現差異。本文件以「店長能用什麼／應該看到什麼」視角撰寫，做為 PM 對齊、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

當配送車到店、實際點收完成，我會發現：

> 「**該收的 50 公斤牛肉餅、實際只有 47 公斤。差 3 公斤是怎麼回事？破損？短發？我要留下紀錄，讓採購跟廠商追。**」

或：

> 「**該 09:00 到、實際 11:30 才到，午餐尖峰被打亂。我要把延誤紀錄下來。**」

「物流配送查詢」看的是「**車到哪了**」，而本功能看的是「**車到了之後，實際 vs 預計差多少**」 — 是事後對帳與求償的依據。

### 1.2 我要做什麼

- 點收完成後：對照預計量與實際到店量，找差異
- 發現短裝／破損／延誤：留下異動紀錄與證據
- 月底對帳：列出當月所有異動，準備內部與廠商對帳會議
- 區經理問：「上個月你們店延誤幾次？」 → 查歷史
- 異常通報：拍照、填原因、提交給採購／物流規劃

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看實際到店量 vs 預計量的差異 | 找短裝、破損 |
| 看實際抵達時間 vs 預計時間 | 找延誤 |
| 留下異動原因 | 採購要對廠商求償 |
| 上傳照片 | 食安／品質證據 |
| 只看自店的異動 | 全公司資料無關 |
| 行動裝置在點收當下使用 | 賣場現場、雙手忙時手機操作 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 自店異動紀錄列表 | 看當月／歷史所有異動 |
| 建立異動紀錄（短裝／破損／延誤／錯送／拒收／取消） | 留下證據 |
| 拍照上傳 | 食安／品質佐證 |
| 編輯實際到店數量 | 點收差異補登 |
| 跳轉到「物流配送查詢」對應的物流單 | 上下文追溯 |
| Excel 匯出當月異動明細 | 月會檢討資料 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 物流異動查詢作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合（PDM）」 |
| 兄弟功能 | 物流管理行事曆、物流規劃查詢、物流配送查詢、物流串接記錄 |
| 主要頁面 | 自店異動列表、異動建立／編輯、照片上傳 |
| 簽核流程 | 待確認（推論：應有採購主管簽核） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在「**收貨之後**」階段，**主動留下異常紀錄**，並串接後續處理（採購對廠商求償、物流公司檢討）。

設計理念：
- **由店長發起**（兄弟功能多為唯讀；本功能是少數**店長可寫**的物流功能）
- **對帳與求償依據** — 紀錄要可追溯、有證據（照片）
- **與物流配送查詢成對** — 配送查詢看「車到了沒」，異動查詢看「到了之後對不對」
- **與庫存入庫聯動** — 短裝會直接影響庫存數字

## 3. 業務邏輯背景

### 3.1 「異動」目前在後端的狀態

底層物流單明細**僅有「實際到店數量」一個欄位**，沒有：

- 異動類型（延誤／短裝／破損／錯送／拒收／取消）
- 異動原因文字
- 短缺量／延誤時間
- 預計抵達時間 vs 實際抵達時間
- 照片附件
- 異動建立人、建立時間（除底層 BaseDO 的時間戳外無變更歷史）

也就是說，**業務上談到的「異動」目前後端沒有結構化紀錄能力** — 必須新增資料欄位／資料表才能滿足。

> 待確認：漢堡王現行「異動」是走紙本還是另有他系統？若已有人工流程，本功能要先做哪些異動類型？

### 3.2 異動類型（業務需求，推論）

| 異動類型 | 案例 |
|---|---|
| 延誤 | 該 09:00 到、實際 11:00 才到 |
| 短裝 | 該 50 公斤、實際 47 公斤 |
| 破損 | 2 箱壓壞，無法入庫 |
| 錯送 | 送來不屬於本店的品項 |
| 拒收 | 品質問題，全車退回 |
| 取消 | 物流未派車就取消 |

### 3.3 異動 vs 庫存入庫的關係

「短裝／破損」一定會影響庫存：

```
預計 50 kg  →  實際入庫 47 kg  →  庫存少 3 kg
                 │
                 └─ 異動單記「短裝 3 kg、破損 0」
```

店長收貨時應**先確認實際數量**，再決定要不要建異動。建議流程：

1. 點收當下發現差異
2. 庫存入庫照「實際量」進
3. 異動單紀錄「預計 vs 實際」與原因
4. 採購／物流規劃接手後續

> 待確認：是否要把「庫存入庫」與「異動建立」做成一鍵操作。

### 3.4 與兄弟物流功能的差異（店長視角）

| 功能 | 看什麼 | 主要時機 |
|---|---|---|
| 物流管理行事曆 | 未來某天有沒有配送 | 排班 |
| 物流規劃查詢 | 派車前的規劃明細 | 細查、單號追蹤 |
| 物流配送查詢 | 派車後車輛狀態 | 收貨當下 |
| **物流異動查詢（本功能）** | **實際到店 vs 預計差異、原因紀錄** | **收貨後對帳、求償佐證** |
| 物流串接記錄 | ERP 給中央倉的批次紀錄 | IT 軌跡 |

> 本功能與「物流配送查詢」共用同一張底層物流單表，差別在「**配送中 vs 配送後**」 — 配送查詢是進行式、本功能是結算式。
>
> 本功能與「物流串接記錄」在後端是同一支 CRUD；差別只是 UI 切換時機與顯示欄位 — 串接記錄看「ERP→中央倉的傳輸動作」、本功能看「車到店之後的數量與原因」。

### 3.5 「我的店」是怎麼判定的

底層物流單明細含「門市 ID／編碼」，店長視角應在明細層先過濾自店。

### 3.6 異動是否要簽核（待確認）

異動可能涉及採購對廠商求償，金額大時應有簽核：

- 短裝金額 < N 元：店長自填、自動結案
- 短裝金額 >= N 元：採購主管簽核
- 拒收（整車退回）：物流主管 + 採購主管雙簽

> 待確認：分流規則由誰決定。

## 4. 情境說明

### 4.1 正常流程 — 短裝建異動

5/19 早上 9:30，BK001 信義店店長小陳點收：

1. 牛肉餅實際只有 47 公斤、預計 50 公斤
2. 庫存入庫進 47 公斤
3. 進「物流異動查詢」→ 建異動
4. 選異動類型「短裝」、填短缺 3 公斤、原因「外箱有水漬，疑似冷藏破損」
5. 拍照 2 張上傳
6. 提交 → 後續採購接手向廠商求償

### 4.2 延誤情境

5/19 該 09:00 到、實際 11:30 才到：

1. 進「物流異動查詢」→ 建異動
2. 選異動類型「延誤」、填預計 09:00／實際 11:30
3. 原因「司機回報路況」
4. 影響範圍「午餐尖峰麵包庫存吃緊」
5. 提交

### 4.3 拒收情境

某日生菜整批變色，店長判斷無法使用：

1. 進「物流異動查詢」→ 建異動
2. 選「拒收」、勾選「全車退回」
3. 拍照 3 張
4. 提交（推論：應走採購主管 + 物流主管雙簽）

### 4.4 月底對帳

月底店長要整理上月異動報表：

1. 切日期：4/1～4/30
2. 列表顯示當月所有異動
3. 按類型分群（延誤 N 筆、短裝 N 筆、破損 N 筆）
4. Excel 匯出做月會資料

### 4.5 區經理問

區經理：「上個月你們店延誤幾次？」

1. 店長進本功能，過濾「延誤」+「上月」
2. 列表顯示 3 筆
3. 即時回覆區經理

### 4.6 異動原始資料來源 — 從配送查詢跳入

從「物流配送查詢」看到某單延誤 → 點「建異動」：

1. 自動帶入該物流單號
2. 自動帶入自店明細
3. 店長只需選類型、填原因

### 4.7 取消情境

物流尚未派車就確定取消（廠商缺貨）：

1. 進本功能 → 建異動
2. 選「取消」、勾關聯預測單號
3. 待確認：取消後是否需自動回寫預測量、是否需通知採購重發

## 5. 操作流程

> 後端目前無店長視角 endpoint，且異動結構欄位多數未實作，以下為「應有的呼叫鏈」（推論）。

```
[店長收貨後發現差異]
  |
  +- 進「物流異動查詢」
  |    +- 鎖店：自店相關物流單
  |
  +- 兩條路徑：
  |    A. 列表 → 「+ 建異動」按鈕（從零開始）
  |    B. 從「物流配送查詢」跳入 → 自動帶單號
  |
  +- 建立頁：
  |    +- 選異動類型（延誤／短裝／破損／錯送／拒收／取消）
  |    +- 依類型動態欄位：
  |        延誤 → 預計時間、實際時間
  |        短裝 → 預計量、實際量、短缺量
  |        破損 → 破損量、破損描述
  |        錯送／拒收／取消 → 原因勾選 + 自由文字
  |    +- 必拍照 N 張（推論）
  |    +- 影響範圍說明
  |
  +- 送出
  |    +- 依金額／類型決定簽核分流
  |    +- 通知採購／物流主管
  |
  +- 結果回饋（求償成功／補件／結案）
```

## 7. 商業邏輯

### 7.1 自動鎖店

同其他店長功能：登入時取得所屬門市、覆寫過濾條件。
讀取時鎖店、建立時也鎖店（不允許店長為其他店建異動）。

### 7.2 異動類型分流

- 「延誤」與「取消」**不影響庫存**，純紀錄性
- 「短裝」「破損」**影響庫存** → 應與庫存入庫關聯（推論）
- 「錯送」「拒收」**衍生退貨流程**（推論：與庫存出庫聯動）

### 7.3 簽核分流（待確認）

按金額與類型走不同簽核路徑（推論）：

- 小額短裝：自動結案（待設金額閥值）
- 大額短裝、破損：採購主管簽核
- 拒收：物流主管 + 採購主管雙簽

### 7.4 照片必填

短裝、破損、錯送、拒收應強制至少 1 張照片（推論）。
延誤、取消可不必填。

> 待確認：照片保留多久（食安政策）。

### 7.5 異動歷史軌跡

異動單建立後，修改應留下軌跡（誰改、何時改、改了什麼）。底層物流單表目前**無此能力**（修改採刪舊插新策略）。

### 7.6 預計量的取得

預計量需從「需求預測明細」反查（依物流單明細的來源預測單號），底層 API 尚未提供此聯動查詢。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店異動 | 建立、編輯（草稿）、查閱、Excel 匯出 |
| 副店長／代班 | 自店異動 | 建立、查閱（推論） |
| 採購主管 | 自區異動 | 簽核、求償處理 |
| 物流主管 | 全公司異動 | 簽核（拒收／重大延誤） |
| 區經理 | 區內各店 | 查閱（推論） |
| 總部物流規劃 | 全公司 | 走總部視角 |

> 後端目前以「具備物流單建立／修改／查詢／匯出權限」的角色控管，未細分異動類別簽核。

## 10. 功能範圍

### 10.1 包含的功能

- 自店異動列表查詢（多條件過濾）
- 建立異動紀錄（多類型）
- 照片附件上傳
- 預計量／實際量比對顯示
- 與「物流配送查詢」跳轉聯動
- 自店 Excel 匯出
- 簽核流程整合（推論）

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端無 `/app/*` 對應路徑 |
| 自店過濾（鎖店） | 既有 API 接受任意條件 |
| 異動類型欄位 | 物流單明細無此欄位 |
| 預計／實際時間欄位（延誤） | 無 |
| 短缺量／破損量欄位 | 無 |
| 異動原因文字欄位 | 無 |
| 照片附件能力 | 物流單表無附件欄位 |
| 異動歷史軌跡 | 編輯採刪舊插新、無變更歷史 |
| 預計量自需求預測明細反查 | 後端無聯動 API |
| 簽核流程綁定 | 物流單無 BPM 整合 |
| 與庫存入庫的聯動（短裝、破損） | 無 |
| 與庫存出庫的聯動（拒收、錯送） | 無 |

### 10.3 不包含

- 修改物流單本身（總部專用）
- 即時配送追蹤（屬「物流配送查詢」）
- 對廠商實際求償流程（屬採購模組）
- 庫存入庫操作（屬「庫存入庫作業」）
