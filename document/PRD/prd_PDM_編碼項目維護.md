## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 PDM 的**編碼體系維護人員**。
「編碼類別」（食材大類、食材中類、食材小類…）已經先在上一張表建好。我這裡要做的事是：**在每個編碼類別底下，建立具體的「項目」**。例如「食材大類」底下有「肉類」「蔬果類」「乳製品」「調味料」這些項目；「食材中類」底下又有「牛肉」「豬肉」「雞肉」等項目；以此類推。每筆項目都有自己的代碼、名稱、可選的父級代碼。

### 1.2 我要做什麼

- **新增編碼項目**：選定要歸屬的編碼類別，給項目一個代碼（在該類別內唯一）、名稱、父級代碼（可選）。
- **修改編碼項目**：調整代碼、名稱或父級。
- **批量刪除**：清理過時項目。
- **食材小類特殊欄位「儲存類型」**：當該編碼項目所屬類別為「食材小類」時，可額外設定儲存類型（LONG 長期儲存 / SHORT 短期儲存），這個欄位存在另一張表 `pdm_ingredient_subcategory_type` 中。
- **查詢**：條件分頁查詢。
- **匯出**：盤點佐證。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 同一個編碼類別底下，代碼不能重複 | 編碼唯一性的最低要求 |
| 同一個編碼類別底下，名稱不能重複 | 避免歧義 |
| 食材小類要能標記「儲存類型」 | 影響採購頻率與庫存策略（長期 vs 短期備料） |
| 父級代碼可選 | 用來表達層級關係（中類項目可指到大類項目） |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 新增編碼項目（含「同類別下唯一性」檢查） | 編碼正確性 |
| 修改編碼項目（同樣檢查；含食材小類儲存類型 upsert） | 編輯重整 |
| 批量刪除 | 清理 |
| 條件分頁查詢（含 storageType 補資料） | 維運盤點 |
| 取得單一項目（含 storageType） | 編輯前回填 |
| 匯出 Excel | 盤點佐證 |

## 2. 功能目的

提供漢堡王 PDM 編碼體系的**第二層具體項目**定義。每筆「編碼項目」掛在某個「編碼類別」底下，記錄該類別下的具體值。例如：

```
編碼類別「食材大類」(category=1)
├── 編碼項目 code=01, name=肉類
├── 編碼項目 code=02, name=蔬果類
├── 編碼項目 code=03, name=乳製品
└── 編碼項目 code=04, name=調味料
```

下游使用者：

- **食材維護作業**：選擇「食材中類」「食材小類」「原型食材」時，從本表取選項
- **單品維護作業**：選擇「單品類別」「單品」時，從本表取選項
- **包材維護作業**：選擇「包材類別」時，從本表取選項

**特殊設計：食材小類專屬「儲存類型」**

當某筆編碼項目所屬類別恰好為「食材小類」時，可額外標註它的儲存類型（LONG / SHORT）。此資訊不存在本表，而存在另一張表 `pdm_ingredient_subcategory_type`。在編碼項目的更新流程中，會 upsert 該關聯表。查詢時透過 LEFT JOIN 補回。

## 3. 業務邏輯背景

### 3.1 主檔欄位

- 編碼類別（category）：對應「編碼類別維護」的代碼或 ID（型別為 String，意義模糊，列入 §11）
- 編碼項目代碼（code）：在該類別內唯一
- 父級代碼（parentCode）：可選，指向同類別或上層類別的某筆編碼項目
- 編碼項目名稱（name）：在該類別內唯一

### 3.2 「同類別下」唯一性檢查（新增與修改皆執行）

| 情境 | 唯一性檢查 |
|---|---|
| 新增 | 同類別（category）下：代碼不重複 AND 名稱不重複 |
| 修改 | 同類別下、排除自己：代碼不重複 AND 名稱不重複 |

任一重複即拋「編碼項目代碼已存在」例外。

> **註**：唯一性檢查的 SQL 使用兩個 OR 子句：第一個查「代碼+類別」重複，第二個查「名稱+類別」重複。命名單一錯誤訊息但實際上代碼或名稱重複都會觸發此錯。錯誤訊息可能誤導（列入 §11）。

### 3.3 食材小類專屬「儲存類型」邏輯

更新時若 `storageType != null` 且該項目所屬類別為「食材小類」：

1. 從 `pdm_code_category` 表查 category ID 對應的類別名稱
2. 若名稱為「食材小類」 → upsert `pdm_ingredient_subcategory_type` 表（記錄 detailId、categoryCode、storageType）

> **註**：
> - 此邏輯**只在修改時觸發**，**新增時不會**儲存 storageType
> - 「食材小類」這個判斷是寫死字串比對，若日後類別名稱變更會失效（列入 §11）
> - `category` 欄位被當作可解析為 Long 的字串使用（`Long.parseLong(categoryCode)`），但 SaveReqVO 為 String、實際呼叫時填的可能是「類別代碼」或「類別 ID」，兩者語意混淆（列入 §11）

### 3.4 「分頁」端點未啟用權限驗證（同編碼類別模組議題）

`/page` 端點 `@PreAuthorize` 被註解掉，任何登入者可呼叫。列入 §11。

### 3.5 查詢與單筆取得帶 storageType（LEFT JOIN）

- `getCodeItem(id)`：呼叫 `selectWithStorageTypeById`，LEFT JOIN `pdm_ingredient_subcategory_type` 取出 storageType
- `getCodeItemPage(reqVO)`：呼叫 `selectPageWithStorageType`，分頁查詢同樣 LEFT JOIN

非食材小類的項目，storageType 為 null。

### 3.6 排序

依「類別」「代碼」雙重升冪排序。

### 3.7 樂觀鎖欄位 revision 存在但未使用（同編碼類別模組）

DO 有 `revision` 欄位但 Service 端未實作樂觀鎖機制。

### 3.8 批量刪除無交易

`/delete` 端點接受 `ids` 清單，迭代呼叫 `deleteCodeItem`。**未包 `@Transactional`**，中途失敗會殘留。

### 3.9 新增時手動設定 createTime / updateTime

Service 在 `createCodeItem` 中**手動** `setCreateTime(LocalDateTime.now())` 與 `setUpdateTime(LocalDateTime.now())`，繞過了 BaseDO 的自動填值機制。這是不必要的（推論：可能是早期程式碼遺留），列入 §11。

### 3.10 `validateCodeItemExists` 是乾淨版本

不像編碼類別模組的版本內部多查一次，本模組的存在驗證是正常實作。

## 4. 情境說明

### 4.1 正常流程 — 新增一筆編碼項目

PDM 維護員小華要建一筆「食材大類」底下的「肉類」：

- 編碼類別：`1`（食材大類）
- 編碼項目代碼：`01`
- 父級代碼：空
- 編碼項目名稱：`肉類`

送出後系統：

1. 唯一性檢查：類別=1 + 代碼=01 是否存在 → 否；類別=1 + 名稱=肉類 是否存在 → 否
2. 寫入主檔，手動填 createTime、updateTime
3. 回傳新 ID

### 4.2 正常流程 — 修改一筆食材小類項目並設儲存類型

小華要把「番茄」（食材小類）標為「短期儲存」：

- 編碼類別：`3`（食材小類）
- 代碼：`0301`
- 名稱：`番茄`
- 儲存類型：`SHORT`

送出後系統：

1. 驗證存在
2. 唯一性檢查（排除自己）
3. 更新主檔
4. 因 `storageType` 非空，且查詢類別 ID=3 對應名稱為「食材小類」→ 條件成立
5. upsert `pdm_ingredient_subcategory_type` 表，記下「detailId=該項目 ID、categoryCode=3、storageType=SHORT」

完成後其他模組查此項目時會帶出 `storageType=SHORT`。

### 4.3 異常情境 — 代碼重複

小華嘗試在「食材大類」底下新建一筆代碼為 `01` 的項目（已存在「肉類」）：

- 唯一性檢查：類別+代碼 已存在 → 拋「編碼項目代碼已存在」

### 4.4 異常情境 — 名稱重複

小華嘗試新建一筆代碼為 `99` 但名稱為「肉類」（已存在）：

- 唯一性檢查：類別+名稱 已存在 → 拋「編碼項目代碼已存在」（**錯誤訊息誤導**，應為「名稱已存在」）

列入 §11。

### 4.5 業務規則分流情境 — 新增食材小類項目時填儲存類型

小華新建一筆食材小類「萵苣」並填 `storageType=SHORT`。送出後：

- 唯一性檢查通過
- 寫入主檔
- **沒有任何儲存類型相關處理**（程式只在 update 路徑檢查 storageType）

結果：主檔建好，但 `pdm_ingredient_subcategory_type` 沒寫入。「萵苣」的儲存類型必須**再做一次修改**才能保存。列入 §11。

### 4.6 業務規則分流情境 — 修改非食材小類項目時填儲存類型

小華手滑在「食材中類」項目的編輯頁填了 `storageType=SHORT`：

- 系統檢查所屬類別不是「食材小類」 → 不 upsert
- 主檔正常更新
- `storageType` 填的值被忽略

### 4.7 業務規則分流情境 — 修改食材小類項目但 storageType 為 null

小華只是想改名稱，沒動 storageType：

- 系統檢查 `storageType == null` → 跳過 upsert
- 既有的關聯記錄保留

### 4.8 異常情境 — 批量刪除中途失敗

同編碼類別模組議題：未包 `@Transactional`，中途失敗會殘留已刪部分。

### 4.9 異常情境 — 刪除一個被下游引用的項目

小華刪除「食材中類-牛肉」項目，但「食材維護作業」中有 23 筆食材主檔的 `category` 指向此項目。系統**不檢查**下游引用 → 刪除成功，23 筆食材變成孤兒。列入 §11。

### 4.10 非預期情境 — 「食材小類」類別名稱被改

IT 把「編碼類別維護」中「食材小類」改名為「食材小類_新」。下次更新編碼項目時 Service 比對「食材小類」字串 → 不匹配 → 不再 upsert 儲存類型。儲存類型永遠寫不進去（無人察覺）。列入 §11。

## 7. 商業邏輯

### 7.1 同類別下的唯一性檢查（核心）

新增與修改都檢查；修改時排除自己。檢查條件以「類別 + 代碼」「類別 + 名稱」兩組同時查重。

### 7.2 儲存類型只在「修改路徑」且「食材小類」才寫入

新增時 storageType 完全被忽略。要設定儲存類型必須：

1. 先新增主檔
2. 再修改該筆，帶上 storageType
3. 系統檢查所屬類別為「食材小類」（字串比對），才會寫入關聯表

### 7.3 「食材小類」是寫死的字串比對

`"食材小類".equals(category.getName())`。若日後類別名稱變更，此邏輯失效。

### 7.4 LEFT JOIN 在查詢時補資料

查詢與單筆取得都透過 XML Mapper 的 `selectWithStorageTypeById` 與 `selectPageWithStorageType` LEFT JOIN 補出 storageType。
