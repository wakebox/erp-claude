# PRD｜系統管理 — 行銷活動公告

> 來源：逆向自 `kingmaker-module-system` 之公告相關後端程式碼（含 `controller/admin/notice/NoticeController.java`、`service/notice/NoticeServiceImpl.java`、`dal/.../notice/NoticeUserDO.java` 等）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

> **本 PRD 範圍**：本程式同時處理三種公告類型（通知 type=1、公告 type=2、行銷活動 type=3），共用同一張資料表與 CRUD 端點。本 PRD 聚焦「行銷活動」（type=3）獨有的「接收人指派、我的公告分頁、已讀狀態追蹤」邏輯；type=1/2 之共通 CRUD 由「公告維護」PRD 描述。

---

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的**行銷活動企劃／HR 福利窗口／門市區經理**。
公司有針對「特定族群」的活動或訊息要發送：例如「中區門市店長季度績效獎金說明」「採購部全體下個月加班費結算公告」「全國店長雙月會議通知」。這類訊息**不適合**發給全公司，只想精準送到該看的人。

### 1.2 我要做什麼

- **發布限定對象訊息**：建一則「行銷活動」類型的公告，指定接收人是「中區所有門市」「採購部與其下所有單位」「特定 12 位店長」之類。
- **修改訊息或調整接收人**：活動延期、加碼、新增 3 位試辦門市等。
- **下架／刪除**：活動結束後關閉。
- **追蹤誰看了**：理論上想看「中區 30 位店長有多少人已讀」（推論：目前未提供查詢 API，列入 §11）。
- **發完自己也想看到**：發完一則「店長獎金活動」要能在自己的「我的公告」中確認內容無誤。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 我能按「部門」一次選一群人，不必逐個勾使用者 | 中區門市有 50+ 家，逐家勾選不切實際 |
| 我選了一個部門後，該部門下的子部門也要一起涵蓋 | 「採購部」實際分國際採購組、在地採購組，我選父部門就應該涵蓋全部 |
| 我也能單獨補幾位特殊使用者 | 例如某個試辦門市的單一店長 |
| 收件人去重 | 若我同時選了部門和個別人，重複的不應該被通知兩次 |
| 我發布完，自己也能在「我的公告」看到 | 確認內容無誤、不需要請別人轉貼給我 |
| 未被指派的人完全看不到這則訊息 | 行銷活動有保密性（例如獎金、人事） |
| 想知道每個接收人有沒有看過 | 後續追蹤、稽核 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 在公告新增／編輯介面選擇接收部門（多選） | 按部門批量指派 |
| 在公告新增／編輯介面選擇接收使用者（多選） | 補單一指派 |
| 系統自動展開「指定部門 + 所有子部門 + 部門下所有使用者」 | 不必逐層展開 |
| 接收人集合自動去重 | 避免重複 |
| 發布人自動加入接收人 | 自己看得到 |
| 「我的公告」端點只回傳「我有被指派的」行銷類型 + 全員可看的通知公告 | 一般員工的閱讀體驗 |
| 每筆接收人關聯帶「已讀／未讀」狀態 | 追蹤閱讀情形（已預留欄位） |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 行銷活動公告 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 公告維護（共用本程式之 type=1/2 分支） |
| 主要頁面 | 行銷活動公告列表頁、新增／編輯頁（含接收部門／使用者多選區塊）、「我的公告」首頁區塊 |
| 簽核流程 | 無（即時生效，不綁簽核） |

## 2. 功能目的

提供一套**可按部門批量指派、可逐人補選、自動去重、發布人自動納入**的精準公告機制。與「通知／公告」全員可見不同，**行銷活動**強調「指派制」：未被列入接收人清單者完全看不到。

設計上靠以下幾項組合達成：

1. 公告主檔的「類型」欄位設為「行銷活動」
2. 一張獨立的「公告接收人」關聯表記錄哪些使用者可見
3. 寫入時做部門→子部門→使用者的兩段展開
4. 「我的公告」端點以「我被指派的行銷類型 + 全員可見的非行銷類型」兩段合併回傳

> **重要過渡背景**：本功能仰賴使用者主檔與部門主檔（含子部門快取）。經系統整合決策（2026-05-16），漢堡王 IT 走向以 Laravel OA 為使用者與組織主檔之單一事實來源。OA 整合後，接收人展開邏輯的人員／部門來源預期改以 OA 為準（推論：階段 5 後重新評估）。

## 3. 業務邏輯背景

### 3.1 「行銷活動」是公告類型的一個分支

從資料模型看，公告主檔同時容納通知（1）、公告（2）、行銷活動（3）三種類型。它們共用相同欄位（標題、類型、內容、狀態）與相同 CRUD 入口；差別只在於：

- 行銷類型在新增／修改時**會額外維護**一張「接收人關聯表」
- 行銷類型在「我的公告」端點走「先取我有的編號清單再分頁查」的路徑
- 非行銷類型在「我的公告」端點走「無條件分頁查」的路徑

### 3.2 接收人指派的兩個來源

新增／修改時，前端可同時傳：

- **接收部門編號清單**：多選；系統會把每個部門展開為「該部門 + 所有後代部門」之集合，再取該集合下所有使用者
- **接收使用者編號清單**：多選；直接加入接收人集合

兩個來源的最終接收人集合用 `Set` 收集，**自動去重**。

### 3.3 發布人自動加入接收人

寫入接收人時，系統會自動把當前登入者（即「發布人」）加入接收人集合。理由推測為：發布完想自己能在「我的公告」看到、確認內容無誤。

> **註**：若 UI 上沒有「發給自己」勾選，使用者可能不知道自己被自動加入。當發布人本身也屬於「目標部門」時無感；當發布人不屬於目標部門時，會出現「公告管理員自己看得到，但業務需求上不應該收到」的情況（列入 §11）。

### 3.4 部門展開使用部門模組的快取

部門→子部門的展開不是即時計算，而是呼叫部門服務的「子部門編號集合（快取版）」端點，過期策略由部門模組維護（任何部門新增／修改／刪除都會清空整批快取）。

### 3.5 接收人關聯帶「已讀狀態」欄位

「公告接收人」表中每筆關聯有 `readStatus` 欄位（已讀 / 未讀），預設未讀。但目前後端**沒有提供任何「標記為已讀」的 API**。前端讀取後若想更新已讀狀態，需要自行新增 API 或直接走資料庫（列入 §11）。

### 3.6 修改時接收人採「全量替換」

修改公告時，伺服端：

1. 先把該公告所有現有接收人關聯**整批刪除**
2. 若新類型仍為行銷活動 → 依新表單的部門／使用者集合重新建立關聯

這意味著：即便修改人只是想改標題，前端表單也必須**把原本的接收人完整帶回來**，否則接收人就被清空。前端如果還沒實作回填（見「公告維護」PRD §11 #7），這幾乎必然出錯（列入 §11）。

### 3.7 「我的公告」是分兩段查再合併

對員工面的「我的公告」端點，邏輯為：

1. 先查「我被指派的行銷類型公告編號」清單
2. 第一段：分頁查所有非行銷類型公告（通知＋公告，全員可見）
3. 第二段：若第 1 步有結果 → 分頁查那批行銷類型公告
4. 合併兩段結果
5. 依「建立時間」倒序排序
6. **程式內手動**依分頁參數切片

每段查詢都帶 `pageNo/pageSize` 條件去資料庫，但只有合併後才切真正的分頁。這在公告數量大時的效能與分頁正確性都有疑慮（列入 §11）。

### 3.8 接收人關聯也是跨租戶

接收人關聯表繼承一般 BaseDO（未繼承多租戶基礎）。公告主檔本身也是跨租戶共用，因此整套行銷活動公告機制不做租戶切分。

## 4. 情境說明

### 4.1 正常流程 — 發給整個部門及其下屬

行銷主管小華要發布「2026 母親節活動企劃公告」給「行銷部」及其下所有單位（含「行銷部—企劃組」「行銷部—設計組」「行銷部—市場研究組」）。她操作：

- 類型：行銷活動
- 標題：`2026 母親節活動企劃`
- 內容：（富文本）
- 接收部門：勾選「行銷部」
- 接收使用者：留空
- 狀態：啟用

送出後系統流程：

1. 公告主檔寫入
2. 取「行銷部」的所有後代部門編號 → 集合：行銷部、行銷部—企劃組、行銷部—設計組、行銷部—市場研究組
3. 取這些部門下所有使用者 → 假設 23 人
4. 加上小華自己 → 24 人（若小華也在這 23 人裡，去重後仍 24 人）
5. 逐筆寫入接收人關聯，已讀狀態預設未讀

完成後行銷部所有人在「我的公告」可以看到這則訊息，其他部門完全看不到。

### 4.2 正常流程 — 部門 + 個別使用者混合指派

公關主管阿珍要發「跨部門年度成果分享會」訊息給「公關部全員 + CFO + 業務副總」。她在新增頁：

- 類型：行銷活動
- 接收部門：勾選「公關部」
- 接收使用者：勾選 CFO、業務副總

送出後系統流程：

1. 公關部展開 → 公關部、公關部—外部公關組、公關部—內部溝通組
2. 取上述部門下所有使用者 → 12 人
3. 加上 CFO、業務副總 → 14 人
4. 加上阿珍自己 → 15 人（若阿珍在這 14 人中則仍 14 人）
5. 寫入接收人關聯

### 4.3 異常情境 — 接收部門編號不存在

阿珍從外部工具呼叫 API，部門編號清單帶了 `[99999]`（不存在的部門）。系統流程：

1. 對 99999 呼叫部門服務的「子部門集合（快取版）」
2. 部門服務從資料庫查不到對應部門，回傳**空集合**
3. 接收人集合在這步沒新增任何人
4. 若使用者清單也是空 → 接收人集合僅含阿珍一人
5. 公告依然建立成功，但實際只發給了阿珍

> **註**：後端**未驗證**部門編號是否存在、是否啟用，因此「打錯部門編號」會默默變成「只發給自己」，沒有錯誤回應（列入 §11）。

### 4.4 異常情境 — 接收使用者編號不存在

同上，使用者編號清單帶 `[99999]`。系統直接把 99999 加入接收人集合並寫入關聯表，留下一筆指向不存在使用者的「孤兒」關聯記錄。
未來查詢時不會 join 出對應使用者，等同無效資料（列入 §11）。

### 4.5 業務規則分流情境 — 一般員工的「我的公告」

員工小李（不屬於行銷部）登入首頁，前端呼叫「我的公告」端點：

1. 查小李對應的行銷類型公告編號清單 → 假設為空（沒被指派任何行銷活動）
2. 分頁查所有非行銷類型公告（通知＋公告）→ 假設 8 筆
3. 因步驟 1 為空，跳過第二段查詢
4. 直接回傳 8 筆非行銷類型公告

### 4.6 業務規則分流情境 — 被指派的員工的「我的公告」

員工小張（屬於行銷部）登入：

1. 查小張對應的行銷類型公告編號清單 → `[100, 102, 105]`
2. 分頁查所有非行銷類型公告 → 假設 8 筆
3. 分頁查編號 `[100, 102, 105]` 的行銷類型公告 → 3 筆
4. 合併 8 + 3 = 11 筆，依建立時間倒序排序
5. 程式內依 `pageNo=1, pageSize=10` 取出前 10 筆回傳，總筆數標 11

### 4.7 業務規則分流情境 — 修改公告但未帶接收人

行銷副理阿凱想把「2026 母親節活動企劃」公告標題從「企劃」改為「企劃 v2」。他在前端編輯頁送出修改，**但前端表單沒有把原本的接收人清單帶上**（接收人欄位空白）。系統流程：

1. 公告主檔覆寫成功
2. **整批刪除**該公告所有現有接收人關聯（23 筆全清掉）
3. 類型仍為行銷活動 → 重新建立關聯
4. 接收部門空、接收使用者空 → 集合為空
5. 加入當前登入者（阿凱）→ 集合僅阿凱一人
6. 寫入單一筆關聯

結果：原本行銷部 24 人能看到此公告，修改後只剩阿凱一人能看到。其他 23 人「我的公告」中該則訊息直接消失（列入 §11）。

### 4.8 異常情境 — 推播給線上接收人

行銷主管小華發完公告後按「推播」。系統透過 WebSocket 把完整公告物件廣播給目前在線的所有**後台類型使用者**。

> **註**：推播沒有過濾「該公告的接收人清單」，是廣播。意即：

- 行銷部以外的線上使用者也會收到 WebSocket 訊息
- 但前端若依「我的公告」端點過濾後渲染，則該訊息不會出現在他們的清單上（保密性靠前端維持）
- 若前端直接顯示 WebSocket 推送內容，會洩露行銷活動給非目標對象

此為設計上的潛在資安風險，列入 §11。

### 4.9 自動處理情境 — 部門剛好停用

行銷主管選了一個剛好被 IT 停用的部門（例如「行銷部—外部公關組—臨時專案組」）作為接收人。系統流程：

1. 對該部門編號呼叫子部門集合（快取）
2. 子部門快取**不過濾啟用狀態**（推論：來源 `DeptServiceImpl.java:174-190`），仍回傳子部門編號
3. 取部門下使用者
4. 寫入接收人關聯

結果：停用部門下的使用者仍會被加入接收人。是否符合業務預期需確認（列入 §11）。

## 5. 操作流程

### 5.1 新增行銷活動公告

```
[填寫公告表單（含類型=行銷活動、接收部門、接收使用者）]
  │
  ▼
[權限檢查：是否具備「系統-公告-新增」]
  │
  ▼
[寫入公告主檔]
  │
  ▼
[類型 = 行銷活動 → 進入接收人寫入流程]
  │
  ▼
[5.4 接收人寫入]
  │
  ▼
[回傳新公告編號]
```

### 5.2 修改行銷活動公告

```
[編輯公告表單，送出]
  │
  ▼
[權限檢查：是否具備「系統-公告-修改」]
  │
  ▼
[該公告必須存在]
  │
  ▼
[整筆覆寫主檔]
  │
  ▼
[刪除該公告所有現有接收人關聯（不論類型）]
  │
  ▼
[新類型 = 行銷活動 → 走 5.4 接收人寫入]
  │
  ▼
[新類型 ≠ 行銷活動 → 結束（即將原為行銷活動之公告改為通知／公告，接收人會被完全清空且不再重建）]
```

### 5.3 刪除行銷活動公告

```
[點選刪除]
  │
  ▼
[權限檢查：是否具備「系統-公告-刪除」]
  │
  ▼
[該公告必須存在]
  │
  ▼
[物理刪除主檔]
  │
  ▼
[依「公告編號」刪除所有接收人關聯]
```

### 5.4 接收人寫入（5.1 / 5.2 共用子流程）

```
[初始化接收人集合（Set，自動去重）]
  │
  ▼
[接收部門清單不為空？]
  ├─ 是：
  │   ├─ 把這些部門編號全部加入「部門總集合」
  │   ├─ 逐個部門呼叫「子部門集合（快取版）」 → 加入「部門總集合」
  │   ├─ 以「部門總集合」呼叫使用者服務「依部門集合取使用者」
  │   └─ 把回傳的所有使用者編號加入接收人集合
  └─ 否：略過
  │
  ▼
[接收使用者清單不為空？]
  ├─ 是 → 直接把這些編號加入接收人集合
  └─ 否：略過
  │
  ▼
[把當前登入者加入接收人集合]
  │
  ▼
[逐筆寫入接收人關聯]
  ├─ 公告編號 = 本次公告
  ├─ 使用者編號 = 集合中的每一筆
  └─ 已讀狀態 = 未讀（預設）
```

### 5.5 我的公告（一般員工使用）

```
[取當前登入者編號]
  │
  ▼
[查該使用者被指派的行銷類型公告編號清單]
  │
  ▼
[分頁查所有非行銷類型公告（通知＋公告）]
  │      條件：標題模糊、狀態（若有指定）
  │      排序：建立時間倒序
  │
  ▼
[行銷類型公告編號清單不為空？]
```

## 7. 商業邏輯

### 7.1 接收人展開的執行順序

1. 處理「接收部門」：先收集所有指定部門編號到一個臨時集合
2. 對每個指定部門呼叫部門服務的「子部門集合（快取版）」→ 加入臨時集合
3. 用臨時集合呼叫使用者服務「依部門集合取使用者」→ 把使用者編號加入接收人集合
4. 處理「接收使用者」：直接把清單上的編號加入接收人集合
5. 加入當前登入者
6. 逐筆寫入接收人關聯

集合用 `Set` 實作，整個過程自動去重。

### 7.2 接收人關聯的逐筆寫入

接收人寫入採**逐筆 insert**（非 batch insert）。當接收人集合很大時（例如整個總部展開有 500+ 人），會產生 500+ 次資料庫 insert，效能堪憂（列入 §11）。

### 7.3 全量替換策略

修改時的接收人處理是「**先清光、再重建**」：

- 即便表單上接收人欄位完全沒動，伺服端仍會清空全部關聯
- 重建時依賴前端把原本的接收人清單完整帶回來
- 若前端未帶回 → 接收人僅剩當前登入者一人

此設計簡化了「差異比對」邏輯，但代價是前端責任變重。

### 7.4 類型變更會清空接收人

若一筆原為「行銷活動」的公告被改為「通知」或「公告」，伺服端：

1. 清空接收人關聯
2. 因新類型不是行銷活動 → **不會重建**

這是合理的（通知／公告不需要接收人），但這代表「不能直接從非行銷類型改回行銷類型再保留原接收人」（列入 §11）。

### 7.5 「我的公告」的查詢拆分

- 第一段：所有非行銷類型公告，不論發布人是誰、是否啟用，全部撈
- 第二段：限定編號清單的行銷類型公告

兩段都不過濾「啟用／停用」狀態（除非呼叫端有帶 status 條件）。因此**已停用的公告若曾被指派**，仍會出現在「我的公告」中（列入 §11）。

### 7.6 推播沒有過濾接收人

「推播公告」端點透過 WebSocket 廣播給所有在線的後台使用者，**不論該使用者是否為該公告的接收人**。前端若不過濾就直接呈現，會洩漏行銷活動內容。

### 7.7 已讀狀態目前無法被更新

接收人關聯的 `readStatus` 欄位寫入時預設「未讀」，但本模組未提供：

- 標記為已讀的 API
- 查詢「某公告的已讀／未讀人數統計」端點
- 查詢「我的未讀公告數」端點

理論上的「未讀紅點」「閱讀率追蹤」需求都未實作（列入 §11）。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| **行銷活動公告管理員（具備「系統-公告-查詢」「-新增」「-修改」「-刪除」全部權限）** | 全部公告（含所有行銷類型） | 新增、修改、刪除行銷類型公告，指派接收人 |
| **任何登入使用者** | 透過「我的公告」端點看到全員可見公告 + 自己被指派的行銷類型 | 僅檢視 |
| **未配發本模組權限者** | 仍可透過「我的公告」「公開分頁」端點看到 | 僅檢視 |

> 行銷活動公告的存取控制屬「設計上交由前端＋查詢端點本身的篩選」處理：未被指派者透過 `selectNoticeIdsByUserId(userId)` 拿不到該公告的編號，因此第二段查詢撈不到資料。資料保密性**仰賴此查詢路徑的正確性**。

## 10. 功能範圍

### 10.1 包含的功能

- 行銷活動類型公告之新增、修改、刪除（與「公告維護」共用入口，類型 = 3）
- 接收人指派：依部門多選（自動展開到後代）＋依使用者多選＋發布人自動納入＋去重
- 「我的公告」分頁查詢（含合併分頁邏輯）
- 接收人關聯之 `readStatus` 欄位寫入（預設未讀）

### 10.2 預留但尚未實作

- **接收人回填**：`NoticeRespVO` 有 `deptIds` / `userIds` 但 Service 未實作回填
- **已讀標記 API**：`NoticeUserDO` 有 `readStatus` 欄位但無更新 API
- **已讀率統計**：無查詢端點
- **接收人查詢**：未提供「某公告的接收人清單」獨立端點
- **接收人驗證**：部門編號、使用者編號皆未驗證存在性／啟用狀態
- **批次寫入優化**：接收人逐筆 insert，缺批次 insert 優化
- **推播過濾**：推播未按接收人清單過濾，廣播給所有在線後台使用者

### 10.3 不包含

- 通知（type=1）、公告（type=2）之 CRUD（在「公告維護」PRD）
- 推播至 Email、簡訊、LINE、Push 通知等外部通道（僅 WebSocket）
- 公告附件上傳（資料表無此欄位）
- 公告排程上下架（無排程機制）
- 公告與選單／角色的可見性綁定（行銷活動的可見性靠接收人關聯，與選單／角色無關）
