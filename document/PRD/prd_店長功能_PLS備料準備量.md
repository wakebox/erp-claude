## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長 / 廚房備料負責人**（例：BK001 信義店）。

每天從開店到收店，廚房後場必須持續「備料」 — 把冷凍牛肉餅退冰、把生菜切好、把麵包預熱、把醬料分裝。**備早了賣不完丟掉、備慢了客人等不到出餐**。PLS（Prep List System，備料準備清單系統）就是給我**「下一小時要備多少」的即時清單**，讓後場一目了然。

> ⚠️ **後端目前沒有這份功能的程式碼**。本 PRD §2～§9 描述「**應該要有什麼**」，§10.2 與 §11 列出「**缺什麼、要確認什麼**」。此功能對「中繼即時訂單 API」與「廚房現場設備」的依賴度比一般 ERP 查詢功能高很多。

### 1.2 我要做什麼

- 開店前（07:00）：看「開店至 10:30 早餐時段」要備多少早餐餅、麵包、雞蛋
- 接近午餐尖峰（10:30）：看「未來 1 小時」需備多少牛肉餅、生菜、麵包
- 全天每 15～30 分鐘：依即時客流動態微調備料
- 客流突增 / 突減時：手動觸發「重算」
- 收店後：對照「實際備量 vs 實際消耗」，回饋預估準確度

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看「下一小時要備多少牛肉餅／麵包／生菜」 | 不知道要備多少就會備過頭或不足 |
| 看「目前已備庫存」與「還要備多少」分開 | 廚房不能再用「總需要量」估，會重複備 |
| 看清楚「為什麼是這個數字」 | 後場師傅不會盲從機器 |
| 在後場大螢幕 / 平板上顯示 | 不可能一邊備一邊翻電腦 |
| 急迫食材要醒目（紅 / 黃 / 綠） | 一眼看到優先級 |
| 可手動覆寫建議量 | 老師傅的直覺有時比演算法準 |
| 客流突變時可手動重算 | 系統的「下一輪」週期太慢 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 即時 / 半即時備料清單（每 15～30 分） | 主畫面 |
| 依未來 1 小時預估銷量推算食材用量 | 演算核心 |
| 扣除「當前已備庫存」算「淨需備量」 | 不重複備料 |
| 急迫度配色（紅 / 黃 / 綠） | 視覺優先級 |
| 食材對應產品說明（這 80 片牛肉餅是給華堡用） | 後場可理解 |
| 手動覆寫 / 立即重算 | 業務彈性 |
| 後場大螢幕 / 平板專用 UI | 廚房動線 |
| 預估準確度回饋（事後比對） | 持續改善 |

## 2. 功能目的

讓門市後場能在無須翻 POS、無須估算的前提下，**自動取得「未來 1 小時各食材還需要備多少」的清單**，並以紅 / 黃 / 綠急迫度配色顯示在後場大螢幕，讓備料動作直接對齊預估銷售。同時讓店長能手動微調建議量，並在事後比對「預估 vs 實際」，逐步提升預估準確度。

設計理念：**事前預估、可覆寫、留軌跡**。系統提出建議備量，但最終決定權在現場師傅；所有覆寫紀錄都會被保留，做為下一輪演算法調校的依據。

## 3. 業務邏輯背景

### 3.1 「備料準備量」是什麼

備料分三層：

1. **食材**（牛肉餅、生菜、麵包、醬料）：本功能管的層
2. **半成品**（已退冰牛肉餅、已切生菜）：本功能可選擴充
3. **成品**（已組好的華堡）：屬出餐管理，非本功能

「準備量」指「**下一小時需要備多少食材，扣掉現在已備好的，還差多少**」。例：未來 1 小時預估賣 80 個華堡，每個用 1 片牛肉餅，目前後場已退冰 30 片 → **還要再退 50 片**。

### 3.2 預估邏輯（推論）

```
下一小時食材需備量 = ( 預估下一小時各產品銷量 × 各產品食譜標準用量 ) - 當前已備量

預估下一小時各產品銷量 = 過去 N 週同期同時段平均 × 加權係數
                       + 即時訂單修正
                       + 手動覆寫值（若有）
```

可能的加權因子（待確認）：

- 天氣（晴雨、溫度）
- 節氣 / 假日 / 活動日
- 上午表現偏差（若早餐高於平均 20%，午餐可能也偏高）
- 區域促銷活動

### 3.3 資料來源（推論）

| 來源 | 提供什麼 | 用途 |
|---|---|---|
| 中繼「按時間區間查訂單」服務 | 過去同期銷售 | 「同期平均」基底 |
| 中繼「即時訂單 / 排隊訂單」服務（未確認存在） | 當前未出餐訂單 | 「即時修正」 |
| ERP 食譜（產品 ↔ 食材標準用量） | 每個華堡用幾片牛肉餅 | 「銷量 → 用量」展開 |
| ERP 庫存系統 | 當前已備食材庫存 | 計算「還要備多少」 |
| ERP 食材主檔 | 食材名、單位 | 顯示用 |

⚠️ **中繼是否提供「即時 / 排隊訂單」未知**。若無，「即時修正」這層只能改用 POS 端推送或暫時放棄，**待確認**。

### 3.4 與其他功能的關係

| 功能 | 時間粒度 | 用途 |
|---|---|---|
| 食材需求預測（PDM） | 週 | 進貨計畫 |
| 每日盤點作業 | 日 | 庫存結餘 |
| 每日時段報表 | 時段（早午晚） | 事後檢視 |
| **PLS 備料準備量（本功能）** | **小時** | **事前現場備料** |

PLS 用每日時段報表的事後資料校準預估、用食材需求預測的 BOM 展開食譜、用每日盤點的庫存做扣除。是這三者的「時間粒度最細、現場使用」的延伸。

### 3.5 「期初庫存」如何取得（推論）

「當前已備量」可有兩種定義：

| 定義 | 優點 | 缺點 |
|---|---|---|
| ERP 庫存表的「當前餘量」 | 系統一致 | 含未退冰 / 含倉儲，非真正「後場已準備」 |
| 早班開店盤點數 + 當日進貨 - 已消耗 | 接近現場真實 | 需手動盤點配合 |

推薦：**前期用 ERP 庫存表近似，後期建議在後場加掛「已備量手動回填」機制**。

### 3.6 重算週期

| 週期 | 適用情境 |
|---|---|
| 15 分 | 尖峰前 30 分啟動 |
| 30 分 | 平時段 |
| 60 分 | 離峰 / 早餐後 |
| 手動立即 | 客流突變、活動異常 |

### 3.7 急迫度配色（推論）

```
紅色（需立刻備）   : 未來 30 分內可能不足
黃色（提前備）     : 未來 60 分內需備齊
綠色（充足）       : 庫存足夠覆蓋 60 分以上
灰色（已超量）     : 已備量超過建議
```

## 4. 情境說明

### 4.1 正常流程 — 午餐尖峰前 30 分

5/19 週二 10:30，BK001 信義店廚房大螢幕自動更新：

```
PLS 備料清單 ｜ 11:00 – 12:00
──────────────────────────
紅 牛肉餅      還要備 50 片  (預估 80, 已備 30)  → 華堡 60、雙層牛 20
紅 生菜絲      還要備 1.2 公斤
黃 起司片      還要備 30 片
黃 番茄片      還要備 25 片
綠 麵包        還要備 0 個    (預估 80, 已備 95)
```

師傅看到紅色項目立即動作：退冰牛肉餅 50 片、切生菜 1.2 公斤。

### 4.2 正常流程 — 開店前早餐備料

7:00 開店前，店長小陳打開 PLS：

1. 預設顯示「7:00 – 10:30 早餐時段」
2. 系統依「上週二早餐銷售」推估
3. 列出：早餐肉餅 35 片、漢堡蛋 40 顆、培根 20 片、薯餅 60 片
4. 小陳依今日陰雨，**手動上調漢堡蛋 +5 顆**（手感）
5. 後場依清單備料

### 4.3 客流突增情境

12:15 午餐尖峰，30 分前的預估是 80 份華堡，但前 15 分已賣 50 份：

1. 店長按「立即重算」按鈕
2. 系統重新拉即時訂單，預估 12:30 – 13:00 的銷量上修至 90
3. 牛肉餅需備量從 50 上修至 60，配色由「紅」變「深紅」
4. 後場立即追加退冰

### 4.4 手動覆寫情境

某日有大型活動下單 50 份華堡套餐（已預訂）：

1. 店長進入 PLS
2. 在「牛肉餅」列點覆寫，加 50 片並備註「11:30 團體訂單」
3. 系統記錄覆寫人、覆寫值、原建議值、備註
4. 重算時不再覆蓋此手動值（直到該時段結束）

### 4.5 異常情境 — 中繼斷線

中繼「即時訂單」API 12:00 突然斷線：

1. 系統降級：改用「純歷史同期平均」做預估
2. 畫面頂端顯示告警 banner「即時資料中斷，依歷史推估，請現場主管研判」
3. 重算頻率不變，但精度下降
4. 中繼恢復後自動切回

### 4.6 異常情境 — 食譜缺失

某新品上線但食譜尚未維護：

1. 系統發現預估銷量含某產品但無食譜
2. 列表底部顯示「⚠️ 以下 1 項產品無食譜，未納入備料計算：XX 新品（預估 12 份）」
3. 店長知道要手動加備

### 4.7 事後比對情境

每晚 23:30 系統排程比對：

1. 取當日各時段「最終建議備量 vs 實際消耗量」
2. 落「PLS 預估校準」表，做為下次演算加權調整
3. 店長可在隔日報表查到「上週牛肉餅平均誤差 -5%」

[後場大螢幕 / 店長平板]
  ├─ 每 15 秒自動拉新清單
  ├─ 紅黃綠急迫度配色
  ├─ 點任一食材 → 顯示「對應產品說明」

## 7. 商業邏輯與設計建議

### 7.1 鎖店過濾

PLS 強鎖店，無跨店概念（廚房不會跨店備料）。

### 7.2 預估演算法分層設計（建議）

```
第一階段（MVP）：純歷史推估
  下小時銷量 = 過去 4 週同期同時段平均
  → 上線快、易驗證

第二階段：加權
  + 即時當日上午偏差係數
  + 假日 / 平日切換
  → 提升精度

第三階段：機器學習
  + 天氣、活動、區域特徵
  → 需大量歷史資料訓練
```

⚠️ 建議**先做第一階段 MVP 並上線驗證 1～2 個月**，再投入第二、三階段。

### 7.3 食譜展開規則

```
1. 取「下小時預估銷量」每個產品
2. 對每產品查食譜（PDM「產品配方」）
3. 對每食材：用量 += 該產品銷量 × 標準用量
4. 多產品共用食材自動加總
5. 食譜缺失之產品列入「未納入計算」清單
```

### 7.4 當前已備量取得

MVP 階段：直接讀 WHS 庫存表「當前餘量」。

長期建議：在後場加設「手動回填當前已備」介面，店員每 30 分鐘輕點一次。

### 7.5 急迫度演算

```
若 (建議備量 ÷ 預估總用量) > 0.7  → 紅（缺口大）
若 (建議備量 ÷ 預估總用量) ∈ (0.3, 0.7] → 黃
若 (建議備量 ÷ 預估總用量) ∈ (0, 0.3]   → 綠
若 已備量 ≥ 預估總用量 × 1.2 → 灰（已超備）
```

數字為示意，需業務確認。

### 7.6 手動覆寫的有效期

```
覆寫值生效時段＝該次預估覆蓋區間（11:00–12:00）
逾時自動失效，下次重算改用系統建議
```

### 7.7 降級機制

若中繼「即時資料」斷線：

```
1. 不阻斷功能，改用「純歷史推估」
2. 畫面 banner 顯示「即時資料中斷，請現場主管研判」
3. 排程仍照常重算
4. 中繼恢復後下一輪重算自動切回
```

### 7.8 重算最小間隔

防呆：「立即重算」按鈕**最小間隔 2 分鐘**，避免店長焦慮按爆 API。

### 7.9 唯讀範圍

店長可：查詢、手動覆寫、立即重算。
店長不可：修改食譜、修改加權係數、修改字典。

## 10. 功能範圍

### 10.1 包含的功能（建議實作）

- 自店 PLS 備料清單即時 / 半即時更新
- 食譜展開（產品 → 食材）
- 當前已備量扣除
- 急迫度配色
- 手動覆寫與軌跡保留
- 立即重算
- 中繼斷線降級
- 事後預估準確度比對
- 後場大螢幕與店長平板雙端 UI

### 10.2 預留但尚未實作

> ⚠️ **整份功能後端未實作**。以下為實作前需建立的基礎：

| 項目 | 證據 |
|---|---|
| 店長視角專屬功能入口 | 無對應 Controller |
| PLS 清單主檔與明細表 | 後端無對應資料表 |
| 重算排程（每 15/30 分） | 排程清單未發現 |
| 中繼「即時 / 排隊訂單」服務 | 既有中繼 client 僅支援「按時間區間查已完成訂單」與「指定日期商品銷量」，**無即時 API**（待中繼方確認） |
| 「預估演算法」（即使是 MVP 同期平均） | 後端無對應 Service |
| 食譜「產品 ↔ 食材標準用量」結構 | PDM 已有「產品配方」基礎可用 |
| 「使用者 ↔ 門市」對應表 | 後端尚無 |
| 後場大螢幕 / 平板 UI | 前端未開發 |
| 預估準確度排程與表 | 無 |

### 10.3 不包含

- 半成品（已退冰、已切）追蹤（建議另立功能）
- 成品（已組好華堡）出餐管理（屬 KMS 廚房系統）
- 食譜維護（屬 PDM 模組）
- 食材進貨建議（屬需求預測模組）
- 廢棄量追蹤（屬庫存異動）
- 即時 POS 端整合（屬中繼系統）
