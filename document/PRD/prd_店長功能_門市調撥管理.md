## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。當我快斷貨、隔壁店有，或我多了一堆食材沒人用，我會發起或接受「店對店調撥」：

> 「2026-05-19 上午 10:30 — 信義店剩 8 公斤生菜、敦化店剩 0.5 公斤已低於安全存量；我（敦化店店長）向信義店發起調撥『請撥 5 公斤』；信義店店長確認可撥；下午 14:00 司機送達；雙方系統紀錄完成。」

### 1.2 我要做什麼

- 缺料時：找鄰近且有庫存的店發起調撥
- 多料時：主動把過剩食材撥給需要的店
- 收到他店的調撥請求時：確認是否可撥（庫存足夠？影響營運？）
- 收貨時：點收 + 寫入庫存（同入庫流程）
- 對帳：每月看本店「撥出 vs 撥入」總量

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看「鄰近有庫存」的店清單 | 第一時間找到能撥的店 |
| 雙方確認流程 | 不能我說撥就撥，撥出方要同意 |
| 即時看雙方目前庫存 | 撥多少才不影響各自營運 |
| 大額要簽核 | 跨店物資移動需有監督 |
| 行動裝置可操作 | 缺料當下站在賣場用手機 |
| 只看「我發起的」或「我接收的」 | 不要看到別人之間的撥動 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「我發起的調撥」列表 | 我的請求進度追蹤 |
| 「我接收的調撥請求」列表 | 待我確認的對方請求 |
| 鄰店推薦（距離 + 庫存） | 找撥方 |
| 雙方即時庫存對照 | 撥多少才合理 |
| 雙方確認狀態機 | 草稿→待確認→運送中→已收貨→完成 |
| 撥出簽核（撥出方店長同意） | 雙向約束 |
| 大額調撥綁區經理簽核 | 風險控管 |
| 完成自動建出入庫單 | 不用兩次手動操作 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 門市調撥管理（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理」 |
| 兄弟功能 | 倉儲查詢、安全存量設定、入庫管理、出庫管理 |
| 主要頁面 | 我發起的、我接收的、發起新調撥、編輯頁、歷史查詢 |
| 簽核流程 | 底層已串簽核流程；店長視角需額外綁「對方店長確認」 |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在缺料／多料當下，**自助發起點對點的食材調撥**，並讓撥出方與撥入方在雙方確認的前提下完成搬移。在門市端建立「靈活補位機制」：總部來不及補貨時，店與店之間先互相幫忙，減少斷貨與報廢。

設計理念：**雙方對等、雙向確認、出入庫聯動、大額必審**。店長**只能看見自己發起或自己接收的單據**（不是全公司的調撥都看得到）。發起與接收雙方均需有明確的角色與動作，狀態機清楚標示「現在等誰」。

## 3. 業務邏輯背景

### 3.1 調撥的雙方角色

| 角色 | 動作 |
|---|---|
| **發起方店長**（撥入方） | 發起調撥 → 等對方確認 → 收貨 → 簽收 |
| **撥出方店長** | 收到請求 → 確認可撥 → 安排出庫 → 等對方收貨 |
| 區經理 | 大額簽核（雙方都需通過） |
| 物流／司機 | 運送（流程外，但可影響時程） |

### 3.2 調撥單的結構

- **單頭**：單據編號、流程狀態、區域、倉別、出庫倉、入庫倉、調撥時間、主旨、物流處理、出入庫旗標（追蹤是否已執行）
- **明細**：每食材的品號、申請數量、確認數量、出庫倉當前庫存、入庫倉當前庫存

### 3.3 雙狀態機（底層既有 + 店長視角應有）

底層目前有兩個狀態欄位：

- **流程狀態**：待處理／待簽核／已簽核／已歸檔
- **出入庫旗標**：null（未執行）／已出庫／已入庫

⚠️ 底層 PRD 已標示**繁簡體混搭風險**：「已簽核／已签核」「已歸檔／已归档」四種字串均可能出現，比對易出錯。

對店長視角，建議擴充為完整狀態機：

```
草稿 → 待對方店長確認 → 待區經理簽核（大額）→ 待出庫 → 運送中 → 待收貨 → 已收貨完成
                ↓                      ↓
             對方拒絕              區經理駁回
                ↓                      ↓
              取消                    退回
```

### 3.4 試算（底層既有）

底層提供「對品號清單算雙方當前庫存」端點，建單時即時顯示：

- 出庫倉當前量
- 入庫倉當前量

> 推論：店長視角應同時顯示「安全存量水位」，幫助雙方判斷撥多少才不影響各自。

### 3.5 與出入庫的串接（底層既有）

調撥單編號的前綴用來識別來源。簽核完成後：

1. 系統用調撥單號生成「出庫批次」 → 觸發批次處理 → 撥出方庫存減少
2. 系統用調撥單號生成「入庫批次」 → 觸發批次處理 → 撥入方庫存增加
3. 兩次批次處理會回寫調撥單的「出入庫旗標」

⚠️ 底層風險：

- 「申請數量 / 確認數量 / 實際出庫量」三層**無校驗**
- 出庫量 ≠ 入庫量（運輸損耗）未分流處理
- 已歸檔且有流程實例的單據仍可修改流程狀態

### 3.6 鎖店規則（待實作）

店長視角的所有讀寫，應在後端強制：

```
1. 解析登入者所屬門市
2. 列表只回「出庫倉屬該店長」或「入庫倉屬該店長」的單據
3. 編輯／確認時驗證身份
4. 不接受參數覆寫
```

> 推論；目前後端未實作。

### 3.7 與總部視角的差異

| 對比項 | 總部／區經理 | 店長 |
|---|---|---|
| 可看 | 全公司／自區 | **只能看自己發起或自己接收** |
| 角色 | 監督／媒合／大額簽核 | **發起／接收／雙方確認** |
| 模式 | 主動撥動 | **點對點協商** |
| 行動裝置 | 桌機為主 | **手機／平板** |
| 雙方確認 | 不參與 | **必要**（撥出方同意才能執行） |
| 簽核 | 視政策 | 大額**建議簽核**（推論門檻見 §7） |

## 4. 情境說明

### 4.1 正常流程 — 缺料找鄰店

5/19 上午 10:30，敦化店店長阿陽發現生菜只剩 0.5 公斤（安全存量 5 公斤），距離下次配送還有 4 小時：

1. 開「我要調撥」→ 選「生菜」「需求 5 公斤」
2. 系統列出鄰近 3 km 內有生菜庫存的店：
   - 信義店：8 公斤
   - 仁愛店：3 公斤
   - 復興店：6 公斤
3. 阿陽選「信義店 5 公斤」
4. 系統建單，狀態「待信義店確認」
5. 信義店店長小陳收到推播
6. 小陳看：信義店生菜 8 公斤，撥 5 後剩 3 公斤（仍高於該店安全水位 2 公斤）→ 同意
7. 狀態「待出庫」→ 信義店執行出庫，庫存 -5
8. 狀態「運送中」（司機 / 工讀生帶過去）
9. 14:00 抵達敦化店，阿陽點收 5 公斤、實收 5 公斤、簽收
10. 狀態「已收貨完成」，敦化店庫存 +5
11. 雙方系統都留下完整紀錄

### 4.2 異常 — 對方拒絕

延續 4.1 場景，但小陳發現信義店今晚有業績比賽，生菜剩 8 公斤已不算多：

1. 小陳「拒絕」+ 留言「今晚活動，無法調撥」
2. 阿陽收到通知，狀態「對方拒絕」
3. 阿陽改向「仁愛店 3 公斤」發新單，或聯絡總部緊急配送

### 4.3 大額調撥情境

板橋店缺 50 公斤牛肉餅，向新莊店調撥：

1. 板橋店店長發單
2. 因金額／數量超過「免簽門檻」（推論：±20 公斤或 NT$3,000）
3. 流程：板橋店店長發單 → 新莊店店長同意 → 區經理簽核 → 出入庫
4. 比一般調撥多一個簽核節點

### 4.4 異常 — 跨店越權

新進店長阿明用網址列直接打共用 API，傳入「信義店 → 板橋店」的調撥單，自己其實是敦化店店長：

⚠️ 目前底層**未拒絕**；阿明可成功建立一張別人的調撥單。

此為**目前已知資安風險**。

### 4.5 物流損耗

5/22 信義店撥 5 公斤生菜到敦化店，運送途中冷藏失效，實到 3 公斤可用、2 公斤報廢：

1. 敦化店點收：實收 3 公斤
2. 系統應：信義店庫存 -5、敦化店庫存 +3、差額 2 公斤進入「不良品報損」流程
3. 連動報廢費用由誰承擔（信義店／物流商）

⚠️ 第 2 步的「差額分流」目前底層**未實作**；目前出庫量 = 入庫量。

### 4.6 雙方確認後流程未推進

5/23 敦化店發起調撥，信義店同意了，但兩天過去無人去取：

1. 系統應有「逾時提醒」推播給雙方
2. 逾時 N 天後自動取消

> 推論；目前底層無逾時機制。

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[發起方店長：缺料 → 開「我要調撥」]
  ├─ 登入身份取得：店長、所屬門市（撥入方）
  ├─ 選食材 + 需求量
  ├─ 系統推薦：鄰近且有庫存的店（距離 × 庫存權重）
  ├─ 選撥出方 + 確認數量
  │
  ├─ 試算：雙方目前庫存、雙方安全存量水位
  │
  ├─ [送出]
  │    ├─ 建單，狀態「待對方店長確認」
  │    └─ 推播通知對方店長
  │
[撥出方店長：收到推播]
  ├─ 開「我接收的調撥請求」
  ├─ 看明細與雙方庫存
  ├─ [同意] → 狀態「待出庫」（大額再進區經理簽核）
  ├─ [拒絕] → 狀態「對方拒絕」，發起方可重發
  │
[執行階段]
  ├─ 撥出方安排出庫
  │    └─ 系統依單號生成出庫批次 → 撥出方庫存減
  ├─ 運送（流程外）
  ├─ 發起方收貨點收
  │    └─ 系統依單號生成入庫批次 → 撥入方庫存加
  ├─ 雙方都完成 → 狀態「已收貨完成」
  │
  └─ 另：自店歷史調撥單查詢（雙向：撥出 + 撥入）

⚠️ 「鎖店」「對方確認狀態」「鄰店推薦」「逾時取消」「物流損耗分流」目前後端均未實作。
⚠️ 流程狀態的繁簡體混搭比對風險仍在底層。
```

## 7. 商業邏輯

### 7.1 鎖店與雙向視角

店長視角的所有讀寫，應在後端強制：

```
1. 解析登入者所屬門市
2. 列表 = (出庫倉 = 自店) OR (入庫倉 = 自店)
3. 建單時驗證：入庫倉必須屬該店長（即「我發起的」）
4. 同意撥出時驗證：出庫倉必須屬該店長
5. 不接受參數覆寫
```

> 推論；目前後端未實作。

### 7.2 鄰店推薦演算（推論）

```
1. 取所有「自店除外」的門市
2. 過濾「該食材庫存 > 申請數量」的店
3. 依「距離 × 0.4 + 庫存充裕度 × 0.6」排序（權重待 PM 定）
4. 回前 5–10 家
```

> 待確認：距離資料來源、是否限同區內、是否排除特定加盟形態。

### 7.3 免簽／必簽門檻（推論）

| 規模 | 流程 |
|---|---|
| 小額（< 5 公斤 or < NT$1,000） | 雙方確認即可 |
| 中額（5–20 公斤 or NT$1,000–NT$3,000） | 雙方確認 |
| 大額（> 20 公斤 or > NT$3,000） | 雙方確認 + 區經理簽核 |
| 跨區調撥 | 必經區經理簽核 |

> 推論；實際門檻待 PM 定義。

### 7.4 出入庫聯動

簽核完成後：

```
1. 撥出方執行出庫：
   - 庫存 ≥ 確認數量？否則拒絕
   - 庫存 -= 確認數量
   - 寫出庫單

2. 撥入方收貨：
   - 輸入實收數量
   - 庫存 += 實收數量
   - 寫入庫單
   - 若實收 < 確認 → 觸發物流損耗單（待實作）
```

⚠️ 底層目前缺：撥出方的庫存擋負值（同出庫管理）、實收 ≠ 確認的損耗分流。

### 7.5 逾時取消（推論）

| 狀態 | 逾時時間 | 動作 |
|---|---|---|
| 待對方店長確認 | 4 小時 | 自動取消 |
| 待出庫 | 24 小時 | 提醒 + 48 小時取消 |
| 運送中 | 24 小時 | 提醒區經理 |

> 推論；待 PM 定義。

### 7.6 狀態字串繁簡相容（既有問題）

底層流程狀態同時可能寫入「已簽核／已签核／已歸檔／已归档」四種字串。對店長視角的影響：

- 列表分群／篩選需做字串正規化
- 建議統一改為枚舉 / 字典代號

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店發起或接收的調撥 | 發起、同意、執行、查詢、匯出 |
| 副店長（自店） | 同店長 | 同店長（待確認大額是否需店長覆核） |
| 區經理 | 區內所有店之間的調撥 | 大額簽核、跨區媒合、唯讀檢視 |
| 倉儲／採購（總部） | 全公司 | 全 CRUD |
| 系統管理員 | 全公司 | 全 CRUD（後門用） |

> 後端目前以「具備調撥權限」的角色控管，未細分發起／接收／簽核，也未鎖店。
