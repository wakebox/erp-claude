>
> ⚠️ 後端**目前無店長視角專屬 endpoint**；既有 API 為總部／物流規劃共用版本。本文件以「店長能用什麼／應該看到什麼」視角撰寫，做為 PM 對齊、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

我的店每週會經過數次配送，**派車之前**這些「我要收什麼」的明細已經算好了。我打開「物流規劃查詢」，是要查 「**未來的某一天，我的店預計會收到的清單**」 — 還沒派車、還沒上路的「**規劃量**」。

簡單說：行事曆告訴我「哪天有」，規劃查詢告訴我「**那天我會收什麼，量多少**」。

### 1.2 我要做什麼

- 排班前一天：列出明天要收的所有品項與數量
- 採購助理問起：「某張預測單的牛肉餅哪天會送？」 → 用單號查
- 月底對帳前：列出某段日期區間我的店所有規劃量
- 異常追蹤：規劃裡有，現場沒收到 → 跳到「物流配送查詢」或「物流異動查詢」

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 看「明天會收什麼」清單 | 排班、備位置、備設備 |
| 依預計配送日查 | 物流是以日為單位 |
| 依來源預測單號查 | 想知道某張預測單何時實際送達 |
| 看數量、單位 | 預估點收時間 |
| 只看自店 | 全公司資料無關 |
| 行動裝置可用 | 賣場現場手機查 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 依預計配送日列出規劃明細 | 主畫面 |
| 依來源單號／食材搜尋 | 反向查詢 |
| 日期區間過濾 | 多日彙整 |
| 配送模式過濾（週／月／直送） | 區分不同物流 |
| 跳轉至配送查詢／異動查詢 | 異常時的下一步 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 物流規劃查詢作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合（PDM）」 |
| 兄弟功能 | 物流管理行事曆、物流配送查詢、物流異動查詢、物流串接記錄 |
| 主要頁面 | 規劃明細列表、單筆展開 |
| 簽核流程 | 無（純查詢） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能從「**列表／表格**」視角，**精準查到某天我的店預計收到的每筆品項**。

設計理念：
- **「規劃」= 派車前**。資料來自需求預測明細，物流公司尚未排車
- **以列表為主、行事曆為輔**（行事曆是入口型總覽，本功能是明細型清單）
- **唯讀＋鎖店**
- 與「物流配送查詢」是同一條鏈的兩個階段：本功能是**派車前的規劃**，配送查詢是**派車後的執行**

---

## 3. 業務邏輯背景

### 3.1 「規劃」階段資料怎麼產生（與店長無關，但要知道）

每張需求預測單／臨時需求單，被建立或簽核後，系統會展開成「需求預測明細」：

```
某張預測單  →  展開出多筆明細
            每筆含：食材、數量、需求日、預計配送日、配送模式、門市
```

「物流規劃查詢」看的就是這份明細表 — 還沒進到物流公司、還沒派車。

### 3.2 規劃 vs 配送 vs 異動 vs 行事曆（店長視角）

| 功能 | 看什麼 | 主要時機 |
|---|---|---|
| 物流管理行事曆 | 未來某天有沒有配送 | 排班、總覽 |
| **物流規劃查詢（本功能）** | **某天具體會送什麼、量多少（派車前）** | **排班前細查、單號追蹤** |
| 物流配送查詢 | 派車後車輛狀態 | 收貨當下 |
| 物流異動查詢 | 實際到店 vs 預計差異 | 收貨後對帳 |
| 物流串接記錄 | ERP 給中央倉的批次紀錄 | IT 軌跡 |

> 本功能與「物流配送查詢」的最大差別：**規劃查詢看的是「該到什麼」，配送查詢看的是「車到哪了」**。

### 3.3 來源單號的兩種類型

- **需求預測單號**：常規每月／每週預測展開的單號
- **臨時需求單號**：臨時加單（活動補貨、預測偏差補單）

兩種單號**並存**，店長搜尋時可能用任一種；後端底層支援「逗號分隔多單號」批次查詢。

### 3.4 「我的店」是怎麼判定的

底層查詢有「指定門市」過濾，但**沒有「依登入者自動鎖店」**。店長視角需在後端：
1. 從登入身份取得所屬門市
2. 強制覆寫門市條件
3. 不允許前端傳其他門市

### 3.5 與行事曆的關係

行事曆每一日格子，「**點開**」就應該跳到本功能的「**當日明細**」。換言之：

- 行事曆 = 入口
- 規劃查詢 = 明細

兩功能應緊密整合，避免店長重複輸入查詢條件。

## 4. 情境說明

### 4.1 正常流程 — 週日排明日班

5/18（日）晚上，BK001 信義店店長小陳：

1. 打開「物流規劃查詢」，預設條件「明日（5/19）、自店」
2. 列表顯示 8 筆明細：
   - 牛肉餅 12 公斤（週配，來自預測單 D-2026-21）
   - 生菜 8 公斤（週配）
   - 麵包 200 個（週配）
   - 雞肉 5 公斤（週配）
   - 等
3. 小陳估算點收時間約 30 分鐘，安排兩名同事

### 4.2 用單號反查 — 追單

採購助理打電話來：「上週送的那張預測單 D-2026-19 的雞肉，預計哪天會送到你們店？」

1. 小陳在「單號」搜尋框輸入 D-2026-19
2. 列表只顯示該單在自店的明細
3. 看到雞肉行：預計配送日 5/22、量 8 公斤
4. 回覆採購助理

### 4.3 日期區間 — 月底對帳

月底店長想清查上月所有規劃量：

1. 切日期區間：4/1 ～ 4/30
2. 列表顯示該月自店所有規劃明細
3. Excel 匯出（限自店）做月報

### 4.4 異常情境 — 「規劃裡有，現場沒收到」

5/19 早上 10:30，店長發現生菜沒到：

1. 進「物流規劃查詢」，5/19 自店列表確認生菜「應該要送」
2. 點該行 → 跳「物流配送查詢」看車輛狀態
3. 若車已抵達但無生菜 → 再跳「物流異動查詢」建短裝紀錄

### 4.5 配送模式過濾 — 只看月配

某月底要對紙袋／清潔用品（月配）庫存：

1. 配送模式選「月配」
2. 列表只顯示月配明細，過濾掉每週都有的食材

### 4.6 公休日後補配 — 跨日合併

店長公休日（5/20）取消的配送，預測單可能合併到隔日（5/21）：

> 待確認：合併規則由誰決定，是否在明細上有標示「原預定 5/20、改 5/21」。

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「物流規劃查詢」]
  |
  +- 取得登入者身份 → 鎖定「自店」
  |
  +- 預設條件：今日／明日、自店、不限配送模式、不限單號
  |
  +- 撈「需求預測明細」 where 自店 + 條件
  |
  +- 列表呈現：
  |    每列：食材、數量、配送模式、預計配送日、來源單號
  |
  +- 互動：
  |    +- 點某行 → 展開該筆細節（食材主檔資訊、廠商）
  |    +- 點「跳配送查詢」→ 帶單號去配送查詢
  |    +- 點「跳異動查詢」→ 帶單號去異動查詢
  |    +- 點「Excel 匯出」→ 限自店
  |
  +- 從行事曆跳入時：日期自動帶當日
```

## 6. 欄位規格

### 6.1 篩選條件

| 欄位（業務名） | 必填 | 說明 |
|---|---|---|
| 預計配送日（起） | 否 | 預設今日 |
| 預計配送日（迄） | 否 | 預設今日，可調區間 |
| 食材 | 否 | 模糊搜尋 |
| 來源單號 | 否 | 預測單或臨時需求單，可逗號分隔多筆 |
| 配送模式 | 否 | 週／月／直送 |
| 門市 | 系統自填 | 鎖定為登入者自店 |

### 6.2 列表欄位

| 欄位（業務名） | 店長視角是否該顯示 |
|---|---|
| 食材名稱 | 是 |
| 食材品號 | 是（次要、可摺疊） |
| 數量＋單位 | 是 |
| 配送模式 | 是 |
| 預計配送日 | 是 |
| 來源預測單號 | 是 |
| 來源臨時需求單號 | 是 |
| 預估銷售量 | 否（屬規劃內部資料，店長不需） |
| 門市名稱 | 否（一律自店） |
| 區域 | 否（一律自區） |

### 6.3 行動詳細頁

| 欄位（業務名） | 說明 |
|---|---|
| 廠商名稱 | 從廠商主檔帶 |
| 食材分類 | 從食材主檔帶 |
| 規劃量／單位 | 同列表 |
| 派車狀態 | 由「物流配送查詢」帶（推論，需聯動） |

## 7. 商業邏輯

### 7.1 自動鎖店

同其他店長功能：登入時取得所屬門市，覆寫過濾條件。

### 7.2 預設查詢區間

- 列表預設「**今日**」單日
- 從行事曆跳入時：帶該日
- 可手動拉成「明日」「本週」「自選區間」

### 7.3 唯讀

店長視角全程**只能查**。底層 API 提供修改（總部視角專用），但店長按鈕應不出現。

### 7.4 多單號查詢

來源單號欄位支援逗號分隔，後端會 split 為列表查詢。前端應提供「多選 + 自動帶逗號」的友善 UI。

### 7.5 跨日合併規則（待確認）

公休日後的合併補配如何呈現尚未確認；推論應在明細上有「原預計／實際預計」兩個日期欄位。

### 7.6 與其他物流功能的跳轉串接

- 規劃 → 配送（同筆明細，帶單號跳）
- 規劃 → 異動（同筆明細，帶單號跳）

> 後端 API 之間需共用「以單號／明細 ID 反查」端點。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店所有預測明細 | 唯讀＋自店 Excel 匯出 |
| 副店長／代班 | 自店所有預測明細 | 唯讀（推論） |
| 區經理 | 區內各店切換 | 唯讀（待確認） |
| 總部物流規劃 | 全公司 | 走總部視角 |

> 後端目前以「具備需求預測查詢權限」的角色控管，未細分店長／總部。店長視角的鎖店過濾需另實作。

## 10. 功能範圍

### 10.1 包含的功能

- 自店預測明細列表查詢
- 多條件過濾（日期、食材、單號、配送模式）
- 單筆展開含廠商與分類資訊
- 跳轉至配送查詢／異動查詢
- 自店 Excel 匯出

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端無 `/app/*` 對應路徑 |
| 自店過濾（鎖店） | 既有 API 接受任意門市條件 |
| 派車狀態顯示 | 需與「物流配送查詢」聯動，目前無欄位 |
| 跨日合併補配標示 | 後端無此欄位（待確認業務規則） |
| 自店 Excel 匯出限制 | 既有 Excel 匯出無門市過濾 |

### 10.3 不包含

- 修改規劃明細（屬需求預測模組）
- 行事曆視圖（屬「物流管理行事曆」）
- 即時車輛狀態（屬「物流配送查詢」）
- 異動建立（屬「物流異動查詢」）
