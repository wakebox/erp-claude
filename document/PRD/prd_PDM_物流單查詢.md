## 1. 功能概覽

### 1.1 我是誰
我是總部 **物流管理人員**，負責管理原物料從廠商出貨到各門市到貨的物流配送記錄。我需要在系統中建立、查詢和維護每一份物流配送單據，確保各門市的食材供應記錄完整可追溯。

### 1.2 我要做什麼
- 每次廠商出貨後，在系統中建立一張物流單，記錄此次配送批次的基本資訊
- 在物流單中登錄各門市、各品項的實際到貨數量
- 查詢特定出貨日、特定廠商的物流記錄，確認配送是否完成
- 匯出物流單資料提供報表或對帳使用
- 如有錯誤，可修正或刪除物流單

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 快速找到某日期、某廠商的物流配送記錄 | 廠商眾多、出貨日頻繁，需多條件篩選縮小範圍 |
| 查看物流單內各門市的實際到貨明細 | 主單只記概況，各門市到貨量需展開明細才能確認 |
| 知道配送的是哪個食材、哪個廠商、哪個門市 | 明細只有代號，需系統自動帶出名稱，避免人工查找 |
| 能匯出 Excel 做對帳 | 財務或採購需要離線報表進行廠商對帳 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 物流單分頁查詢（含多條件篩選） | 快速定位特定批次的配送記錄 |

## 2. 功能目的

本功能管理原物料配送批次單據（物流單），完整記錄每一次廠商出貨到多家門市的配送資訊，包含出貨日、配送廠商、配送模式，以及各門市各品項的實際到貨數量。

設計核心理念為「一張主單對應多筆明細」——主單記錄此次配送批次的共通屬性，明細逐筆記錄每個品項到每個門市的到貨量，並可追溯關聯到的需求預測單或臨時需求單。

---

## 3. 業務邏輯背景

### 3.1 物流單的業務位置

物流單是物流配送結果的記錄性單據，記錄廠商「已出貨」的事實。它與需求端的關聯如下：

- 明細中的**需求預測關聯單號**（`demandRelationDoc`）指向需求預測預估單
- 明細中的**臨時需求關聯單號**（`tempRelationDoc`）指向臨時需求預估單
- 一張物流單可同時包含來自常規需求和臨時需求的品項

### 3.2 配送模式與物流類型

物流配送模式（如「直送」、「配送」）的主檔資料維護於物流類型維護表（`pdm_logistics_type`）。物流單的配送模式欄位值應對應物流類型中的模式設定，但系統目前以字串儲存，無程式碼層級的強制對應。

### 3.3 廠商資訊的來源

明細中的廠商代號在查詢時，系統會去採購管理模組的廠商報價維護表（`pmm_vendor_quote_maintenance`）查找對應廠商名稱並回傳。主表的配送廠商代號目前僅儲存代號字串，查詢時不自動補帶廠商名稱。

### 3.4 區域資訊的取得方式

明細查詢時，區域名稱係透過「找出 `area` 欄位等於該區域ID的門市記錄，取其名稱」的方式間接取得，而非直接查詢區域主檔。

## 4. 情境說明

### 4.1 正常流程 — 查詢特定廠商的物流配送記錄

物流管理人員小陳在 5/25 收到廠商 `V-001` 的出貨通知，欲確認系統中的配送記錄。她在物流單查詢頁面輸入廠商代號 `V-001` 並選擇出貨日 `2026-05-25`，系統回傳 3 筆物流單（單號如 `LOG-2026-0521`、`LOG-2026-0522`、`LOG-2026-0523`）。點選任一單號展開明細後，系統自動帶出各品項的食材名稱、廠商名稱、門市名稱，顯示如「牛肉餅 100 公斤 → 台北信義店」。

### 4.2 異常情境 — 查詢不存在的單據 ID

系統管理員透過 API 傳入不存在的物流單 ID（如 id=9999），系統回傳錯誤訊息「物流查詢作業不存在」（錯誤碼 1004002000），不回傳空白資料。

### 4.3 操作情境 — 修改物流單明細數量

入庫完成後，小陳發現實際到貨數量有誤，需更正台北板橋店的牛肉餅到貨量從 80 公斤改為 95 公斤。她點選該物流單進入編輯模式，修改對應明細列後送出。系統以**全量取代**方式更新明細：刪除原有所有明細後，重新批次寫入新的明細清單（包含修正後的數量）。主單資訊同步更新。

### 4.4 操作情境 — 刪除錯誤物流單

誤建的物流單（如廠商代號打錯）需要刪除。小陳選擇該筆單據執行刪除，系統先確認單據存在，再於同一交易中刪除主表記錄與所有對應明細，確保不留孤立明細。

### 4.5 操作情境 — 匯出 Excel 對帳

月底廠商對帳時，小陳設定篩選條件（如 5 月份、特定配送廠商）後，點選「匯出 Excel」，系統將所有符合條件的物流單一次匯出為 `.xls` 檔（無分頁限制，全量取出），欄位包含：ID、單據編號、配送模式、配送廠商代號、出貨日、建立時間。

## 5. 操作流程

### 5.1 分頁查詢流程

```
[查詢請求進入]
  ├─ 權限檢查：是否具備「PDM 物流單查詢」
  ├─ 篩選條件組合（全部為精確比對，未填即忽略）
  │    ├─ 單據編號
  │    ├─ 配送模式
  │    ├─ 配送廠商代號
  │    ├─ 出貨日
  │    └─ 建立時間
  ├─ 依 ID 遞減排序
  └─ 回傳分頁結果
```

### 5.2 明細查詢流程（含關聯補查）

```
[明細查詢，傳入主單 ID]
  ├─ 權限檢查：是否具備「PDM 物流單查詢」
  ├─ 查明細主表（pdm_raw_material_logistics_dtl）
  ├─ LEFT JOIN 食材規格表 → 補食材名稱（by 品號）
  ├─ LEFT JOIN 廠商報價維護表 → 補廠商名稱（by 廠商代號，取第一筆）
  ├─ LEFT JOIN 門市資料表 → 補門市名稱（by 門市ID，取第一筆）
  ├─ LEFT JOIN 門市資料表 → 補區域名稱（by area = 區域ID，取第一筆）
  └─ 回傳明細列表（含所有補查名稱欄位）
```

### 5.3 建立流程

```
[建立物流單]
  ├─ 寫入主表（pdm_raw_material_logistics）
  ├─ 批次寫入明細（pdm_raw_material_logistics_dtl）
  │    └─ 每筆明細綁定主單 ID
  └─ 交易提交（整批成功或整批失敗）
```

### 5.4 修改流程

```
[修改物流單]
  ├─ 驗證主單存在 → 不存在則拋出「物流查詢作業不存在」
  ├─ 更新主表欄位
  ├─ 刪除原有全部明細
  ├─ 重建新明細清單（清空 id / 更新者 / 更新時間，避免衝突）
  └─ 交易提交
```

### 5.5 刪除流程

```
[刪除物流單]
  ├─ 驗證主單存在 → 不存在則拋出「物流查詢作業不存在」
  ├─ 軟刪除主表記錄
  ├─ 軟刪除對應全部明細
  └─ 交易提交
```

## 7. 商業邏輯

### 7.1 明細全量更新策略

修改物流單時，明細採**全量刪除後重建**策略，而非逐筆比對差異。業務含義為：每次修改視為重新定義整批明細清單。重建時，系統會清空每筆明細的主鍵與更新者資訊，由資料庫重新指派流水號，避免主鍵衝突或時間戳記凍結。

（來源：`RawMaterialLogisticsServiceImpl.java:111`）

### 7.2 單據存在性驗證

執行更新或刪除前，系統先以主單 ID 查詢主表；若記錄不存在（包含已被軟刪除的），回傳錯誤「物流查詢作業不存在」（錯誤碼 1004002000），操作中止。

（來源：`RawMaterialLogisticsServiceImpl.java:75`）

### 7.3 聯動刪除

刪除主單時，同一交易中一併軟刪除所有對應明細，不會留下孤立的明細記錄。

### 7.4 匯出無分頁上限

Excel 匯出時，系統強制將每頁筆數設為「不限制」，一次取出全部符合篩選條件的資料，直接寫入 `.xls` 檔。操作量大時應注意記憶體使用。

（來源：`RawMaterialLogisticsController.java:92`）
