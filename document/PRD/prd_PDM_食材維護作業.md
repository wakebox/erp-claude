## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 PDM（產品資料管理）的**食材建檔人員**，所屬部門通常是研發部或商品部。
公司每隔幾週就要上一個新菜單或新品項，背後牽動一連串新食材：例如「夏季限定芒果」「新開發的鬆露蛋黃醬」「換廠商後的新規格番茄」。我得把這些食材完整建檔到 ERP 裡：規格（廢棄率、加工方式、存儲方式、品號）、跟其他食材的相容性（相生相剋）、營養成分（讓行銷與法規揭示用）。建好之後送簽核，相關部門核可後才會歸檔生效，後續才能被 BOM、需求預測、採購使用。

### 1.2 我要做什麼

- **新增食材**：選分類、設規格、設相生相剋對象、設營養成分後送單，由系統自動產出單據編號（如 `IGR250520001`）並送 BPM 簽核。
- **修改食材**：在簽核完成「歸檔」前可以反覆編輯；歸檔後就鎖住不能再改（要改得走再次簽核或別的流程）。
- **追蹤審批進度**：在「待辦」頁看自己參與的食材單據目前的流程狀態（待處理、待簽核、待歸檔、已歸檔）。
- **刪除食材**：尚未生效或測試資料可刪；連同其規格、相生相剋、營養成分四張表一起清掉。
- **批次刪除**：年度盤點時把過時資料一次清理。
- **查詢**：照中類、小類、原型食材篩選查詢，看到中類／小類／原型食材的中文名稱以便閱讀。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 食材分類要按既定的「原則 → 大類 → 中類 → 小類 → 原型 → 食材」架構走 | 公司有完整編碼體系，不能讓人亂分類 |
| 一筆食材可以對應多個「規格」（同一種食材不同來源、不同加工、不同單位） | 例如「番茄」可能有「整顆冷藏」「切丁冷凍」兩種規格，採購單號不同 |
| 一筆食材可以記錄與多筆其他食材的「相生／相剋」關係 | 食物搭配安全與口味考量 |
| 一筆食材可以記錄多項營養成分（熱量、蛋白、糖、鈉…） | 法規揭示、菜單卡片標示 |
| 系統自動產出單據編號 | 避免人工編號錯誤 |

| 同一張表單能一次完成主檔＋三張子表 | 不希望分四個畫面建檔 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |

## 2. 功能目的

提供一套**主檔加三張子表、走簽核流程歸檔生效、編號自動產出**的食材管理機制。設計上：

1. **一筆主檔**統整一個業務概念的食材（例：「夏季芒果」）。
2. **三張子表**從三個角度補完該食材：規格（買進來、處理後長什麼樣、用什麼單位、品號為何）、相容性（與其他食材的相生相剋）、營養成分（每份含量、單位）。
3. **單據編號**靠「選單管理」之取號機制產出，前綴、日期格式、流水長度等規則設定在「食材維護作業」這條選單上。
4. **簽核流程**由「選單管理」之「啟用流程＋流程 Key」綁定，建立食材時自動觸發；歸檔後就鎖住不可改。

> **重要過渡背景**：本功能屬於 PDM 業務模組，與 OA 使用者整合過渡關聯較低。但「建立人員」「建立部門」欄位將來會以 OA 為準（推論：階段 5 後 mapping）。簽核流程的簽核者來源也仰賴使用者主檔。本 PRD 描述的是過渡期內、ERP 自管使用者主檔時的需求。

## 3. 業務邏輯背景

### 3.1 「主檔 + 三張子表」的關聯結構

```
食材主檔（pdm_ingredient）
  ├── 1:N 食材規格（pdm_ingredient_specs）           ← 一個食材可以有多種規格
  ├── 1:N 食材相生相剋（pdm_ingredient_compat）       ← 一個食材可以與多筆其他食材記錄相生相剋
  └── 1:N 食材營養成分（pdm_ingredient_nutritional_contents）  ← 一個食材可以有多種營養成分項目
```

四張表的關聯欄位都是「食材主檔 ID」。

### 3.2 編碼層級：原則 → 大類 → 中類 → 小類 → 原型 → 食材

| 層級 | 欄位 | 說明 |
|---|---|---|
| 原則代碼 | structure | 系統建檔時**寫死為 5**（推論：本功能對應的編碼原則） |
| 大類 | mainCategory | 系統建檔時**寫死為 5**（推論：本功能對應的食材大類） |
| 中類 | category | 使用者選；對應「編碼項目維護」中的食材中類 |
| 小類 | subcategory | 使用者選 |
| 原型食材 | source | 使用者選；對應「編碼項目維護」中的原型食材列舉 |
| 食材本身 | id | 本功能建出的記錄 |

> **註**：寫死的「原則代碼=5、大類=5」是程式行為（`IngredientServiceImpl.java:67-68、112-113`），意義對應到「編碼類別維護」「編碼原則維護」中的設定值，本 PRD 不展開。

### 3.3 流程狀態的四階段

| 狀態 | 觸發點 |
|---|---|
| 待處理 | 新增完成、流程尚未啟動 |
| 待簽核 | BPM 流程已啟動、待簽核中（推論：實際值可能不同，列入 §11） |
| 待歸檔 | 簽核已通過、尚未寫回主檔（推論） |
| 已歸檔 | 簽核通過後監聽器更新主檔的 `processStatus` 為「已歸檔」 |

### 3.4 「已歸檔保護」的精確條件

更新時的驗證是：

- 該食材**必須存在**，否則拋「食材主類資料不存在」
- 若該食材**未綁流程實例 ID** 且 流程狀態 = 「已歸檔」 → 拒絕，拋「食材維護表-食材主表已歸檔，不能再進行更新」

意即：

- 沒走流程而被人為改為「已歸檔」的單據 → 鎖住
- 走流程的單據（不論狀態為何，已綁 processInstanceId） → 不會被此驗證擋下，即可繼續修改

> **註**：此驗證邏輯較寬鬆——綁過流程的單據即使流程狀態為「已歸檔」仍可被改。是否符合預期需確認（列入 §11）。

### 3.5 單據編號的產出

新增時呼叫 `menuService.generateSignCode("食材維護作業")`：

- 系統由「選單管理」中「食材維護作業」這條選單取得：前綴、日期格式、流水長度、上次留存值
- 按「選單管理」§7.5 的取號邏輯產出新編號
- 寫到食材主檔的 `signCode` 欄位

未設定取號規則時，會拋「該菜單功能的單據編號配置信息不完整」（來源於選單管理模組）。

### 3.6 BPM 流程綁定

新增完成後，呼叫 `menuFlowProcessInstanceHelper.createProcessInstanceIfFlowOpen()`：

- 由「食材維護作業」這條選單讀取「啟用流程＋流程 Key」
- 若啟用：發起 Flowable 流程實例、把實例 ID 寫回主檔的 `processInstanceId`
- 若未啟用：略過，食材 `processInstanceId` 保持空

新增當下流程狀態必為「待處理」（程式預設值），由 BPM 流程自身的狀態事件後續驅動。

### 3.7 簽核結果監聽

`IngredientStatusListener` 監聽 BPM 流程狀態事件：

- 動態從「選單管理」拿到當前綁定的流程 Key 作為比對依據
- 業務鍵格式為 `{食材表單 path}:{食材主檔 ID}`，前綴對得起來才處理
- 流程結果為「通過」（APPROVE）→ 更新該食材的流程狀態為「已歸檔」
- 流程結果為「不通過」（REJECT）→ **程式碼被註解、未實作**，列入 §11
- 流程結果為「取消」（CANCEL）→ **程式碼被註解、未實作**，列入 §11

### 3.8 子表更新採全量替換（但「為 null」與「為空陣列」行為不同）

修改食材時，三張子表的處理：

| 子表 | 表單值為 null | 表單值為空陣列 | 表單值有內容 |
|---|---|---|---|
| 規格 | 不刪不重建 | **不刪不重建**（程式判斷 `!isEmpty()`，空陣列直接跳過） | 全刪 → 重建 |
| 相生相剋 | 不刪不重建 | **全刪、不重建**（程式只判斷 `!= null`） | 全刪 → 重建 |
| 營養成分 | 不刪不重建 | **全刪、不重建**（程式只判斷 `!= null`） | 全刪 → 重建 |

> **註**：規格子表與另外兩張子表的「空陣列」處理不一致。是 bug 或刻意行為待確認（列入 §11）。

### 3.9 刪除的順序

- **單筆刪除**：先刪主檔、再刪三張子表（程式順序）。刪除過程中若任一步回傳 < 0 才算失敗。
- **批量刪除**：先刪三張子表、再刪主檔（順序正確）。

> **註**：單筆刪除的順序與批量刪除不一致；單筆若主檔刪除失敗，可能殘留子表（推論，列入 §11）。

### 3.10 兩支查詢端點的差異

| 端點 | 用途 | 過濾邏輯 |
|---|---|---|
| `/page` 一般分頁 | 維運盤點 | 按中類、小類、原型食材 eq 過濾；joinDTO 補出中類／小類／原型食材中文名 |
| `/todo-page` 待辦分頁 | 簽核者作業視角 | 先呼叫 BPM 取得「當前登入者有任務」的流程實例 ID 清單，再用此清單過濾本表 |

待辦分頁若 BPM 回傳空清單，直接回傳空結果（不查資料庫）。

### 3.11 子表的常見欄位

**食材規格（一個食材有 N 筆）**：

- 食材名稱、廢棄率(%)、食材狀態、基礎加工、調味處理、存儲方式
- 流水碼一、流水碼二、品號（最終 SKU）
- 計數單位、前置作業時間、啟停狀態
- 單一規格、單一規格單位
- 關聯漢堡王原料 ID（與漢堡王中繼 API 對接用，推論）

**食材相生相剋（一個食材有 N 筆）**：

- 中類、小類、原型食材（指向另一個食材的分類定位）
- 說明、相生相剋（0=相生、1=相剋）

**食材營養成分（一個食材有 N 筆）**：

- 營養成分定義 ID（指向「營養成分定義維護」表）
- 每份含量、計量單位

## 4. 情境說明

### 4.1 正常流程 — 新增一個食材

PDM 建檔人員小張要建立「夏季限定芒果」：

**主檔**：
- 中類：水果類
- 小類：熱帶水果
- 原型食材：芒果
- 產季：5-9 月
- 建議烹煮方式：「直接食用 / 切丁」
- 前處理注意事項：「冷藏 4°C 以下保存，去皮去籽後使用」
- 主旨：「2026 夏季限定芒果引入」
- 建立部門：「商品部」

**規格 2 筆**：
- 規格 A：「整顆冷藏／廢棄率 35%／品號 IGR-MANGO-001／計數單位個」
- 規格 B：「切丁冷凍／廢棄率 0%／品號 IGR-MANGO-002／計數單位公克」

**相生相剋 1 筆**：
- 與「香蕉」相生／說明「果香互補」

**營養成分 3 筆**：
- 熱量 65 kcal / 每 100g
- 維生素 C 36 mg / 每 100g
- 糖 14 g / 每 100g

送出後系統：

1. 呼叫「食材維護作業」選單取號 → 產出 `IGR260520001`
2. 主檔欄位：原則=5、大類=5（系統寫死）、狀態=「待處理」
3. 寫入主檔
4. 寫入兩筆規格、一筆相生相剋、三筆營養成分（都帶上剛產生的主檔 ID）
5. 呼叫 BPM 判斷選單是否綁流程
   - 若已啟用 → 發起流程實例、把實例 ID 寫回主檔
   - 若未啟用 → 略過
6. 回傳新主檔 ID 給前端

### 4.2 正常流程 — 簽核通過後自動歸檔

小張的「夏季芒果」單據送出後，BPM 簽核流程：

- 商品主管核 → 通過
- 採購主管核 → 通過
- 財務主管核 → 通過
- 流程結束、結果 = APPROVE

監聽器接到事件：

1. 判斷流程定義 Key 與「食材維護作業」選單綁定的一致 → 是
2. 業務鍵 `ingredients:1024` 開頭符合 → 是
3. 結果 APPROVE → 取出主檔 ID `1024`，呼叫 `updateIngredient` 把流程狀態改為「已歸檔」

完成後該食材正式生效，可以被 BOM、需求預測、採購引用。

### 4.3 異常情境 — 簽核不通過

某筆食材送簽後被退回。BPM 事件結果 = REJECT。

監聽器走到 REJECT 分支 → **程式碼被註解、不處理**。
結果：該食材的流程狀態欄位**不會回到「待處理」**（保留為「待簽核」），表示退回流程未閉環。實際業務影響：使用者要如何重新編輯送單未定義（列入 §11）。

### 4.4 異常情境 — 已歸檔後想修改

小張幾天後想把「夏季芒果」的前處理注意事項補一段話。他在編輯頁修改後送出：

- 系統檢查食材存在 → 是
- 檢查「未綁流程實例 ID 且 已歸檔」→ 該食材有流程實例 ID（曾走過流程）→ **不擋**，允許修改

> **註**：依目前實作，曾走過流程的食材即使狀態為「已歸檔」也可修改。是否要進一步擋下需確認（列入 §11）。

### 4.5 異常情境 — 直接被人為設成「已歸檔」但沒走流程

假設某筆食材是測試資料，IT 直接寫資料庫把流程狀態改成「已歸檔」（流程實例 ID 仍為空）。任何人嘗試修改時：

- 系統檢查存在 → 是
- 檢查「未綁流程 + 已歸檔」→ 條件成立 → 拒絕「食材維護表-食材主表已歸檔，不能再進行更新」

### 4.6 業務規則分流情境 — 編輯時規格傳空陣列

小張想把某食材的所有規格全刪掉（暫時保留主檔，下個月再補新規格）。他在表單中把規格清單清空後送出：

- 系統處理規格時：判斷 `list != null && !isEmpty()` → 進入分支，但 `isEmpty()` 為 true、`!isEmpty()` 為 false → **整段邏輯被跳過**
- 結果：規格沒被刪、原本的兩筆規格仍在

> **註**：與表單預期不符。要清空規格目前要透過後端 API 直接操作、或在 UI 上採其他流程。列入 §11。

### 4.7 業務規則分流情境 — 編輯時相生相剋傳空陣列

同上情境，但對象是相生相剋子表：

- 系統處理：判斷 `list != null` → 進入分支
- 先刪除既有所有相生相剋
- 後續 `!isEmpty()` 為 false → 不重建
- 結果：原本的一筆相生相剋被清空

意即「相生相剋與營養成分」可以靠傳空陣列清空，「規格」不行。

### 4.8 待辦情境 — 簽核者的視角

商品主管阿凱開啟「食材待辦」頁，篩選流程實例狀態 = `待簽核`：

1. 前端呼叫 `/todo-page`
2. 服務呼叫 BPM 取「阿凱有任務」的流程實例 ID 清單 → 取得 3 筆
3. 用此清單到食材表分頁查 → 取出 3 筆食材主檔（含三類分類名稱）
4. 回傳給前端，阿凱可以挑選其中一筆進入簽核操作

### 4.9 多來源情境 — 「中類」「小類」「原型食材」實際是字典／編碼項目

新增食材時，使用者在前端看到的「中類」「小類」「原型食材」下拉，背後是 PDM 的「編碼項目維護」資料。三個欄位都是 `Long`，存的是該編碼項目的 ID。

> **註**：後端**未驗證**所選的中類、小類、原型食材是否存在、是否啟用、是否屬於合理的層級組合（中類 → 小類 → 原型）。打錯值會默默寫入。列入 §11。

### 4.10 異常情境 — 單筆刪除中途失敗

阿明刪除某筆食材 `1024`，但資料庫剛好有暫時性連線問題。`deleteById` 主檔已刪、要刪規格時失敗：

- 主檔已被軟刪除
- 規格、相生相剋、營養成分仍在
- 系統拋例外，但**沒有交易包覆**（程式中未見 `@Transactional`），導致資料殘留

> **註**：列入 §11。批次刪除同樣未見 `@Transactional`。

### 4.11 多筆建檔情境 — 批量刪除過時測試資料

PDM 年度盤點，小張勾選 12 筆過時測試食材按「批次刪除」：

- 系統先刪三張子表（IN 條件）
- 再刪主檔（deleteByIds）
- 12 筆食材完整移除

## 7. 商業邏輯

### 7.1 主檔欄位的「寫死值」

新增與修改時：

- 原則代碼 = 5
- 大類 = 5

這兩個值由業務 Service 強制覆寫表單值（即便前端帶來別的值也會被覆蓋）。對應到「編碼原則維護」「編碼項目維護」之中具有特定意義（推論為「PDM 食材」之代號），列入 §11 確認對應關係。

### 7.2 子表寫入皆綁主檔 ID

寫入三張子表前，逐筆迭代把主檔 ID 灌進每一筆子表記錄。即便前端有帶其他 ID，也會被覆蓋。

### 7.3 更新時「已歸檔保護」的寬鬆條件

如 §3.4 所述。實際擋到的只是「未綁流程 + 流程狀態為已歸檔」的特殊狀況。曾走過流程的單據即便狀態已歸檔，仍可修改。

### 7.4 子表更新「不一致」的處理邏輯

| 子表 | 程式判斷 | 後果 |
|---|---|---|
| 規格 | `if (list != null && !isEmpty())` | 空陣列不動 |
| 相生相剋 | `if (list != null)` | 空陣列會先 delete 然後不重建 |
| 營養成分 | `if (list != null)` | 同上 |

### 7.5 BPM 業務鍵格式

監聽器期待業務鍵為 `{表單路徑}:{主檔 ID}`。表單路徑由 `FormPathUniqueEnum.INGREDIENTS.getPath()` 提供（推論為「ingredients」或類似字串）。新增時呼叫 `createProcessInstanceIfFlowOpen` 傳的應該也是這個 path（程式邏輯一致則才會被監聽器接到）。

### 7.6 監聽器使用「動態流程定義 Key」

不同於早期硬編碼，本監聽器透過「選單管理」查當前綁定的流程 Key 作為比對基準。意即：

- IT 在選單管理改了流程綁定 → 監聽器下一次事件比對立刻生效
- 不需改程式與重啟

### 7.7 流程被退回的處理目前是「空的」

REJECT 與 CANCEL 分支在 listener 中**被註解掉**，原註解打算把狀態退回「待處理」，但並未實作。實務上若簽核被退回，食材的流程狀態欄位會留在「待簽核」（或當下任何 BPM 寫入的值），需要人為處理才能再次修改與送單（列入 §11）。

### 7.8 刪除無交易包覆

`deleteIngredientById` 與 `deleteByIds` 皆未見 `@Transactional`。中途失敗會殘留資料。

## 10. 功能範圍

### 10.1 包含的功能

- 食材主檔與三張子表的一次性新增、修改
- 自動取號（透過選單管理）
- 自動發起 BPM 簽核（若選單綁定流程）
- 簽核通過後自動歸檔（透過 BPM 事件監聽）
- 已歸檔保護（限定為「未綁流程 + 已歸檔」狀態）
- 一般分頁查詢（含分類名稱 JOIN）
- 待辦分頁查詢（含 BPM 流程實例篩選）
- 取得單一食材
- 單筆／批量刪除（含三張子表的連動清理）

### 10.2 預留但尚未實作

- **簽核退回／取消的處理**：REJECT 與 CANCEL 分支於監聽器中被註解、未實作
- **後端欄位驗證**：中類、小類、原型食材是否存在／啟用、層級組合是否合理皆未檢查
- **取得單一食材時的子表回填**：`/getOne` 僅回主檔，三張子表須前端另呼叫（推論：是否有對應端點未核對，列入 §11）
- **PageReqVO 上的「單據編號」「建立時間」欄位**：定義了但 Mapper 未取用
- **修改時子表處理的一致性**：規格子表與另外兩張的「空陣列」行為不一致
- **刪除交易**：未包覆 `@Transactional`，中途失敗會殘留

### 10.3 不包含

- 食材中類、食材小類、原型食材這些「分類」本身的維護（在「編碼項目維護」「編碼類別維護」）
- 編碼原則的維護（在「編碼原則維護」）
- 營養成分定義的維護（在「營養成分定義維護」）
- 單位定義的維護（在「單位定義維護」「單位轉換維護」）
- BPM 簽核流程定義本身（在 BPM 模組，由 Flowable 維護）
- 取號規則設定（在「選單管理」之單據編號設定區塊）
- 將食材引用為 BOM 之原料（在「單品維護作業」與需求預測模組）
