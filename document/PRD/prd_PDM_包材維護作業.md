## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 PDM 的**包材建檔人員**，所屬部門通常是商品部或物料部。
公司每樣販售品項都需要包裝：紙盒、紙袋、塑膠杯、吸管、餐墊紙、外帶手提袋。每種品項可能有多種規格（大杯、中杯、小杯），每種規格都要有自己的品號、單位、流水編號。我負責把這些包材完整建檔到 ERP，讓採購知道要備什麼、需求預測可以算每店要多少、店長備料時可以對照規格。

### 1.2 我要做什麼

- **新增包材主類**：填主檔（包材類別、主旨）＋ 子表（多個包材明細，每筆包含名稱、品號、單位、規格）。
- **修改包材主類**：在簽核完成「歸檔」前可反覆編輯；歸檔後鎖住。
- **追蹤審批進度**：在「待辦」頁看自己參與的包材單據目前的流程狀態。
- **刪除包材**：取消單一筆或批量刪除。
- **匯出**：依目前篩選匯出 Excel。
- **查詢**：依單據編號、包材類別、主旨、單據日期、流程狀態篩選。

## 2. 功能目的

提供一套**主檔加一張子表、走簽核流程歸檔生效、編號自動產出**的包材管理機制。設計上：

1. **包材主類主檔**只記錄基本分類（包材類別、主旨、流程狀態），結構極簡。
2. **包材明細子表**記錄每筆具體的包材規格：名稱、品號、流水編號、單位、單一規格、狀態。
3. **單據編號**靠「選單管理」之取號機制產出。
4. **簽核流程**由「選單管理」綁定；歸檔後鎖住不可改。

> **重要過渡背景**：本功能屬於 PDM 業務模組，與 OA 整合過渡關聯較低。建立人員、簽核者來源仰賴 ERP 使用者主檔；階段 5 後改以 OA 為準（推論）。

> **本模組設計上的特點**：與食材維護作業、單品維護作業相比，包材的主檔欄位非常少（只有「包材類別、主旨、流程狀態」），實際業務資訊都集中在子表。**且本模組沒有 BPM 監聽器**（程式中找不到對應的 listener）。

## 3. 業務邏輯背景

### 3.1 「主檔 + 一張子表」的關聯結構

```
包材主檔（pdm_packing_materials）
  └── 1:N 包材明細子表（pdm_packing_materials_dtl）  ← 每筆包材的具體規格
```

子表的關聯欄位是「包材主檔 ID（packingMaterialsId）」。

### 3.2 主檔欄位極簡

- ID
- 單據編號（系統取號）
- 包材類別（指向編碼項目維護）
- 主旨（自由文字，簽核標題用）
- 流程狀態
- 流程實例 ID

僅此而已。實際業務資訊都在子表。

### 3.3 子表欄位

每筆包材明細記錄：

- 包材類別 ID（指向編碼項目維護）
- 包材名稱
- 流水編號（6 碼，業務上的識別碼）
- 品號
- 計數單位 ID（指向單位定義）
- 單一規格、單一規格單位
- 狀態（字典 status 的值）

> **註**：子表也有「包材類別」欄位，與主檔的「包材類別」可能重複或不同層級。需確認語意（列入 §11）。

### 3.4 流程狀態

同其他模組：「待處理 / 待簽核 / 待歸檔 / 已歸檔」。

### 3.5 「已歸檔保護」

更新時的驗證：

- 該包材主類必須存在
- 若**未綁流程實例 ID** 且 流程狀態 = 「已歸檔」 → 拒絕，拋「包材維護表已歸檔，不能再進行更新」

與食材、單品同樣的寬鬆條件：曾走過流程的單據即便已歸檔仍可被修改（列入 §11）。

### 3.6 單據編號的產出

新增時呼叫 `menuService.generateSignCode("包材維護作業")`，由選單管理取號。

### 3.7 BPM 流程綁定

新增時呼叫 `menuFlowProcessInstanceHelper.createProcessInstanceIfFlowOpen()`：

- 由「包材維護作業」選單讀取「啟用流程＋流程 Key」
- 若啟用：發起 Flowable 流程實例、寫回 `processInstanceId`
- 業務鍵格式：`packagingMaterials:{主檔 ID}`

### 3.8 **本模組沒有 BPM 監聽器**

從程式碼搜尋：**找不到** `PackingMaterials*Listener.java`。意即：

- BPM 流程簽核完成（APPROVE）後，**沒有任何程式碼**會自動把流程狀態改為「已歸檔」
- 流程退回、取消同樣不會處理
- 流程狀態的變更必須依賴**其他機制**：例如人工修改、或在 BPM 流程設定中的「結束事件」呼叫某 API（推論：本模組未啟動簽核流程？或流程是另一種架構？）

這與食材、單品兩模組的設計明顯不同。列入 §11。

### 3.9 子表更新採「先全刪、再批量重建」

修改時：

- 若清單**非 null**（包含空陣列） → 進入分支
- 依 packingMaterialsId 刪除既有
- 每筆 `setId(null)、setUpdater(null)、setUpdateTime(null)`
- 批量寫入

> **註**：判斷條件用 `!Objects.isNull()`，與單品（`isNotEmpty`）不同。意即：

- 空陣列 → 先刪除，但寫入時 forEach 不執行 → 等同清空
- null → 不動

意即包材的子表**可以**透過傳空陣列清空，**不**像單品的子表卡在「非空才動」的限制（列入 §11 比較）。

### 3.10 刪除採「主檔先、子表後」

- 單筆刪除：先刪主檔、再刪子表
- 批量刪除：迭代呼叫單筆刪除

> **註**：批量刪除迭代呼叫單筆刪除，**包在同一個 `@Transactional`** 中，任一筆失敗整批回滾。比食材模組（無交易）更可靠。

### 3.11 提供「全部子表」端點

`/packing-materials-dtl/list-all-packing-materials` 一次取得**全系統所有包材子表**清單。給下游模組（單品配方、需求預測、報價）作選項用。

> **註**：此端點未過濾「已歸檔」狀態，可能撈到簽核中或未啟用狀態的包材。列入 §11。

## 4. 情境說明

### 4.1 正常流程 — 新增一筆包材主類

PDM 建檔人員小華要建立「外帶杯系列」：

**主檔**：
- 包材類別：外帶餐具類
- 主旨：「2026 年新外帶杯規格」

**子表 3 筆**：
- 大杯：品號 `PKG-CUP-L`、流水編號 `100001`、單位「個」、單一規格 700ml
- 中杯：品號 `PKG-CUP-M`、流水編號 `100002`、單位「個」、單一規格 500ml
- 小杯：品號 `PKG-CUP-S`、流水編號 `100003`、單位「個」、單一規格 350ml

送出後系統：

1. 取號：「包材維護作業」→ `PKG260520001`
2. 流程狀態 = 「待處理」
3. 寫入主檔
4. 寫入 3 筆子表
5. 呼叫 BPM 判斷是否啟用流程 → 若啟用發起流程、寫回 `processInstanceId`
6. 回傳新主檔 ID

### 4.2 異常情境 — 簽核通過後流程狀態無人更新

小華送出後 BPM 流程簽核通過。但本模組**沒有 listener**：

- 流程結束事件發出 APPROVE，無監聽器接收
- 主檔的 `processStatus` 保留「待處理」（或當下任何 BPM 寫入的值）
- 包材永遠不會自動變成「已歸檔」狀態

這代表「已歸檔保護」基本上**永遠不會觸發**（除非有人手動把 processStatus 改成「已歸檔」且 processInstanceId 同時為空，這通常是非預期情境）。實質效果：

- 已綁流程實例 ID 的包材：永遠可改
- 未綁流程實例 ID 但被人為改 processStatus = 已歸檔的：擋下

可能的解釋：

- BPM 流程在「包材維護作業」選單上未啟用 → 沒有流程要走，processStatus 由人為設定
- 或本模組仍在開發中，listener 待補
- 或業務流程設計就是「不需要簽核」

需確認。列入 §11。

### 4.3 異常情境 — 已歸檔後想修改

如上所述，「曾走過流程」的包材即便流程狀態為「已歸檔」也可改（與食材、單品同議題）。

### 4.4 業務規則分流情境 — 編輯時子表傳空陣列

小華想清空某包材主類的所有明細：傳空陣列子表清單送出：

- 系統判斷 `!Objects.isNull()` → true → 進入分支
- 先刪除既有所有子表
- `list.forEach()` 因清單為空不執行
- 結果：原本 3 筆子表被清空、無新記錄

與單品的「空陣列不動」行為不同，本模組支援透過空陣列清空子表。

### 4.5 異常情境 — 批量刪除中途失敗

小華批量刪除 3 筆包材，其中第 2 筆不存在：

- 第 1 筆：刪除成功
- 第 2 筆：拋「包材維護表資料不存在」
- 整個交易回滾 → 第 1 筆已刪的也被回滾
- 結果：3 筆全部保留

`@Transactional` 包覆，任一筆失敗整批回滾。

### 4.6 異常情境 — 子表「包材類別」與主檔不同

小華建檔時主檔填「外帶餐具類」，但子表的「大杯」明細手滑填了「內用餐具類」。系統**未驗證**兩者一致，照常寫入。結果：

- 主檔分類：外帶餐具類
- 子表「大杯」明細：內用餐具類

兩者不一致，後續查詢時可能造成混亂。列入 §11。

### 4.7 異常情境 — 未驗證的子表單位 ID

子表 `unitId` 指向「單位定義」，但後端**未驗證**該單位 ID 是否存在、是否啟用。填錯不會擋下。列入 §11。

### 4.8 多來源情境 — 下游模組要全部子表清單

需求預測模組要列出所有可用包材作為選項。呼叫 `/packing-materials-dtl/list-all-packing-materials`：

- 取得全系統所有 `pdm_packing_materials_dtl` 記錄
- 不過濾包材主類的流程狀態
- 不過濾子表的「狀態」欄位

下游可能撈到「未歸檔」的包材或「停用」的子表記錄。列入 §11。

## 5. 操作流程

### 5.1 新增包材主類

```
[填寫主檔 + 子表]
  │
  ▼
[權限檢查：是否具備「PDM-包材-新增」]
  │
  ▼
[呼叫選單取號：「包材維護作業」→ signCode]
  │
  ▼
[組裝主檔]
  ├─ 流程狀態 = 「待處理」
  └─ 其他依表單值
  │
  ▼
[寫入主檔]
  │
  ▼
[子表（若非空）]
  ├─ 每筆綁主檔 ID
  └─ 批量寫入
  │
  ▼
[呼叫 BPM：判斷選單流程是否啟用]
  ├─ 啟用 → 發起流程實例 → 寫回 processInstanceId
  └─ 未啟用 → 略過
  │
  ▼
[回傳新主檔 ID]
```

### 5.2 修改包材主類

```
[編輯表單，送出]
  │
  ▼
[權限檢查：是否具備「PDM-包材-修改」]
  │
  ▼
[驗證存在]
  │
  ▼
[已歸檔保護：未綁流程 + 已歸檔 → 拒絕]
  │
  ▼
[更新主檔]
  │
  ▼
[子表處理]
  ├─ 為 null → 不動
  ├─ 為空陣列 → 全刪不重建（清空）
  └─ 有內容 → 全刪、setId(null)、批量寫入

```
[點選刪除]
  │
  ▼

  │

  ▼

  │
  ▼

  │

## 7. 商業邏輯

### 7.1 新增、修改、刪除（含批量）皆有 `@Transactional`

整段業務邏輯包在交易內，任一步失敗整批回滾。比食材模組（無交易）穩定。

### 7.2 子表更新的 ID 重新分配

同單品模組：每次更新都 `setId(null)`，導致子表 ID 在每次更新後變動。若有外部引用會斷掉（列入 §11）。

### 7.3 子表處理判斷條件與單品不同

| 模組 | 判斷 | 效果 |
|---|---|---|
| 單品 | `isNotEmpty(list)` | 空陣列不動 |
| 包材 | `!isNull(list)` | 空陣列也會先刪除既有，等同清空 |

### 7.4 主檔包材類別與子表包材類別並存

兩個欄位都叫「包材類別」，但分屬主檔與子表。語意是否相同、層級是否一致、是否需驗證一致性目前都未定義。列入 §11。

### 7.5 已歸檔保護的實質效果

由於本模組**無 BPM 監聽器**：

- processStatus 不會自動變「已歸檔」
- 「未綁流程 + 已歸檔」這個條件實際上很難自然發生
- 已歸檔保護幾乎是 dead code

### 7.6 刪除前不檢查是否被下游引用

刪除一筆包材主類時，**不檢查**是否有單品配方、需求預測等下游已引用該包材的子表 ID。直接刪除會留下孤兒引用。列入 §11。

## 10. 功能範圍

### 10.1 包含的功能

- 包材主類與包材明細子表的一次性新增、修改、刪除
- 自動取號
- 自動發起 BPM 簽核（若選單啟用）
- 已歸檔保護（同其他模組之寬鬆條件，且因無 listener 實質效果有限）
- 一般分頁查詢
- 待辦分頁查詢
- 取得單一主檔
- 取得單一主檔之子表清單
- 取得全部子表清單
- 匯出 Excel
- 單筆刪除、批量刪除（皆含子表連動）
- 完整的 `@Transactional` 包覆

### 10.2 預留但尚未實作

- **BPM 監聽器**：無 listener 接收簽核結果事件，自動歸檔無法觸發
- **後端欄位驗證**：包材類別、單位 ID 等皆未驗證存在性／啟用狀態
- **主檔與子表的包材類別一致性檢查**：未檢查
- **下游引用檢查**：刪除前不檢查是否被下游引用
- **「全部子表」端點的過濾**：未按啟用狀態或已歸檔過濾

### 10.3 不包含

- 包材類別本身的維護（在「編碼項目維護」）
- 編碼類別／原則的維護（在各自的維護作業）
- 單位定義的維護（在「單位定義維護」）
- BPM 簽核流程定義本身
- 取號規則設定（在「選單管理」）
- 將包材引用為單品配方一部分（在「單品維護作業」）
