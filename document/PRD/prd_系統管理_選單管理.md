## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員（資訊／IT 角色）。
公司剛把多個業務模組（PDM、需求預測、採購、倉儲、物流、店長功能）一次擠進 ERP，每個模組底下又有幾十個畫面與按鈕。誰能看哪個畫面、能不能按下「刪除」「匯出」「審批」按鈕，全靠「選單管理」這個地方統一定義。

### 1.2 我要做什麼

- **每次上線新功能**：當開發團隊新增了一支畫面（例如「廠商報價維護」），我要在選單樹裡幫它建一條對應的「選單」紀錄，掛在正確的父目錄底下、給它一個權限代號、設定圖示與顯示順序，這樣前端才會在側邊欄看到它。
- **每次上線新按鈕**：開發團隊在某張畫面新增一顆「核准」按鈕，我要在該畫面底下再建一筆「按鈕」型節點，並指派專屬權限代號（如 `pmm:quote:approve`），讓角色管理可以勾選授權。
- **每次模組調整**：把某個畫面從一個目錄搬到另一個目錄、改顯示順序、改圖示、暫時停用某個尚未準備好的功能。
- **每次有單據型功能**：採購單、請購單、驗收單這類需要單據編號的功能，我要在對應選單上設定「單據編號規則」（前綴、日期格式、流水長度），讓系統能自動產出 `RM251029001` 這樣的編號。
- **每次要綁簽核流程**：某個單據需要走簽核（例如「請購單」），我要在該選單上勾「啟用流程」並選擇對應的流程 Key（例如 `paq`），業務系統才會在送單時觸發簽核。
- **每次有人問「為什麼我看不到 XX 畫面」**：先看該選單是否啟用、再去角色管理確認其角色有沒有勾選那個選單。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一個地方能看見整套系統所有畫面與按鈕的樹狀結構 | 模組多、頁面多，沒有總覽很容易遺漏或重複命名 |
| 新建選單時要擋下常見錯誤（亂選父節點、名稱撞名、組件撞名） | 一旦資料壞掉很難追查，前端會出現「點不開」或「畫面跑錯位置」的怪 bug |
| 刪除時不能刪掉還有子節點的目錄 | 否則殘留的子節點變成孤兒，前端側邊欄會錯亂 |
| 設「停用」之後該選單本身及其所有下層都不該出現在使用者眼前 | 半成品功能要能隨時關閉，又不想真的刪掉 |
| 單據編號規則要能設定，並由系統自動遞增 | 採購、請購、驗收等流程的單據編號不能讓業務人員手填 |
| 哪些選單已綁簽核流程要看得到 | 排查「為什麼這張單沒進審批」時的第一線資訊 |
| 修改後新權限要立刻生效，不必重啟系統 | IT 維運成本 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 新增／修改／刪除選單 | 隨業務上線維護選單樹 |
| 列出全部選單（含模糊查名稱、依狀態篩選） | 維運盤點與快速找節點 |
| 取得單一選單明細 | 編輯前的回填 |
| 切換選單啟用／停用 | 隨時關閉半成品功能 |
| 設定選單的簽核流程綁定 | 把單據型功能與 BPM 簽核 Key 關聯 |
| 依「流程 Path」反查選單 | 簽核引擎需要由 path 找回對應選單以判斷「該功能是否要走簽核」 |
| 依選單產生下一個單據編號 | 採購、請購等流程取號 |
| 提供精簡選單清單（給「角色配選單」勾選用） | 角色管理頁需要可勾選的選單樹 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 選單管理 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 使用者管理、角色管理、部門管理、字典管理、操作日誌、登入日誌 |

## 2. 功能目的

提供一個**集中、樹狀、可細到按鈕粒度**的權限與導覽資源定義中心。一筆「選單」紀錄同時負責三件事：

1. **導覽**：決定前端側邊欄／路由要不要出現、出現在哪、用什麼圖示。
2. **授權**：每個選單／按鈕節點都帶一個「權限代號」，搭配角色管理把代號掛到角色上，後端 API 即可依此驗證；前端按鈕也以此決定要不要顯示。
3. **業務元資料**：對於單據型功能，額外存「單據編號前綴／日期格式／流水長度」，以及「是否綁簽核、綁哪個流程」。

> **重要過渡背景**：經系統整合決策（見 `erp-claude` 記憶 `project_user_integration_decision.md`，2026-05-16），漢堡王整體 IT 走向以 Laravel OA 為使用者單一事實來源、ERP 不再自簽 JWT。**選單／權限**為 ERP 過渡期保留之 RBAC 機制，長期規劃中將在搬遷階段 5 後逐步交棒給 OA 統一控管（角色—選單對應仍可能保留在 ERP 端）。本 PRD 描述的是過渡期內、ERP 仍自管選單與權限定義時的需求。

## 3. 業務邏輯背景

### 3.1 選單分三種「類型」，各自有不同義務

| 類型 | 用途 | 必填欄位（除共同欄位外） | 不需要的欄位 |
|---|---|---|---|
| **目錄** | 樹狀分組（例如「系統管理」「PMM 採購」） | 圖示、路由 | 組件、權限代號 |
| **選單** | 真正對應一支畫面（例如「廠商資料維護」） | 路由、組件路徑、組件名（若要快取） | 通常不需要權限代號（在此節點僅控「能否進入該頁」） |
| **按鈕** | 某張畫面內的細粒度操作（例如「核准」「匯出」） | 權限代號 | 路由、組件、圖示、頁面網址（系統會自動清空） |

按鈕在系統內**沒有實體頁面**，純粹是給「角色管理」勾選用的權限掛勾。新增或修改一筆按鈕型選單時，系統會自動把它的路由、組件路徑、組件名、圖示這幾個欄位**強制清空**。

### 3.2 父子關係的硬性約束

- 不能把自己設成自己的父節點。
- 父節點必須存在。
- 父節點的類型必須是「目錄」或「選單」；不能把節點掛在「按鈕」底下。
- 沒有父節點時，`父選單 ID` 設為根節點代號 0。

### 3.3 同層唯一性

- **選單名稱**：同一個父節點底下，子節點名稱不可重複。不同父節點下可以同名（例如「總部—請購單」與「門市—請購單」是允許的）。
- **組件名稱**：全系統唯一（不論在哪個父節點底下）。組件名稱對應前端 Vue 元件名，撞名會導致前端路由 keep-alive 機制壞掉。

### 3.4 狀態判斷會沿父鏈往上看

選單的「啟用／停用」不只看自己。對於前端要不要顯示，邏輯是：

1. 自己若停用 → 不顯示
2. 父節點若停用 → 即使自己啟用也不顯示
3. 一路檢查到根節點都啟用 → 顯示

此邏輯只作用於「給角色勾選的精簡清單」端點，目的是不讓角色管理頁面勾到一個其實前端不會出現的選單。維運用的「全部清單」則不過濾，方便管理員看到完整樹況。

### 3.5 單據編號的生成規則

某些選單（採購、請購、驗收等）背後對應的是單據型功能，這類選單會額外設定：

- **前綴**：例如 `RM`
- **日期格式**：例如 `yyMMdd`
- **流水長度**：例如 3（產出 `001`、`002`…）

當業務模組（如採購單）要新增單據時，會由功能名稱反查這筆選單、再呼叫產號邏輯：

- 若上次取的單據編號（最後一次留存值）的「前綴 + 日期段 + 流水段」總長正確且前綴對得起來：
  - 取出其中的日期段，若與今日相同 → 流水段 +1
  - 若日期段不同（已跨日） → 流水歸 1
- 否則（最後一次留存值是空的、或規則與上次留存不符代表規則已變更）→ 流水歸 1

新編號組好後**即時回寫**到該選單的「最後一次留存值」欄位，作為下一次取號的依據。

### 3.6 簽核流程綁定

每筆選單可選擇是否「啟用流程」並指定一個「流程 Key」。當業務模組準備建立單據時，會以該模組對應選單的這兩個欄位來決定要不要呼叫 Flowable 開啟簽核實例。本功能不負責執行簽核，只負責**定義**：「這個畫面對應的單據要走哪一個簽核流程」。

### 3.7 兩種快取，由本功能負責清

- **權限代號 → 選單編號清單**：權限驗證的高頻查詢，新增／修改／刪除選單都會清理對應快取。權限代號新建或修改時若會涉及新舊兩個權限代號，做法是直接清掉整個快取空間，簡單但完整。
- **路由 path → 選單**：簽核引擎以 path 反查選單時走的快取，由「更新流程資訊」入口維護（更新流程設定時主動 evict 對應 path）。

### 3.8 多租戶過濾

當系統開啟多租戶模式時，**精簡清單**端點會額外按「該租戶套餐有開通的選單」過濾。一般的全部清單不過濾。

## 4. 情境說明

### 4.1 正常流程 — 新增一個畫面選單

IT 阿明收到通知：「採購模組新增了一支廠商報價維護畫面，請建立選單」。阿明操作：

- 在樹中找到父目錄「採購管理」，點「新增子選單」
- 選單名稱：`廠商報價維護`
- 類型：選單
- 顯示順序：`30`
- 路由地址：`/pmm/vendor-quote`
- 組件路徑：`pmm/vendorQuote/index`
- 組件名：`VendorQuote`（讓前端可以做頁面快取）
- 圖示：`vendor-quote.svg`
- 狀態：啟用
- 是否可見：是
- 是否快取：是
- 是否總是顯示：是

送出後系統檢查：父節點「採購管理」存在且為目錄類型 → 通過；同層下沒有別的子節點叫「廠商報價維護」 → 通過；組件名「VendorQuote」全系統沒撞名 → 通過。建立後新節點立刻可以被角色管理掛載，相關使用者重新整理頁面後即可在側邊欄看到。

### 4.2 正常流程 — 為某張畫面新增一顆按鈕

「廠商報價維護」需要一顆「核准」按鈕。阿明在該選單底下「新增子節點」：

- 名稱：`核准`
- 類型：按鈕
- 權限代號：`pmm:vendor-quote:approve`
- 顯示順序：`10`

送出時系統依「按鈕類型」自動把組件路徑、組件名、圖示、路由全部清空，只保留名稱與權限代號。後續到「角色管理」可勾選此項，授權者才會看到並能按下「核准」。

### 4.3 異常情境 — 想刪除一個仍有子節點的目錄

阿明想把「採購管理」整個目錄刪掉，但底下還掛著「廠商資料維護」「請購單管理」等子節點。系統回應「存在子菜单，无法删除」並拒絕刪除。阿明改成先逐一刪子節點才能刪父目錄。

### 4.4 異常情境 — 把選單掛在「按鈕」下面

阿明手滑，把「子選單」的父節點選成了一顆按鈕。系統回應「父菜单的类型必须是目录或者菜单」並擋下。按鈕不可作為父節點。

### 4.5 異常情境 — 名稱撞名

阿明在「採購管理」底下新增第二個叫「廠商報價維護」的選單。系統回應「已经存在该名字的菜单」並拒絕。同層不可重名。

### 4.6 異常情境 — 組件名撞名

阿明在「PDM」底下新增一個叫「食材維護」的選單，組件名填 `VendorQuote`。系統回應「已经存在该组件名的菜单」並拒絕（即使中文名不一樣，組件名全系統唯一）。

### 4.7 業務規則分流情境 — 按鈕類型自動清空欄位

阿明從「按鈕」改回「選單」時忘了清空原先按鈕用不到的欄位，順手在按鈕表單填了一個 path。送出後系統儲存前主動把按鈕的 component、componentName、icon、path 全部置空，最終存進去的這些欄位都是空字串。

### 4.8 業務規則分流情境 — 停用後子節點不會出現在角色選項

阿明把「物流管理」整個目錄停用。當角色管理員小李去設定「物流主管」這個角色的選單權限時，發現連同「物流規劃查詢作業」「物流配送查詢作業」這些原本啟用的子節點也都消失在勾選清單裡。理由：父節點停用會沿樹遮蔽所有後代。

### 4.9 自動計算情境 — 取下一個單據編號（同日）

採購模組今早第一張請購單取號：

- 對應選單的設定為：前綴 `RM`、日期格式 `yyMMdd`、流水長度 3、最後一次留存值 `RM26051910`
- 今天是 2026-05-19
- 系統檢查：最後一次留存值總長 = 2 + 6 + 3 = 11 ✓、前綴對得起來 ✓ → 規則未變
- 取出日期段 `260519`，與今日 `260519` 相同 → 流水段 `010` +1 = `011`
- 產出 `RM260519011`，並寫回最後一次留存值

### 4.10 自動計算情境 — 取下一個單據編號（跨日）

採購模組今早第一張：

- 對應選單的設定為：前綴 `RM`、日期格式 `yyMMdd`、流水長度 3、最後一次留存值 `RM260518023`
- 今天是 2026-05-19
- 系統檢查：總長正確、前綴正確 → 規則未變
- 取出日期段 `260518` ≠ 今日 `260519` → 流水歸 1
- 產出 `RM260519001`

### 4.11 自動計算情境 — 取單據編號時設定不完整

某模組剛建好選單但忘了設前綴、日期格式或流水長度。第一次取號時系統回應「該菜單功能的單據編號配置信息不完整,請檢查該菜單的設置」並拒絕產號。

### 4.12 簽核綁定情境 — 啟用請購單的簽核流程

採購流程確認「請購單必須走兩級簽核」，BPM 工程師已在 Flowable 設好流程定義 Key `purchase-apply-quote`（即 `paq`）。阿明開啟「請購單管理」選單的「流程綁定」設定：

- 是否啟用流程：是
- 流程 Key：`paq`

送出後系統更新該選單，並順手把該 path 對應的快取清掉。當業務人員下一次新增請購單時，業務模組會由其對應選單（依 path 反查）讀到「啟用流程 = 是、流程 Key = paq」，據此觸發 Flowable 發起簽核實例。

### 4.13 異常情境 — 修改一個不存在的選單編號

阿明拿到一個來源不明的選單編號 `99999` 並嘗試更新。系統回應「菜单不存在」並拒絕。

## 5. 操作流程

### 5.1 新增選單

```
[填寫選單表單]
  │
  ▼
[權限檢查：是否具備「系統-選單-新增」]
  │
  ▼
[父節點驗證]
  ├─ 父節點 = 根（0）→ 跳過後續父節點檢查
  ├─ 父節點 ≠ 自己（新增時 N/A）
  ├─ 父節點必須存在
  └─ 父節點類型必須為「目錄」或「選單」
  │
  ▼
[同層唯一性：相同父節點下，名稱不可重複]
  │
  ▼
[組件名稱唯一性：全系統不可重複（若有填）]
  │
  ▼
[類型微調：若為「按鈕」→ 清空組件、組件名、圖示、路由]
  │
  ▼
[寫入選單主檔]
  │
  ▼
[清掉「該權限代號 → 選單編號清單」之快取]
  │
  ▼
[回傳新選單編號]
```

### 5.2 修改選單

```
[編輯選單表單，送出]
  │
  ▼
[權限檢查：是否具備「系統-選單-修改」]
  │
  ▼
[該選單必須存在]
  │
  ▼
[父節點驗證]
```

## 7. 商業邏輯

### 7.1 新增／修改的前置檢核

新增時 3 關，修改時 4 關（多一關「目標選單必須存在」）：

1. **目標選單必須存在**（僅修改）
2. **父節點合法**：不可自指、必須存在、必須是目錄或選單
3. **同層名稱唯一**
4. **組件名稱全系統唯一**（若有填）

不符任一關即整體拒絕，已驗證過的欄位不會落地。

### 7.2 按鈕類型強制清欄位

新增與修改時，只要類型 = 按鈕，系統會在寫入前主動把以下欄位清為空字串：

- 組件路徑
- 組件名稱
- 圖示
- 路由地址

意義是：即便表單帶值來，存進資料庫的就是空字串。理由是按鈕只是權限掛勾、不對應頁面。

### 7.3 父節點停用會遮蔽整個子樹（僅限「精簡清單」端點）

「精簡清單」用遞迴方式計算每個節點實際是否可見：

1. 自己若停用 → 直接判定為不可見
2. 自己啟用但父節點停用 → 仍為不可見
3. 一直回到根節點都啟用 → 才判為可見

中途有快取（同一張表內的暫存集合），同一筆計算過的不可見節點不再重算。

維運用的「全部清單」**不**做此過濾，目的是讓管理員能看到完整樹況。

### 7.4 刪除的連動處理

刪除採物理刪除（直接 `deleteById`，由框架的軟刪除欄位機制處理），並做兩件事：

1. **先擋**：若該選單仍有子節點，整體拒絕，不會物理刪除。
2. **連動清角色**：呼叫權限服務的「處理選單刪除」鉤子，把所有授予此選單之角色關聯清掉。

整個刪除動作包在資料庫交易內，任一步失敗整批回滾。

### 7.5 取號的關鍵判斷

「規則是否變更」決定流水歸 1 還是 +1：

- **留存值總長 ≠ 前綴長 + 日期格式長 + 流水長度** → 視為規則已變、流水歸 1
- **留存值不以設定的前綴開頭** → 視為規則已變、流水歸 1
- 兩條都通過 → 規則未變，再比日期段：
  - 日期段 = 今日 → 流水 +1
  - 日期段 ≠ 今日 → 流水歸 1

組合完之後**即時**寫回「最後一次留存值」。這個寫入沒有額外的並發鎖；同一個功能在同一秒內極短時間取兩次號，理論上存在兩次取到同一個流水的風險（推論，列入 §11）。

### 7.6 快取的維護

| 觸發點 | 快取空間 | 動作 |
|---|---|---|
| 新增選單 | 權限代號 → 選單編號清單 | 依新增的權限代號 evict 該鍵 |
| 修改選單 | 權限代號 → 選單編號清單 | 整批 evict（不知道修改後該 ID 對應哪個權限代號，最簡單做法清光） |
| 刪除選單 | 權限代號 → 選單編號清單 | 整批 evict |
| 更新流程綁定 | path → 選單 | evict 該選單原本的 path 鍵 |

> 「整批清掉」會在大流量情境下對權限驗證造成短暫延遲，因為下一個請求要重新去資料庫查。列入 §11 待確認。

### 7.7 多租戶過濾（僅限「精簡清單」端點）

當系統啟用多租戶時，精簡清單會再過一道過濾：移除「該租戶套餐」沒開通的選單。維運用的「全部清單」不做此過濾。

### 7.8 流程綁定更新採「傳了才更新」策略

`MenuFlowUpdateVO` 的兩個欄位都是選填，伺服端用 `null` 判斷是否需要寫入。這代表：

- 只想改「是否啟用流程」，不傳流程 Key → 流程 Key 保留原值
- 想把流程 Key 清空，要傳空字串 `""`（不是 null）

> 此語意對前端可能造成「想清空但結果沒清掉」的誤解。列入 §11 待確認。

## 10. 功能範圍

### 10.1 包含的功能

- 選單（含目錄／選單／按鈕三種類型）之新增、修改、刪除
- 全部清單查詢（支援名稱模糊、狀態篩選；維運用）

- 選單刪除時自動連動清理「角色—選單」關聯

### 10.2 預留但尚未實作

- **權限代號修改時的快取清理粒度**：修改時直接整批清空，未針對「舊代號 / 新代號」精準 evict。技術可行性已有，但目前以「簡單可靠」為優先（來源：`MenuServiceImpl.java:76-77` 註解）。

### 10.3 不包含

- 角色掛載哪些選單（在「角色管理」之「設定角色選單權限」功能）
- 簽核流程本身的定義（流程 Key 對應的 BPM 流程在 Flowable 維護）
- 前端側邊欄根據使用者角色組合可見選單樹的呈現邏輯（前端責任）
- 取號被使用後的「單據編號重用／作廢」機制（業務模組責任）
