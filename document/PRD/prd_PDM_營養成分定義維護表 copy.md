## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 PDM 的**營養資訊維護人員**，所屬部門通常是商品部或品保。
法規要求菜單卡片要揭示每樣品項的「熱量、蛋白質、脂肪、碳水化合物、鈉…」等營養成分。每個成分有自己的單位（kcal、g、mg…）、建議攝取量、英文名（給國際版菜單用）。我負責把這些成分定義一筆筆建好，下游的「食材維護作業」「單品維護作業」才能在它們各自的「營養成分子表」中引用這些定義。

### 1.2 我要做什麼

- **新增營養成分定義**：給一個項目名稱（中文）、英文名、單位、平均每餐建議攝取量、預設成分標記、顯示排序。
- **修改既有定義**：調整名稱、單位、排序。
- **刪除（單筆／批量）**：清理過時項目。
- **查詢**：依名稱、英文名、預設成分篩選。
- **匯出**：盤點佐證。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一個營養成分要有中文與英文名 | 國際版菜單 |
| 要有單位 | 蛋白質用 g、鈉用 mg、熱量用 kcal |
| 可以標「預設成分」 | 建檔食材時，預設成分自動帶入 |
| 顯示排序 | 菜單卡片上的呈現順序固定 |
| 編輯時要知道這筆定義有沒有被下游用過 | 已被引用的不應隨便刪 |

| 功能 | 解決的問題 |

| 新增、修改、刪除、批量刪除 | 完整生命週期 |

## 2. 功能目的

提供一份**營養成分項目的字典**，作為食材與單品的「營養成分子表」之引用來源。每筆定義包含：

- 項目名稱（中文＋英文）
- 單位（指向「單位定義維護」）
- 平均每餐建議攝取量
- 是否為預設成分
- 顯示排序

下游：

- **食材維護作業**之營養成分子表的 `nutritionalDefinitionId` 引用此表
- **單品維護作業**之營養成分子表的 `nutritionalDefinitionId` 引用此表

## 3. 業務邏輯背景

### 3.1 主檔欄位（七個業務欄位）

| 欄位 | 型別 | 必填 |
|---|---|---|
| 項目名稱 | 文字 | 是（`@NotEmpty`） |
| 英文項目名稱 | 文字 | 否 |
| 單位 | 數字 | 是（`@NotNull`） |
| 平均每餐建議攝取量 | 文字 | 否 |
| 預設成分 | 布林 | 否 |
| 顯示排序 | 數字 | 是（`@NotNull`） |

### 3.2 排序

查詢時依「顯示排序」升冪。

### 3.3 「editable」欄位是分頁端點補上的標記，**目前永遠為 true**

`getNutritionalDefinitionsPageWithEditable` 對每筆定義設定 `editable=true`，並有 TODO 註解：

> TODO 待[食材維護作業]和[食譜維護作業]的營養成分子表建立之後補上此處邏輯

意義推論為：未來要檢查此定義是否被下游引用，若有則 `editable=false`，前端據此禁用刪除／修改按鈕。**目前邏輯未實作**，所有定義永遠 `editable=true`。列入 §11。

### 3.4 新增、修改**完全無驗證**

- 新增：直接 insert
- 修改：僅驗證自身存在
- 不檢查項目名稱唯一性
- 不檢查英文名唯一性
- 不檢查單位 ID 是否存在於「單位定義維護」
- 不檢查預設成分數量上限（理論上可以全部都是預設）

### 3.5 刪除採物理刪除＋不檢查下游引用

- 單筆刪除：驗證存在 → `deleteById`
- 批量刪除：直接 `deleteByIds`（不迭代、不驗證每筆存在性）

兩者**都不檢查**該定義是否被食材／單品的營養成分子表引用。刪除後下游可能留下「指向不存在定義」的孤兒記錄。列入 §11。

### 3.6 批量刪除無存在性驗證

`deleteBatchNutritionalDefinitions(List<Long> ids)` 直接 `deleteByIds`，不對每筆驗證存在。混入不存在的 ID 也不會報錯。

### 3.7 匯出 Excel 強制不分頁

`pageSize = PAGE_SIZE_NONE`。資料量大時可能 OOM（同其他模組議題）。

### 3.8 匯出端點呼叫了「不含 editable」版本

`exportNutritionalDefinitionsExcel` 呼叫 `getNutritionalDefinitionsPage`（無 editable），而非 `getNutritionalDefinitionsPageWithEditable`。Excel 不會包含 editable 欄位（推論為合理設計，因 RespVO 主要用於展示）。

## 4. 情境說明

### 4.1 正常流程 — 新增一個營養成分定義

PDM 維護員小華要建立「蛋白質」：

- 項目名稱：`蛋白質`
- 英文名稱：`Protein`
- 單位：`1`（克 g）
- 平均每餐建議攝取量：`18`
- 預設成分：是
- 顯示排序：`1`

送出後系統直接 insert（無驗證）。

### 4.2 異常情境 — 項目名稱重複

小華手滑要建第二筆「蛋白質」。系統**不擋**，兩筆並存。列入 §11。

### 4.3 異常情境 — 單位 ID 不存在

小華手滑單位填 `9999`（不存在於「單位定義維護」）。系統**不擋**。列入 §11。

### 4.4 異常情境 — 修改一筆已被下游引用的定義

小華修改「蛋白質」的「單位」從 g 改成 mg。系統**不擋**。但下游食材／單品的「每份含量」原本是依 g 填的，現在語意全部變了。列入 §11。

### 4.5 異常情境 — 刪除一筆已被下游引用的定義

小華刪除「蛋白質」（id=1）。系統**不擋**。下游食材主檔的營養成分子表中，所有 `nutritionalDefinitionId=1` 的記錄變成孤兒。列入 §11。

### 4.6 業務規則分流情境 — editable 欄位

前端列表頁取得分頁時，每筆都帶 `editable=true`。前端據此渲染「可編輯／刪除」按鈕。**永遠可編輯／刪除**，無下游保護。

### 4.7 多來源情境 — 食材維護作業要列出可選的營養成分

食材建檔頁要顯示營養成分子表的下拉。前端呼叫此模組分頁端點（或專屬「全部清單」端點，但本模組未提供）取得選項。

> **註**：未提供「全部清單」（不分頁取全部）端點。下游須自行用大 pageSize 撈或多次呼叫。列入 §11。
