## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。每天開店前、收班前、補貨前，我都要看一次自己門市的當前庫存：

> 「信義店今天牛肉餅剩 32 公斤、麵包剩 80 個、生菜剩 4 公斤（低於安全存量 8 公斤，要叫貨）。」

### 1.2 我要做什麼

- 開店前：確認所有食材庫存是否足以撐到下次到貨
- 中段：看哪些食材已低於安全存量，安排叫貨或調撥
- 收班前：核對當日異動是否合理（與紙本對照）
- 週末／月底：整體掃一次「我這家店」的存貨水位

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一眼看當前庫存量 | 補貨決策的基礎 |
| 只看自店資料 | 不要被全公司資料淹沒 |
| 與安全存量對照 | 知道哪些低於安全水位 |
| 顯示低於安全存量的醒目標示 | 補貨決策不能漏 |
| 支援行動裝置 | 在賣場現場操作 |
| Excel 匯出 | 與紙本盤點單對照 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「自店」當前庫存量列表 | 一眼看自店所有食材的當前量 |
| 安全存量並列顯示 + 低於水位醒目標示 | 補貨決策視覺化 |
| 食材分類／品名搜尋 | 上百樣食材時快速定位 |
| 日期切換（今日／指定日） | 跨日對帳 |
| Excel 匯出（限自店） | 對帳、稽核 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 倉儲查詢作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理」 |
| 兄弟功能 | 安全存量設定、入庫管理、出庫管理、門市調撥 |
| 主要頁面 | 自店當前庫存列表、Excel 匯出 |
| 簽核流程 | 無（純查詢） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在不打擾總部與倉儲規劃的前提下，**自助查出「我這家店現在還有多少存貨、哪些已經低於安全存量、需不需要叫貨」**。在門市端建立「庫存可視性」，是補貨、調撥、防止缺料的第一線決策依據。

設計理念：**唯讀＋鎖店＋警示優先**。店長不主動修改庫存值（庫存值由驗收、入庫、出庫、調撥、盤點等業務行為自動寫入），只負責看、判斷與決定下一步行動。

## 3. 業務邏輯背景

### 3.1 「當前庫存」是怎麼來的

每樣食材在每個倉庫各有一筆「當前庫存量」紀錄，由以下業務行為自動維護：

| 行為 | 對庫存的影響 |
|---|---|
| 驗收歸檔（總部採購流程） | 加庫存 |
| 物流配送收貨入庫 | 加庫存 |
| 調撥入庫（自他店撥入） | 加庫存 |
| 廚房領料／報廢／員工餐 出庫 | 減庫存 |
| 調撥出庫（撥到他店） | 減庫存 |
| 盤點盈虧調整 | 加或減 |

店長本身**不直接改庫存數字**，只是「看」與在其他作業（入出庫、調撥）中觸發變動。

### 3.2 「我的店」是怎麼判定的

當前庫存量端點在共用 API 中已有「依登入者門市過濾」的雛形：登入者帶有所屬門市時，會被自動限定為該門市對應的倉。

⚠️ 但底層 PRD 已指出兩個風險：

- 「店」與「倉」概念混淆 — 登入者門市 ID 直接當倉庫 ID 使用，**無法表達「一店多倉」**（例：信義店冷凍倉 + 冷藏倉 + 乾貨倉應為三筆）
- 一般分頁端點**不套**此過濾；如果店長視角誤接到一般分頁，會看到全公司資料

### 3.3 與安全存量的關係

「安全存量設定」由總部／店長分別維護一張「該店 × 各食材」的閾值清單。當前庫存查詢時，**建議**並列顯示「目前量」「安全存量」與差距，協助店長快速判斷。

> 待確認：目前底層 API 是否已 join 安全存量、或仍由前端自行串接（兩個 API 在 UI 端合併）。

### 3.4 店長視角看「在途量」？

採購已下單、物流尚未到店的數量稱為「在途量」。對店長補貨決策來說，「在途量」應與當前庫存並列才完整（例：當前 32 公斤、在途 10 公斤、安全存量 50 公斤 → 仍需補 8 公斤）。

> 待確認：是否要把在途量納入店長視角；目前查詢端點未提供此欄位。

### 3.5 與總部視角的差異

| 對比項 | 總部／倉儲人員 | 店長 |
|---|---|---|
| 預設可看範圍 | 全公司／自區 | **只能看自己門市** |
| 倉庫過濾 | 可選任一倉 | 自動鎖定該門市對應的倉 |
| 倉庫多選 | 支援 | 通常單店單倉，不需多選 |
| 安全存量警示 | 可選擇是否顯示 | **預設顯示**（操作決策） |
| 可寫入 | 視權限可寫 | **唯讀** |
| Excel 匯出 | 全欄位 | 簡化欄位（業務友善） |

## 4. 情境說明

### 4.1 正常流程 — 開店前掃一遍

5/19 早上 7:30，BK001 信義店店長小陳打開「我的庫存」：

1. 系統自動鎖定信義店
2. 列表顯示信義店所有食材的當前量
3. 第一頁就把「低於安全存量」的品項放最上面，醒目標示
4. 小陳看到生菜 4 公斤（安全 8 公斤）、可可粉 1 公斤（安全 3 公斤）兩項需補
5. 切到「門市調撥」找鄰店調撥，或通報總部採購

### 4.2 中段補貨評估

12:30 午餐尖峰結束，小陳評估下午茶段是否要再補：

1. 列表預設今日當下
2. 看「華堡相關食材」分類群組
3. 牛肉餅 14 公斤、起司 3 公斤、麵包 60 個
4. 對照預估下午銷量，判斷麵包不夠，發起調撥

### 4.3 異常情境 — 看到全公司資料

新進店長阿明剛開通帳號，登入查詢：

1. 系統未正確帶入所屬門市
2. 列表回傳全公司所有門市庫存（沒有鎖店）
3. 阿明看到上萬筆資料無從下手

⚠️ 此為**目前已知風險**（底層共用 API 鎖店邏輯仰賴登入者門市欄位正確設定）。

### 4.4 月底盤點對照

月底 5/31，小陳要做月盤：

1. 切到 5/31 當天
2. 匯出 Excel
3. 與紙本盤點單對照
4. 差異列入下月盤虧／盤盈調整

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「我的庫存」]
  ├─ 登入身份取得：店長、所屬門市
  ├─ 系統強制鎖店：以登入者門市過濾倉庫
  ├─ 查詢當前庫存量列表
  │    └─ 列：食材分類 ｜ 品號 ｜ 食材名 ｜ 當前量 ｜ 單位 ｜ 安全存量 ｜ 狀態
  ├─ 狀態欄位：正常／偏低／嚴重不足／（待確認：在途）
  ├─ 篩選：食材分類、品名、僅顯示低於安全存量
  └─ 點「Excel 匯出」→ 限自店資料

⚠️ 「自店過濾」目前由共用 API 雛形提供，依賴登入者門市欄位正確設定；若帳號未綁門市，會看到全公司資料。
```

## 6. 欄位規格

### 6.1 列表主要欄位

| 欄位（業務名） | 說明 | 店長視角是否顯示 |
|---|---|---|
| 品號 | 食材品號 | 是 |
| 食材名稱 | 從食材主檔帶 | 是 |
| 食材分類 | 食材主檔分類 | 是（用於分組） |
| 當前庫存量 | 當前可用數量 | 是 |
| 單位 | 公斤／個／包等 | 是 |
| 規格 | 單一規格資訊（如：1 公斤／包） | 是 |
| 安全存量 | 來自「安全存量設定」 | 是 |
| 狀態 | 正常／偏低／嚴重不足 | 是（醒目顏色） |
| 在途量 | 已下單尚未到店 | 待確認 |
| 倉庫 | 該庫存所在倉位 | 多倉時顯示 |
| 區域 ID／名稱 | 區域 | 否（店長無跨店） |
| 門市 ID／門市名 | 鎖店欄位 | 否（用於過濾） |

### 6.2 店長視角缺少／待補的欄位（推論／待確認）

- 在途量（已下單尚未到店）
- 最近一次入庫日（協助判斷新鮮度）
- 最近一次出庫日
- 預計下次到貨日（與物流行事曆綁定）
- 多倉時的「合計」列

## 7. 商業邏輯

### 7.1 鎖店過濾規則

店長視角查詢時，應在後端強制以登入者所屬門市過濾，**不接受**使用者從前端傳入的門市參數。

> 推論：目前實作雖有雛形（登入者門市自動帶入），但**未拒絕**使用者覆寫；若前端有 BUG 傳入別店 ID，可能造成越權。

### 7.2 狀態欄位的判定（推論）

```
若 當前量 < 安全存量 × 50% → 嚴重不足（紅）
若 當前量 < 安全存量       → 偏低（黃）
否則                       → 正常（綠）
```

> 50% 為示意門檻；實際門檻待 PM 定義。

### 7.3 唯讀

店長視角**不應有**新增、修改、刪除按鈕。底層 API 開放這三項，但權限應只給總部 IT／倉儲規劃。

### 7.4 預設排序

建議：先按「狀態」（嚴重不足→偏低→正常），再按「食材分類」，最後按「品名」。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店所有食材當前量 | 唯讀＋匯出 |
| 副店長（自店） | 同店長 | 唯讀＋匯出 |
| 區經理 | 區內所有店 | 唯讀＋匯出 |
| 採購／倉儲規劃（總部） | 全公司 | 全 CRUD（既有 API） |
| 系統管理員 | 全公司 | 全 CRUD（後門用，業務上不建議） |

> 後端目前以「具備倉儲查詢權限」的角色控管，未細分店長／總部。店長視角的鎖店過濾仰賴使用者身份的門市欄位是否正確設定。

## 10. 功能範圍

### 10.1 包含的功能

- 自店當前庫存量列表（待補強制鎖店）
- 安全存量並列顯示與狀態判定
- 食材分類／品名搜尋
- Excel 匯出（推論：沿用既有，需限制為自店）

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端無 `controller/app/` 對應路徑，僅 `controller/admin/` |
| 強制鎖店（拒絕越權） | 共用 API 僅以登入者門市做粗略過濾，未拒絕參數覆寫 |
| 一店多倉的正確處理 | 「店」與「倉」概念混淆 |
| 在途量欄位 | 共用 API 未提供 |
| 多倉合計列 | 未實作 |
| 行動裝置適配 UI | 未實作 |

### 10.3 不包含

- 修改庫存值（由其他作業自動寫入）
- 食材主檔維護（屬基礎主檔模組）
- 倉庫主檔維護（屬基礎主檔模組）

## 11. 待確認事項

| 議題 | 為何要確認 |
|---|---|
| 一個店長能管理多個門市嗎？（代理／跨店支援／區經理） | 影響鎖店是單值還是多值清單 |
| 店長能不能看到「自己店之外」的倉（中央倉、區域倉）？ | 影響鎖店範圍 |
| 「店」與「倉」是 1:1 還是 1:N？目前底層把店 ID 當倉 ID 用，一店多倉的場景無法表達 | 資料模型重大議題 |
| 「低於安全存量」的醒目門檻（50%？業務字典？固定？） | UI / Service 規則 |
| 是否要顯示「在途量／待驗收量」？ | UI 完整性 |
| 是否要顯示「預計下次到貨日」（綁物流行事曆）？ | 跨模組整合 |
| 是否支援「常用食材」標記（每店有固定關注的 20–30 項）？ | 提效 |
| Excel 匯出是否要限制只能匯自店 | 資安 |
| 多語系（中／英／越）需求 | 政策 |
| 共用 API 的鎖店是否拒絕參數覆寫？目前是否有越權風險？ | 資安 |
