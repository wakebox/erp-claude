## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

每天早班、午班、晚班交接時，我必須知道**我的店在每個時段「賣了什麼、賣了多少、跟以往比是高是低」**。台北市區的午餐尖峰、市郊店的晚餐尖峰、假日 vs 平日的客流差異 — 這些「時段曲線」決定我下一週的排班、每日的備料、現場的促銷話術。

> ⚠️ **後端目前沒有這份報表的程式碼**。本 PRD §2～§9 描述「**應該要有什麼**」，§10.2 與 §11 列出「**缺什麼、要確認什麼**」。

### 1.2 我要做什麼

- 每日下班前（21:00～23:00）：看「今日各時段曲線」，知道今天高峰落在幾點
- 隔日開店前（06:30）：對照「昨日同時段」與「上週同日同時段」，調整今日備料與排班
- 每週週會前：彙整一週的時段表現，跟區經理回報
- 月底：列上月各時段平均，做為下月排班依據
- 異常即時應變：午餐時段突降 30% 時，主動調人、調備料

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 我要看「今天」各時段賣多少 | 即時掌握當日營運節奏 |
| 我要看「同期比較」（上週同日、上月同日） | 判斷今日是正常 / 異常 |
| 我要看各時段的「熱銷單品」 | 知道哪個品項是高峰主力，備料才不會缺 |
| 我要看「我自己店」的數字，不要看別人 | 不要被全公司資料淹沒 |
| 我要能匯出 Excel 給區經理 | 週會報告 |
| 我要在手機 / 平板看 | 在賣場現場用 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「自店」每日時段銷售查詢頁 | 一眼看到今天時段曲線 |
| 時段定義（早餐／午餐／下午茶／晚餐／宵夜） | 跟漢堡王業務用語對齊 |
| 同期對比（上週同日／上月同日／去年同月） | 判斷異常與趨勢 |
| 各時段熱銷單品 TOP N | 備料與促銷依據 |
| 訂單金額／訂單筆數雙指標 | 不只看「賣幾個」，也看「賣多少錢」 |
| Excel 匯出 | 報表外呈 |
| 行動裝置可看 | 賣場現場使用 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 每日時段報表 |
| 所屬模組 | Excel 列「店長功能」（後端尚未歸位，推論應落於「需求集合」或新建之「店長報表」模組） |
| 兄弟功能 | PLS 備料準備量、每日盤點作業、物流串接記錄作業 |

## 2. 功能目的

讓店長能在無須打擾總部 IT、不需要登入 POS 後台的前提下，**自助查出「我這家店在指定日期內各時段的銷售曲線、熱銷單品、與同期差異」**，做為當日營運應變、隔日備料調整、週月排班分析的依據。

設計理念：**唯讀、鎖店、時段粒度標準化**。店長不能修改任何銷售數字，僅能查詢與匯出；查詢結果一律強制過濾為登入者所屬門市；時段切分採用業務字典統一定義，避免每位店長自訂導致週週對不起來。

## 3. 業務邏輯背景

### 3.1 「時段」是什麼

漢堡王業界與餐飲業實務上的時段切分通常如下（推論，待業務確認實際版本）：

| 時段代號 | 業務名稱 | 預估時間區間 |
|---|---|---|
| BR | 早餐 | 06:00 – 10:30 |
| LU | 午餐 | 10:30 – 14:00 |
| TE | 下午茶 | 14:00 – 17:00 |
| DI | 晚餐 | 17:00 – 21:00 |
| NI | 宵夜 | 21:00 – 隔日 02:00 |

> 此切分需業務確認。漢堡王台灣可能依「全日早餐」「市郊店無宵夜」等差異有店型差別切分，**待確認**。

### 3.2 資料來源（推論）

報表底層資料**不在 ERP 內**，而需向漢堡王中繼系統取得：

| 來源 | 提供什麼 | ERP 端用法 |
|---|---|---|
| 中繼「按時間區間查已完成訂單」服務 | 指定門市 + 指定起訖時分秒，回傳訂單摘要與商品銷量 | 拆時段呼叫多次，組成時段曲線 |
| 中繼「指定日期商品銷量」服務 | 指定門市 + 指定日期，回傳「整日」商品銷量 | 做為「整日合計」對帳數字 |
| ERP 端「門市」「商品」主檔 | 顯示用 | 將中繼回傳的代號翻成中文門市名／商品名 |

⚠️ 中繼是否已支援「時段粒度」直接彙整未知；但**至少可由「時間區間查詢」拆成多時段呼叫合成**，因此功能可行（推論）。

### 3.3 資料保存策略（推論）

時段報表有兩種可能做法：

| 做法 | 優點 | 缺點 |
|---|---|---|
| **即時呼叫中繼** | 永遠最新；不佔 ERP 空間 | 中繼響應慢時頁面卡；月度比較需多次拉 |
| **每日排程落 ERP 本地表** | 查詢快；可做歷史比對 | 中繼修正當日資料後本地會不一致 |

推薦混合：**當日資料即時拉，跨日比對改讀 ERP 本地表**。本地表建議於每日 23:30 排程，把今日 5 個時段的銷售匯總寫入「店長時段報表彙總」表。

### 3.4 與其他功能的關係

| 功能 | 差異 |
|---|---|
| 每日盤點（事後資料） | 看「期末庫存」，本功能看「期間銷售」 |
| 食材需求預測（事前） | 預測「下週要進多少貨」，本功能看「實際各時段賣多少」 |
| PLS 備料準備量（事前） | 預估「下一小時要備多少」，本功能是「事後對照」 |

時段報表往前餵 PLS（事後資料用於修正預估）、往後餵需求預測（每週時段曲線做為下週進貨依據）。

## 4. 情境說明

### 4.1 正常流程 — 下班前檢視今日表現

5/19（週二）21:30，BK001 信義店店長小陳結束晚班，打開「每日時段報表」：

1. 系統預設今日（2026-05-19）、自店（BK001 信義店）
2. 顯示 5 個時段柱狀圖：早餐 45 份、午餐 220 份、下午茶 80 份、晚餐 195 份、宵夜尚未結束
3. 旁邊顯示「上週二（5/12）同時段」對比：午餐 +12%、下午茶 -8%
4. 點開午餐時段：熱銷 TOP 5 — 華堡 60、雙層牛 35、雞塊 6 入 28、薯條 90、可樂 110
5. 小陳發現「下午茶 -8%」是因今日下雨；薯條銷量上升表示「外送量提升」，記在交接本

### 4.2 開店前對照備料

5/20 早上 6:30，早班店長小林開店前：

1. 進入時段報表，切到 5/19（昨日）
2. 看到昨日午餐華堡賣 60 份；今日預估下雨、用昨日數字 ×1.0 倍做備料
3. 也看 5/13（上週二同日）的早餐：賣 50 份；今日 6:30 已備 45 份，臨時加備 10 份

### 4.3 異常情境 — 午餐銷量突降

5/19 12:30，店長現場巡視，午餐時段已過半但訂單明顯偏少。打開報表：

1. 切「今日 11:00–12:30」即時數字
2. 與上週二同期相比 -45%
3. 系統旁顯示「同區域其他店」是否也下跌（待確認此功能是否提供）
4. 若僅自店下跌 → 確認設備故障 / 排隊動線；若整區下跌 → 大環境因素

### 4.4 跨日對帳情境

月初店長要核對上月：

1. 切日期範圍 4/1 – 4/30
2. 列表每日每時段一行（最多 30 × 5 = 150 行）
3. Excel 匯出，給區經理彙整

### 4.5 多店店長情境（推論）

某些「副店長」可能輪調支援 2～3 家店：

1. 預設帶入「我管的所有店」
2. 可切換單店或彙總顯示
3. 待確認後端「使用者 ↔ 多門市」關係如何維護

## 7. 商業邏輯

### 7.1 鎖店過濾規則

店長視角必須在查詢入口強制過濾自店：

```
1. 取登入者身份
2. 查「使用者 ↔ 門市」關係表（推論需建表）
3. 若使用者為單店店長 → 強制 storeId = 該店；UI 隱藏門市選單
4. 若使用者為多店副店長 → 顯示其管轄門市清單
5. 若使用者為區經理 → 顯示其管轄區域內全部門市
```

### 7.2 時段切分演算

```
給定日期 D 與時段字典：
  for each 時段 T in 時段字典：
      startTime = D + T.startTime
      endTime   = D + T.endTime
      if T 跨日（如宵夜 21:00–次日 02:00）：
          endTime = D+1 + T.endTime
      呼叫中繼「按時段查訂單」(自店, startTime, endTime)
      彙總筆數、件數、金額
```

⚠️ 中繼回應時間估計每時段 1～2 秒；5 時段 × 1 日約 5～10 秒。**建議前端非同步分時段顯示**，或於每日 23:30 排程預先落地。

### 7.3 同期對比演算

```
今日數字 / 同期數字 - 1 = 差異百分比

同期定義（業務確認）：
- 上週同日：D - 7 日
- 上月同日：D - 28 日（取最近 4 個 7 日）或月對日
- 去年同月同日：D - 365 日（閏年需處理）
```

### 7.4 客單價計算

```
客單價 = 銷售金額 ÷ 訂單筆數（除以 0 時顯示「-」，不丟例外）
建議保留小數 1～2 位
```

### 7.5 跨日宵夜時段處理

宵夜時段橫跨兩個日曆日（21:00 – 次日 02:00）。報表呈現時應「歸屬營業日」（21:00 那一天），而非歸屬日曆日，避免「次日宵夜被算進次日早餐之前」造成誤解。

### 7.6 即時 vs 排程的切換閾值

- 查當日（D = 今天）：即時呼叫中繼
- 查昨日及更早：讀本地表（前提：23:30 排程已執行；若排程失敗，回退到即時呼叫）

### 7.7 唯讀

店長視角僅提供查詢與匯出，無新增、修改、刪除按鈕。

|---|---|
| 是否實作此功能？業務優先級為何？ | 整份功能存廢前提 |
