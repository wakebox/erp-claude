# PRD｜系統管理 — 設定角色資料權限

> 來源：逆向自 `kingmaker-module-system` 之權限相關後端程式碼（`controller/admin/permission/PermissionController.java` 之 `assign-role-data-scope` 端點、`service/permission/RoleServiceImpl.java` 之 `updateRoleDataScope`、`PermissionServiceImpl.java` 之 `getDeptDataPermission` 解析邏輯、`vo/permission/PermissionAssignRoleDataScopeReqVO.java`、`enums/permission/DataScopeEnum.java`）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員。
我已經幫每個角色設定好「可以看到哪些頁面與按鈕」（選單權限）。但這還不夠 — 即使「採購助理」角色能看到「請購單列表」頁面，台北的助理也不應該看到台中的請購單；門市店長更只應該看到自己這家店的資料。

我需要為每個角色額外設定「在這些可以看到的頁面裡，到底能看到哪一個範圍的資料」，這就是「資料權限」。

### 1.2 我要做什麼

- **新角色建立後**：設完選單權限，緊接著設資料權限，否則該角色預設只能看到自己經手的資料（過於受限）。
- **組織異動時**：部門合併、分拆、新設區域時，要重新檢視「指定部門」類型的角色是否仍對應正確部門集合。
- **角色性質改變時**：例如「採購代理人」原本只能看本部門，後來改為支援跨部門，要把資料範圍從「本部門資料」改為「全部資料」。
- **稽核盤點時**：核對每個角色目前的資料範圍是否符合最小權限原則。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一次選一種範圍類型（不要勾來勾去） | 範圍互斥，例如「全部」與「僅本人」邏輯上不能同時 |
| 選「指定部門」時可挑多個部門 | 跨幾個部門的督導／代理人需要 |
| 不允許我改到「系統內建角色」 | 內建角色（超級管理員等）的資料範圍是系統假設 |
| 設定後立即生效 | 不想等快取過期才生效 |
| 設定不會誤觸發其他角色 | 一次只動一個角色 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 為指定角色設定資料範圍類型（5 選 1） | 一次性指派該角色的資料可視邊界 |
| 若選「指定部門」，可額外挑一組部門 | 支援跨部門但非全公司的角色 |
| 儲存時擋下對系統內建角色的操作 | 防止破壞系統假設 |
| 儲存後清除該角色之快取 | 下次驗權即用新設定 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 設定角色資料權限 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 角色管理（增刪改）、設定角色選單權限、設定使用者角色、部門管理 |
| 主要頁面 | 角色管理列表頁中，每個角色列上之「資料權限」動作；點開為彈窗，內含「範圍類型」選單與「部門挑選器」 |
| 簽核流程 | 無 |

## 2. 功能目的

為每個角色設定一個**資料可視邊界**，使「擁有相同選單權限」的兩個角色仍能看到不同範圍的業務資料。設計上把資料權限分為 5 種互斥型態（從寬到嚴）：全部、指定部門、本部門、本部門及以下、僅本人。

> 經系統整合決策（2026-05-16），漢堡王 IT 走向以 Laravel OA 為使用者單一事實來源、ERP 不再自簽 JWT。**「角色—資料範圍」對應關係長期將由 OA 或新權限模型統一管理**，本模組屬於 ERP 過渡期保留功能；長期規劃中將在搬遷階段 5 後廢棄。本 PRD 描述的是過渡期內、ERP 仍自管角色之資料範圍時的需求。

## 3. 業務邏輯背景

### 3.1 五種資料範圍互斥

| 範圍類型 | 意思 | 是否需要選部門 |
|---|---|---|
| 全部資料 | 不設限，可看到系統內所有資料 | 否 |
| 指定部門資料 | 僅可看到特定幾個部門 | **是**（需挑一組部門） |
| 本部門資料 | 僅可看到自己所屬的那一個部門 | 否（系統依使用者所屬部門自動代入） |
| 本部門及以下資料 | 自己所屬部門及其下層部門 | 否（系統依使用者部門自動展開） |
| 僅本人資料 | 只能看到自己經手／建立的單據 | 否 |

「本部門」「本部門及以下」「僅本人」是**動態解析**的 — 同一個角色掛在不同使用者身上，會根據該使用者所在部門得出不同的可視範圍。

### 3.2 一個使用者可能有多個角色，最終資料範圍是「最寬鬆的聯集」

使用者通常會被掛上一到多個角色。系統解析其資料權限時：

- **任一**角色為「全部資料」→ 該使用者可看全部，其他角色的限制被忽略
- **多個**角色為「指定部門」→ 部門集合做聯集
- **同時**有「指定部門」與「本部門及以下」→ 兩組部門 union
- 「指定部門」會**自動補上「使用者自己所在部門」** — 避免登入時連自己的資料都看不到
- 任一角色為「僅本人」→ 在 union 結果之上額外標記 `self=true`（可與部門範圍並存）
- 若使用者**完全沒有任何角色**或所有角色之資料範圍欄位為空 → 只能看到自己（self=true）

### 3.3 系統內建角色不可改

「系統內建角色」（如超級管理員、預設角色）是系統初始化時建立、被多處邏輯依賴的角色，**不允許修改**資料權限。嘗試對這類角色設定時系統會回覆「不能操作類型為系統內置的角色」並拒絕。

> 與 [[設定角色選單權限]] 行為不同 — 選單權限不檢查角色類型，本功能會檢查。

### 3.4 預設值：新建角色等於「全部資料」

新建角色時系統自動把資料範圍設為「全部資料」（見 [[角色管理]]）。意義是：**剛建好的自訂角色預設不受資料權限限制**，要靠本功能再收斂到實際需要的範圍。

### 3.5 「指定部門」儲存的是部門編號集合

選擇「指定部門資料」時，系統把選到的部門編號集合以 JSON 形式存進角色資料。後續使用該角色的使用者登入時，系統會解析此集合用於資料過濾。**該集合不會自動更新** — 若某部門被刪除或合併，集合內可能殘留無效部門編號（推論）。

### 3.6 儲存後清除該角色快取

每次儲存後，系統清除該角色的快取項目（**只清這一個角色，不像 [[設定角色選單權限]] 那樣清整個快取空間**）。下次有任何使用者透過此角色驗權時，會從資料庫重抓新設定。

## 4. 情境說明

### 4.1 正常流程 — 設定區域督導為「指定部門」

5/18 阿明剛建好「北區區域督導」角色。該角色預設資料範圍是「全部資料」（從新建時繼承），但實際上督導只應該看北區的三家門店與北區辦公室。

阿明點該角色之「資料權限」按鈕，彈出設定彈窗：

- 範圍類型：選「指定部門資料」
- 部門挑選器：彈出部門樹，勾選「台北信義店」「台北中山店」「桃園機場店」「北區辦公室」
- 按儲存

系統檢查：該角色為「自訂」類型，可以修改。更新資料範圍與部門集合，清掉該角色快取。督導下次登入即只能看到上述四個部門的資料。

### 4.2 正常流程 — 設定門市店長為「本部門資料」

「門市店長」角色將被掛載到全國 80 多家店長帳號上。阿明不希望逐店設定，於是把該角色的資料範圍設為「本部門資料」（不需要選部門）。

- 信義店店長登入 → 系統自動以該店長所屬部門代入 → 只看到信義店資料
- 中山店店長登入 → 同一個角色設定 → 但代入自己的部門 → 只看到中山店資料

一個設定服務所有店長，這是「動態解析」型範圍的價值。

### 4.3 異常情境 — 嘗試修改系統內建角色

阿明手滑點到「超級管理員」之「資料權限」並按儲存。系統回覆「不能操作類型為系統內置的角色」並拒絕，資料庫不變動。

### 4.4 邊界情境 — 多角色合併

員工小李同時擁有兩個角色：

- 「採購助理」→ 資料範圍「本部門資料」（採購部）
- 「跨部門代理人」→ 資料範圍「指定部門資料」（行政部、財務部）

登入時系統解析其資料權限：

- 兩組合併為「採購部、行政部、財務部」
- 因為「指定部門」自動補上「自己所在部門」 → 加上小李自己的部門編號（採購部，已在集合內）

最終可看到三個部門的資料。

### 4.5 邊界情境 — 多角色含「全部」

員工大華同時擁有「採購助理」（本部門）與「集團稽核員」（全部資料）兩個角色。系統解析時看到「全部資料」即直接判定可看全部，「採購助理」的部門限制被忽略。

### 4.6 邊界情境 — 同時有「僅本人」與「指定部門」

員工小陳同時擁有「採購助理」（指定部門：採購部）與「個人助理代簽」（僅本人）兩個角色。系統解析時：

- 部門集合 = {採購部}
- self 標記 = true

最終結果是「採購部 OR 自己經手」的聯集，由各業務模組查詢時自行套用此規則。

### 4.7 使用者鍵入錯誤情境 — 範圍類型缺值

阿明在前端忘記選範圍類型直接送出。伺服端驗證攔下，回覆「資料範圍不能為空」，未變動資料庫。

### 4.8 使用者鍵入錯誤情境 — 範圍類型非合法值

若前端傳入非 1-5 之資料範圍代碼，伺服端驗證攔下，回覆「資料範圍必須是 [1, 2, 3, 4, 5]」並拒絕。

### 4.9 推論性情境 — 部門被刪除後的殘留

阿明在 3 月把「北區督導」角色之「指定部門」設為含「桃園機場店」。5 月「桃園機場店」門店收掉，部門被軟刪除。本功能未自動清除角色內的部門集合，集合內仍有該部門編號。

下次督導登入時，系統嘗試展開「桃園機場店」對應的資料 → 該部門已不存在 → 無資料可看（推論：與資料權限解析側之容錯有關，本 PRD 不展開）。§11 列為待確認。

### 4.10 即時生效情境 — 快取自動失效

任何儲存後該角色快取立即被清除。使用者下次操作即用新設定，不需重新登入。

## 5. 操作流程

```
[使用者點某角色之「資料權限」按鈕]
  │
  ▼
[前端彈出彈窗，預先載入該角色現有設定]
  ├─ 範圍類型下拉
  └─ 部門挑選器（若現值為「指定部門」則啟用）
  │
  ▼
[使用者調整範圍類型]
  ├─ 選「全部資料」「本部門」「本部門及以下」「僅本人」→ 部門挑選器停用
  └─ 選「指定部門」→ 部門挑選器啟用，需挑至少一個部門
  │
  ▼
[按儲存]
  │
  ▼
[權限檢查：是否具備「系統-權限-設定角色資料權限」]
  │
  ▼
[輸入驗證]
  ├─ 角色編號必填
  ├─ 範圍類型必填、必須是 5 種之一
  └─（部門編號清單若範圍類型非「指定部門」，目前不檢查，仍照傳照存）
  │
  ▼
[角色存在性檢查 + 內建角色檢查]
  ├─ 角色不存在 → 拒絕，回覆「角色不存在」
  └─ 角色為「系統內建」類型 → 拒絕，回覆「不能操作類型為系統內置的角色」
  │
  ▼
[更新角色之「資料範圍」與「指定部門集合」欄位]
  │
  ▼
[清除該角色之快取]
  │
  ▼
[回傳成功]
```

## 7. 商業邏輯

### 7.1 儲存前的兩道校驗

1. **角色存在性**：依角色編號查詢，不存在則拒絕（回覆「角色不存在」）
2. **角色類型限制**：若該角色之類型為「系統內建」則拒絕（回覆「不能操作類型為系統內置的角色」）

> 與 [[設定角色選單權限]] 不同 — 選單權限不檢查角色類型，本功能會檢查。（來源：`RoleServiceImpl.java:163-173`）

### 7.2 不檢查「指定部門集合」與「範圍類型」的一致性

當前實作**不會檢查**：

- 範圍類型 = 「指定部門資料」但部門清單為空 → 不擋下，會儲存空集合，最終該角色看不到任何部門資料
- 範圍類型 = 其他四種（不需要部門）但部門清單有值 → 不擋下，部門清單仍會被存進去

§11 列為待確認。前端應作補強。

### 7.3 不檢查「指定部門」是否為有效部門

提交的部門編號集合**不會逐筆校驗**該部門是否存在、是否啟用。理論上可傳入不存在或已停用的部門編號，系統會原樣儲存。（推論：來源 `RoleServiceImpl.java:95-105`）

### 7.4 「指定部門」自動補使用者自身部門（解析階段，非儲存階段）

注意：儲存階段不會自動補。**儲存時存什麼就是什麼**。

但**解析階段**（使用者實際登入用此角色時）系統會自動把「使用者自己所在部門」加入可視部門集合，避免登入流程被自己的部門過濾擋下。（來源：`PermissionServiceImpl.java:303-308`）

對 PM 與管理員的意義：「指定部門」設定中不需要刻意把每位使用者所在部門也勾進去，系統解析時會自動加。

### 7.5 多角色合併規則（解析階段）

| 情境 | 結果 |
|---|---|
| 任一角色為「全部資料」 | result.all = true（其他規則被忽略） |
| 多個角色為「指定部門」 | 部門集合做聯集 + 自動補使用者部門 |
| 多個角色為「本部門及以下」 | 各自展開後聯集 + 自動補使用者部門 |
| 角色為「本部門資料」 | 自動補使用者部門 |
| 任一角色為「僅本人」 | result.self = true（可與部門範圍並存） |
| 全部角色之資料範圍欄位為空 | 跳過該角色（不貢獻任何範圍） |
| 使用者完全無角色 | result.self = true（只能看自己） |

（來源：`PermissionServiceImpl.java:276-331`）

### 7.6 儲存後僅清該角色快取

`@CacheEvict` 設定為僅清「該角色編號」對應的快取項目，不影響其他角色。較 [[設定角色選單權限]] 的「清整個快取空間」精細。（來源：`RoleServiceImpl.java:94`）

### 7.7 無操作日誌

此動作未標註 `@LogRecord`，**不會留下「誰、何時、把哪個角色之資料權限從 X 改為 Y」的稽核軌跡**。權限變動是高敏感操作。§11 列為待確認。

### 7.8 交易範圍

儲存只動一張資料表（角色主檔之兩個欄位），單一 SQL 即可完成，**未標 `@Transactional`**。實務上單筆 update 由資料庫保證原子性，無一致性風險。（來源：`RoleServiceImpl.java:95-105`）

## 8. 使用角色與權限

| 角色 | 可看／可操作 |
|---|---|
| **具備「系統-權限-設定角色資料權限」權限者**（通常為系統管理員） | 對任一非內建角色設定資料權限 |
| **僅具備「角色查詢」權限者** | 可看到角色列表，但看不到／無法觸發「資料權限」動作 |
| **未配發系統管理權限者** | 看不到系統管理選單 |

此權限與「設定角色選單權限」分開掛載（兩個不同的權限字串），可分別配發給不同角色 — 例如可讓某位審計人員只能改資料範圍而不能改選單。

## 9. 畫面需求／視覺規範

後端僅提供 API。預期前端在角色管理列表頁的每列操作區提供：

- 「資料權限」動作按鈕
  - 對「系統內建」類型之角色建議直接停用此按鈕（伺服端也會擋下，但前端先擋體驗較好）
- 點擊後彈出設定彈窗
  - 範圍類型下拉（5 選 1）
  - 部門挑選器：僅當範圍類型 = 「指定部門資料」時啟用，建議為樹狀勾選 UI（多選）
  - 範圍類型切換時，若從「指定部門」切到其他，部門挑選器停用且建議清空
  - 範圍類型切到「指定部門」時，建議強制至少選一個部門才能儲存
- 儲存／取消按鈕
- 對「系統內建」角色觸發此功能時，前端可直接顯示提示「系統內建角色之資料範圍由系統決定，無法調整」

## 10. 功能範圍

### 10.1 包含的功能

- 為指定角色設定資料範圍類型（5 選 1）
- 「指定部門」類型額外接收部門編號集合
- 儲存前檢查角色存在與非「系統內建」
- 儲存後清除該角色之快取以即時生效

### 10.2 預留但尚未實作

- 範圍類型與部門清單的**一致性校驗**未實作（例如「指定部門」但傳空集合會被原樣存）
- 部門編號集合中**部門有效性**未校驗（不存在或已停用的部門編號會被原樣存）
- 此動作**無操作日誌**（無 `@LogRecord` 標註），無法稽核「誰改了誰的資料權限」

### 10.3 不包含

- 角色本身的維護（在「角色管理」功能）
- 選單可見性設定（在「設定角色選單權限」功能）
- 使用者—角色之掛載（在「設定使用者角色」功能）
- 部門本身的維護（在「部門管理」功能）
- 多角色合併規則之解析（屬於使用者登入時的權限解析，自動執行）
