> 底層 PRD：[prd_庫存管理_出庫作業管理.md](prd_庫存管理_出庫作業管理.md)

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長 / 副店長 / 倉管員**（例：BK001 信義店）。出庫的常見場景：

> 「2026-05-19 — 廚房早班領料 8 公斤牛肉餅、20 個麵包；中午發現 2 包生菜過期報廢；下午撥 5 公斤雞塊到敦化店；晚班員工餐用了 1 公斤雞塊。」

每筆出庫都要在系統登記，當前庫存才會跟著減少。

### 1.2 我要做什麼

- 廚房領料：從倉位拿食材到工作站
- 報廢登記：過期、破損、品質不合的銷帳
- 員工餐／試吃／公關用：內部消耗
- 跨店調撥的出庫端：對應「門市調撥」功能的撥出方
- 盤點盤虧的銷帳：盤點完成後系統自動建立出庫單

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 快速建立出庫單（手機友善） | 每天多次操作，UI 卡頓會降效 |
| 出庫類型分類 | 領料 / 報廢 / 員工餐應分別統計 |
| 庫存不足要擋下來 | 防止庫存變負 |
| 報廢要拍照、留原因 | 食安稽核要證據 |
| 只看自店資料 | 不要被全公司資料淹沒 |
| 出庫後庫存即時更新 | 下游決策需即時 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「快速出庫」一頁式 UI | 多次操作的效率 |
| 出庫類型分類（領料／報廢／員工餐／試吃／…） | 統計與成本分析 |
| 庫存不足擋下、明確錯誤訊息 | 防呆 |
| 報廢照片＋原因（必填） | 食安稽核 |
| 自店歷史出庫單查詢 | 對帳 |
| 月底依類型統計（領料 vs 報廢 vs 員工餐） | 成本分析 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 出庫作業管理（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理」（與入庫共用一套程式碼） |
| 兄弟功能 | 倉儲查詢、安全存量設定、入庫管理、門市調撥 |
| 主要頁面 | 快速出庫頁、自店歷史出庫單、月度類型統計 |
| 簽核流程 | 報廢類**建議綁簽核**（待確認；底層自動歸檔多） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在現場操作的當下，**自助登記「我這家店出了什麼、出了多少、為什麼出」**，並讓系統把對應庫存即時減去。在門市端建立「庫存變動的完整記錄與責任歸屬」：什麼時候、誰、為什麼、出了多少。

設計理念：**手機友善、出庫前嚴格擋負值、報廢必留證、自動依類型分流走簽核**。出庫比入庫更頻繁（一天可能 10–20 次），UI 摩擦力低於入庫；同時報廢類型對食安稽核敏感，需要更多防呆與證據。

## 3. 業務邏輯背景

### 3.1 出庫的來源與類型（店長視角）

| 出庫類型 | 觸發方式 | 是否需簽核（推論） |
|---|---|---|
| 廚房領料 | 店長／倉管手動建單 | 否 |
| 報廢／損耗 | 店長／倉管手動建單 + 必拍照 | **是**（食安） |
| 員工餐 | 店長／倉管手動建單 | 否（小額） |
| 試吃／行銷活動 | 店長／倉管手動建單 | 視金額（待確認） |
| 跨店調撥（撥出方） | 對應「門市調撥」流程觸發 | 已在調撥流程中簽核 |
| 盤點盤虧調整 | 盤點流程完成後系統建單 | 已在盤點流程中簽核 |
| 不良品銷帳 | 不良品流程觸發 | 已在不良品流程中簽核 |

### 3.2 出庫單的結構

出庫與入庫共用同一張表，以「出入庫類型」欄位區分：

- **單頭**：單據編號、流程狀態、來源單據編號（若有）、區域／倉別／倉名、出入庫類型（出庫=0）、入庫時間（共用欄位，對出庫意義不明）、主旨、入庫事由（共用名稱，對出庫實為「出庫事由」）
- **明細**：品號、食材名、數量、單位、出庫事由

### 3.3 出庫時的庫存變動

簽收（單據歸檔）的當下：

1. 對每明細，找出該店該食材的當前庫存
2. 從當前庫存量減去出庫量
3. 若來源單據是調撥／盤點／不良品，回寫上游狀態

⚠️ 底層 PRD 已標示**多項風險**：

- **不檢查減後是否為負**：庫存可能變負值（隱性錯誤）
- **不檢查出庫量為正**：若傳負值會反向變加法
- 並行出庫無樂觀鎖：可能讀-改-寫競爭，結果錯誤
- 批次更新無交易保護：中途失敗留下半套資料

對店長視角而言，**第一點是最危險的**：店長因疏忽多輸了一個 0，庫存從 10 公斤瞬間變 -90 公斤，沒有任何警告。

### 3.4 出庫事由（字典）

底層以代號維護（例：採購入庫、調撥、盤點、不良品等）。店長視角應顯示中文名稱，並依出庫類型限縮選項。

### 3.5 與總部視角的差異

| 對比項 | 總部／倉儲人員 | 店長 |
|---|---|---|
| 可看單據 | 全公司／自區 | **只能看自己門市** |
| 主要操作 | 全流程 CRUD | **快速建單**為主 |
| 出庫類型 | 多元 | 領料／報廢／員工餐／試吃／調撥／盤虧 |
| 行動裝置 | 桌機為主 | **手機／平板**（現場操作） |
| 即時性 | 可批次補登 | **當下即時登記** |
| 簽核 | 視政策 | 報廢類必簽核 |
| 數量檢查 | 寬鬆 | **嚴格擋負值** |

## 4. 情境說明

### 4.1 正常流程 — 早班領料

5/19 上午 7:30，信義店店長小陳開店：

1. 打開「快速出庫」
2. 選「領料」類型
3. 加入：牛肉餅 8 公斤、麵包 30 個、生菜 1 公斤
4. 確認當前庫存：牛肉餅 40 → 32、麵包 80 → 50、生菜 5 → 4
5. 確認 → 系統檢查每項都 ≥ 0 → 寫入
6. 庫存即時更新

### 4.2 報廢情境

11:30 小陳發現 2 包生菜（共 2 公斤）已過期：

1. 「快速出庫」選「報廢」類型
2. 食材：生菜 2 公斤
3. 報廢原因：「效期 5/18，已過期」（必填）
4. 拍照（必填）
5. 提交 → 進入區經理簽核佇列
6. 區經理通過 → 庫存減 2 公斤、報廢成本入帳

### 4.3 異常情境 — 庫存不足

5/19 13:00，廚房急需 5 公斤雞塊，但庫存只剩 3 公斤：

1. 「快速出庫」選「領料」、雞塊 5 公斤
2. 提交 → 系統拒絕：「雞塊庫存不足（剩 3 公斤）」
3. 小陳改為領 3 公斤、同時發起「門市調撥」找鄰店補

⚠️ 第 2 步「拒絕」目前底層程式**未實作**；目前會直接讓庫存變負。**這是店長視角最關鍵的補強項**。

### 4.4 異常情境 — 多打一個 0

小陳要報廢 1 公斤生菜，誤輸入 10 公斤：

1. 「快速出庫」選「報廢」、生菜 10 公斤
2. 當前庫存 3 公斤，10 > 3
3. 系統應拒絕；目前底層不會擋
4. 庫存變 -7 公斤（隱性錯誤）

### 4.5 跨店調撥的撥出方

對方店發起調撥單請求 5 公斤雞塊：

1. 信義店店長收到調撥簽核請求
2. 通過簽核 → 系統自動建立一筆「調撥出庫」單
3. 庫存減 5 公斤
4. 此時店長**通常不需手動建單**，由調撥流程驅動

### 4.6 員工餐情境

晚班結束，剩 0.5 公斤雞塊做員工餐：

1. 「快速出庫」選「員工餐」、雞塊 0.5 公斤
2. 無需拍照（推論），無需簽核
3. 提交 → 庫存減 0.5 公斤

  │
  ├─ [提交]
  │    ├─ 庫存量 ≥ 出庫量？否則拒絕 ⚠️ 待實作

## 7. 商業邏輯

### 7.1 鎖店規則

店長視角的所有讀寫，應在後端強制：

```
1. 解析登入者所屬門市
2. 出庫時驗證：要扣庫存的倉必須屬於該店長
3. 列表只回該店單據
4. 不接受參數覆寫
```

> 推論；目前後端未實作。

### 7.2 庫存防超扣（**最關鍵**）

提交當下：

```
對每明細：
  1. 出庫量必須 > 0（否則拒絕）
  2. 找出該店該食材的當前庫存
  3. 當前庫存 ≥ 出庫量？否則拒絕並顯示「剩餘量」
  4. 才允許扣庫存
```

⚠️ 第 1、3 步底層**未實作**，是店長視角的首要補強項。

### 7.3 報廢類簽核（推論）

```
若 出庫類型 = 報廢：
  ├─ 必填原因與照片
  ├─ 提交後狀態 = 待簽
  └─ 進入區經理簽核佇列
否則：
  └─ 直接落表並扣庫存
```

### 7.4 出庫類型的統計分流

月底依出庫類型聚合：

- 領料總成本（屬正常營運）
- 報廢總成本（屬損耗，需檢討）
- 員工餐／試吃（屬員工福利／行銷）
- 調撥（屬倉位調整）
- 盤虧（屬盤點調整）

### 7.5 出庫不可逆

歸檔後不可由店長直接刪除／編輯。如需更正，走盤點調整。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店所有出庫單 | 建立、提交、查詢、匯出 |
| 副店長（自店） | 同店長 | 同店長 |
| 倉管員（自店） | 同店長 | 建立非報廢類；報廢類由店長提交（待確認） |

| 倉儲／採購（總部） | 全公司 | 全 CRUD |
| 系統管理員 | 全公司 | 全 CRUD（後門用） |

> 後端目前以「具備出入庫權限」的角色控管，未細分店長／總部，也未鎖店。

## 10. 功能範圍

### 10.1 包含的功能

- 自店「快速出庫」建單頁（待補強制鎖店、待補擋負值）

- 報廢類拍照＋必填原因（待實作）

- 自店歷史出庫單查詢（待補強制鎖店）

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端無 `controller/app/` 對應路徑 |
| 強制鎖店 | 共用 API 無此邏輯 |
