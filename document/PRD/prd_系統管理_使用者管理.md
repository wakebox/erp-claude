# PRD｜系統管理 — 使用者管理

> 來源：逆向自 `kingmaker-module-system` 之使用者相關後端程式碼（含 `controller/admin/user/`、`service/user/`、`dal/.../user/`）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

---

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員（資訊／IT 角色）。
公司剛導入這套 ERP，門市、總部採購、PDM、倉儲、財務都會用到，我負責讓「對的人能進系統、看對的東西、做對的操作」。

### 1.2 我要做什麼

- **每天**：接到新進員工通知，幫他在 ERP 開帳號、設定所屬部門、配崗位、配角色；接到離職通知，把該帳號停用。
- **每天**：處理「我忘記密碼了」「我登不進去」這類請求，重設密碼。
- **每週**：盤點目前啟用中的帳號，看哪些人已經離職或調離但帳號還在；確認門市店長對應的帳號是不是綁對門店。
- **每月**：協助新門市開幕，批次匯入該店相關員工帳號。
- **不定期**：接到主管請求「請把某某調到 OO 部門」「把這個人改成兼職崗位」，更新該員工資料。
- **不定期**：HR 索取系統帳號清冊作為稽核佐證，把帳號清單匯出成 Excel 給對方。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 新人到職當天就要能登入做事 | 採購、門市備料時程很緊，慢一天就影響營運 |
| 不能讓離職員工還能登入 | 資安與稽核風險（食材成本、廠商價是敏感資訊） |
| 我不想記每個人的所屬部門、崗位、店長身份在哪填、要不要分開維護 | 過去靠 Excel 維護權限矩陣很容易出錯 |
| 重設密碼必須留下紀錄 | 出事時要查得到誰在何時動了誰的帳號 |
| 主管能透過我快速把整批新店員工匯入 | 一家新店開幕常常一次就 20-30 個帳號要建 |
| 想知道一個帳號最後何時登入 | 判斷帳號是不是已經沒在用、可以停用 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 新增／修改／停用／刪除使用者 | 完整的帳號生命週期管理 |
| 重設使用者密碼 | 處理忘記密碼、入職初始密碼設定 |
| 啟用／停用使用者 | 離職或休假時暫停存取，不必刪除以便日後復職 |
| 條件查詢使用者分頁清單 | 依帳號、手機、狀態、建立時間、部門、角色快速找人 |
| 部門下拉同時涵蓋下層部門 | 主管查自己部門時想看整個一級部以下所有人 |
| 匯出使用者清冊（Excel） | 應稽核要求、月度盤點 |
| 匯入使用者（Excel） | 新店開幕批次建立帳號 |
| 取得目前登入者個人資料 | 前端顯示「我的帳戶」資訊 |
| 個人資料維護（暱稱／信箱／手機／頭像／自己改密碼） | 員工自助修改，不必每件事都找 IT |
| 綁定店長所管理的門市 | 門市店長角色需要對應一家具體的門市，否則無法做門市權限控管 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 使用者管理 |
| 所屬模組 | 系統管理 |

## 2. 功能目的

提供一套**集中、可追溯**的內部員工帳號管理機制，讓 IT 在單一介面完成「建立 → 配權 → 重設密碼 → 停用 → 刪除」的全生命週期動作，並支援 Excel 匯入／匯出以因應新店開幕、稽核盤點等批次場景。設計上把「組織歸屬（部門）」「角色任職（崗位）」「門市權責（店長綁店）」三件事與帳號主檔分開維護但綁定關聯，讓人事異動時不必動到帳號本體。

> **重要過渡背景**：經系統整合決策（見 [`erp-claude` 記憶 `project_user_integration_decision.md`]，2026-05-16），漢堡王整體 IT 走向以 Laravel OA 為使用者單一事實來源、ERP 不再自簽 JWT。本模組屬於 ERP 過渡期保留功能；長期規劃中，本模組將在搬遷階段 5 後廢棄。本 PRD 描述的是過渡期內、ERP 仍自管使用者主檔時的需求。

## 3. 業務邏輯背景

### 3.1 一個人在系統裡同時有「組織歸屬」「崗位」「角色」「門市」四種身份維度

- **組織歸屬（部門）**：每位使用者最多歸屬一個部門。決定他屬於哪個事業單位、影響資料權限（例如「只能看本部門以下的人」）。
- **崗位**：一位使用者可同時有多個崗位（例如「採購主管 + 採購代理人」）。屬於職務描述，不直接決定權限。
- **角色**：透過角色管理另行授權，本模組對角色僅提供「依角色篩選使用者」的查詢能力，不在本頁直接掛角色。
- **門市（店長綁店）**：當該使用者是門市店長時，需綁定他所管理的那一家門市。一位使用者最多綁一家門市。非店長角色不需要綁。

### 3.2 帳號唯一性以三個欄位個別控管

「使用者帳號」「手機號碼」「電子郵件」三者各自全系統唯一，不可重複。建立或修改時若任一欄位已被其他人使用，會擋下並回報具體原因。

### 3.3 部門查詢自動涵蓋下層

在使用者分頁查詢若指定部門條件，系統會自動把該部門的**所有下層部門也一併納入**，使用者只要選「總部」就能看到總部下面所有單位的人，不必逐層展開。

### 3.4 密碼從不以明文儲存

密碼一律以單向雜湊（業界標準演算法）加密儲存，連 IT 也無法反查。忘記密碼只能由 IT 走「重設密碼」流程指定新密碼；使用者也可在個人中心輸入舊密碼後自行更換。

### 3.5 初始密碼由系統參數決定

Excel 匯入時，新建立的帳號不會逐筆指定密碼，而是統一套用「系統參數設定」裡的初始密碼。若該參數未設定值，整批匯入會直接拒絕。

### 3.6 帳號上限受租戶配額管制

每個租戶（公司／組織）在系統中設有最大可建立帳號數。建立新帳號前會檢查目前使用量，若已達上限會拒絕新增並提示配額數字。（推論：與 [[多租戶機制]] 對接，本 PRD 範圍內不展開）

### 3.7 帳號狀態只有「啟用」「停用」兩態

沒有「凍結中」「待審核」這類中間狀態。離職員工的處理方式有兩種：
- 短期休假／可能復職 → 停用
- 已確定不會再使用 → 刪除（連同其關聯崗位資料一併移除）

## 4. 情境說明

### 4.1 正常流程 — 新進員工開帳號

5/20 新到職的採購助理小王要開帳號。系統管理員阿明開啟使用者管理頁，點「新增」，輸入：

- 帳號：`wangsmall`
- 暱稱：王小明
- 部門：選「採購部」
- 崗位：勾「採購助理」
- 信箱：`wang@bk.com.tw`
- 手機：`0912345678`
- 性別：男
- 密碼：`Bk2026!`（先給一個初始密碼，告知小王首次登入後到「個人中心」自己換）
- 店長綁店：留空（小王不是店長）

送出後系統建立帳號、預設「啟用」狀態，回到列表頁可看到該筆新資料。小王當天即可登入。

### 4.2 正常流程 — 門市店長開帳號

新開的「台北信義店（BK001）」要設定店長李大華。阿明新增帳號時除了基本資料外，特別在「店長管理的門市」欄選擇「台北信義店」。送出後系統同時建立帳號並寫入店長—門市對應，李大華登入後即僅能看到 BK001 的資料。

### 4.3 異常情境 — 帳號重複

阿明要幫新員工開帳號 `chenmm`，送出時系統回應「使用者帳號已經存在」。阿明改用 `chenmm2` 後成功。同理若信箱、手機已被他人用走，會分別回報「邮箱已经存在」「手机号已经存在」。

### 4.4 使用者鍵入錯誤情境 — 密碼長度

阿明在新增時打了 3 個字的初始密碼 `Bk1`，前端送出後系統回應「密碼長度為 4-16 位」並擋下，未建立帳號。

### 4.5 異常情境 — 達租戶上限

某子公司租戶設定上限 50 人，目前已有 50 個帳號，阿明嘗試新增第 51 人時系統回應「创建用户失败，原因：超过租户最大租户配额(50)！」並拒絕建立。

### 4.6 業務規則分流情境 — 修改使用者不改密碼

阿明要把員工小李的部門從「總務部」改成「行政部」。他點「編輯」，改部門後送出。系統明確不會更動小李的密碼（即使前端表單帶了 password 欄位也會被服務端忽略）。要改小李密碼必須走另一個「重設密碼」入口。

### 4.7 異常情境 — 嘗試刪除有資料依賴的帳號

阿明刪除離職員工小陳。系統執行：刪除使用者主檔 → 移除其角色關聯 → 移除其崗位關聯。小陳建立的歷史單據（請購單、採購單等）不受影響，仍以建檔人欄位的方式保留紀錄。（推論：商業邏輯依各模組之 `creator` 欄位設計，本 PRD 不展開）

### 4.8 多來源資料情境 — 新店一次匯入 25 人

新門市「台中清水店」開幕前一週，阿明拿到 HR 寄來的員工清單。他先用「下載匯入模板」取得 Excel 範本，把 25 筆資料貼進去（含登入名稱、用戶名稱、部門編號、信箱、手機、性別、狀態）並上傳。

系統逐筆處理：

- 18 筆是全新帳號 → 建立，密碼套用系統初始密碼。
- 5 筆登入名稱與既有帳號重複，但阿明上傳時勾了「支援更新」→ 系統以該登入名稱比對到既存帳號，覆寫其他欄位（密碼不動）。
- 2 筆因為信箱與其他人重複 → 失敗，列入失敗清冊並顯示原因。

完成後系統回傳「建立成功 18 筆 / 更新成功 5 筆 / 失敗 2 筆」，阿明把失敗清冊發給 HR 處理。

### 4.9 系統自動處理情境 — 使用者最後登入紀錄

每次使用者通過認證後，系統會自動寫入該使用者本次的登入 IP 與時間。阿明在列表頁可看到每位使用者「最後登入時間」，藉此判斷哪些帳號超過 90 天未登入、可考慮停用。

### 4.10 個人自助情境 — 員工自己換密碼／頭像

員工小張覺得 IT 給的初始密碼不安全，他登入後進「個人中心」，輸入舊密碼一次、新密碼兩次（系統會比對舊密碼，比對成功才更新）。他也順便上傳了自己的大頭照，系統將圖片存進檔案服務後寫回他帳號的頭像欄位。

## 5. 操作流程

### 5.1 新增使用者

```
[填寫使用者表單]
  │
  ▼
[點選「儲存」]
  │
  ▼
[權限檢查：是否具備「系統-使用者-新增」]
  │
  ▼
[租戶配額檢查：目前帳號數 < 上限]
  │
  ▼
[唯一性檢查]
  ├─ 帳號是否已存在
  ├─ 手機是否已存在
  └─ 信箱是否已存在
  │
  ▼
[關聯資料檢查]
  ├─ 所選部門必須為啟用狀態
  ├─ 所選崗位必須為啟用狀態
  └─ 所綁門市必須為啟用狀態（若有填）
  │
  ▼
[寫入]
  ├─ 主檔：使用者基本資料（密碼以加密形式存）
  ├─ 預設狀態 = 啟用
  ├─ 寫入使用者-崗位關聯（若有勾選崗位）
  └─ 寫入使用者-門市關聯（若有選店長綁店）
  │
  ▼
[記錄操作日誌：新增使用者]
  │
  ▼
[回傳新使用者編號]
```

### 5.2 修改使用者

```
[編輯使用者表單，送出]
  │
  ▼
[權限檢查：是否具備「系統-使用者-修改」]
  │
  ▼
[忽略表單帶入的密碼欄位 — 此入口不改密碼]
  │
  ▼
[同 5.1 之唯一性、部門、崗位、門市啟用檢查]
  │
  ▼
[更新主檔]
  │
  ▼
[同步崗位關聯]
  ├─ 新增的崗位 → 寫入關聯
  └─ 移除的崗位 → 刪除關聯
  │
  ▼
[同步門市關聯]
  ├─ 若本次選了門市 → 寫入新關聯
  └─ 刪除該使用者既有所有門市關聯
  │
  ▼
[記錄操作日誌：修改使用者（含前後差異）]
```

> **註**：5.2 中門市同步邏輯目前為「先寫新的 → 再刪舊的」，且未過濾「新舊相同則不動」。本行為來源 `AdminUserServiceImpl.java:193-206`，列入 §11 待確認。


### 5.7 Excel 匯入

```
[使用者上傳 Excel + 是否允許更新旗標]
  │
  ▼
[權限檢查：是否具備「系統-使用者-匯入」]
  │
  ▼
[讀取 Excel → 解析為使用者清單]
  │
  ▼
[清單不可為空]
  │
  ▼
[讀取系統參數中的初始密碼 → 不可為空]
  │
  ▼
[逐筆處理]
  ├─ 欄位格式驗證（帳號、信箱、手機、暱稱長度等）
  │     └─ 失敗 → 記入失敗清冊，繼續下一筆
  ├─ 業務檢查（部門啟用、信箱手機唯一）
  │     └─ 失敗 → 記入失敗清冊，繼續下一筆
  ├─ 依登入名稱比對是否已存在
  │     ├─ 不存在 → 套用初始密碼 + 寫入新帳號 → 列入「建立成功」
  │     ├─ 存在 + 允許更新 → 覆寫資料 → 列入「更新成功」
  │     └─ 存在 + 不允許更新 → 列入失敗清冊
  ▼
[整批回傳：建立清單 / 更新清單 / 失敗清單]
```

> **註**：整批匯入包在交易內，若中途出現非業務性錯誤（程式錯誤、資料庫異常）會回滾整批。業務性失敗（重複、格式錯）僅記入失敗清冊不會回滾。

## 7. 商業邏輯

### 7.1 新增／修改的前置檢核（必過 5 關）

1. **帳號／手機／信箱唯一性** — 任一重複即拒絕，回報具體欄位
2. **所屬部門啟用中** — 若部門已停用不可選
3. **所屬崗位啟用中** — 多崗位逐一檢查
4. **店長綁店啟用中** — 若有填則該門市必須啟用
5. **租戶配額（僅新增）** — 目前帳號數需 < 該租戶上限

唯一性檢查時系統會暫時忽略資料權限，以確保跨部門也能正確判斷是否重複。

### 7.2 修改不變動密碼

從一般「修改使用者」入口送出表單，即使表單帶 password 欄位，伺服端會在處理時清空該欄位（`AdminUserServiceImpl.java:157`）。要改密碼必須走獨立的「重設密碼」入口。

### 7.3 崗位關聯增量更新

修改時不是「刪除舊有 → 全部重寫」，而是計算差集：

- 新表單有、舊資料沒有 → 新增關聯
- 舊資料有、新表單沒有 → 刪除關聯
- 兩邊都有 → 不動

### 7.4 門市關聯更新（待確認）

修改時的處理順序：

1. 若本次表單有指定門市 → 寫入新關聯
2. 把該使用者既有的全部門市關聯刪除（包含剛剛寫入的）

此邏輯在「先寫後刪」順序下，將不會留下任何門市綁定（推論：與預期不符，列入 §11 待確認）。來源 `AdminUserServiceImpl.java:193-206`。

### 7.5 部門查詢自動下展

查詢條件帶部門時，後端會：

1. 取該部門的所有子部門集合
2. 加入該部門本身
3. 以此集合做 IN 查詢

故 UI 上指定「採購部」可以查到「採購部—國際採購組」「採購部—在地採購組」等下層所有人。

### 7.6 密碼處理

- 儲存：以單向加密儲存，鹽值由演算法處理，系統不另存
- 比對舊密碼：以加密器之 `matches` 比對，不解密
- IT 重設：直接覆寫（不需舊密碼）
- 個人改密碼：必須提供舊密碼且比對成功

### 7.7 刪除使用者的連動

刪除時連動處理：

1. 主檔刪除
2. 該使用者所有角色關聯一併清除（透過權限服務的「處理使用者刪除」鉤子）
3. 該使用者所有崗位關聯一併清除

> 該使用者所有「店長—門市」關聯目前在刪除流程中未明確處理（推論：來源 `AdminUserServiceImpl.java:279-292`），列入 §11 待確認。

### 7.8 匯入：交易與失敗處理

- 整批包在資料庫交易內 → 程式錯誤回滾整批
- 業務性失敗（重複、格式錯）→ 不回滾、記入失敗清冊繼續
- 初始密碼來自系統參數（「使用者初始密碼」）→ 若未設定，整批直接拒絕

### 7.9 簡易使用者清單

提供一個「簡易清單」端點，僅回傳「啟用中」的使用者，給前端做下拉選單用。停用的使用者不會出現在下拉選項。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| **系統管理員（具備「系統-使用者-查詢」「-新增」「-修改」「-刪除」「-重設密碼」「-匯入」「-匯出」全部權限）** | 全部使用者（含其他部門） | 新增、修改、刪除、停用、重設密碼、匯入、匯出 |

> 權限細節由角色管理掛載；本模組僅在後端對每個動作驗證對應之權限字串。資料權限部分（誰能看誰）由資料權限框架另外控管，本 PRD 不展開。

## 10. 功能範圍

### 10.1 包含的功能

- 使用者主檔新增、修改、刪除
- 啟用／停用
- IT 重設他人密碼
- 個人中心：自查、自改資料、自改密碼、上傳頭像
- 條件分頁查詢（含部門下展、角色篩選）
- Excel 匯入（支援「更新既有」選項）／匯入模板下載
- Excel 匯出
- 簡易使用者清單（給下拉用，僅啟用中）
- 綁定使用者所管理的門市（店長綁店）

### 10.2 預留但尚未實作

- **頭像欄位**：主檔有頭像欄位（`avatar`），但只有「個人中心—上傳頭像」入口會更新它，IT 端的新增／修改入口雖接受 avatar 欄位，但 UI 是否提供未確認。
- **備註欄位**：主檔有 `remark`，前端是否顯示與利用待確認。

### 10.3 不包含

- 角色本身的維護（在「角色管理」模組）
- 部門與崗位本身的維護（在「部門管理」「崗位管理」模組）
- 角色—使用者的批次掛載（在「角色管理」模組）
- 登入／登出／JWT 簽發機制
- 社交帳號綁定（個人中心僅顯示既有綁定）
