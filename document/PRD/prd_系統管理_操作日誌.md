> 來源：逆向自 `kingmaker-module-system` 之操作日誌相關後端程式碼（含 `controller/admin/logger/OperateLogController.java`、`service/logger/OperateLogServiceImpl.java`、`dal/.../logger/OperateLogDO.java`、`framework/operatelog/.../LogRecordServiceImpl.java`）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的**資安／稽核／IT 主管**，或被指派處理「誰動了那筆資料」的調查窗口。
ERP 系統裡每天有人在改使用者、改菜單、改廠商報價、開請購單、刪單。一旦出事（例如：某筆採購單金額被人改成 0 元、某帳號被刪了又建），我要能在最短時間找出「誰、什麼時候、在哪支畫面、把什麼從什麼改成什麼」。

### 1.2 我要做什麼

- **日常稽核**：每月撈一次操作日誌做隨機抽查，確認沒有異常操作。
- **稽核外稽**：四大會計師事務所或母公司稽核要求提供「2026 年 Q1 採購單金額異動紀錄」，我要能依模組、操作名、時間區間撈出來。
- **事件調查**：被通報「某廠商報價被無故改低」，要查那筆業務編號的所有操作紀錄。
- **問題排查**：開發或 IT 想知道「某使用者今天下午 14:00 後做了哪些操作」協助重現問題。
- **匯出佐證**：把查到的紀錄匯出成 Excel 給稽核者或留檔。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 系統要自動把所有重要操作記下來，不用我介入 | 我不可能要每位 IT／業務都主動寫日誌 |
| 日誌要有「人、時、地、操作、前後值」 | 稽核佐證最低要求 |
| 即便操作者在後續刪掉了某筆資料，日誌也要留住前後值 | 「刪除」常常是調查的重點 |
| 日誌要能依模組、操作名、操作明細、業務編號、人、時間區間做模糊查詢 | 不知道是誰幹的時候要能反查 |
| 同一次請求引發的多筆日誌要能串起來 | 一次 API 呼叫可能觸發多個 Service 動作，要能用同一個 TraceId 串聯 |
| 可以匯出 Excel | 提供給外稽 |
| 寫入日誌**不能拖慢主要業務** | 寫日誌不該是業務的關鍵路徑 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 業務 Service 在執行關鍵操作時自動寫入操作日誌（無侵入） | 不必每個 Service 自己寫 insert |
| 寫入採異步、寫失敗不影響業務 | 不拖慢業務 |
| 日誌帶 TraceId、使用者、模組、操作名、業務編號、操作明細、擴充欄位、Request 資訊（method、URI、IP、UA） | 滿足稽核欄位需求 |
| 操作日誌分頁查詢（含多條件） | 維運盤點 |
| 匯出 Excel | 外稽佐證 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 操作日誌 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 登入日誌、使用者管理、角色管理、選單管理、字典管理 |
| 主要頁面 | 操作日誌查詢頁（含篩選列、分頁列表、匯出按鈕） |
| 簽核流程 | 無 |

## 2. 功能目的

提供一套**無侵入、自動記錄、異步寫入、可稽核可匯出**的操作軌跡追蹤機制。設計上分兩塊：

1. **寫入面（被動、隱性）**：業務 Service 在重要動作上掛一個註解，框架自動攔截、組裝、異步寫入。業務開發者不必手寫 insert，也不必擔心日誌失敗會回滾業務。
2. **查詢面（主動、顯性）**：在系統管理介面提供分頁查詢與匯出，供資安、稽核使用。

兩塊是分離的：寫入由框架 starter 統一處理，查詢由 system 模組提供。

> **重要過渡背景**：本模組屬於 ERP 內部稽核機制，與 OA 整合過渡關聯較低。OA 整合後本模組仍可獨立存在；但「使用者編號」欄位的解讀（指向 ERP user 還是 OA user）長期需要對齊（推論：階段 5 後若使用者主檔搬遷至 OA，此欄位的「翻譯來源」要切換）。

## 3. 業務邏輯背景

### 3.1 日誌寫入由註解觸發，不需業務 Service 自己寫

業務 Service 在方法上掛上「日誌註解」（例如 `@LogRecord(type="使用者管理", subType="刪除使用者", bizNo="{{#id}}", success="刪除使用者[{{#user.nickname}}]")`），框架攔截器（基於 mzt-biz-log 套件）在方法執行成功後：

1. 從 SpEL 表達式解析出 `type`、`subType`、`bizNo`、`action`、`extra`
2. 從當前登入上下文取得使用者編號、使用者類型
3. 從當前 HTTP Request 取得：請求方法（GET/POST/...）、請求網址、使用者 IP、瀏覽器 UA
4. 從 Tracer 取得當前 TraceId
5. **異步**呼叫「建立操作日誌 API」寫入

整個過程業務 Service 不知道也不需要知道。

### 3.2 寫入失敗不影響業務

日誌寫入採異步（`@Async`）。整個寫入路徑被 `try/catch` 包覆，任何例外只會打 log 不會丟回給主執行緒。這代表：

- 業務交易不會因為日誌寫入失敗而回滾
- 業務回應時間不受日誌寫入耗時影響
- 但**日誌可能丟失**：服務重啟、執行緒池滿、資料庫短暫不可用都可能讓某些日誌沒寫進去（列入 §11）

### 3.3 日誌欄位的兩種來源

| 欄位類別 | 來源 | 範例 |
|---|---|---|
| **業務面** | 註解上的 SpEL 表達式由業務 Service 設定 | `type=使用者管理`、`subType=刪除使用者`、`bizId=1024`、`action=「刪除使用者[王小明]」`、`extra={"deptId":3}` |
| **環境面** | 框架攔截時自動補 | `traceId`、`userId`、`userType`、`requestMethod`、`requestUrl`、`userIp`、`userAgent` |
| **建檔面** | 由 `BaseDO` 自動補 | `createTime`、`updateTime`、`creator`、`updater`、`deleted` |

### 3.4 TraceId 的角色

每個請求進入系統時會由攔截器產生唯一的 `traceId`，貫穿該請求生命週期內所有日誌、錯誤、Logger 輸出。一個 HTTP 請求若觸發多筆 `@LogRecord`（例如：刪除使用者連動刪除其角色、崗位關聯，每個 Service 都掛了註解），這些日誌會帶相同的 `traceId`。

### 3.5 操作明細欄位（action）是業務語句

`action` 是給人看的中文敘述，不是 JSON。範例：「修改編號為 1024 的使用者資料，將性別從『男』改為『女』，將暱稱從『芋道』改為『源碼』」。

這個敘述是業務 Service 透過 SpEL 在註解中拼出來的；前後值比對也是業務側做的（mzt-biz-log 提供 `DIFF` 函式對前後物件做差異）。

### 3.6 拓展欄位（extra）是 JSON

當業務需要記錄結構化資訊（例如「這次刪除使用者連帶清掉了哪些角色關聯」），可在註解的 `extra` 表達式中產出 JSON 字串。

### 3.7 查詢的過濾條件支援模糊

「模組類型」「操作名」「操作明細」均採模糊比對（SQL `like`）；「使用者編號」「業務編號」採精確比對；「建立時間」採區間。

### 3.8 「使用者編號」用翻譯註解轉成暱稱

`OperateLogRespVO` 在「使用者編號」欄位上掛了翻譯註解（`@Trans`），回傳分頁／匯出 Excel 時會根據編號自動補上對應使用者暱稱欄位。即便日後使用者主檔搬遷至 OA，這個翻譯只要替換實作即可。

### 3.9 匯出本身會被記錄為操作日誌

「匯出操作日誌」這支端點本身又掛了 `@ApiAccessLog(operateType = EXPORT)` 註解，匯出動作也會被記下來。意即：誰、什麼時候匯出了多少筆操作日誌，自己也會留下紀錄。

> **註**：此處用的是 `@ApiAccessLog` 而非 `@LogRecord`，存的是另一張「存取日誌」表，不是這張「操作日誌」表（推論，待確認與「存取日誌」的差異與重疊）。

## 4. 情境說明

### 4.1 正常流程 — 業務 Service 自動寫日誌

IT 阿明在「使用者管理」頁刪除使用者「王小明」（編號 1024）：

1. 前端送出 `DELETE /system/user/delete?id=1024`
2. `UserController.deleteUser(1024)` → 進入 `AdminUserServiceImpl.deleteUser(1024)`
3. 該 Service 方法掛了 `@LogRecord(type="使用者管理", subType="刪除使用者", bizNo="{{#id}}", success="刪除使用者[{{#user.nickname}}]")`
4. 方法執行成功
5. 框架攔截器：
   - SpEL 解析出 `type=使用者管理`、`subType=刪除使用者`、`bizId=1024`、`action=刪除使用者[王小明]`
   - 自動補：`userId=阿明的編號`、`userType=後台`、`traceId=abc-123`、`requestMethod=DELETE`、`requestUrl=/system/user/delete`、`userIp=10.1.2.3`、`userAgent=Mozilla/...`
6. 異步呼叫「建立操作日誌 API」→ 寫入資料庫

完成後阿明感受到的回應時間不受日誌寫入影響。

### 4.2 正常流程 — 資安主管查詢

資安主管小華懷疑「採購廠商報價有人惡意調低」。她開啟操作日誌查詢頁：

- 模組類型（模糊）：`採購` 或 `廠商報價`
- 操作名（模糊）：`修改`
- 業務編號：留空
- 操作明細（模糊）：留空
- 使用者編號：留空
- 建立時間：`2026-05-01 00:00:00 ~ 2026-05-19 23:59:59`

送出。系統依以上條件分頁查詢（依日誌編號倒序），回傳 38 筆紀錄。每筆顯示：操作時間、操作人暱稱、模組、操作名、業務編號、操作明細、IP。

### 4.3 正常流程 — 鎖定某筆業務編號的所有操作

小華從上一步看到一筆異常的廠商報價編號 `2087`，她改用：

- 模組類型：`採購`
- 業務編號：`2087`（精確）

系統回傳 7 筆紀錄：建立、3 次修改、2 次匯出、1 次刪除。她依「操作明細」逐筆比對前後值，找到第 3 次修改把單價從 320 元改成 32 元，並抓到當時的使用者編號、IP。

### 4.4 正常流程 — 匯出給稽核

小華準備提供給外稽。她在查詢條件填好後按「匯出」：

1. 系統檢查權限「操作日誌-匯出」
2. 強制改用「不分頁」（PAGE_SIZE_NONE）撈全部符合條件的紀錄
3. 透過翻譯機制把「使用者編號」轉成暱稱
4. 輸出 Excel：`操作日志.xls`
5. 本次匯出動作本身也被存取日誌記下（誰、何時、匯出條件）

### 4.5 一次請求多筆日誌情境 — TraceId 串聯

阿明刪除使用者「王小明」，此動作連動觸發：

- 主檔刪除（`AdminUserServiceImpl.deleteUser`） → 1 筆日誌
- 清除角色關聯（`PermissionService.processUserDeleted`） → 1 筆日誌
- 清除崗位關聯（`PostService.processUserDeleted`） → 1 筆日誌

三筆日誌的 `traceId` 都相同（例如 `abc-123`）。日後若懷疑某次刪除有問題，可以用 traceId 撈到完整關聯動作。

### 4.6 異步失敗情境 — 服務重啟

阿明刪除使用者後 0.1 秒服務重啟。日誌寫入的異步任務在執行緒池中尚未執行就被中斷，這筆日誌**永久丟失**。業務本身已成功，但稽核軌跡缺一筆。

> **註**：此為設計權衡。若要 100% 可靠，需改為同步寫入或落地到佇列。列入 §11。

### 4.7 非 Web 觸發情境 — 排程任務寫日誌

某個排程任務（例如「凌晨重算安全存量」）在 Service 中也掛了 `@LogRecord`，會觸發日誌寫入。攔截器流程：

- 從 Tracer 取 TraceId → 通常排程也有自己的 traceId
- 從登入上下文取使用者 → **取不到**（排程沒有登入使用者）→ `userId`、`userType` 留空
- 從 Request 取請求資訊 → **取不到**（不是 HTTP）→ `requestMethod`、`requestUrl`、`userIp`、`userAgent` 留空
- 業務面 `type`、`subType`、`bizId`、`action` 仍然有

結果是一筆「使用者未知、來源未知」的日誌。

### 4.8 異常情境 — 日誌寫入時部分必填欄位是空

「建立操作日誌 API」的 DTO 對 `userId`、`userType`、`type`、`subType`、`bizId`、`action`、`requestMethod`、`requestUrl`、`userIp`、`userAgent` 都標註了 `@NotNull` / `@NotEmpty`。但攔截器在「非 Web 情境」或「未登入情境」會留空。

實際執行時 Service `createOperateLog` 沒有對 DTO 做 `@Valid` 校驗，DTO 上的驗證註解形同虛設，仍會插入到資料庫（推論：列入 §11）。

## 5. 操作流程

### 5.1 業務操作自動觸發寫入日誌

```
[業務 Service 方法執行成功]
  │
  ▼
[框架攔截器（mzt-biz-log）攔截 @LogRecord]
  │
  ▼
[解析 SpEL：type / subType / bizNo / action / extra]
  │
  ▼
[補使用者資訊]
  ├─ 從 SecurityFrameworkUtils 取登入者
  └─ 未登入 → userId、userType 留空
  │
  ▼
[補請求資訊]
  ├─ 從 ServletUtils 取 Request
  └─ 非 Web → 全部留空
  │
  ▼
[補 TraceId（從 TracerUtils）]
  │
  ▼
[異步呼叫 operateLogApi.createOperateLogAsync(reqDTO)]
  │
  ▼
[Service 內 BeanUtils 轉 DO → mapper.insert]
  │
  ▼
[（過程中任何異常 → log.error 不重拋）]
```

### 5.2 分頁查詢

```
[輸入查詢條件]
  │
  ▼
[權限檢查：是否具備「操作日誌-查詢」]
  │
  ▼
[組成 SQL 條件]
  ├─ 使用者編號 = ?
  ├─ 業務編號 = ?
  ├─ 模組類型 LIKE %?%
  ├─ 操作名 LIKE %?%
  ├─ 操作明細 LIKE %?%
  └─ 建立時間 BETWEEN ? AND ?
  │
  ▼
[依日誌編號倒序分頁]
  │
  ▼
[BeanUtils 轉成回應 VO]
  │
  ▼
[（翻譯機制在輸出時把使用者編號 → 暱稱）]
  │
  ▼
[回傳分頁結果]
```

### 5.3 匯出 Excel

```
[輸入查詢條件]
  │
  ▼
[權限檢查：是否具備「操作日誌-匯出」]
  │
  ▼
[強制 pageSize = PAGE_SIZE_NONE（不分頁）]
  │
  ▼
[同 5.2 之查詢條件取得完整清單]
  │
  ▼
[執行翻譯（使用者編號 → 暱稱）]
  │
  ▼
[輸出 Excel：操作日志.xls，工作表「数据列表」]
  │
  ▼
[本次匯出動作被 @ApiAccessLog 記入存取日誌]
```

## 6. 欄位規格

### 6.1 操作日誌主檔欄位

| 欄位（業務名） | 型別／格式 | 來源 | 規則 | 備註 |
|---|---|---|---|---|
| 日誌編號 | 數字 | 系統產生 | 全系統唯一 | — |
| TraceId | 文字 | 攔截器自動補 | 由 Tracer 產生 | 同一次請求多筆日誌共享 |
| 使用者編號 | 數字 | 攔截器自動補 | 由登入上下文取 | 非 Web／未登入時留空 |
| 使用者類型 | 整數 | 攔截器自動補 | 對應使用者類型列舉（後台／會員等） | 非 Web／未登入時留空 |
| 模組類型 | 文字 | 業務 Service 註解 | 例如「使用者管理」「採購單」 | 大粒度模組分類 |
| 操作名 | 文字 | 業務 Service 註解 | 例如「刪除使用者」「修改報價」 | 更細粒度 |
| 業務編號 | 數字 | 業務 Service 註解（SpEL） | 對應業務主鍵 | 用於依筆查詢 |
| 操作明細 | 文字 | 業務 Service 註解（SpEL） | 中文敘述，含前後值 | 給人看 |
| 拓展欄位 | 文字 | 業務 Service 註解（SpEL） | JSON 字串 | 結構化補充 |
| 請求方法 | 文字 | 攔截器自動補 | GET / POST / PUT / DELETE | 非 Web 時留空 |
| 請求網址 | 文字 | 攔截器自動補 | Request URI | 非 Web 時留空 |
| 使用者 IP | 文字 | 攔截器自動補 | 由 X-Forwarded-For / RemoteAddr 取得 | 非 Web 時留空 |
| 瀏覽器 UA | 文字 | 攔截器自動補 | User-Agent header | 非 Web 時留空 |
| 建立時間 | 日期時間 | BaseDO 自動補 | — | 操作發生時間 |

> 來源：`OperateLogDO.java`、`OperateLogCreateReqDTO.java`、`LogRecordServiceImpl.java`

### 6.2 查詢輸入欄位

| 欄位 | 規則 |
|---|---|
| 使用者編號 | 精確比對 |
| 業務編號 | 精確比對 |
| 模組類型 | 模糊比對 |
| 操作名 | 模糊比對 |
| 操作明細 | 模糊比對 |
| 建立時間 | 區間（起 / 訖） |

> 來源：`OperateLogPageReqVO.java`、`OperateLogMapper.java`

### 6.3 回傳欄位（包含 Excel 匯出）

`OperateLogRespVO` 同 6.1 之主檔欄位，並額外提供：

| 欄位 | 來源 |
|---|---|
| 使用者暱稱 | 由翻譯機制把「使用者編號」轉成「對應使用者的暱稱」 |

> Excel 匯出僅包含部分欄位（`@ExcelProperty` 標註的）：日誌編號、操作人、模組類型、操作名、業務編號、操作明細、…

## 7. 商業邏輯

### 7.1 寫入採異步

- 攔截器解析完 `@LogRecord` 後呼叫 `operateLogApi.createOperateLogAsync()`（標註 `@Async`）
- 寫入發生在獨立執行緒池
- 業務主流程不等待日誌寫入完成

### 7.2 寫入失敗以 log.error 吞掉

整個 `record()` 方法被 `try/catch Throwable` 包覆，任何異常都只打 `log.error` 不重拋。理由是異步呼叫已脫離原請求 context，若拋異常會被執行緒池吞噬或破壞穩定性，倒不如記下來人為跟進。

### 7.3 業務 Service 不必感知日誌的細節

`@LogRecord` 用 SpEL 表達式從方法參數、回傳值、上下文中取資料：

- `{{#id}}` 從方法參數 `id` 取
- `{{#user.nickname}}` 從 ApplicationContext 取得（mzt-biz-log 的 SpEL 求值機制）
- `success="..."` 是方法成功時的訊息範本；亦可指定 `fail="..."` 給失敗時用

這代表業務 Service 只需要關心「我想記錄什麼業務語意」，框架負責拼字串、攔截、寫入。

### 7.4 查詢只能依固定欄位

查詢條件固定 6 個：使用者編號、業務編號、模組類型、操作名、操作明細、建立時間。**未提供**：

- 依 TraceId 反查（要查同一請求的所有日誌目前不行，列入 §11）
- 依使用者 IP 篩選
- 依請求網址篩選
- 依模組類型多選

### 7.5 匯出時強制不分頁

匯出端點把 `pageSize` 設為 `PAGE_SIZE_NONE`，意即無視前端傳的分頁大小、撈全部符合條件的紀錄。當資料量大時：

- 記憶體：整批載入 + 翻譯 → 可能 OOM
- 回應時間：完成前端會卡住

實務上若篩選條件過寬（例如「2026 年全年」），可能造成系統不可用（列入 §11）。

### 7.6 翻譯機制

`OperateLogRespVO.userId` 掛了翻譯註解，告訴翻譯框架：

- 用 `userId` 去 `AdminUserDO` 表查
- 取出 `nickname` 欄位
- 寫到 `userName` 欄位

這是查詢／匯出時做的「補資料」，不影響存儲。日後若使用者主檔搬遷至 OA，此翻譯來源替換即可。

### 7.7 操作日誌**不**通過 BPM 簽核

本模組屬於系統管理裡的純查詢與匯出功能，新增、修改、刪除日誌的端點**完全不存在**（只有透過框架攔截的隱性寫入）。資料不可由 UI 編輯，也不可由 UI 刪除（資料保存於資料庫的軟刪除欄位機制以外不再說明）。

### 7.8 「使用者類型」決定本日誌屬於哪個使用者體系

ERP 同時有後台使用者（admin）與會員使用者（member）兩種體系。`userType` 欄位區分本日誌的 `userId` 應該對到哪一張使用者表。本系統主要為後台使用，會員體系預計過渡期內不啟用。

## 10. 功能範圍

### 10.1 包含的功能

- 業務操作的自動寫入（透過 `@LogRecord` 註解，由 mzt-biz-log 框架攔截）
- 異步寫入（不阻塞業務主流程）
- 操作日誌分頁查詢（6 個固定條件）
- 操作日誌 Excel 匯出
- 翻譯機制（使用者編號 → 暱稱）
- 匯出動作本身被存取日誌記錄

### 10.2 預留但尚未實作

- **依 TraceId 反查**：欄位有，但查詢條件未提供。要查「同一請求的所有日誌」目前要直接看資料庫。
- **依 IP / 請求網址查詢**：欄位有，查詢條件未提供。
- **多選模組類型**：目前只支援單一模糊比對。
- **DTO 驗證生效**：`OperateLogCreateReqDTO` 上的 `@NotNull` / `@NotEmpty` 在 Service 端未被 `@Valid` 觸發，必填欄位失守。

### 10.3 不包含

- 登入日誌（屬另一份功能：登入日誌）
- 存取日誌 / API 訪問日誌（由 `@ApiAccessLog` 觸發、存於另一張表，本 PRD 不涵蓋）
- 業務 Service 上的 `@LogRecord` 註解使用規範（屬開發手冊，不屬 PRD 範疇）
- 對日誌的修改／刪除（不提供，符合稽核原則）
- 日誌歸檔／清理／長期保存策略（推論：目前無排程清理，列入 §11）
