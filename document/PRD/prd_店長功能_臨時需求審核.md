# PRD｜店長功能 — 臨時需求審核

> 來源：逆向掃描 `kingmaker-module-pdm` 之 `tempreq`（`TempReqController` / `TempReqServiceImpl`）。
>
> 後端**目前無店長視角專屬 endpoint**，但底層 service 已有「依登入者區域／門市自動過濾」邏輯（推論可被店長視角共用）。既有 API 為總部／行銷／簽核者共用版本（見底層 PRD [prd_需求集合_臨時需求審核.md](prd_需求集合_臨時需求審核.md)）。本文件以「店長能用什麼／應該看到什麼」視角撰寫，做為 PM 對齊、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

常規需求預測（食材／物料試算表，見兄弟 PRD）跑的是「依過去銷量推估的常態用量」，但門市常碰到突發狀況：

> 「2026-05-19 下午 3 點 — 接到電話，明天 5/20 中午有 200 人團體訂位、要 200 個華堡套餐，正常週訂的牛肉餅與麵包都不夠了，要緊急加單。」

這時候我就要建一張「臨時需求單」說明：要加哪個餐點、加多少份、什麼時候要到貨、為什麼。送出後由區經理或採購主管簽核，核可後會被下游的「物料需求預測試算表」自動納入合併（見兄弟 PRD），最終轉成採購單派車送來。

### 1.2 我要做什麼

- 突發狀況發生 → 立刻發起一張臨時需求單，以「**單品**」（華堡、薯條、雞塊）為單位填要加的份數
- 系統會自動展開該單品對應的食材清單與用量試算給簽核者看（多少牛肉餅、多少麵包…）
- 送出進入 BPM 簽核流程 → 區經理或採購主管在「待簽分頁」收到
- 簽核期間在「我發起的單」追進度，必要時打電話催
- 已歸檔的單視為定案、無法再改
- 多筆同活動 → 一張單可同時放多個單品

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 緊急狀況要能立刻發起加單 | 走完整週訂流程來不及 |
| 用「華堡 200 個」這種業務語講需求 | 我不會講「牛肉餅 200 片＋麵包 200 個＋生菜 10 公斤」 |
| 系統幫我展開食材清單 | 讓簽核者知道影響範圍、好做判斷 |
| 一張單放多個單品 | 同活動可能同時要加華堡、雞塊、薯條 |
| 必須走簽核 | 我不能自由加單，要主管把關 |
| 已歸檔的不能改 | 否則下游採購會跟著錯 |
| 只看自己門市的單 | 我不需要看其他店的活動加單 |
| 簽核狀態看得到 | 緊急時要主動催 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 一鍵發起臨時需求單 | 緊急場景的最低門檻入口 |
| 以單品為單位的明細填寫 + 自動展開食材試算 | 申請者與簽核者用同一張單對齊認知 |
| BPM 簽核流程 | 區經理／採購主管把關 |
| 我發起的單一覽 | 追進度 |
| 編輯（未歸檔）／作廢（推論） | 弄錯了還能改 |
| 自店資料過濾 | 不要被其他店資料淹沒 |
| 行動裝置可發起 | 突發場景常在賣場現場 |
| 簽核者推播通知（待補） | 緊急場景需主管小時級回應 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 臨時需求審核（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合」 |
| 兄弟功能 | 食材需求預測試算表（店長視角）、物料需求預測試算表（店長視角） |
| 主要頁面 | 新增臨時需求單、我發起的單一覽、單據詳情（含食材展開） |
| 簽核流程 | 有（既有 BPM 流程已綁定） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在突發狀況下，**在門市現場（含行動裝置）以最低門檻發起一張「緊急加單」**，把申請理由、單品、份數送上簽核流程，並讓主管能在最短時間內看到「這筆加單會多耗多少食材、影響哪些採購」。

設計理念：**店長發起、主管把關、系統自動展開影響範圍**。店長不能自由加單（必須簽核），但發起門檻夠低（單品 + 數量即可）。

## 3. 業務邏輯背景

### 3.1 「單品 vs 食材」雙層結構

| 層 | 誰填／看 |
|---|---|
| 單品層 | 店長填：要加哪個餐點（華堡、薯條…）、加多少份 |
| 食材層 | 系統自動展開：每個單品對應的食材清單與每食材的用量試算（標準用量 × 份數）|

> 為什麼這樣設計？因為店長與行銷的語彙是「200 個華堡」，但採購與簽核者要評估的是「200 個華堡會多耗 200 片牛肉餅、200 個麵包、10 公斤生菜」。

### 3.2 食材展開的計算

```
每食材的臨時需求量 = 該食材的標準用量 × 該單品的申請數量（四捨五入到小數第 2 位）
```

例：華堡的標準用量「生菜 0.05 kg」、申請 200 個 → 臨時需求量 = 0.05 × 200 = 10.00 kg

> 注意：底層程式有「臨時需求計量」分支邏輯目前被註解，所以最終數量永遠等於上式（推論不影響店長視角使用，但下游採購可能需要釐清）。

### 3.3 一張單一店

每張臨時需求單只能指定一個門市（單頭欄位）。**店長視角下，門市應自動鎖為自店**。若行銷活動跨多店，要分別建多張單（推論：建議從 PM 角度確認此設計是否合理）。

### 3.4 簽核流程

1. 店長提交 → 流程狀態「待處理」
2. 系統若已綁定簽核流程 → 自動發起 BPM 流程實例，分派給區經理／採購主管
3. 主管在「待簽分頁」看到 → 點開查看單頭 + 食材展開
4. 主管核准 → 狀態推進到「待簽核」（推論：可能還有多階）→ 最終「已歸檔」
5. 「已歸檔」即定案，店長不能再改
6. 已核可的單會被下游「物料需求預測試算表（非 BOM）」自動撈出，加進當期物料合併試算

### 3.5 建單後的明細補強

提交時，系統會幫每筆明細補上：

- 單品品號（從食譜主檔對照）
- 食材品號（從食材規格主檔對照）

**對照不到的明細會被靜默跳過**（底層既有行為）。例如剛建立、品號還沒設好的單品，可能整筆從明細裡消失（店長視角看不到任何錯誤）。店長視角**建議改為顯示警示**（見 §11）。

### 3.6 必填欄位（後端未驗證，待補）

底層 VO **沒有任何必填驗證**，全靠前端把關。店長視角應強制：

- 申請單位（推論：自動帶「自店」）
- 門市（自動帶自店）
- 需求週起訖（必填）
- 主旨（必填，例「5/20 中午 200 人團訂」）
- 明細至少 1 筆（必填）

### 3.7 已歸檔保護

「已歸檔」單據不能修改，否則拋「已歸檔，禁止更新」。

> 注意：底層判斷邏輯有點怪（同時要求「無流程實例 ID」+「已歸檔」才擋），實際情境下大部分單據都有流程實例 ID，可能擋不到。見 §11。

### 3.8 資料權限自動過濾

底層 service 已有「依登入者區域／門市自動過濾」：登入者若有門市權限且未指定門市，會強制套用為登入者門市。**店長視角應確認此鎖店行為被正確套用**（待確認）。

## 4. 情境說明

### 4.1 正常流程 — 接到團體訂位緊急加單

5/19（一）下午 3:05，信義店店長小陳接到電話：「明天中午有 200 人團體訂位、要 200 份華堡套餐」。小陳開手機 App：

1. 點「新增臨時需求單」
2. 系統自動帶入：
   - 申請單位：信義店
   - 門市：信義店（鎖定，不可改）
   - 需求週起訖：本週（推論）
3. 主旨：「5/20 中午 200 人團訂華堡套餐」
4. 明細點「加單品」 → 從下拉選「華堡」 → 填「200」
5. 系統即時試算下方顯示：
   - 華堡 200 份
     - 牛肉餅 → 臨時需求 200（個）
     - 麵包 → 臨時需求 200（個）
     - 生菜 → 臨時需求 10.00（kg）
     - 起司 → 臨時需求 200（個）
6. 小陳檢視 → 點「提交」
7. 系統建單、啟動 BPM 流程 → 區經理收到推播（推論，待補）
8. 區經理 15 分鐘後核准 → 流程進到「待簽核 → 已歸檔」
9. 隔天 5/20 早上配送車已多帶 200 份食材

### 4.2 典型業務 — 一張單多單品

行銷活動「下週六買一送一」同時涵蓋華堡、薯條、雞塊：

1. 店長建一張單
2. 主旨：「5/30 板橋分店買一送一活動」
3. 明細：
   - 華堡 200
   - 薯條 150
   - 雞塊 100
4. 系統自動展開所有食材試算給簽核者看

### 4.3 異常情境 — 簽核期間發現填錯

小陳剛提交發現華堡份數該是 250 不是 200：

1. 進「我發起的單」找到該單，狀態「待處理」（流程剛啟動）
2. 點「編輯」 → 改華堡為 250
3. 系統用「刪舊插新」策略重新寫明細（底層既有行為）
4. 區經理刷新看到的是新數字（推論）

若狀態已是「已歸檔」 → 不能改，只能另外建一張補單

### 4.4 異常情境 — 系統靜默丟失明細

小陳選了一個新進的「特餐 Z」單品，這品的食譜還沒維護完整、品號還沒設好：

1. 小陳填 50
2. 提交後系統補品號 → 對照不到
3. **該明細靜默被丟棄**，沒有任何錯誤訊息
4. 簽核者看到的單只有其他明細，「特餐 Z」整行消失
5. 小陳完全不知道、可能還以為被核准了

> 風險高。店長視角強烈建議補錯誤訊息（見 §11）。

### 4.5 規則分流 — 已歸檔不能改

兩天前提的單已經跑完流程到「已歸檔」：

- 進編輯 → 拋「已歸檔，禁止更新」
- 唯一選項：再建新單

### 4.6 跨日對帳 — 我發起的單一覽

小陳每週一早上看一次：

- 上週發起 3 張，全部「已歸檔」
- 本週 1 張「待簽核」 → 看流程節點卡在「採購主管」 → 打電話催

### 4.7 異常情境 — 主管退回

區經理收到單後覺得「200 份太多、改 150」：

1. 在 BPM 流程點「退回」並寫退回原因
2. 小陳收到通知（推論）
3. 小陳改明細後重新提交
4. 流程重跑

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「臨時需求審核」]
  ├─ 入口 1：新增臨時需求單
  │    ├─ 1. 系統自動帶：申請單位、區域、門市（鎖店）
  │    ├─ 2. 填：需求週起訖、主旨
  │    ├─ 3. 明細：加單品（從下拉選）+ 申請數量
  │    │    └─ 即時試算：展開該單品對應的食材清單與用量
  │    ├─ 4. 提交 → 後端：
  │    │    ├─ 產生簽核單號
  │    │    ├─ 流程狀態 = 待處理
  │    │    ├─ 建單頭、批次建明細（單品品號／食材品號對照不到者靜默跳過）
  │    │    └─ 啟動 BPM 流程（若選單綁定）
  │    └─ 5. 回單號、跳到詳情頁
  │
  ├─ 入口 2：我發起的單一覽
  │    ├─ 1. 自動過濾：自店、本人
  │    ├─ 2. 列表：單據編號、主旨、需求週、流程狀態、建立時間
  │    └─ 3. 點某張 → 詳情頁（含食材展開）
  │
  ├─ 入口 3：詳情頁
  │    ├─ 單頭資訊
  │    ├─ 單品 → 食材 三層展開
  │    ├─ 簽核流程節點視覺化（推論）
  │    └─ 操作：編輯（未歸檔）／作廢（推論）／匯出
  │
  └─ 入口 4：（多店店長／代理主管）待簽分頁
       ├─ 1. 取「分派給我的」流程實例 ID
       └─ 2. 列出待我處理的單
```

> **自店過濾**：底層 service 已有此邏輯（且明確支援 storeId），店長視角實作上應較直接，但目前後端仍走 `controller/admin/*`，未驗證店長能否被鎖到自店。

## 6. 欄位規格

### 6.1 單頭（店長視角輸入）

| 欄位 | 店長視角預設 | 可否修改 |
|---|---|---|
| 申請單位 | 自店店名 | 否（推論） |
| 區域 | 自店所屬區域 | 否 |
| 門市 | 自店 | 否 |
| 需求週起訖 | 當週（推論） | 是 |
| 主旨 | 由店長填 | 是 |

### 6.2 明細（店長視角輸入）

| 欄位 | 說明 |
|---|---|
| 單品 | 從食譜下拉選（華堡、薯條、雞塊…） |
| 申請臨時需求量 | 整數（份數／個數） |

### 6.3 即時試算結果（系統自動展開）

每選一個單品 + 填數量，系統即時顯示：

| 欄位 | 說明 |
|---|---|
| 食材名稱 | 該單品對應食材 |
| 標準用量 | 1 份單品用多少 |
| 臨時需求量 | 標準用量 × 申請數量（系統算） |

> 注意：底層的「臨時需求計量」分支邏輯被註解，目前僅算「計數」一條路徑。店長視角看到的是這條路徑的結果。

### 6.4 我發起的單一覽（單頭）

| 欄位 | 說明 |
|---|---|
| 單據編號 | 系統產生 |
| 簽核單號 | 系統產生 |
| 主旨 | 我填的 |
| 需求週 | 起～訖 |
| 流程狀態 | 待處理／待簽核／已歸檔 |
| 建立時間 | 我提的時間 |

## 7. 商業邏輯

### 7.1 鎖店過濾（**底層支援，店長視角待驗證**）

底層 service 在分頁查詢時已會「自動套登入者門市」。店長視角應確保：

```
1. 新增單時：申請單位、區域、門市 自動帶為自店且不可改
2. 查詢時：自動過濾為自店且本人發起的單
3. 詳情頁：若單據不屬於自店或非本人發起 → 拒絕存取（推論）
```

### 7.2 必填驗證的店長視角補強

底層 VO 無必填驗證，店長視角應補：

- 主旨非空
- 需求週起訖非空且起 ≤ 訖
- 明細至少 1 筆
- 每筆明細的單品、申請數量非空

### 7.3 食材展開的計算

每食材臨時需求量 = 標準用量 × 申請數量，四捨五入到小數第 2 位。

### 7.4 已歸檔保護

任何單據一旦狀態為「已歸檔」 → 編輯／刪除一律拒絕，拋「已歸檔，禁止更新」。

> 底層判斷有缺陷（需同時「無流程實例 ID」+「已歸檔」才擋），實務上有流程實例 ID 的歸檔單可能擋不到。建議從店長視角補一層保護（見 §11）。

### 7.5 對照不到的明細處理

底層既有行為是「單品品號或食材品號對照不到 → 該筆明細靜默跳過」。店長視角建議改為：

```
1. 仍寫入明細，但標記「主檔不完整」
2. 提交時顯示警示彈窗：「以下單品的食譜尚未維護完整，可能影響採購：[單品名稱清單]，是否仍要送出？」
3. 給店長選擇繼續送或取消
```

### 7.6 編輯採「刪舊插新」

底層更新邏輯是「刪掉所有舊明細、重新插入新明細」。店長視角影響：

- 編輯後舊明細 ID 變動 → 若有外部引用會壞（推論：對店長視角影響小）
- 明細歷史軌跡丟失（如建立時間、建立者）

### 7.7 唯讀範圍

店長不應有「食譜維護」「食材主檔維護」「簽核流程設定」按鈕。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店且本人發起的單 | 新增、編輯（未歸檔）、查詢、追蹤 |
| 區經理（簽核者） | 區內所有店的單 | 同店長，加待簽分頁簽核 |
| 採購主管（簽核者） | 全公司 | 待簽分頁簽核、查詢、匯出 |
| 行銷部（申請者） | 自身發起的單 | 與店長類似（屬「申請者」角色） |

> 底層權限字串前綴與 PDM 其他模組不一致（屬底層問題）。店長視角應一律具備「建立／編輯／查詢」對應權限。

## 10. 功能範圍

### 10.1 包含的功能

- 自店發起臨時需求單（單頭 + 單品明細）
- 自動展開食材試算（單品 → 食材 → 臨時需求量）
- BPM 簽核流程綁定
- 我發起的單一覽（自動過濾為自店、本人）
- 詳情頁含食材展開
- 編輯（未歸檔）
- 已歸檔保護
- 與物料需求預測（非 BOM）下游對接

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端僅有 `controller/admin/tempreq/*`，無 `/app/*` 對應路徑 |
| 自店過濾的店長視角驗證 | 共用 API 已有過濾邏輯，未針對店長路徑驗證 |
| 必填驗證 | 後端 VO 無 `@NotNull`／`@NotEmpty` |
| 對照不到的明細警示 | 目前靜默跳過 |
| 「臨時需求計量」分支 | 程式碼註解掉，最終數量等同計數 |
| 行動裝置 UI | 推論：店長使用情境主要在門市現場 |
| 推播通知簽核者 | 後端無推播管道實作 |
| 取消／作廢臨時需求 | 後端僅有刪除（無「作廢」狀態） |
| 自動批准門檻（如 < 某金額自動過） | 後端無此分流 |
| 同活動跨店 | 一張單僅能一個門市 |
| 與門市調撥的優先序提示 | 後端無此整合 |

### 10.3 不包含

- 食譜（BOM）維護（屬於單品維護作業）
- 食材主檔維護（屬於食材維護作業）
- 簽核流程節點設定（屬於 BPM 模組）
- 物料需求合併（屬於兄弟 PRD：物料需求預測試算表）
- 採購單建立（屬於採購管理；本單是上游輸入）
- 入庫管理（屬於庫存管理）
- 簽核者視角的「待簽分頁」（本 PRD 以申請者視角為主，待簽分頁為其延伸）

## 11. 待確認事項

| 議題 | 為何要確認 |
|---|---|
| 「登入者 → 所屬門市」對應如何取得？是否已有單一門市鎖定？多店店長如何切換？ | 鎖店實作前提 |
| 簽核者層級（區經理 → 採購主管 → 副總？）與分派規則 | BPM 設計 |
| 簽核 SLA（如 30 分鐘內必須回應）是否要設定？ | 緊急場景的服務時效 |
| 是否有「自動批准門檻」（如 < NT$1000 自動過、< 50 份自動過）？ | 業務政策 |
| 推播管道（漢堡王是否已有店長 App／簡訊平台）？ | 跨系統 |
| 是否要支援「取消／作廢」臨時需求？目前後端只有「刪除」 | UX |
| 「臨時需求計量」分支邏輯被註解 — 是否應啟用？對店長視角的數字呈現會否變？ | 計算邏輯 |
| 單品品號／食材品號對照不到時靜默跳過 — 店長視角是否要顯示警示？ | 漏單風險 |
| 申請單位欄位是否要字典化（避免「行銷部／行銷／行銷企劃」並存）？ | 主檔 |
| 「申請臨時需求量」目前只支援整數，無法填半份（如 0.5 個套餐）— 是否合理？ | 業務 |
| 一張單只能對應一個門市 — 跨店活動如何處理？ | 業務流程 |
| 同主旨重複建單檢查（避免行銷重複建）是否需要？ | UX |
| 必填欄位（主旨、需求週、明細）後端無驗證 — 店長視角是否強制前端把關？ | 資料品質 |
| 已歸檔保護的判斷邏輯有缺陷（同時要求無流程實例 ID）— 是否要在店長視角加強？ | 保護機制 |
| 編輯採「刪舊插新」會丟失明細歷史 — 對審計影響大嗎？ | 審計 |
| 簽核退回後店長收到的通知管道？ | 通知機制 |
| 與「鄰店調撥」（庫存模組）的優先順序：先撥後採？或讓店長選？ | 業務規則 |
| 緊急採購是否走特殊廠商／加價單價？ | 採購規則 |
| 高頻發臨時需求的店長是否會被預警（預測能力差）？ | 績效 |
| 失敗／廠商無法供貨如何處理？ | 例外覆蓋 |
| 行動裝置 / PWA 是否為必需？ | 開發範疇 |
