## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員。
ERP 系統內有幾十個功能頁面（採購單、請購單、庫存查詢、報表⋯），各部門看得到、按得動哪些功能由「角色」決定。我負責配權：採購主管能看採購相關功能、門市店長能看門市相關功能、財會只能看報表⋯。當公司新增一個角色（例如「區域督導」）或某個既有角色要調整可看到的功能時，我要進這個畫面打勾／取消勾。

### 1.2 我要做什麼

- **每週**：依各部門主管請求調整角色可看到的功能（如「請把採購助理也能看到請購單」）。
- **新增角色之後**：建好角色本身只是空殼，必須立刻進「選單權限」勾選該角色能看哪些頁面與按鈕，否則該角色登入後是一片空白。
- **新功能上線時**：新增的選單預設沒有任何角色掛載，要手動把它加進對應角色的權限樹中。
- **稽核前**：核對每個角色目前的選單權限是否與權限政策一致。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 我希望一次看清楚這個角色目前能看到哪些頁面、按鈕 | 不能讓使用者「看得到頁面但按鈕按不動」之類的破碎權限 |
| 我希望以樹狀勾選介面操作，不要逐筆設定 | 系統可能有數十甚至上百個選單，逐筆操作不可行 |
| 我希望儲存後立即生效 | 主管臨時請求，等隔天就太慢 |
| 我希望儲存時系統只動「有差異」的部分 | 避免每次都把整張表刪掉重建影響歷史紀錄 |
| 我希望錯點儲存時不會把整批權限弄亂 | 必須是「比對差異後增量更新」而非「先全砍再寫」 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 取得指定角色目前已掛載的選單清單 | 進畫面後可預先勾選出現況 |
| 提交一組選單清單覆寫該角色之權限 | 一次儲存所有調整 |
| 儲存時系統自動計算差集，只動有變動的部分 | 避免不必要的資料庫操作、保留未變動的歷史記錄 |
| 儲存後同步清除權限快取，下次驗權立刻生效 | 不需重啟、不需等快取過期 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 設定角色選單權限 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 角色管理（增刪改）、設定角色資料權限、設定使用者角色、選單管理、部門管理 |
| 主要頁面 | 角色管理列表頁中，每個角色列上之「選單權限」動作；點開為樹狀勾選彈窗 |
| 簽核流程 | 無 |

## 2. 功能目的

讓系統管理員為**指定角色**勾選一組「該角色可看到的選單／按鈕」，作為前端選單顯示與後端 API 權限判斷的依據。設計上強調：

1. **整批提交、增量更新** — 使用者一次送出選單清單，伺服端與資料庫現況比對差集，只插入新增的、只刪除取消的，未變動的不動。
2. **多租戶模式下自動過濾** — 若該租戶未開通某個選單，即使前端誤傳也會被擋下。
3. **儲存後快取立即失效** — 確保下一次驗權即用新權限。

> 經系統整合決策（2026-05-16），漢堡王 IT 走向以 Laravel OA 為使用者單一事實來源、ERP 不再自簽 JWT。**「角色—選單」對應關係長期將由 OA 或新權限模型統一管理**，本模組屬於 ERP 過渡期保留功能；長期規劃中將在搬遷階段 5 後廢棄。本 PRD 描述的是過渡期內、ERP 仍自管角色—選單關聯時的需求。

## 3. 業務邏輯背景

### 3.1 選單分三種層級

ERP 內的「選單」並非單純的左側導航項目，而是含三類：

- **目錄** — 父節點，用於分組（如「採購管理」這個分類）
- **頁面** — 真正可點擊進入的功能頁（如「請購單管理」）
- **按鈕** — 頁面內的特定動作（如「新增請購單」「匯出」「審核」按鈕）

三者統一以「選單」抽象，**設定角色選單權限時是同一棵樹一起勾**。沒勾的按鈕在前端不會顯示、後端 API 也會擋下。

### 3.2 多租戶下的選單可見性

若系統以多租戶模式運作，每個租戶會被綁定一個「租戶套餐」，套餐中規範該租戶**可以使用哪些選單**。設定角色選單權限時：

- 系統租戶（內建總管）→ 可看到全系統選單
- 一般租戶 → 只能看到該租戶套餐內的選單

提交時若使用者勾了「該租戶未開通的選單」，伺服端會自動把那些選單從清單中移除再儲存，**不會擋下整次提交**。

### 3.3 增量更新而非全砍重寫

儲存時系統的處理方式：

1. 撈出該角色目前已掛載的選單編號集合
2. 比對提交的新清單
3. 算出「新增的」「移除的」「不動的」三組
4. 只對新增與移除執行資料庫操作

此設計的好處：

- 未變動的關聯保留其原始建立時間／建立人（推論）
- 減少資料庫寫入量
- 若有觸發器或事件監聽，僅針對實際變動的關聯觸發

### 3.4 超級管理員的特殊處理

當查詢某個角色目前的選單清單時，若該角色屬於「超級管理員」類型，系統會**忽略資料庫中的角色—選單關聯表**，直接回傳系統內所有選單作為其權限。換言之：

- 超級管理員的權限不是靠勾選維護，是系統內建的「全部」
- 嘗試對超級管理員「取消勾選某選單」也不會真的限縮其權限

§11 列為待確認：使用者介面是否要禁止對超級管理員觸發此功能。

### 3.5 儲存後清除選單—角色關聯快取

每次儲存角色選單權限後，系統會**清空所有「選單對應角色」的快取**（不是只清這個角色的）。原因是該快取以「選單」為 key、值是「擁有此選單的角色集合」，當某角色的選單變動時，受影響的選單清單只能整批清掉重建。

## 4. 情境說明

### 4.1 正常流程 — 為新角色「區域督導」配權

5/18 公司新增「區域督導」角色。阿明先在角色管理頁建好角色（此時該角色登入後是空白頁）。接著在該角色列上點「選單權限」，彈出選單樹勾選介面。系統先查詢該角色目前的選單清單（回傳空集合），所以樹上沒有任何項目被勾。

阿明依督導工作範圍勾選：

- 「採購管理」目錄
  - 「請購單管理」頁面（含查詢、檢視按鈕，不含新增與刪除）
- 「報表中心」目錄
  - 「採購週報」頁面（僅檢視）

按儲存。系統比對：原本 0 個，新表單 7 個（兩個目錄、兩個頁面、三個按鈕），全部執行新增。儲存成功，督導登入後即可看到上述功能。

### 4.2 正常流程 — 調整既有角色

「採購助理」角色原本不能看到請購單。採購主管請阿明加上。阿明點該角色「選單權限」，彈出時看到目前已勾選 12 項。他在「請購單管理」頁面與其 3 個按鈕（查詢、檢視、列印）打勾，按儲存。

系統比對：原本 12 個，新表單 16 個，差集為新增 4 個、移除 0 個、不動 12 個。資料庫只插入 4 筆，未動原本 12 筆。

### 4.3 異常情境 — 提交清單為空

阿明誤觸「清空全部」並儲存。系統比對：原本 16 個、新表單 0 個，差集為新增 0、移除 16。資料庫刪除該角色所有選單關聯。該角色下次登入會看到空白頁。

> §11 待確認：是否應在儲存前提示「您即將清除該角色所有選單權限，確定？」

### 4.4 多租戶情境 — 勾到租戶未開通的選單

「漢堡王台中分公司」是一個獨立租戶，僅開通了採購、庫存兩個套餐。阿明（系統管理員）誤把「PDM 模組」下的選單也勾進該租戶下的角色。儲存時：

- 系統取得該租戶套餐允許的選單清單
- 把不在清單內的 PDM 選單從提交資料中移除
- 用剩餘的選單進行差集計算與儲存

儲存成功，但 PDM 選單實際上未掛載。阿明可能不會察覺直到使用者反應「我看不到 PDM」。

> §11 待確認：是否要回傳「已忽略 X 個非本租戶選單」之訊息。

### 4.5 邊界情境 — 對超級管理員設定選單

阿明手滑點到「超級管理員」角色的「選單權限」按鈕。系統開啟勾選樹時直接顯示「全部選單均已勾選」（不論資料庫中是否有對應的角色—選單關聯）。阿明把幾個選單取消勾選並儲存。系統執行差集儲存（從「全部」減去使用者取消的部分）。

但超級管理員下次登入仍可看到所有選單，因為**查詢階段該角色直接回傳全選單**而忽略資料庫關聯（見 §3.4）。視覺上似乎被修改、實際無效，這是**已知設計**而非 bug。

### 4.6 自動處理情境 — 快取失效

阿明儲存任一角色的選單變更後，使用者下次登入或操作即時生效，無需等待。

## 5. 操作流程

### 5.1 進入畫面：取得該角色目前的選單清單

```
[使用者點某角色之「選單權限」按鈕]
  │
  ▼
[權限檢查：是否具備「系統-權限-設定角色選單」]
  │
  ▼
[依角色編號查詢]
  ├─ 若該角色是超級管理員 → 回傳系統內所有選單編號
  └─ 否則 → 從角色-選單關聯表撈出該角色掛載之選單編號集合
  │
  ▼
[回傳選單編號集合，前端據以勾選樹]
```

### 5.2 儲存：賦予角色一組選單

```
[使用者勾選完成 → 按儲存]
  │
  ▼
[權限檢查：是否具備「系統-權限-設定角色選單」]
  │
  ▼
[多租戶過濾]
  ├─ 取得該租戶允許之選單編號集合
  └─ 把提交清單中不在允許範圍的選單移除
  │
  ▼
[撈出該角色資料庫中目前掛載之選單編號集合]
  │
  ▼
[計算差集]
  ├─ 新表單有、資料庫沒有 → 待新增
  ├─ 資料庫有、新表單沒有 → 待移除
  └─ 兩邊都有 → 不動
  │
  ▼
[批次新增「待新增」之關聯]
  │
  ▼
[批次刪除「待移除」之關聯]
  │
  ▼
[清空「選單→擁有此選單的角色集合」之全域快取]
  │
  ▼
[回傳成功]
```

## 7. 商業邏輯

### 7.1 取得角色選單時的超級管理員特例

當欲取得的角色之中包含「超級管理員」類型時，系統不查角色—選單關聯表，直接回傳「系統內所有選單編號」。此設計反映「超級管理員享有全部選單，不需要、也不應該被個別勾選限制」之假設。（來源：`PermissionServiceImpl.java:185-196`）

### 7.2 多租戶過濾的時機

多租戶過濾發生在**進入服務層之前**（在 Controller 內執行），且採用「直接修改原請求物件」的方式：

- 系統取得該租戶允許的選單清單
- 把提交清單中不在允許範圍的選單**從原請求中移除**
- 服務層只看到過濾後的清單

此設計意味著：

- 服務層不知道「使用者實際勾了什麼，但有 N 個被擋下」
- 不會拋錯、也不會在回應中告知使用者哪些被忽略

（來源：`PermissionController.java:51-52`）

### 7.3 增量更新邏輯（核心）

```
新增集合 = 新表單 - 資料庫
移除集合 = 資料庫 - 新表單
不動集合 = 新表單 ∩ 資料庫
```

只對「新增集合」執行批次插入、只對「移除集合」執行批次刪除。

「不動集合」對應的關聯紀錄完全不動 — 不會被 update、不會被 delete-then-insert。這保留了原始的建立時間／建立人，且減少對其他相依資料（如審計日誌）的干擾。（來源：`PermissionServiceImpl.java:142-161`）

### 7.4 空集合提交 = 清除該角色所有選單

若使用者送出空集合，差集計算結果為「新增 0、移除全部」，資料庫會把該角色的選單全部刪除。**伺服端不會擋下空集合**，亦無提示。（來源：同上）

### 7.5 不檢查選單之啟用狀態

提交的選單清單未檢查每個選單是否為「啟用」狀態。理論上可把「已停用」的選單也掛到角色上（推論）。當該選單後續被重新啟用時，掛載即生效。

### 7.6 不檢查角色之啟用狀態與類型限制

`assign-role-menu` 不檢查目標角色是否為「啟用」狀態、不檢查是否為「系統內建角色」。對內建角色（含超級管理員）操作不會拋錯，但因 §7.1 的特例，對超級管理員的設定實質上**儲存了卻無作用**。

### 7.7 快取失效範圍

儲存後清除的是「選單→擁有該選單的角色集合」之快取，且是**清空整個快取空間**（allEntries），而非僅清相關鍵。原因是該快取以選單為 key，無法只清「受本次變動影響的選單」對應的鍵。

### 7.8 交易範圍

`assignRoleMenu` 方法**未明確標註 `@Transactional`**（推論：依賴上層或非交易執行），意味著若「批次新增」成功、「批次刪除」失敗，可能出現「新權限已加上、舊權限未清除」之中間狀態。§11 待確認此風險。

## 8. 使用角色與權限

| 角色 | 可看／可操作 |
|---|---|
| **具備「系統-權限-設定角色選單」權限者**（通常為系統管理員） | 可查詢任一角色之選單清單；可對任一角色提交新的選單清單 |
| **僅具備「角色查詢」權限者** | 可看到角色列表，但看不到／無法觸發「選單權限」動作 |
| **未配發系統管理權限者** | 看不到系統管理選單 |

查詢與儲存共用**同一個權限字串**，無法分開（例如不能只給「查詢權限」而不給「修改權限」）。§11 待確認此設計是否符合稽核需求。

## 9. 畫面需求／視覺規範

後端僅提供 API。預期前端在角色管理列表頁的每列操作區提供：

- 「選單權限」動作按鈕
- 點擊後彈出選單樹勾選介面
  - 樹形結構（目錄 → 頁面 → 按鈕三層）
  - 預先勾選該角色目前掛載之節點
  - 父節點勾選邏輯（推論：勾父應自動勾子、取消子應級聯處理）
  - 儲存／取消按鈕
- 儲存後顯示成功提示
- 對「超級管理員」角色可考慮直接停用此按鈕並提示「超級管理員享有全部選單，無需設定」（§11 #5）

## 10. 功能範圍

### 10.1 包含的功能

- 查詢指定角色目前掛載之選單編號集合
- 提交一組選單清單覆寫該角色之選單權限（採差集增量更新）
- 多租戶下自動過濾非本租戶選單
- 儲存後清空選單—角色相關之全域快取

### 10.2 預留但尚未實作

- 多租戶過濾後**未通知使用者**哪些選單被忽略
- 對超級管理員之「儲存」實質無效（查詢階段被特例覆蓋）但仍會寫入資料庫，造成「設了等於沒設」的混淆

### 10.3 不包含

- 選單本身的維護（在「選單管理」功能）
- 角色本身的維護（在「角色管理」功能）
- 使用者—角色之掛載（在「設定使用者角色」功能）
- 角色之資料權限設定（在「設定角色資料權限」功能）
- 權限變更之歷史記錄（程式中無 `@LogRecord` 標註此動作）
