## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

每天系統會把當日需各門市的食材，組成一張「物流串接單（PR + 出貨時間戳）」，自動產出 CSV 並送到 MSS（中央倉／配送系統）。我需要知道：**「我這家店今天會收到什麼？以哪張串接單派車？實際到店多少？」**

### 1.2 我要做什麼

- 每日早班開店前：查當日「我的店」是否已被排入物流串接單
- 收貨前：確認物流串接單上「我的店」要收的品項與數量
- 收貨後：實際到店數量回填（或對照核帳）
- 異常時：標出哪些品項沒到／少到，做為通報維運／採購的依據

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 知道今天會收到哪些品項、多少量 | 預排點收人手、空間 |
| 知道是哪張串接單帶來的 | 與配送司機／客服對單 |
| 看到實際到店 vs 預計到店的差異 | 對帳、補單依據 |
| 只看自己店的資料 | 不要被全公司資料淹沒 |
| 行動裝置可看 | 在賣場現場操作 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「自店」物流串接單列表 | 一眼看到今天我的店要收幾張單 |
| 串接單明細展開 | 看當日具體品項／數量／廠商 |
| 實際到店數量欄位顯示 | 對帳、找差異 |
| 預設今日／可切換日期 | 跨日對帳 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 物流串接記錄作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合（PDM）」 |
| 兄弟功能 | 物流管理行事曆、物流規劃查詢、物流配送查詢、物流異動查詢 |
| 主要頁面 | 自店串接單列表、串接單明細 |
| 簽核流程 | 無 |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在不打擾總部與物流公司的前提下，**自助查出「今天我的店要從中央倉收到哪些東西、是哪張單派的、預計／實際到貨多少」**。在門市端建立「物流可視性」，避免漏收、錯收、無人在場簽收。

設計理念：**唯讀＋鎖店**。店長不主動建立串接單（串接單由排程每天 1 次自動產出），只負責看與比對。

## 3. 業務邏輯背景

### 3.1 「串接單」是怎麼產生的（與店長無關，但要知道）

每天由系統排程在固定時間執行一次：

1. 撈出「今天預計配送日」的需求預測明細，且該食材對應之物流類型為「配送（中央倉送）」
2. 依品號、廠商、店號、需求量組成一筆 CSV
3. 把 CSV 透過 SFTP 上傳到 MSS 中央倉系統
4. 同時在 ERP 的「物流串接單」寫入 1 張單頭 + N 張明細
5. 單頭固定：配送模式＝配送、配送廠商代號＝「MSS」、出貨日＝執行時間
6. 單據編號格式：`PR` + 執行時間（年月日時分秒）

⚠️ 上傳 CSV 與寫入 DB 不在同一交易：CSV 失敗時，DB 仍可能有資料；反之亦可能。店長視角看到的串接單，**只代表 ERP 已知，不代表 MSS 已收到**（待確認：是否該對店長隱藏未確認上傳成功的單）。

### 3.2 串接單的結構

- **單頭**：每天 1 張（同一執行批次），包含：單據編號、配送模式、配送廠商代號、出貨日
- **明細**：每張單頭多筆，含品項、廠商、需求門市、原始需求預估單號、實際到店數量、訂單類別等

> 重要：明細欄位中有「實際到店數量」，但**目前由系統把「需求量」直接帶入**（推論：實際到店資料尚未自 MSS 回傳；待確認）。

### 3.3 與其他「物流查詢」功能的差異（店長視角）

| 功能 | 看什麼 | 主要時機 |
|---|---|---|
| 物流規劃查詢 | 還沒派車前的「規劃量」 | 預備期 |
| 物流配送查詢 | 已派車、在配送中的車輛狀態 | 收貨當下 |
| 物流異動查詢 | 實際 vs 預計的差異 | 收貨後對帳 |
| **物流串接記錄（本功能）** | **ERP 已向 MSS 中央倉送出的批次紀錄** | **每日對帳、訊息軌跡** |

> 推論：對店長而言，本功能與「物流異動查詢」高度重疊，差別只在資料來源（本功能是「ERP→MSS 同步批次」、異動查詢是「車輛實際到店」）。建議 PM 確認是否真有兩個畫面。

### 3.4 「我的店」是怎麼判定的

明細表中有「門市 ID」與「門店編碼」欄位；店長視角應**強制過濾**為登入者所屬門市。

> 待確認：後端是否已建立「登入使用者 → 所屬門市」對應；目前掃描未發現店長視角專屬過濾邏輯。

## 4. 情境說明

### 4.1 正常流程 — 早班店長查當日到貨

5/19 早上 7:30，BK001 信義店店長小陳開店前打開「物流串接記錄」：

1. 系統預設今日（2026-05-19）
2. 自動過濾「我的店」
3. 顯示 1 張串接單：`PR20260519030000`（凌晨自動產出）
4. 點開明細，看到 4 個品項：牛肉餅 12 公斤、生菜 8 公斤、麵包 200 個、起司 5 公斤
5. 小陳安排 8:30 兩名同事點收

### 4.2 異常情境 — 該收沒收

11:00 牛肉餅還沒到，小陳重新進入：

1. 在串接單明細上看到牛肉餅明細仍存在
2. 但「實際到店數量」與「需求量」相同（系統預設帶值），**無法分辨已到 / 未到**
3. 小陳改用「物流異動查詢」或打電話給配送司機

> 待確認：本功能是否要顯示「MSS 是否已確認接收」狀態欄；目前資料表無此欄位。

### 4.3 跨日對帳情境

月初店長要核對上月帳：

1. 切到 4/1～4/30
2. 列出每日 1 張串接單
3. 點任一張看當日明細
4. Excel 匯出比對紙本簽收單

### 4.4 我的店今天沒被排入串接情境

某日該店無需求（門市公休／本日所有食材皆走直送非中央倉）：

1. 列表為空
2. 顯示「今日無中央倉配送」訊息
3. 店長不需處理

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「物流串接記錄」]
  ├─ 登入身份取得：店長、所屬門市 ID
  ├─ 預設帶入今日日期（可改）
  ├─ 查詢串接單列表（鎖店：只回傳含自店明細的單頭）
  │    └─ 列：單據編號 ｜ 出貨日 ｜ 配送模式 ｜ 廠商代號
  ├─ 點某張單 → 展開明細（再次鎖店）
  │    └─ 列：品號 ｜ 食材名 ｜ 廠商名 ｜ 需求量 ｜ 實際到店量 ｜ 訂單類別
  └─ 點「匯出」→ Excel 下載（僅自店明細）
```

> ⚠️ 「自店過濾」尚未在後端實作（待確認）。若直接沿用既有 API，會看到全公司資料。

## 6. 欄位規格

### 6.1 串接單（單頭）

| 欄位（業務名） | 說明 | 來源 |
|---|---|---|
| 單據編號 | `PR` + 年月日時分秒，例 `PR20260519030000` | 系統產生 |
| 配送模式 | 固定「配送」 | 系統產生 |
| 配送廠商代號 | 固定「MSS」（中央倉） | 系統產生 |
| 出貨日 | 排程執行當下時間 | 系統產生 |
| 建立時間 | 寫入 DB 時間 | 系統 |

### 6.2 串接單明細

| 欄位（業務名） | 說明 | 店長視角是否該顯示 |
|---|---|---|
| 配送模式 | 配送 | 否（永遠相同） |
| 品號 | 食材品號 | 是 |
| 食材名稱 | 從食材主檔帶 | 是 |
| 廠商代號 | 食材實際供應廠商 | 否（次要） |
| 廠商名稱 | 從廠商報價帶 | 是（顯示給店長辨識） |
| 實際到店數量 | 目前等於需求量（待確認） | 是 |
| 訂單類別 | 固定「R」（推論：常規） | 否 |
| 門店編碼／門市 ID | 鎖店欄位 | 否（用於過濾） |
| 門店名稱 | 顯示用 | 是（多店店長才需） |
| 需求預測預估單號／臨時需求預估單號 | 回溯來源 | 是（追查用） |
| 區域 ID／名稱 | 區域 | 否（店長無跨店） |
| 出貨日 | 同單頭 | 是 |

### 6.3 店長視角缺少的欄位（推論／待確認）

- 上傳 MSS 狀態（成功／失敗／重試中）
- MSS 回傳的實際出車時間、到店時間
- 點收人簽名／簽收時間

## 7. 商業邏輯

### 7.1 鎖店過濾規則（**未實作，待補**）

店長視角查詢時，應在明細層先過濾出**自店相關明細**，再反推單頭：

```
1. 取登入者門市 ID
2. 查「串接單明細」where 門市 ID = 自店
3. 由明細的「主表 ID」反查單頭
4. 回傳「該店在當日／當期所有串接單」與其明細
```

### 7.2 預設時間範圍（推論）

- 列表預設今日（出貨日 = 今天）
- 可改為「最近 7 日」「指定日期區間」

### 7.3 唯讀

店長視角**不應有**新增、修改、刪除按鈕。目前後端 API 開放這三項，但權限應只給總部 IT／物流規劃。

### 7.4 與「物流異動查詢」共用底層

該兩功能在後端為同一套 CRUD；差別僅在過濾預設與顯示欄位。建議前端用同一頁、依進入入口切換預設條件。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店相關明細＋其單頭 | 唯讀＋匯出（待實作） |
| 區經理 | 區內所有店 | 唯讀＋匯出（推論） |
| 採購／物流規劃（總部） | 全公司 | 全 CRUD（既有 API） |
| IT 維運 | 全公司 | 全 CRUD＋對帳 |

> 後端目前以「具備物流單查詢／編輯／匯出權限」的角色控管，未細分店長／總部。店長視角的鎖店過濾需另實作。

## 10. 功能範圍

### 10.1 包含的功能

- 自店串接單分頁查詢（待補鎖店）
- 串接單明細展開（含食材／廠商／門店名稱關聯）
- 日期過濾
- Excel 匯出（推論：沿用既有，需限制為自店）

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端無 `/app/*` 對應路徑 |
| 自店過濾 | 既有 API 無門市條件 |
| 「實際到店數量」由 MSS 回傳 | 目前系統把需求量直接寫入 |
| 上傳 MSS 狀態欄位 | 單頭無此欄位 |
| 失敗重發按鈕 | 後端無對應 API |

### 10.3 不包含

- 串接單建立（由系統排程產出，店長不操作）
- 修改串接單（總部專屬）
- 與 MSS 端的回傳對接（外部系統）
- 物流配送車輛追蹤（屬「物流配送查詢」）

## 11. 待確認事項

| 議題 | 為何要確認 |
|---|---|
| **店長真的需要看這份報表嗎？** Excel 將其歸類為店長功能，但本質是「ERP→MSS 同步軌跡」，對店長用處有限 | 範圍確認 |
| 店長視角與「物流異動查詢」的差別 — 兩者均涉及實際到店數量 | 避免重複功能 |
| 登入者門市的判定方式（單一門市？多店店長？區經理？） | 鎖店實作 |
| 「實際到店數量」目前是系統預填需求量；店長能否手動修正？或要由 MSS 回傳？ | 資料正確性 |
| 是否需顯示「上傳 MSS 成功 / 失敗」狀態 | 異常排查能力 |
| 是否需「失敗重發」按鈕（推論：應為總部權限，不開放店長） | 權限切分 |
| 行動裝置／PWA 是否為必需 | 開發範疇 |
| Excel 匯出是否需限制只能匯自店 | 資安 |
| 排程執行時間（目前為固定每日 1 次）是否固定，店長能否信任「早上一定查得到」 | 操作節奏 |
| 公休日 / 多批次 / 補單情境如何呈現 | 例外覆蓋 |
