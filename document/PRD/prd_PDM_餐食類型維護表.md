## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣總部的 **PDM（產品資料管理）人員**，同時也常被營養師、行銷與 KMS 點餐系統的同事問到：「華堡可以掛素食標籤嗎？」「兒童餐要怎麼歸類？」。為了讓食材／單品在系統內能被一致地貼上飲食偏好標籤（素食、無麩質、清真、兒童餐…），我需要先在這張表把所有可被選用的「餐食類型」建立起來。

### 1.2 我要做什麼

- 維護整個 ERP 內可被引用的餐食類型清單（例：素食 / Vegan、奶蛋素 / Lacto-Ovo、無麩質 / Gluten-Free、清真 / Halal、兒童餐 / Kids…）

## 2. 功能目的

餐食類型是 PDM 模組中的字典型基礎資料，用來描述「一道單品 / 一個食材適合哪種飲食偏好或客群」。其設計目的：

1. **建立跨模組統一的飲食標籤語彙**：食材維護、單品 / 食譜維護、行銷活動、KMS 點餐顯示都引用同一份清單，避免命名分裂
2. **以「狀態」做軟啟停**：試辦或季節性類型（例如「低 GI 月」）下架時只需停用、不必刪除，歷史資料仍可顯示
3. **預留引用保護的旗標**：回應 VO 上的 `editable` 旗標已預埋，待食材／食譜模組對接後改為「被引用即鎖定」（來源：`MealTypeServiceImpl.java:86` TODO）

## 3. 業務邏輯背景

### 3.1 狀態值的字典依賴

`status` 欄位型別為字串，並透過 `@DictFormat("status")` 與 `DictConvert` 連到「系統字典」中字典類型 `status` 的字典項。一般 ERP 內 `status` 字典常用 `0=停用 / 1=啟用`，但本功能的字典類型未在程式碼內固化（仍是 TODO 註解），實際值要看資料庫的字典資料為準（來源：`MealTypeRespVO.java:30`）。

**含義**：

- 後端不做有效值約束，只要非空字串即可寫入。前端應透過字典 API 取得可選值並做下拉
- Excel 匯出時，狀態欄位會自動轉換為字典 label（中文）

### 3.2 沒有「分類」「父子層級」「排序」

不像「編碼類別維護」會有父子節點，餐食類型是 **單層、平面** 的清單。也沒有顯示排序欄位 — 預設按 ID 倒序（最新建立的在最上面）排列（來源：`MealTypeMapper.java:27` `orderByDesc(getId)`）。

### 3.3 「狀態」與「軟刪除」的差別

- **狀態 = 停用**：仍存在於資料庫，仍可被舊資料引用，但前端應在新建食材／食譜時不再顯示在下拉
- **刪除**：軟刪除（由 BaseDO 機制處理），通常表示「這個類型完全是建錯了」

目前後端沒有阻止「刪除一個還在使用中的類型」，需靠未來的 editable 邏輯補強（見 §10.2）。

### 3.4 編輯保護的預留邏輯（尚未啟用）

`MealTypeServiceImpl.getMealTypePageWithEditable()` 在分頁查詢後對每筆附加 `editable` 旗標，**目前固定設為 true**。TODO 註解寫的是「待[食材維護作業]和[食譜維護作業]的營養成分子表建立之後補上此處邏輯」（來源：`MealTypeServiceImpl.java:86`） — 這段註解疑似從營養成分定義複製貼上而未改寫；對餐食類型而言，正確的引用源應該是 **食材 / 食譜的餐食類型欄位** 或 **單品-餐食類型的多對多關聯表**，而非「營養成分子表」（見 §11）。

## 4. 情境說明

### 4.1 正常流程 — 新增「無麩質」類型

行銷要在 Q3 推「無麩質」系列，需要在系統把這個標籤建起來。PDM 人員李小姐進入「PDM > 餐食類型維護表」，點「新增」，填入：

- 項目名稱：無麩質
- 英文名稱：Gluten-Free
- 狀態：啟用

送出後系統回傳建立成功。李小姐之後到單品維護作業，能在華堡（無麵包版）的餐食類型多選欄位看到「無麩質」並勾選。

### 4.2 典型業務 — 暫停試辦類型

去年試辦的「低 GI 餐」這季先不推，但歷史單品仍掛著這個類型。李小姐進入清單頁，搜尋名稱「低 GI」，點編輯，把狀態改為「停用」。狀態變更後，前端建檔下拉不再出現此選項，但歷史單品上的標籤仍正常顯示為「低 GI」。

### 4.3 查詢情境 — 看本月新增了哪些類型

主管想看 5 月有沒有新增餐食類型。李小姐在清單頁的「創建時間」帶入 `2026-05-01 00:00:00 ~ 2026-05-31 23:59:59`，按搜尋。系統用 `between` 條件回傳該區間建立的餐食類型，依 ID 倒序排列。

### 4.4 使用者鍵入錯誤 — 漏填必填欄位

新進人員建檔「兒童餐」時忘了選狀態就送出，系統回傳「狀態不能为空」；若忘了填項目名稱，回傳「項目名稱不能为空」（驗證來源：`MealTypeSaveReqVO.java:15、22`）。

### 4.5 異常情境 — 編輯或刪除已不存在的 ID

若 A 同事剛把「無糖月」刪掉，B 同事的清單還是 cache 狀態並嘗試編輯，系統會以錯誤碼 `1_003_000_008 / MEAL_TYPE_NOT_EXISTS`（訊息：「餐食類型不存在」）拒絕（來源：`ErrorCodeConstants.java:39`、`MealTypeServiceImpl.java:62-66`）。

### 4.6 規則分流 — 批次刪除不做存在性檢查

批次刪除直接呼叫 `deleteByIds`，**不檢查每筆是否存在**（來源：`MealTypeServiceImpl.java:57-60`）。前端送入混入不存在的 ID，會被靜默忽略。

### 4.7 匯出情境

行銷部要做「品項可掛飲食偏好總表」，點「匯出 Excel」。系統不分頁、全量匯出，並把狀態欄位自動由 `1` 轉成「啟用」這類字典 label。檔名「餐食類型.xls」，欄位順序見 §6（來源：`MealTypeController.java:100-114`、`MealTypeRespVO.java`）。

## 5. 操作流程

```
[PDM 人員進入「餐食類型維護表」]
  │
  ├─ 分頁查詢 GET /pdm/meal-type/page
  │    ├─ 權限檢查：pdm:meal-type:query
  │    ├─ 過濾：名稱 like、英文名稱 like、狀態 =、創建時間 between
  │    ├─ 排序：ID desc（無顯示排序欄位）
  │    └─ 補欄位：editable = true（固定，未來改為動態）
  │
  ├─ 取得單筆 GET /pdm/meal-type/get?id=
  │    └─ 權限檢查：pdm:meal-type:query
  │
  ├─ 新增 POST /pdm/meal-type/create
  │    ├─ 權限檢查：pdm:meal-type:create
  │    ├─ 必填驗證：name、status
  │    └─ 直接 insert，回傳新 ID
  │
  ├─ 更新 PUT /pdm/meal-type/update
  │    ├─ 權限檢查：pdm:meal-type:update
  │    ├─ 必填驗證：name、status
  │    ├─ 存在性檢查：查不到 → 拋 MEAL_TYPE_NOT_EXISTS
  │    └─ updateById
  │
  ├─ 單筆刪除 DELETE /pdm/meal-type/delete?id=
  │    ├─ 權限檢查：pdm:meal-type:delete
  │    ├─ 存在性檢查
  │    └─ deleteById（軟刪除）
  │
  ├─ 批次刪除 DELETE /pdm/meal-type/batch-delete?ids=1,2,3
  │    ├─ 權限檢查：pdm:meal-type:delete
  │    └─ deleteByIds（不檢查存在性）
  │
  └─ 匯出 Excel GET /pdm/meal-type/export-excel
       ├─ 權限檢查：pdm:meal-type:export
       ├─ 套用同分頁查詢條件，但不分頁
       └─ 寫出 Excel：狀態欄自動轉成字典 label
```

## 6. 欄位規格

### 6.1 主資料欄位（對應 `pdm_meal_type` 資料表）

| 欄位 | 中文業務語 | 型別 | 必填 | 說明 |

| id | 餐食類型 ID | Long | 系統產生 | 主鍵（PostgreSQL sequence） |
| name | 項目名稱 | 字串 | ✅ | 例：素食、無麩質、兒童餐。前端查詢使用模糊比對 |
| engName | 英文名稱 | 字串 | ✕ | 例：Vegan、Gluten-Free、Kids。前端查詢使用模糊比對 |
| status | 狀態 | 字串 | ✅ | 透過字典 `status` 取值，常見為 `1=啟用 / 0=停用`，實際以字典內容為準 |

## 7. 商業邏輯

### 7.1 新增 / 更新流程

1. 攔下 Request VO，由 Jakarta Validation 跑必填驗證（name、status）
2. 更新時先查 ID 是否存在，不存在拋 `MEAL_TYPE_NOT_EXISTS`
3. 用 BeanUtils 把 VO 複製成 DO
4. 直接 insert / updateById（無其他業務檢查）

**未實作的業務檢查**：

- 名稱唯一性（中英文皆無）
- 狀態值是否屬於字典 `status` 的合法 key（後端不檢查；非合法值會寫入並讓前端顯示異常）
- 停用前是否還有食材／食譜在引用

### 7.2 刪除流程

- 單筆：檢查存在 → 軟刪除
- 批次：直接呼叫 `deleteByIds`，不檢查存在性、也不檢查是否被引用
- 預期未來會加上「被引用則禁止刪除」（見 §10.2）

### 7.3 分頁查詢的 editable 加工

`getMealTypePageWithEditable()` 對每筆資料附加 `editable = true`。Controller 的 `/page` 端點呼叫的就是這個方法，因此目前前端永遠看到 editable=true。**待補**：用該餐食類型 ID 反查食材／食譜，若有引用則設 false。TODO 註解原文寫「待…營養成分子表建立之後補上」，但本功能的引用源應為食材 / 食譜的餐食類型欄位（非營養成分子表），註解疑似貼錯（見 §11）。

### 7.4 排序

依 ID 倒序，新建在最上方。**沒有顯示排序欄位** — 若行銷要求按特定順序排列（例如「素食」永遠在第一個），需在前端硬編或加開「排序」欄位。
