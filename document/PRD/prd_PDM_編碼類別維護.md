## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 PDM 的**編碼體系維護人員**，負責建立 ERP 中各種「類別」的編碼規則。
公司各種主檔（食材、單品、包材）的編號背後都有一套「類別代碼 + 流水」的結構。我要先在這裡建立各種「類別」：例如「食材大類」「食材中類」「食材小類」「單品大類」「單品類別」等等，每個類別有自己的代碼、名稱、碼數長度。

### 1.2 我要做什麼

- **新增編碼類別**：定義一個新類別（例如「食材中類」），給它代碼、名稱、碼數長度。
- **修改編碼類別**：調整既有類別的代碼、名稱或碼數。
- **刪除編碼類別**：刪除已不使用的類別。
- **批量刪除**：多筆一次清理。
- **查詢**：依代碼、名稱、碼數篩選查詢。
- **取得全部**：給「編碼項目維護」「編碼原則維護」等模組做選項用。
- **匯出 Excel**：盤點 / 稽核佐證。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 類別名稱不能重複 | 避免歧義 |
| 類別代碼不能重複 | 是編碼體系的基礎 |
| 碼數要可調 | 不同類別位數可能不同（例如食材大類用 2 位、食材小類用 4 位） |
| 可以隨時新增 | 業務擴張時要能彈性加新類別 |

## 2. 功能目的

提供漢堡王 PDM 編碼體系的**最上層分類定義**。每筆「編碼類別」是一個「類別維度」，例如：

- 「食材大類」（會配 2 碼）
- 「食材中類」（4 碼）
- 「單品類別」（5 碼）
- 「包材類別」（3 碼）

下游：

- **編碼項目維護**：在每個「編碼類別」底下建立具體的項目（例如「食材大類」底下有「肉類」「蔬果類」「乳製品」等項目）
- **編碼原則維護**：定義不同編號的拼接規則（例如：原則代碼 + 食材大類 2 碼 + 食材中類 4 碼 + 流水）

> **本模組設計上的特點**：
> - 結構非常簡單：只有「代碼、名稱、碼數」三個欄位
> - 無簽核、無流程、無單據編號
> - 是純字典維護
> - 但程式有些**特殊行為值得注意**（見 §3）

## 3. 業務邏輯背景

### 3.1 三個核心欄位

- **代碼（code）**：編碼類別的代號（短文字）
- **類別名稱（name）**：給人看的中文名
- **碼數（len）**：底下「編碼項目」的編碼長度，1–9 位

### 3.2 唯一性檢查（僅新增時）

新增時系統檢查：

- 名稱（name）必須在全表唯一，否則拋「編碼類別名稱已經存在」
- 代碼（code）必須在全表唯一，**但出錯時拋的是同一個「名稱已存在」訊息**（程式 bug，列入 §11）

修改時**不檢查**唯一性。意即修改名稱或代碼撞名也不會擋。列入 §11。

### 3.3 刪除前不檢查下游引用

刪除一筆編碼類別前**不檢查**：

- 是否有「編碼項目」屬於此類別
- 是否有「編碼原則」引用此類別

刪除後可能造成下游孤兒記錄。列入 §11。

### 3.4 「分頁」端點未要求權限（控制器中 `@PreAuthorize` 被註解）

`/page` 端點上面有條註解掉的 `@PreAuthorize`。意即：任何登入者皆可呼叫此端點。其他端點都要求權限。列入 §11。

### 3.5 「取得全部」即「不分頁的查詢」

`/getAll` 將 `pageSize` 設為 `PAGE_SIZE_NONE`，呼叫一般的分頁邏輯，效果等於「無分頁取全部」。給下游模組做選項用。

### 3.6 「樂觀鎖」欄位 revision 存在但未使用

DO 有 `revision` 欄位（樂觀鎖機制慣用），但 Service 端未見任何 `@Version` 或樂觀鎖機制。意即此欄位目前是**dead column**。列入 §11。

### 3.7 「碼數」的範圍限制

查詢時的 `len` 標註 `@Range(min=1, max=9)`，但**新增／修改的 SaveReqVO 上沒有此驗證**。意即新增時可填 0 或 99（驗證不一致）。列入 §11。

### 3.8 排序

查詢時依「代碼（code）」升冪排序。

### 3.9 匯出 Excel 端點被 `@Hidden`

`/export-excel` 標註了 `@Hidden`，Swagger UI 不會顯示此端點。表示可能仍可呼叫但希望不公開。列入 §11。

### 3.10 批量刪除的實作

`/delete` 端點接受 `ids` 清單，迭代呼叫 `deleteCodeCategory(id)`。**未包 `@Transactional`**，中途失敗會殘留。列入 §11。

### 3.11 `validateCodeCategoryExists` 內部邏輯異常

驗證存在的方法寫法異常：

```java
if (codeCategoryMapper.selectById(id) == null) {
    PdmCodeCategoryDO pdmCodeCategoryDO = codeCategoryMapper.selectById(id);  // 又查一次（== null）
    log.info("編碼類別不存在，pdmCodeCategoryDO = {}", pdmCodeCategoryDO);  // 印 null
    throw exception(CODE_CATEGORY_NOT_EXISTS);
}
```

第一次 select 為 null 之後又 select 一次（必為 null），純粹是浪費一次 DB 查詢。屬程式碼瑕疵但功能正常。列入 §11。

## 4. 情境說明

### 4.1 正常流程 — 新增一筆編碼類別

PDM 維護員小明要建一個「食材小類」：

- 代碼：`FD-S`
- 類別名稱：`食材小類`
- 碼數：`4`

送出後系統：

1. 檢查名稱「食材小類」未被使用 → 通過
2. 檢查代碼「FD-S」未被使用 → 通過
3. 寫入新記錄
4. 回傳新 ID

### 4.2 異常情境 — 名稱重複

小明手滑要建第二個「食材小類」。系統檢查名稱已存在 → 拋「編碼類別名稱已經存在」並拒絕。

### 4.3 異常情境 — 代碼重複

小明改名字、但代碼 `FD-S` 已被使用。系統檢查代碼已存在 → 拋「編碼類別名稱已經存在」（**錯誤訊息誤導，應該是「代碼已存在」**）。列入 §11。

### 4.4 業務規則分流情境 — 修改時改名為已存在的名稱

小明把「食材中類」的名稱改為「食材小類」（已被別筆占用）。系統**不檢查**修改時的唯一性 → **修改成功**。資料庫出現兩筆「食材小類」。列入 §11。

### 4.5 異常情境 — 刪除一個被下游引用的類別

小明刪除「食材中類」，但「編碼項目維護」表中還有 38 筆項目歸屬此類別。系統**不檢查** → 刪除成功。38 筆編碼項目失去歸屬，變成孤兒。列入 §11。

### 4.6 異常情境 — 批量刪除中途失敗

小明批量刪除 3 筆，第 2 筆不存在：

- 第 1 筆：成功
- 第 2 筆：拋例外
- 第 3 筆：未執行
- 因為**未包 `@Transactional`**，第 1 筆**仍被刪除**（無法回滾）

列入 §11。

### 4.7 非預期情境 — 新增時填碼數 0

小明手滑碼數填 0。SaveReqVO **無 @Range 驗證**，會接受。寫入成功但實際應用時可能出錯。列入 §11。

### 4.8 下游使用情境 — 編碼項目維護想列出可選類別

「編碼項目維護」頁面要顯示「歸屬類別」下拉。前端呼叫 `/getAll` 取得全部編碼類別清單。

### 4.9 公開分頁情境 — 未配發本模組權限者

某使用者沒有「PDM-編碼類別-查詢」權限，但仍可呼叫 `/page` 拿到所有編碼類別清單。因為控制器上 `@PreAuthorize` 被註解掉。列入 §11。

## 5. 操作流程

### 5.1 新增編碼類別

```
[填寫表單]
  │
  ▼
[權限檢查：是否具備「PDM-編碼類別-新增」]
  │
  ▼
[檢查名稱唯一性 → 重複則拒絕]
  │
  ▼
[檢查代碼唯一性 → 重複則拒絕（但錯訊為「名稱」）]
  │
  ▼
[寫入]
  │
  ▼
[回傳新 ID]
```

### 5.2 修改編碼類別

```
[編輯表單，送出]
  │
  ▼
[權限檢查：是否具備「PDM-編碼類別-修改」]
  │
  ▼
[驗證存在]
  │
  ▼
[直接覆寫主檔（不檢查唯一性）]
```

### 5.3 批量刪除

```
[輸入 IDs 清單]
  │
  ▼
[權限檢查：是否具備「PDM-編碼類別-刪除」]
  │
  ▼
[迭代每個 ID：驗證存在 → 物理刪除]
  │
  ▼
[（無 @Transactional，中途失敗則殘留）]
```

### 5.4 分頁查詢

```
[輸入查詢條件：代碼、名稱、碼數]
  │
  ▼
[（未要求權限）]
  │
  ▼
[依條件查（代碼 LIKE、名稱 LIKE、碼數 =），依代碼升冪]
  │
  ▼
[回傳分頁]
```

### 5.5 取得全部

```
[呼叫 /getAll]
  │
  ▼
[權限檢查：是否具備「PDM-編碼類別-查詢」]
  │
  ▼
[強制 pageSize = PAGE_SIZE_NONE]
  │
  ▼
[呼叫一般分頁查詢回傳全部]
```

### 5.6 匯出 Excel（@Hidden）

```
[輸入查詢條件]
  │
  ▼
[權限檢查：是否具備「PDM-編碼類別-匯出」]
  │
  ▼
[強制 pageSize = PAGE_SIZE_NONE]
  │
  ▼
[輸出 Excel：編碼類別.xls]
  │
  ▼
[本次匯出被 @ApiAccessLog 記入存取日誌]
```

## 7. 商業邏輯

### 7.1 新增唯一性檢查（只查名稱與代碼）

- 名稱重複 → 拋 `CODE_NAME_ALREADY_EXISTS`
- 代碼重複 → 也拋 `CODE_NAME_ALREADY_EXISTS`（程式錯用同一個錯誤代碼）

### 7.2 修改採直接覆寫

驗證存在後直接 `BeanUtils.toBean` 整筆覆寫，不檢查任何業務唯一性。

### 7.3 刪除採物理刪除＋驗證存在

- 驗證存在（內含一個多餘的 select）
- `deleteById`
- 無交易包覆

### 7.4 分頁查詢

- 條件用 `likeIfPresent`、`eqIfPresent`
- 依「代碼」升冪排序

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| **PDM 維護員（具備「PDM-編碼類別-新增」「-修改」「-刪除」「-查詢」「-匯出」全部權限）** | 全部編碼類別 | 全部操作 |
| **僅具備「PDM-編碼類別-查詢」「-匯出」者** | 全部編碼類別 | 僅查詢、匯出 |
| **任何登入使用者** | 全部編碼類別（透過未啟用權限的 `/page`） | 僅瀏覽 |
| **下游模組（編碼項目、編碼原則）** | 透過 `/getAll` 取選項 | 不可改 |

> `/page` 端點未啟用權限驗證（程式中 `@PreAuthorize` 被註解），任何登入者皆可呼叫。
