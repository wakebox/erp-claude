## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。

每週四下午我要為下週的食材備料做準備。系統會把過去幾週的銷售資料拉出來，依照單品的食譜（BOM）展開為食材，告訴我「下週牛肉餅大概要備多少公斤、麵包多少個、生菜多少公斤」。我需要這份試算結果做為向中央採購提報的依據。

> 「下週母親節週末加加班、信義店牛肉餅平日約 80 公斤／週末約 120 公斤、需要從 5/22 早上前到貨；麵包週用量 1200 個、生菜 50 公斤…」

### 1.2 我要做什麼

- 每週週訂前：開試算頁、預設選好自家門市 → 觀察下週各食材建議備量
- 對照行銷檔期、節慶、天氣：手動微調某項食材的預測量並註明原因
- 把當前庫存帶進來看：長期庫存食材（如冷凍肉、醬料）若庫存充足就不算進這週補貨
- 試算結果定案 → 提交給區經理或採購主管覆核
- 在「待處理／待簽核／已歸檔」清單裡查我自己過去提報的單

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 系統幫我推估下週各食材該備多少 | 用憑直覺猜常會偏多或偏少，造成積壓或斷貨 |
| 平日與假日要分開算 | 信義店週末客流明顯比平日大 |
| 食譜變動要立即反映 | 行銷把華堡的牛肉餅從 1 片改 2 片時，預測也要跟著動 |
| 長期庫存的東西不要每週都被算進去 | 醬料、冷凍肉庫存還很多時，下週其實不用補 |
| 預測結果要能調整 | 試算 100 公斤，行銷說下週有活動，要手動加到 120 |
| 微調要寫原因 | 區經理才知道為什麼我比系統算的多／少 |
| 試算完要走簽核 | 不能拍腦袋下單；須讓主管確認 |
| 只看自己門市 | 不要被全公司資料淹沒 |
| 已歸檔的單不能再被改 | 否則和下游採購對不上 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 「我的門市」食材預測即時試算 | 一頁看到下週每項食材的平日／假日建議備量 |
| 單筆食材重算 | 手動改某項食材的銷量基準後即時看新結果，不用全部重算 |
| 長期庫存判斷顯示 | 標出「目前庫存夠，不用補」的食材 |
| 提交為需求預測單（含明細） | 試算結果定案、進入簽核流程 |
| 我發起的單一覽 | 看「待處理／待簽核／已歸檔」進度 |
| 待簽分頁 | 若我有跨店或多店身份，看分派給我簽的單 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 食材需求預測試算表（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「需求集合」 |
| 兄弟功能 | 物料需求預測試算表（店長視角）、臨時需求審核（店長視角） |
| 主要頁面 | 自店試算頁、我的單分頁、待簽分頁 |
| 簽核流程 | 有（沿用既有 BPM 流程） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能在不依賴總部協助的前提下，**自助看到「下週我的店每項食材該備多少」**，並根據自店實際情況（活動、客流、庫存）微調後送出簽核。在門市端建立「需求預測的第一線責任」，避免總部單一視角產生的偏差。

設計理念：**鎖店 + 可微調 + 必簽核**。店長不能跨店看，不能直接定案，但可以發起、可以調整、可以追蹤。

## 3. 業務邏輯背景

### 3.1 試算的核心輸入

| 輸入 | 來源 | 店長視角預設 |
|---|---|---|
| 區域 | 中繼區域主檔 | 自店所屬區域（自動帶入，不可改） |
| 門市 | 中繼門店主檔 | 自店（自動帶入，不可改） |
| 銷售資料區間 | 中繼銷售歷史 | 通常為過去 28 天（推論） |
| 預測週起訖 | 使用者選 | 預設下週週一到週日 |
| 預測加成 | 使用者選 | 預設 0%（無加成） |

### 3.2 試算的核心邏輯（文字版）

系統對自店每個有銷售的單品（華堡、薯條、雞塊…）做下列事：

1. 從中繼系統拉自店在銷售資料區間內的每日銷量
2. 對每個單品查食譜（BOM），展開成每樣食材的標準用量
3. 用「每日每萬元銷售額對應多少食材」作為穩定口徑，乘上「預測週的預估銷售額」反推每樣食材的預測量
4. 平日（週一至四）與假日（週五至日）分開算，得到兩組數值
5. 若食材屬「長期庫存」（如冷凍肉、醬料）且有設定安全存量：先查自店當前庫存
   - 庫存充足（≥ 安全存量）→ 跳過萬元計算，標「判斷庫存」（不需補）
   - 庫存不足 → 走完整預測，並回傳「目前庫存」給店長對照
   - 查無安全存量或查無庫存資料 → 一律視為需補（保守處理）
6. 對需補的食材，依品號 + 自店查最新廠商報價，補上廠商名稱與「箱換算」（預測量 ÷ 單箱裝數），方便採購下單

### 3.3 BOM 展開

「BOM」即食譜：一個單品由多項食材組成（華堡 = 牛肉餅 + 麵包 + 生菜 + 起司 + 醬…）。店長視角不維護食譜，只是「拿食譜當算式」。食譜的維護屬於單品維護作業（不在本 PRD 範圍）。

### 3.4 平日 vs 假日

- 平日：週一至週四
- 假日：週五至週日

系統內部以這個切分計算兩組數值。若連假、補班、節日的處理規則需另外定義（待確認）。

### 3.5 長期庫存的智慧跳過

「長期庫存」是食材主檔的分類屬性。對這類食材：庫存夠就不算進這週補貨，避免冷凍肉、醬料每週重複下單造成倉儲爆滿。

### 3.6 試算結果的人工微調

店長可以對單一食材手動調整「平日銷量」或「假日銷量」的基準值（例如把牛肉餅的平日銷量從 80 改成 100），系統會用同一公式把該行的預測量重算。其他食材不受影響。

### 3.7 從試算到定案

1. 試算結果先在頁面呈現，不寫進資料庫
2. 店長微調完畢 → 點「儲存」 → 系統建立「需求預測單」（單頭 + 明細）
3. 單據流程狀態自動設為「待處理」
4. 若該選單已綁定簽核流程 → 自動發起 BPM 流程，分派給區經理或採購主管
5. 流程推進時，狀態依序變「待簽核 → 已歸檔」
6. 一旦「已歸檔」，店長就不能再改這張單

### 3.8 資料權限自動過濾

底層共用 API 已實作「依登入者區域／門市自動過濾」：登入者若有門市權限且未指定門市，會強制套用為登入者門市。店長視角理論上會被自動鎖到自店，**但此鎖店行為未經驗證**（待確認，見 §11）。

## 4. 情境說明

### 4.1 正常流程 — 週四下午做下週預測

5/14（週四）下午 4 點，信義店店長小陳要做 5/18–5/24 的食材需求預測：

1. 進「食材需求預測試算」頁
2. 系統自動鎖定「北一區／信義店」（推論），銷售資料區間預設過去 4 週
3. 選預測週 5/18–5/24、加成 0%
4. 點「試算」
5. 結果顯示：
   - 華堡 → 牛肉餅 平日 80 kg／假日 120 kg、麵包 600 個／900 個、生菜 30 kg／45 kg、起司 平日 40 kg／假日 60 kg
   - 雞塊 → 雞肉 平日 25 kg／假日 38 kg、醬料「判斷庫存」（庫存還很多，不用補）
6. 小陳點「儲存」 → 系統建單號、進入待簽核

### 4.2 典型業務 — 行銷活動加碼

小陳得知下週六母親節，行銷做「華堡買一送一」：

1. 一樣跑試算
2. 看到華堡的牛肉餅假日預測 120 kg
3. 點該食材的「假日銷量」欄位 → 從 1500 個改成 2200 個
4. 系統重算該行 → 牛肉餅假日預測量更新為約 176 kg
5. 在備註填「5/26 母親節華堡活動加碼」
6. 提交

### 4.3 異常情境 — 長期庫存食材庫存充足

冷凍牛肉現在自店庫存 12 kg、安全存量 5 kg：

- 試算結果這行顯示「判斷庫存」（灰色文字，不是數字）
- 店長看一眼就知道「這項不用補」
- 該食材不進預測量總計

若庫存掉到 3 kg < 5 kg：
- 該行恢復顯示預測量
- 並顯示「目前庫存：3 kg／安全存量：5 kg」提示

### 4.4 異常情境 — 試算結果與直覺差異很大

系統算出生菜下週要 50 kg，小陳憑經驗認為頂多 30 kg：

1. 直接改該行的銷量基準
2. 重算後變 30 kg
3. 備註寫「上週新進員工，銷量被新人收錯訂單拉高，已通報修正」
4. 提交給區經理覆核
5. 區經理可在待簽分頁看到備註、決定核准或退回

### 4.5 異常情境 — 已歸檔的單想再改

兩天前提的單已經被區經理簽完、流程跑到「已歸檔」：

- 小陳開單 → 系統顯示「已歸檔，禁止更新」
- 若還要調整 → 必須走「臨時需求審核」（另一張單）做加碼

### 4.6 跨日對帳 — 我發起的單一覽

每週一早上小陳開「我的單」分頁：

- 看到上週發起的 4 張單，狀態：3 張「已歸檔」、1 張「待簽核」
- 點開那張「待簽核」的單，看簽核流程卡在區經理那節
- 直接打電話催

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「食材需求預測試算」]
  ├─ 1. 取登入者身份 → 自動帶入：區域 ID、門市 ID（鎖店）
  ├─ 2. 選預測週起訖（預設下週）、預測加成（預設 0）
  ├─ 3. 點「試算」 → 後端：
  │    ├─ 拉自店銷售歷史
  │    ├─ 對自店有銷量的單品展開食譜
  │    ├─ 長期庫存食材：查安全存量＋自店當前庫存
  │    ├─ 對需補的食材：查最新廠商報價、算箱換算
  │    └─ 回傳三層結構：自店 → 各單品 → 各食材（含平日／假日兩組數值）
  ├─ 4. 店長微調某行銷量 → 重算該行
  ├─ 5. 點「儲存」 → 建立需求預測單（單頭 + 明細）
  │    ├─ 流程狀態 = 待處理
  │    └─ 啟動 BPM 流程（若選單綁定）
  ├─ 6. 進入「我的單」分頁追蹤狀態
  └─ 7. 流程推進：待處理 → 待簽核 → 已歸檔
```

> **自店過濾**：底層 service 已有「登入者門市自動過濾」邏輯，但店長視角是否真能被鎖到自店未經驗證（待確認）。

## 7. 商業邏輯

### 7.1 鎖店過濾（**未驗證，建議補檢核**）

店長視角進入試算時：

```
1. 取登入者門市 ID
2. 強制：區域、門市條件以登入者為準（即使前端傳了別的也覆寫）
3. 試算過程：所有「廠商報價」「庫存」查詢一律以該門市為條件
4. 列表查詢：自動加門市過濾
```

底層程式已有此設計，但目前後端只有「總部共用」端點，沒有「店長 app 專屬」端點；前端若呼叫共用端點未必能保證鎖店行為一致（待確認）。

### 7.2 微調的判斷分流（推論）

- 微幅微調（在系統預測量 ±10% 內，**門檻待確認**）→ 直接寫入 → 走標準簽核
- 大幅微調（超過 ±10%）→ 強制需填原因 → 標記為「特殊」單 → 簽核流程可能升級
- 目前後端**無此分流邏輯**，僅有單一儲存路徑（推論：店長視角應補）

### 7.3 已歸檔保護

任何單據一旦進入「已歸檔」狀態，店長都不能修改，否則拋出「已歸檔，禁止更新」錯誤。

### 7.4 預設值

- 預測週：下週週一到週日
- 銷售資料區間：過去 28 天（推論）
- 預測加成：0%（無加成）

### 7.5 唯讀範圍

店長不應有「廠商報價維護」「食材主檔維護」「食譜維護」「安全存量維護」等按鈕，這些都屬於上游基礎資料，由總部維護。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店所有預測單、自店所有試算 | 試算、微調、儲存（提交簽核）、追蹤 |
| 區經理 | 區內所有店 | 同店長，且可在待簽分頁看到分派的單做簽核 |
| 採購規劃／總部 | 全公司 | 全 CRUD（已實作於既有共用 API） |
| 店長兼任多店者 | 多店資料 | 試算需可切換門市（待確認） |

> 後端目前以「具備食材需求預測查詢／編輯／建立／刪除」的角色控管，未細分店長／總部。店長視角的鎖店與權限切分需另實作。

```
│  ▼ 雞塊（銷量：平日 800／假日 1100）
│    醬料 X     判斷庫存（目前 12 kg／安全 5 kg）
│
│  [儲存為需求預測單]
```

行動裝置：每個單品一張卡片，展開後顯示食材列表。

## 10. 功能範圍

### 10.1 包含的功能

- 自店食材需求即時試算（待補鎖店）
- 平日／假日分流呈現
- 長期庫存的智慧跳過顯示
- 廠商與箱換算顯示
- 單行微調 + 重算
- 儲存為需求預測單（含明細）
- BPM 簽核流程綁定
- 我的單分頁追蹤
- 已歸檔保護

### 10.2 預留但尚未實作

| 項目 | 證據 |
|---|---|
| 店長視角專屬 endpoint | 後端僅有 `controller/admin/demand/*`，無 `/app/*` 對應路徑 |
| 自店過濾的店長視角驗證 | 共用 API 已有過濾邏輯，但未針對店長路徑驗證 |
| 大幅微調的差異門檻與分流 | 後端無此邏輯 |
| 微調原因強制填寫 | 後端無此欄位／驗證 |
| 行動裝置 UI | 推論：店長使用情境包含現場操作，但目前無前端 |
| 試算結果與當前庫存的並排對照 | 目前僅長期庫存顯示，短期庫存未顯示 |
| 試算結果緩存 | 每次重算，大區域試算耗時長 |

### 10.3 不包含

- 食材主檔維護（屬於食材維護作業）
- 單品／食譜維護（屬於單品維護作業）
- 廠商報價維護（屬於廠商報價維護作業）
- 安全存量設定（屬於庫存管理）
- 物料需求（包材、清潔用品等非 BOM 物料 → 另一張試算表）
- 臨時加單（突發大量訂位 → 走臨時需求審核）
- 採購單建立（屬於採購管理）

## 11. 待確認事項

| 議題 | 為何要確認 |
|---|---|
| 「登入者 → 所屬門市」對應如何取得？是否已有單一門市鎖定？多店店長如何切換？ | 鎖店實作前提 |
| 「大幅微調」的門檻（百分比？絕對量？）由 PM 還是設定檔決定？ | 簽核分流規則 |
| 微調時是否強制填寫原因？原因要不要進待簽分頁的關鍵欄位？ | 簽核者判斷依據 |
| 預設「銷售資料區間」是過去 28 天還是過去 4 週同期？ | 試算邏輯 |
| 連假、補班、節日：平日／假日定義是否要重新分類？ | 計算口徑 |
| 預測加成的格式：使用者填 10 表示 10%、還是填 1.10？ | UX 與計算 |
| 店長視角是否需要顯示「同類店預測對照」（防離群）？ | UX |
| 新店無歷史銷售時用什麼基線？ | 演算法 |
| 大區域試算很慢，是否要把店長視角限縮為「只算自店」以避免效能問題？ | 效能 |
| 店長能否查看區內其他店的預測（如代班、跨店支援場景）？ | 權限 |
| 與物料需求預測（非 BOM）是否要在前端合併為單一介面？ | UX |
| 與臨時需求審核的關係（突發加單）：店長要先評估走哪個？ | 業務流程 |
| 行動裝置 / PWA 是否為必需？ | 開發範疇 |
| 若試算結果暫時不想提交，是否要支援「暫存草稿」？ | UX |
| 預測週的長度：固定一週、還是支援雙週／月？ | 業務政策 |
