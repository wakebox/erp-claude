# PRD｜系統管理 — 設定使用者角色

> 來源：逆向自 `kingmaker-module-system` 之權限相關後端程式碼（`controller/admin/permission/PermissionController.java` 之 `list-user-roles` 與 `assign-user-role` 端點、`service/permission/PermissionServiceImpl.java` 之 `assignUserRole`、`dal/dataobject/permission/UserRoleDO.java`、`vo/permission/PermissionAssignUserRoleReqVO.java`）。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽

### 1.1 我是誰

我是漢堡王 ERP 的系統管理員。
角色本身已經配好「能看哪些頁面」（選單權限）「能看哪些資料」（資料權限），但個別員工**還沒實際擁有任何角色**。新人到職、職務異動、員工兼任、離職復職⋯各種人事變動最終都會收斂到一件事：「這個人應該掛哪些角色」。

我負責把使用者與角色掛上鉤，讓他登入後真的能用 ERP 做事。

### 1.2 我要做什麼

- **每天**：新進員工開好帳號後，立即指派初始角色組合（採購助理、門市店員、財會等）。
- **每週**：人事異動 — 員工調動部門時把舊角色拿掉、加上新角色；員工升遷時加上主管類角色。
- **每月**：盤點代理人狀況 — 某主管請長假，把他的「主管」角色暫時加掛到代理人帳號上；歸來時還原。
- **不定期**：HR 或法遵要求「請把某某的權限暫時降為最基本」，把所有角色拿掉只留基礎角色。
- **不定期**：員工兼任跨部門職務時，加上多個角色。

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一個使用者可以同時掛多個角色 | 兼任、代理、跨部門職務在實務上很常見 |
| 進畫面要先看到該使用者目前已有哪些角色 | 不知道現況就改不好 |
| 改動後立刻生效 | 主管臨時請假，代理立刻要能登入接手 |
| 即使我手滑亂改，系統有交易保證不留中間狀態 | 兩步驟（新增＋移除）任一步驟失敗都不應留下一半 |
| 我可以一次拿掉所有角色，讓某使用者「暫時停權」但保留帳號 | 比刪帳號或停用帳號更可逆 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 取得指定使用者目前掛載之角色清單 | 進畫面後可預先勾選現況 |
| 提交一組角色清單覆寫該使用者之角色 | 一次儲存所有調整 |
| 儲存時系統自動計算差集，只動有變動的部分 | 避免不必要的資料庫操作、保留未變動的歷史 |
| 儲存包在交易內 | 「新增＋移除」兩步驟同存同亡 |
| 儲存後同步清除該使用者之角色快取 | 下次驗權即用新角色 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 設定使用者角色 |
| 所屬模組 | 系統管理 |
| 兄弟功能 | 使用者管理、角色管理、設定角色選單權限、設定角色資料權限 |
| 主要頁面 | 使用者管理列表頁中，每個使用者列上之「分配角色」動作；點開為勾選清單彈窗 |
| 簽核流程 | 無 |

## 2. 功能目的

把「使用者」與「角色」**多對多關聯**起來。一個使用者可以同時擁有多個角色，所有角色的權限（選單與資料）在使用者登入時自動合併（最寬鬆者勝出）。本功能負責維護這層關聯，是 ERP 權限模型的最後一塊拼圖。

設計上強調：

1. **整批提交、增量更新** — 一次送出角色清單，伺服端與資料庫現況比對差集，只插入新增的、只刪除取消的，未變動的不動。
2. **交易保護** — 「批次新增」「批次刪除」包在交易內，任一步驟失敗整個回滾。
3. **快取精準失效** — 儲存後僅清除該使用者之角色快取，不影響其他使用者。

> 經系統整合決策（2026-05-16），漢堡王 IT 走向以 Laravel OA 為使用者單一事實來源、ERP 不再自簽 JWT。**「使用者—角色」對應關係長期將由 OA 統一管理**，本模組屬於 ERP 過渡期保留功能；長期規劃中將在搬遷階段 5 後廢棄。本 PRD 描述的是過渡期內、ERP 仍自管使用者—角色關聯時的需求。

## 3. 業務邏輯背景

### 3.1 使用者—角色為多對多

一個使用者可以掛多個角色、一個角色也可以被掛到多個使用者。資料以一張關聯表記錄每一組「使用者編號 ↔ 角色編號」對應。

### 3.2 多角色登入時權限自動合併

當使用者擁有多個角色，登入後系統自動合併：

- **選單可見性** — 所有角色之選單聯集（任一角色看得到，該使用者就看得到）
- **資料範圍** — 依照 [[設定角色資料權限]] §3.2 合併規則（任一角色「全部資料」勝出；多個「指定部門」聯集；「僅本人」可與部門範圍並存）

**權限模型偏寬鬆** — 多角色實質擴大可看／可做的範圍，不會互相收緊。

### 3.3 已停用之角色會被自動忽略

使用者可被掛載「已停用」的角色（系統不擋下），但登入時系統解析權限會自動把已停用角色過濾掉，等於沒掛。**該角色重新啟用時即自動生效**，不需重新指派。

對 IT 的意義：停用某角色等於暫停所有掛此角色之使用者的對應權限，比逐一移除快。

### 3.4 沒有任何角色 ≠ 沒有權限

一位使用者若完全無掛角色：

- 選單可見性：無，登入後是空白頁
- 資料權限：被視為「僅本人」，即只能看自己經手／建立的資料

「沒角色」與「角色全停用」結果相同。

### 3.5 增量更新而非全砍重寫

儲存時系統的處理方式：

1. 撈出該使用者目前掛載之角色編號集合
2. 比對提交的新清單
3. 算出「新增的」「移除的」「不動的」三組
4. 只對新增與移除執行資料庫操作

未變動的關聯保留其原始建立時間／建立人（推論），與 [[設定角色選單權限]] 同樣設計。

### 3.6 儲存後僅清該使用者之角色快取

`@CacheEvict` 按使用者編號**精準清快取**，不像 [[設定角色選單權限]] 那樣清整個快取空間。原因是「使用者→角色集合」之快取以使用者編號為 key，可定點清除。

### 3.7 交易保護

整個 `assignUserRole` 動作包在多資料源交易內，「批次新增」與「批次刪除」兩步驟若任一失敗，已執行的部分會回滾。**與 [[設定角色選單權限]] 行為不同** — 該功能未標交易，本功能標了。

## 4. 情境說明

### 4.1 正常流程 — 新人到職指派角色

5/18 新到職的採購助理小王已建好帳號（見 [[使用者管理]]）。阿明接著進使用者列表，點小王列上的「分配角色」按鈕。彈窗載入小王目前的角色清單（空），阿明勾選：

- 「採購助理」
- 「全體員工」（基礎角色，可看公告、個人中心）

按儲存。系統比對：原本 0 個、新表單 2 個，全部執行新增。儲存包在交易內成功完成，該使用者快取被清除。小王下次登入即擁有兩個角色合併的權限。

### 4.2 正常流程 — 兼任跨部門

採購主管小陳兼任「集團採購策略推進小組」副召集人。阿明在小陳之既有角色「採購主管」「全體員工」上，加掛「策略推進小組成員」。

系統比對：原本 2 個、新表單 3 個，差集為新增 1、移除 0、不動 2。資料庫只插入 1 筆。小陳兼任時刻起即可看到推進小組的相關選單與資料。

### 4.3 正常流程 — 部門調動

員工小李從採購部調到行政部。阿明把小李之「採購助理」拿掉、加上「行政專員」、保留「全體員工」。

系統比對：原本 2 個（採購助理、全體員工）、新表單 2 個（行政專員、全體員工），差集為新增 1、移除 1、不動 1。資料庫插入 1 筆、刪除 1 筆，兩動作包在交易內。

### 4.4 異常情境 — 暫時停權

員工小郭因內控檢視被暫時降為基礎權限。阿明進其角色設定，把所有實質角色取消，只留「全體員工」。

系統比對：原本 4 個、新表單 1 個，差集為新增 0、移除 3。小郭下次操作所有採購類功能即被擋下。

### 4.5 邊界情境 — 提交空集合 = 拔掉所有角色

阿明誤把某使用者所有角色都取消並按儲存。系統比對：原本 N 個、新表單 0 個，差集為新增 0、移除 N。該使用者下次登入會看到空白頁，且只能看到自己經手的資料。**伺服端不擋下空集合**。

> §11 待確認：是否要在前端加上「您即將拔掉該使用者所有角色」之確認。

### 4.6 隱性風險情境 — 指派超級管理員角色

阿明（具備「系統-權限-設定使用者角色」權限）可以為任意使用者勾選「超級管理員」角色。**系統不擋下也不警告**，儲存後該使用者立即擁有全系統權限。

> §11 待確認：是否要對「超級管理員」角色之指派加上額外保護（例如：僅特定主管帳號可指派、需二次確認、需簽核流程）。

### 4.7 邊界情境 — 指派已停用角色

阿明把一個「已停用」的角色掛到小張身上。系統儲存成功（不擋下）。小張登入時系統解析權限自動過濾掉該角色，等於沒掛。日後該角色被重新啟用，小張即自動取得對應權限。

> §11 待確認：是否要在 UI 上以視覺標示「此角色目前已停用，掛上不會生效」。

### 4.8 推論性情境 — 對已刪除使用者或角色操作

若被指派的使用者已被刪除（但呼叫方不知道）：

- 系統不檢查使用者存在性，會在 `system_user_role` 表中插入指向不存在使用者的關聯。
- 此資料不會被任何登入流程使用（因該使用者無法登入），等於垃圾資料。

若提交的角色 ID 中有不存在的：

- 系統不檢查角色存在性，會原樣插入。
- 此資料在權限解析時會找不到對應角色被忽略。

§11 列為待確認。

### 4.9 異常情境 — 中間步驟失敗

假設「批次新增」成功後，「批次刪除」階段因資料庫鎖等待逾時失敗。本功能標有交易，整個動作回滾，已新增的關聯也被撤銷。最終狀態與動作前一致。**這是本功能與 [[設定角色選單權限]] 的關鍵差異** — 該功能未標交易，可能殘留中間狀態。

### 4.10 自動處理情境 — 該使用者被刪除時

若某使用者透過 [[使用者管理]] 之刪除流程被移除，系統會自動連動清除該使用者的所有角色關聯，避免殘留資料。

## 5. 操作流程

### 5.1 進入畫面：取得該使用者目前的角色清單

```
[使用者點某員工之「分配角色」按鈕]
  │
  ▼
[權限檢查：是否具備「系統-權限-設定使用者角色」]
  │
  ▼
[從使用者-角色關聯表撈出該使用者掛載之角色編號集合]
  │
  ▼
[回傳角色編號集合，前端據以勾選]
```

### 5.2 儲存：賦予使用者一組角色

```
[使用者勾選完成 → 按儲存]
  │
  ▼
[權限檢查：是否具備「系統-權限-設定使用者角色」]
  │
  ▼
[開啟交易（多資料源）]
  │
  ▼
[撈出該使用者資料庫中目前掛載之角色編號集合]
  │
  ▼
[計算差集]
  ├─ 新表單有、資料庫沒有 → 待新增
  ├─ 資料庫有、新表單沒有 → 待移除
  └─ 兩邊都有 → 不動
  │
  ▼
[批次新增「待新增」之關聯]
  │
  ▼
[批次刪除「待移除」之關聯]
  │
  ▼
[任一步驟失敗 → 整個交易回滾]
  │
  ▼
[清除該使用者之「使用者→角色集合」快取]
  │
  ▼
[回傳成功]
```

## 7. 商業邏輯

### 7.1 無前置存在性／狀態檢核

儲存時**不會檢查**：

- 使用者編號對應之使用者是否存在
- 使用者編號對應之使用者是否為「啟用」狀態
- 角色編號集合中之每個角色是否存在
- 角色編號集合中之每個角色是否為「啟用」狀態
- 角色編號集合中是否含「系統內建」類型角色（含超級管理員）

任一情況皆會原樣儲存。系統依賴**權限解析階段**之過濾來忽略無效資料（例如已停用角色於解析時被自動排除）。（來源：`PermissionServiceImpl.java:209-229`）

### 7.2 增量更新邏輯（核心）

```
新增集合 = 新表單 - 資料庫
移除集合 = 資料庫 - 新表單
不動集合 = 新表單 ∩ 資料庫
```

只對「新增集合」執行批次插入、只對「移除集合」執行批次刪除。

「不動集合」對應的關聯紀錄完全不動 — 保留原始建立時間／建立人。（來源：同上）

### 7.3 空集合提交 = 清除該使用者所有角色

若使用者送出空集合，差集計算結果為「新增 0、移除全部」，資料庫會把該使用者的角色關聯全部刪除。**伺服端不會擋下空集合**，亦無提示。

### 7.4 交易保護

整個方法標有多資料源交易（`@DSTransactional`）。「批次新增」與「批次刪除」兩動作任一失敗，整個交易回滾，已執行的部分被撤銷。**這是本功能優於 [[設定角色選單權限]] 之處**（該功能未標交易）。（來源：`PermissionServiceImpl.java:207`）

### 7.5 快取失效精準

`@CacheEvict` 以使用者編號為 key 清除該使用者的角色快取，不影響其他使用者。（來源：`PermissionServiceImpl.java:208`）

### 7.6 與「刪除使用者」的連動

使用者透過 [[使用者管理]] 被刪除時，系統會在內部呼叫權限清理流程，自動刪除該使用者所有角色關聯。此清理動作亦會清除該使用者之角色快取。**本功能與該清理流程共用同一張關聯表**。

### 7.7 與「刪除角色」的連動

角色透過 [[角色管理]] 被刪除時，系統會自動連動刪除「該角色關聯之所有使用者—角色」資料以及「該角色關聯之所有角色—選單」資料。**角色刪除會影響到原本掛此角色的所有使用者**，這些使用者於下次登入時對應權限消失。

### 7.8 無操作日誌

此動作未標註 `@LogRecord`，**不會留下「誰、何時、把哪個使用者之角色從 X 改為 Y」的稽核軌跡**。權限變動屬高敏感操作。§11 列為待確認，與 [[設定角色選單權限]] §11 #8、[[設定角色資料權限]] §11 #1 為同一類問題。

## 8. 使用角色與權限

| 角色 | 可看／可操作 |
|---|---|
| **具備「系統-權限-設定使用者角色」權限者**（通常為系統管理員） | 可查詢任一使用者之角色清單；可對任一使用者提交新的角色清單（含指派超級管理員） |
| **僅具備「使用者查詢」權限者** | 可看到使用者列表，但看不到／無法觸發「分配角色」動作 |
| **未配發系統管理權限者** | 看不到系統管理選單 |

查詢與儲存共用**同一個權限字串**，無法分開（與 [[設定角色選單權限]] 同樣設計）。§11 待確認此設計是否符合稽核需求。

## 9. 畫面需求／視覺規範

後端僅提供 API。預期前端在使用者管理列表頁的每列操作區提供：

- 「分配角色」動作按鈕
- 點擊後彈出角色勾選介面
  - 多選清單（或下拉多選），列出所有可用角色
  - 預先勾選該使用者目前掛載之角色
  - 已停用之角色建議以視覺標示（灰階／徽章）並可附說明「目前停用中，掛上不會生效」
  - 「超級管理員」之類高敏感角色建議特別標示，並可考慮加上勾選時的二次確認
  - 儲存／取消按鈕
- 儲存後顯示成功提示

## 10. 功能範圍

### 10.1 包含的功能

- 查詢指定使用者目前掛載之角色編號集合
- 提交一組角色清單覆寫該使用者之角色（採差集增量更新）
- 儲存包在交易內，新增/刪除同存同亡
- 儲存後精準清除該使用者之角色快取

### 10.2 預留但尚未實作

- 對「超級管理員」之類高敏感角色之指派**無額外保護**（任何具備本功能權限者都可指派）
- 無前置存在性檢核（使用者／角色是否存在、是否啟用皆不檢查）
- 此動作**無操作日誌**（無 `@LogRecord` 標註），無法稽核「誰改了誰的角色」

### 10.3 不包含

- 使用者本身的維護（在「使用者管理」功能）
- 角色本身的維護（在「角色管理」功能）
- 角色之選單權限設定（在「設定角色選單權限」功能）
- 角色之資料權限設定（在「設定角色資料權限」功能）
- 多角色合併規則之解析（屬於使用者登入時的權限解析，自動執行）
- 使用者刪除時的角色關聯清理（由「使用者管理」之刪除流程自動連動）
- 角色刪除時的關聯清理（由「角色管理」之刪除流程自動連動）
