## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長 / 副店長 / 倉管員**（例：BK001 信義店）。

每天打烊後我要清點冰箱、冷凍倉、乾貨架，把實際剩多少寫進系統，順便跟「系統算出來應該剩多少」做比對。這是日常結算，不是月度大盤。

> 「2026-05-19 信義店打烊後：牛肉餅剩 31 公斤、起司剩 18 公斤、生菜剩 5 公斤、麵包剩 76 個。系統說我應該剩 32／20／6／80，差異 1／2／1／4。」

### 1.2 我要做什麼

- 每天閉店後（或隔日開店前）建立一張「今日盤點單」
- 系統幫我從 POS 拉當日銷量，順便展開食譜算「理論該用掉多少」
- 我對冷凍／冷藏／乾貨倉每個品項輸入「實盤數量」
- 系統比對差異，列出超出容忍度的項目
- 必要時填差異原因（報廢／員工餐／客訴退／不明損耗），送主管簽核

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 一打開就是「我的店、今天」 | 不要再從整公司資料裡找自店 |
| 系統幫我先把該有的食材清單列出來 | 不要每筆都自己加列 |
| 顯示「理論應剩」當作參考值 | 知道差異有沒有不合理 |
| 行動裝置可操作 | 冷凍倉裡用平板填，不可能跑回辦公室 |
| 大字數字鍵盤 | 戴手套也能按 |
| 大差異強制要原因 | 規避私下調整、稽核留痕 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 自店、今日預設的盤點單入口 | 一鍵開始 |
| 銷量試算（POS 銷量 × 食譜 → 理論消耗） | 提供差異參考線 |
| 食材清單自動帶（依倉別、區域） | 不漏項 |
| 實盤輸入欄（每食材一列） | 主要操作 |
| 差異自動計算 + 容忍度檢查 | 警示異常 |
| 差異原因（必填／字典化） | 留痕 |
| 簽核流程銜接 | 大額差異需主管核可 |
| 當日盤點清單／補登 | 忘了昨晚盤、隔日補 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 每日盤點作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理（WHS）」 |
| 兄弟功能 | 盤點計劃執行管理（店長視角）、出庫作業、入庫作業 |
| 主要頁面 | 自店當日盤點單編輯頁、自店盤點單列表、待簽列表 |
| 簽核流程 | 有（底層已串 BPM；超出容忍度的差異建議走簽核） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |
| 後端鎖店過濾 | **無**（資料表單頭不存「門市 ID」，僅存區域／倉別／倉名） |

## 2. 功能目的

讓店長在閉店或開店前能 **5–15 分鐘** 完成自店的日結庫存核對，得到三項輸出：

1. **每食材的實盤量** — 為「庫存帳實一致」提供日層級的對帳資料
2. **超出容忍度的異常項目** — 第一時間發現報廢未登、員工偷竊、食譜誤差、收貨少算
3. **簽核軌跡** — 大額差異留下原因說明 + 主管核可紀錄

設計理念：**高頻、輕量、強鎖店、行動優先**。店長只關心自己門市、只關心今天，畫面不該出現其他門市資料。

## 3. 業務邏輯背景

### 3.1 與「盤點計劃執行」的差別（店長視角）

|  | 每日盤點 | 盤點計劃執行 |
|---|---|---|
| 誰發起 | 店長自己 | 總部排計畫、分派到店 |
| 頻率 | 每天 | 月／季／年 |
| 範圍 | 重點品項（高週轉、易耗損） | 計畫指定的全部品項 |
| 嚴謹度 | 操作層級的對帳 | 結帳基礎、可進稽核 |
| 簽核 | 大額差異才需 | 必走簽核 |
| 時效 | 當日完成 | 計畫排定的時段內 |
| 與庫存連動 | 推論：直接調整 | 透過盤盈虧單調整 |

### 3.2 「應有庫存」是怎麼算的

系統會在編輯頁提供一個「載入今日銷量」動作，背後做的事是：

1. 取登入者所屬區域、門市、目標日期
2. 從漢堡王中繼系統拉「今日 POS 銷量」（每商品賣了幾份）
3. 對每個商品展開食譜（PDM 模組維護），得到「每食材的理論用量」
4. 加總到食材層級，呈現「今日理論該消耗的量」

> 推論：店長視角應再進一步顯示「理論應剩 = 期初庫存 + 期間入庫 - 理論消耗」當作對照欄位。底層目前提供的是「理論消耗」，是否再算「應剩」需待確認。

### 3.3 兩張表

- **每日盤點單頭**：每張單一個區域、一個倉別、一個盤點日期，記流程狀態
- **每日盤點明細**：一張單頭 N 筆食材，每筆記品號、標準用量、庫存計量／計數、預計銷售計量、賬面計量、盤點計量（實盤）、實際出庫計量

> 重要：明細欄位中已有「賬面計量」（系統算的應有值）與「盤點計量」（店長實盤值）兩個欄位，**設計上就是要做差異比對**；但目前未掃描到自動的「差異警示／鎖定」邏輯。

### 3.4 「我的店」是怎麼判定的

**目前資料表單頭沒有「門市 ID」欄位**，只有「區域 ID／區域名稱／倉別／倉名／倉名名稱」。

> 待確認：漢堡王門市與倉別／倉名是否一對一？若一店對一倉，則「倉名」可當門市過濾鍵；若一店多倉（冷凍／冷藏／乾貨分倉），則一張盤點單只代表一個倉，店長一天要建多張單，或要改為「單頭含店 + 多倉」的結構。**此設計差異會直接影響店長日常操作頻率。**

### 3.5 跨模組依賴

| 依賴 | 用途 |
|---|---|
| 漢堡王中繼 API（每日銷量） | POS 銷量來源 |
| PDM 食譜／BOM | 銷量 → 食材消耗的展開規則 |
| WHS 庫存主檔 | 期初庫存、入庫紀錄、理論應剩 |
| BPM 簽核 | 大額差異審批 |
| 系統使用者主檔 | 登入者 → 所屬門市的對應（推論：尚未建立） |

## 4. 情境說明

### 4.1 正常情境 — 閉店後快速盤點

5/19 晚 22:00 BK001 信義店打烊，店長小李拿著平板走進冷凍倉：

1. 開「每日盤點作業」，畫面預設「信義店／2026-05-19／冷凍倉」
2. 點「載入今日銷量」，系統秀出 6 個食材的理論消耗（牛肉餅 8 kg、雞胸 4 kg、薯條 12 kg、起司 2 kg、培根 1 kg、漢堡排 5 kg）
3. 對每個食材磅秤後填實盤值
4. 系統即時算差異，4 個食材在 ±3% 內、1 個食材（薯條）差 8%
5. 系統強制要小李在薯條那列填「差異原因」，小李選「冷凍機故障 1 包報廢」並附照片
6. 全部填完 → 送簽核 → 區經理隔天上班核可
7. 小李離開冷凍倉、關燈、下班

### 4.2 異常情境 — 中繼還沒同步銷量

7:30 隔日開店前小李才想到昨晚沒盤：

1. 進系統補登 2026-05-19 信義店
2. 點「載入今日銷量」，系統回「2026-05-19 中繼銷量尚未同步」
3. 小李改填「未含理論消耗」的盤點（手動列食材清單），系統允許但在單上標「未核對銷量」
4. 中繼資料晚到後，系統補回理論消耗，差異欄位事後補算

> 待確認：上述流程是否被允許？或必須等中繼資料才能盤？

### 4.3 異常情境 — 差異過大

某日小李發現牛肉餅實盤少 5 公斤（差 20%）：

1. 系統強制要原因，小李選「不明損耗」
2. 自動送區經理 + 食安主管雙簽
3. 區經理打回去重盤，小李再清點一次冰箱底層，發現一箱被推到角落
4. 重新提交，差異降為 1%，免簽通過

> 待確認：「打回重盤」的動作是否允許？或被打回就必須立另一張單？

### 4.4 跨日對帳

月底店長要核對上月帳：

1. 進「我的盤點單列表」
2. 切日期範圍 5/1～5/31
3. 看到 31 筆（理想）／少於 31 筆（有漏盤）
4. 把漏盤的日期補上，或在月結報表標註

## 5. 操作流程

> 後端目前無店長視角 endpoint，以下為「應有的呼叫鏈」（推論）。

```
[店長打開「每日盤點作業」]
  │
  ├─ 1. 登入身份取得：店長、所屬門市、所屬區域、可用倉別
  │     └─ ⚠️ 目前後端無「使用者 → 門市」對應，需另實作
  │
  ├─ 2. 預設「自店、今日、第一個倉別」
  │     └─ 若該店該日該倉已有盤點單 → 進入編輯；否則 → 開新單
  │
  ├─ 3. 載入今日銷量試算（取得每食材理論消耗）
  │     └─ ⚠️ 鎖店：禁止傳入其他店的 storeId
  │
  ├─ 4. 載入食材清單（依倉別過濾，可搜尋）
  │
  ├─ 5. 對每食材輸入實盤計量／計數
  │     └─ 即時顯示差異 = 賬面 - 實盤
  │
  ├─ 6. 差異處理
  │     ├─ 在容忍度內 → 不需原因
  │     └─ 超容忍度 → 必填原因（字典）+ 可選拍照
  │
  ├─ 7. 提交送簽核
  │     └─ BPM 走「每日盤點」流程
  │
  └─ 8. 簽核通過 → 自動調整庫存主檔（推論，待確認）
```

## 7. 商業邏輯

### 7.1 鎖店過濾規則（**未實作，待補**）

```
1. 取登入者 → 所屬門市 ID
2. 查盤點單（單頭） where 門市 ID = 自店
3. 若門市未在單頭，改以「倉名 / 倉 ID ∈ 該店可用倉」過濾
4. 編輯／新增單據時，倉別下拉只列自店可用倉
5. 中繼銷量試算的 storeId 強制帶自店
```

### 7.2 唯一性

- 推論：每店、每日、每倉 **最多一張**盤點單
- 目前後端**未檢查**唯一性 → 待確認是否需擋

### 7.3 容忍度（**目前未實作**）

- 容忍度應在食材主檔或盤點規則表設定（例：高週轉食材 ±5%、低週轉食材 ±2%）
- 差異 ≤ 容忍度 → 直接寫入 + 自動調整庫存（推論）
- 差異 > 容忍度 → 必填原因 + 進簽核

### 7.4 與「出庫作業」報廢的關係

- 店長若已在「出庫作業」做了報廢登錄，理論上該量應從「賬面計量」中先扣掉，避免雙計
- 推論：當日報廢出庫單在計算「賬面應剩」時應一併納入
- 待確認：目前邏輯是否已含此扣抵

### 7.5 與「盤點計劃執行」的並行

- 若同日同店既有計劃盤又有每日盤 → 哪個為準？是否互斥？
- 待確認：建議該日有計劃盤則停用每日盤

### 7.6 唯讀切換

- 簽核中、已歸檔 → 唯讀
- 退回 → 可編輯
- 已過 N 日（推論：3 日）→ 強制鎖定不可補
