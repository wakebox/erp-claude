## 1. 功能概覽

### 1.1 我是誰

我是漢堡王台灣 **某分店店長**（例：BK001 信義店）。每季／每月／行銷活動前，我會調整自己門市的安全存量水位：

> 「6 月母親節主打華堡買一送一，預估牛肉餅銷量翻倍；先把信義店牛肉餅安全存量從 30 公斤拉到 60 公斤，避免活動期間斷貨。」
>
> 「冷飲季結束，可可粉用量下降；安全存量從 5 公斤降到 2 公斤，避免長期擱置過期。」

### 1.2 我要做什麼

- 季節更替前：調整即將進入旺／淡季的食材閾值
- 行銷活動前：拉高活動主打商品相關食材
- 新品上市時：為新增食材設定第一個安全存量
- 異常事件後（颱風、缺料、客訴）：重新檢視

### 1.3 我有什麼需求

| 我的需求 | 背後的問題 |
|---|---|
| 系統建議一個合理起始值 | 上百樣食材一個一個拍腦袋訂太花時間 |
| 依「我這家店」的過去銷量推算 | 信義店與板橋店銷量結構不同，用全公司平均不準 |
| 看到目前實際庫存對照 | 設定時知道現在還有多少 |
| 只能改自己店的 | 不要動到別店、不要被別店動到 |
| 改動有審核 | 避免拉太高綁住採購、拉太低造成斷貨 |
| 保留歷史紀錄 | 月底回顧調整成效 |

### 1.4 因此，需要以下功能

| 功能 | 解決的問題 |
|---|---|
| 自店食材閾值列表 | 主要編輯視圖 |
| 系統試算建議值（基於該店過去銷量） | 起始值參考 |
| 並列顯示「目前實際庫存」 | 設定時的對照 |
| 提交→簽核流程 | 防呆 |
| 歷史版本／調整軌跡 | 稽核 |

### 1.5 功能基本資訊

| 項目 | 內容 |
|---|---|
| 功能中文名 | 安全存量設定作業（店長視角） |
| 所屬模組 | Excel 列「店長功能」；底層實作在「庫存管理」 |
| 兄弟功能 | 倉儲查詢、入庫管理、出庫管理、門市調撥 |
| 主要頁面 | 自店食材閾值列表、試算頁、編輯頁 |
| 簽核流程 | **建議綁定**（待確認；底層目前未串簽核） |
| 後端店長專屬 endpoint | **無**（推論：未實作；現用總部共用 API） |

## 2. 功能目的

讓店長能依自己門市的銷量結構，**自助維護「我這家店每樣食材的最低庫存水位」**。這個水位是下游模組（需求預測、請購試算、補貨建議）的關鍵輸入：低於水位即觸發補貨建議。設計理念：**店長提供「該店實際營運感受」，系統提供「銷量數據試算」，最終值經簽核確認**。

由於安全存量會反向影響採購決策（拉太高會綁住總部採購、佔倉儲現金）、也會影響營運（拉太低會斷貨），這是比「倉儲查詢」更敏感的設定，**強烈建議綁簽核**。

## 3. 業務邏輯背景

### 3.1 安全存量是怎麼用的

每家門市對每樣食材維護一個閾值。下游：

| 下游 | 用法 |
|---|---|
| 需求預測 | 當預測庫存低於安全存量，納入補貨建議 |
| 請購試算 | 庫存試算公式以此為其中一個輸入 |
| 倉儲查詢 | 低於安全存量時醒目標示 |
| 補貨建議 | 自動拉清單 |

### 3.2 系統試算的邏輯（底層共用）

底層提供「試算」按鈕：依產品銷量歷史 + 食材標準用量推算每個食材的建議安全存量。

⚠️ 底層 PRD 已標示**明顯計算 bug**：

- 程式取用「12 個月銷量總和」而非「每日平均銷量」
- 結果是建議值大 365 倍（例：應為 30 公斤，實際算出約 10,950 公斤）

> 待確認：店長視角應**等底層修正後**再啟用試算，或前端用「÷365」臨時修正？

此外，底層的「加權日數」欄位（例：3 天份、7 天份）**未在試算中使用**。

### 3.3 「我這家店」的判定

安全存量設定的單頭目前綁「倉庫」，明細按食材展開。

⚠️ 問題：

- 底層把「店」與「倉」混為一談（同其他 WHS 功能）
- 店長視角應把「我這家店」對應到「我這家店的所有倉」（若有多倉）
- 一個店長應**僅能設定自己門市對應的倉**；目前後端無此限制

### 3.4 試算來源是誰的銷量

底層試算使用「全公司銷量歷史」展開。

> 推論：店長視角合理的試算應使用「該門市過去 N 個月銷量」而非全公司。
>
> 待確認：中繼系統是否提供「分門市」的歷史銷量資料。

### 3.5 簽核與權限

底層程式預留「流程狀態」欄位，但**未串簽核流程**。對店長視角而言：

- 建議綁簽核，由區經理或總部採購規劃審核
- 大幅調整（例：±50% 以上）必審
- 微調（例：±10% 內）可考慮直接生效

### 3.6 與總部視角的差異

| 對比項 | 總部／採購規劃 | 店長 |
|---|---|---|
| 可設範圍 | 任意門市 | **只能設自己門市** |
| 試算基礎 | 全公司銷量 | **自店銷量**（推論） |
| 寫入頻率 | 季／月度 | 行銷活動、季節變動 |
| 簽核 | 視政策 | **建議必審** |
| 一次調整品項數 | 大批 | 小批（針對性） |
| Excel 匯出 | 全店 | 只自店 |

## 4. 情境說明

### 4.1 正常流程 — 母親節活動前調整

5/10，信義店店長小陳為 6 月母親節活動準備：

1. 進入「我的安全存量」
2. 系統列出信義店現有設定（牛肉餅 30 公斤、麵包 60 個、起司 5 公斤……）
3. 小陳點「試算」→ 系統用信義店過去 3 個月銷量推算
4. 系統建議：牛肉餅 45 公斤（含活動加成）
5. 小陳手動把牛肉餅再拉到 60 公斤（覆蓋系統建議）
6. 提交 → 進入區經理簽核佇列
7. 區經理通過 → 6/1 生效

### 4.2 新品上市

新品「松露華堡」上市，使用新食材「松露醬」：

1. 系統偵測到信義店食材清單新增「松露醬」但無安全存量
2. 列表上標示「未設定」
3. 小陳點「試算」→ 因為新品無歷史，系統建議「同類食材的平均」（推論演算法）
4. 小陳填 0.5 公斤 → 提交

### 4.3 異常情境 — 跨店越權

新進店長阿明用網址列直接打共用 API，傳入板橋店的倉庫 ID：

⚠️ 目前底層**未拒絕**；阿明可成功修改板橋店的安全存量。

此為**目前已知資安風險**。

### 4.4 試算 bug 情境

5/15 小陳試算可可粉：

1. 系統建議 365 公斤（實際應為 1 公斤）
2. 小陳警覺數字明顯異常，手動填 1 公斤
3. 提交

⚠️ 若店長照單全收，會綁住採購過量備料。**強烈建議底層修正後再開放給店長使用試算**。

### 4.5 微調情境

8/10 小陳發現牛肉餅最近常剩太多：

1. 進入列表，找到牛肉餅當前安全存量 60 公斤
2. 改為 50 公斤
3. 因幅度 -16.7%（在 ±20% 內，推論的免簽門檻）→ 直接生效
4. 系統記錄調整歷史

## 6. 欄位規格

### 6.1 列表與編輯欄位

| 欄位（業務名） | 說明 | 店長視角是否顯示 |
|---|---|---|
| 食材品號 | 食材主檔 | 是 |
| 食材名稱 | 食材主檔 | 是 |
| 食材分類 | 食材主檔 | 是（用於分組） |
| 目前安全存量 | 該店目前閾值 | 是 |
| 建議安全存量 | 試算結果 | 試算後顯示 |
| 當前實際庫存 | 來自倉儲查詢 | 是（並列對照） |
| 上次調整日 | 歷史記錄最近一筆 | 是 |
| 加權日數 | N 天份的緩衝 | 待確認（底層欄位存在但未用） |
| 申請單位 | 申請部門 | 否（店長無此概念，預設帶「門市」） |
| 主旨 | 該批調整的說明 | 是（提交時填） |

### 6.2 試算附加欄位

| 欄位 | 說明 |
|---|---|
| 自店過去日均銷量 | 試算基礎 |
| 標準用量 | 該食材每個產品用多少 |
| 建議理由 | 例「過去 90 天日均 5 公斤 × 3 天份 = 15 公斤」 |

### 6.3 店長視角缺少／待補的欄位

- 生效日（指定未來日期生效）
- 失效日（活動結束後自動降回）
- 歷史版本對照（前後差異）
- 簽核狀態
- 簽核者意見

## 7. 商業邏輯

### 7.1 鎖店規則

店長視角的所有讀／寫，應在後端強制：

```
1. 解析登入者所屬門市
2. 寫入時驗證：要設定的門市（或對應倉）必須屬於該店長
3. 不接受參數覆寫
```

> 推論；目前後端未實作。

### 7.2 試算公式（底層既有，含 bug）

底層目前：

```
每日預計銷量 = 該食材所屬產品「過去 12 個月總銷量」（⚠️ 應為日均）
建議安全存量 = 標準用量 × 每日預計銷量
```

修正方向（待 PM 確認）：

```
每日預計銷量 = 該食材所屬產品「過去 N 天日均銷量」
建議安全存量 = 標準用量 × 每日預計銷量 × 加權日數
其中：
  N 推論為 30 或 90 天
  加權日數預設 3 天（例：3 天份的緩衝）
```

### 7.3 免簽／必簽門檻（推論）

| 調整幅度 | 流程 |
|---|---|
| ±10% 內 | 直接生效 |
| ±10%–±50% | 區經理簽核 |
| ±50% 以上 | 區經理 + 總部採購規劃簽核 |
| 新增（首次設定） | 區經理簽核 |

> 推論；實際門檻待 PM 定義。

### 7.4 歷史保留

每次調整視為一個新版本，保留全部歷史。下游模組查當下值。

> 待確認：底層目前無版本概念，採「覆蓋」更新。

### 7.5 同一食材多筆設定

底層當「一張單同時包含華堡與雙層華堡」時，會展開出「牛肉餅」明細兩筆（兩個產品都用）。下游讀取時取第一筆（順序未保證）。

⚠️ 對店長視角而言：「我這家店這個食材」應只有**一個閾值**，不該因產品展開而出現多筆。建議店長視角的列表以「食材」為唯一鍵收斂顯示。

## 8. 使用角色與權限

| 角色 | 可看資料 | 可操作 |
|---|---|---|
| 店長（自店） | 自店所有食材閾值 | 編輯＋提交簽核 |
| 副店長（自店） | 同店長 | 編輯＋提交簽核（待確認是否需店長覆核） |
| 區經理 | 區內所有店 | 編輯＋簽核 |
| 總部採購規劃 | 全公司 | 全 CRUD（既有 API） |
| 系統管理員 | 全公司 | 全 CRUD（後門用） |

> 後端目前以「具備安全存量設定權限」的角色控管，未細分店長／總部，也未鎖店。
