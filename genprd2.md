### 寫作鐵則
 
#### 1. 這不是技術文件
主體章節（§1 ~ §11）避免出現：
- Java 類別、方法名、annotation（`StockController`、`@PreAuthorize`、`@TableName`）
- 資料表名與欄位英文名（`whs_stock`、`store_id`）
- API URL、權限 key 字面（`/whs/stock/page`、`whs:stock:query`）
- BigDecimal scale、RoundingMode 等技術細節寫法
- 字典 key 字面（`dict_type 'safe_stock_setting'`）
- 後端模組短代號（`whs`、`pdm`、`bpm`）— 用中文模組名取代
 
翻譯成業務語言。例：
- ❌ 「`@PreAuthorize("whs:stock:query")`」 ✅ 「具備『倉儲查詢-查看』權限的角色」
- ❌ 「`dict_type 'safe_stock_setting'`」 ✅ 「安全存量分類字典」
- ❌ 「`BigDecimal.divide(qty, 4, HALF_UP)`」 ✅ 「除算後四捨五入到小數第 4 位」
- ❌ 「`whs_stock` 資料表 join `whs_stock_record`」 ✅ 「當前庫存表與庫存異動表」
- ❌ 「`stockServiceImpl.getStockPage` 呼叫 `pdmFeignClient.getMaterial`」 ✅ 「庫存查詢時順帶取得物料主檔資訊」
 
**允許保留的英文／識別字**：
- 業務代號範例（門市編號 `BK001`、單號 `BK-2026-001`）
- 公認縮寫（`ERP`、`BPM`、`PDM`、`SQL`）
- 附錄／來源說明的檔案路徑
 
#### 2. 標註原則
- **若有程式碼證據**：標註來源檔案路徑（檔案路徑可保留英文）
  例：「（來源：`StockServiceImpl.java:128`）」
- **若內容是推論**：標註「推論」
- **若無法確認**：標註「待確認」
 
#### 3. 具體不抽象
情境要有具體場景、具體物件、具體數字（即使是示意數字）。
 
例：
- ❌ 「使用者查詢倉儲庫存」
- ✅ 「總部採購人員小李在 5/18 早上要備料前，查詢台北信義店牛肉餅當前庫存，系統回傳 32 公斤，安全存量 50 公斤，狀態標『低於安全存量』」
 
#### 4. ASCII 流程圖
§5 操作流程用 ASCII 樹狀圖。後端無 UI，可改以「API 呼叫鏈」或「狀態流轉」表達。例：
```
[查詢請求進入]
  ├─ 權限檢查：是否具備「倉儲查詢-查看」
  ├─ 鎖區／鎖店判斷（依使用者層級）
  ├─ 主表查詢 → 補上物料主檔資訊
  └─ 回傳分頁結果
```

- §10.2「預留但尚未實作」無內容寫「無」
- 沒有的情境（§4.x）就不寫，不要硬補
