# 反轉 ERP 規格

逆向掃描 `kingmaker-module-*` 的 Java 程式碼，產出 PM／SA／QA 可讀的中文 PRD。**不是技術文件** — 主體章節必須翻成業務語言。

## 啟動時的輸入

呼叫此 skill 時，使用者通常給：

- **模組功能名**（必要） — 例如「系統管理 — 使用者管理」「WHS — 庫存查詢」「PMM — 採購單」
- **Controller 子目錄關鍵字**（可推） — 例如 `user`、`stock`、`requisition`；若未給，從模組功能名翻譯／搜尋
- **後端程式碼根目錄**（可推） — 預設 `d:\burgerking\erp-spring`
- **輸出檔案路徑**（必要） — 使用者通常會說「輸出到 `xxx` 目錄」

缺哪項就直接從上下文／既有結構推；推不出來才問。

## 掃描順序

每個 `-biz` 模組標準結構：

| 順序 | 路徑 | 抽取什麼 |
|---|---|---|
| 1 | `controller/admin/{feature}/*Controller.java` | API 端點清單、`@PreAuthorize` 權限字串、`@Operation.summary` 摘要、Request/Response 型別 |
| 2 | `controller/admin/{feature}/vo/**/*.java` | Req/Resp/Page VO 欄位、驗證規則（`@NotBlank`、`@Size`、`@Email`、`@Pattern`） |
| 3 | `service/{feature}/*ServiceImpl.java` | 業務邏輯主場：驗證關卡、唯一性檢查、計算公式、分支判斷、交易範圍、`@LogRecord`、上游服務呼叫 |
| 4 | `dal/dataobject/{feature}/*DO.java` | `@TableName` 對應資料表、欄位、`@TableField` 特殊處理 |
| 5 | `dal/mysql/{feature}/*Mapper.java` | 查詢條件組合、join、過濾、排序 |
| 6 | `enums/ErrorCodeConstants.java` | 業務錯誤碼與訊息（可作為 §4 異常情境的素材） |
| 7 | （若有）`convert/*Convert.java` | 欄位對應、補資料規則 |
| 8 | （若有）`listener/`、`job/`、流程綁定 | 事件監聽、排程、BPM 簽核 |

## 抽取要素

從程式碼中找出：

1. **API 清單**：每支端點做什麼、需要什麼權限、是否涉及檔案上傳／匯出
2. **資料表**：對應的資料表、表間關係、特殊欄位（如 JSON、加密、軟刪除）
3. **核心計算與分支**：BigDecimal scale／RoundingMode、type/category 分流、上限／下限
4. **權限**：`@PreAuthorize` 字串 → 翻成「具備 XX 權限的角色」
5. **字典依賴**：`@DictFormat` 對應的字典類型
6. **唯一性／啟用狀態驗證**：哪些欄位 unique、哪些上游資料要 enable
7. **特殊處理**：交易範圍、回滾條件、忽略某欄位（如修改時清空 password）、隱性業務規則

## 翻譯鐵則

主體章節（§1–§11）**禁止出現**下列技術細節：

| ❌ 不可寫 | ✅ 改寫成 |
|---|---|
| `StockController`、`@PreAuthorize`、`@TableName` | 業務語言敘述 |
| `whs_stock`、`store_id`（資料表名／英文欄位） | 「當前庫存表」「門市編號」 |
| `/whs/stock/page`、`whs:stock:query`（URL／權限字面） | 「具備『倉儲查詢-查看』權限的角色」 |
| `BigDecimal.divide(qty, 4, HALF_UP)` | 「除算後四捨五入到小數第 4 位」 |
| `dict_type 'safe_stock_setting'` | 「安全存量分類字典」 |
| `whs`、`pdm`、`bpm`（模組短代號） | 「倉儲管理」「PDM」「審批流」 |

**允許保留的識別字**：

- 業務代號範例（門市編號 `BK001`、單號 `BK-2026-001`）
- 公認縮寫（`ERP`、`BPM`、`PDM`、`SQL`）
- §11 待確認 或 標註來源用的檔案路徑（用反引號）

## 標註原則

- **有程式碼證據**：在 §7、§11 或敏感段落附「（來源：`StockServiceImpl.java:128`）」
- **推論**：標註「（推論）」
- **無法確認**：列入 §11 待確認事項

## 輸出章節結構

依下列骨架輸出 Markdown：

```
# PRD｜<模組> — <功能>

> 來源：逆向自 ... 的後端程式碼。本文件為 PM 對齊需求、SA 拆 task、QA 寫測試案例之工作文件。

## 1. 功能概覽
   1.1 我是誰              ← 第一人稱、預設主要使用者角色視角
   1.2 我要做什麼          ← 第一人稱、條列每週／每日工作任務
   1.3 我有什麼需求        ← 表格「我的需求 ｜ 背後的問題」
   1.4 因此，需要以下功能  ← 表格「功能 ｜ 解決的問題」
   1.5 功能基本資訊        ← 表格：功能中文名、所屬模組、兄弟功能、主要頁面、簽核流程

## 2. 功能目的              ← 1–2 段業務目標陳述，含關鍵設計理念
                              （若該模組與 OA 整合過渡相關，需在此標註）

## 3. 業務邏輯背景          ← 解釋 domain，非操作面
   3.x  業務分類議題／資料來源／核心計算邏輯／關鍵設計理念
        （依該功能需要增減；不要硬補）

## 4. 情境說明              ← 敘事式 user story，禁用 Gherkin GWT
   4.1 正常流程
   4.x 典型業務／異常／使用者鍵入錯誤／合理高偏離／規則分流／自動計算／多來源
        （沒有的情境就不寫）

## 5. 操作流程              ← ASCII 樹狀（API 呼叫鏈或狀態流轉）

## 6. 欄位規格              ← 由 ReqVO / RespVO / DO 推得，欄位名用中文業務語

## 7. 商業邏輯              ← Service 內計算公式、分支、邊界條件，翻成業務語

## 8. 使用角色與權限        ← 角色 × 可看資料 × 可操作（@PreAuthorize 翻成業務語）

## 9. 畫面需求/視覺規範      ← 後端無 UI 細節寫「待前端對照」或「無 UI」

## 10. 功能範圍
   10.1 包含的功能
   10.2 預留但尚未實作       ← 有欄位但 Service 未使用、有 enum 但無分支；無則寫「無」
   10.3 不包含               ← 只列功能本身的範圍邊界，不列其他模組／引擎類

## 11. 待確認事項            ← 表格：議題、為何要確認、證據來源
```

## 寫作風格

### 1. 具體不抽象

每個情境要有具體場景、具體物件、具體數字（即使是示意數字）。

範例：

- ❌「使用者查詢倉儲庫存」
- ✅「總部採購人員小李在 5/18 早上要備料前，查詢台北信義店牛肉餅當前庫存，系統回傳 32 公斤，安全存量 50 公斤，狀態標『低於安全存量』」

### 2. ASCII 流程圖

§5 用 ASCII 樹狀圖表達操作流程／API 呼叫鏈。

```
[查詢請求進入]
  ├─ 權限檢查：是否具備「倉儲查詢-查看」
  ├─ 鎖區／鎖店判斷（依使用者層級）
  ├─ 主表查詢 → 補上物料主檔資訊
  └─ 回傳分頁結果
```

中文混 ASCII 時**不畫右邊框**（中文字寬不一致導致對齊困難），只用左邊框與上下分隔。

### 3. 章節省略原則

- 後端無 UI → §9 寫「待前端對照」或「無 UI」
- §10.2 無內容寫「無」
- 沒有的 §4.x 子情境就不寫，不要硬補

## 與專案規範的對齊（漢堡王 ERP 特有）

寫 PRD 時務必對齊以下專案決策：

### 範圍從嚴

§10.3「不包含」只列**功能本身的範圍邊界**（例如「使用者管理」不含「角色維護本身」），**不要列**：

- 其他業務模組的功能（會由各自的 PRD 涵蓋）
- 引擎類模組（BPM／Flowable／Infra／Codegen／Quartz）
- 樣板殘留警示

### OA 整合過渡背景

`system` 模組（使用者、部門、角色、選單等）相關 PRD，**必須**在 §2 標註：

> 經系統整合決策（2026-05-16），漢堡王 IT 走向以 Laravel OA 為使用者單一事實來源、ERP 不再自簽 JWT。本模組屬於 ERP 過渡期保留功能；長期規劃中將在搬遷階段 5 後廢棄。本 PRD 描述的是過渡期內、ERP 仍自管使用者主檔時的需求。

業務模組（PDM／WHS／PMM）通常無此標註；但若涉及 user/dept/role 引用，可在 §3 簡述「人員與部門資訊長期將以 OA 為準」。

### 凍結模組不寫

`kingmaker-module-bhm`（漢堡王基礎資料）目前凍結，**不要為此模組產 PRD**；若使用者要求請先提示。

## 執行步驟（給未來的自己）

1. **確認輸入**：模組功能名、Controller 子目錄關鍵字、後端根目錄、輸出檔名。缺則推或問。
2. **定位檔案**：用 Glob 找出 Controller + Service + DO + VO + Mapper 五大檔案。
3. **讀 Controller**：抽 API 清單、權限字串、Operation summary。
4. **讀 ServiceImpl**：抽核心業務邏輯（這是 PRD 的肉）。
5. **讀 DO + VO**：抽欄位規格、驗證規則。
6. **讀 Mapper**：抽查詢條件、特殊 join。
7. **讀 ErrorCodeConstants**：抽錯誤訊息（§4 異常情境素材）。
8. **檢查上下文記憶**：是否有相關專案決策（如 OA 整合）需要標註。
9. **依章節結構撰寫**：套用翻譯鐵則，業務語化；具體不抽象；ASCII 樹狀無右邊框。
10. **§11 列出疑點**：發現的潛在 bug、未檢查的邊界、未實作的預留欄位、政策性問題（閾值、流程歸屬）等。
11. **輸出到指定路徑**：Markdown 檔，檔名建議 `prd_<模組>_<功能>.md`。

## 參考範例

`d:\burgerking\erp-claude\document\PRD\prd_系統管理_使用者管理.md` 為一次完整的輸出範例，可作為章節飽和度與寫作風格的對照。
