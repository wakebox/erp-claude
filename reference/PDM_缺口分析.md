# PDM 功能缺口分析

## ① 食材基本資料管理

| # | 缺少的功能 | 業務影響 |
|---|---|---|
| 1-1 | **BOM 修改為全刪重建，無歷史快照**：BOM 子表每次修改都是整批刪除再重寫，系統不保留舊版記錄。財務面影響：當月成本計算使用現在的 BOM，若配方在月中調整（換食材或改用量），系統無法重現修改前的 BOM，導致毛利分析對不上、跨期成本比較失真，稽核時也無法舉證特定時間點的標準成本依據。 | 財務成本可信度、稽核風險 |
| 1-2 | **規格停用後，下游 BOM 未自動排除**：食材主檔下可掛多筆規格（如同一種番茄有「整顆冷藏」和「切丁冷凍」兩種規格，各有獨立品號）。規格標記停用後，系統未限制 BOM 子表繼續引用該規格，採購或備料計算時可能抓到已無廠商供應的品號。 | 採購錯誤、備料異常 |

## ② RBI / EMC 串接 ＋ NCR 雙後台對應

| # | 缺少的功能 | 業務影響 |
|---|---|---|
| 2-1 | **PID 主檔維護功能完全缺失**：PDM 無從 EMC 同步或手動建立 PID 的介面，單品主檔僅預留一個欄位，無任何維護 UI。 | PDM 無法成為 PID 唯一資料來源 |
| 2-2 | **EMC 串接設定缺失**：無 API 同步排程、差異比對、衝突處理介面，PID 資料須人工維護。 | 人工維護易出錯，資料同步無保障 |
| 2-3 | **NCR 中繼端串接設定缺失**：100+ 家門市使用 NCR 中繼端，PDM 無對應欄位或同步機制，大多數門市品項資料無法集中管理。 | 大多數門市品項資料無法在 PDM 統一管理 |
| 2-4 | **雙後台對照表缺失**：同一單品在 EMC 有一個 PID、在 NCR 中繼端有另一組 ID，兩套 ID 體系完全獨立。PDM 需在單品主檔同時維護「EMC PID」與「NCR ID」兩欄，否則跨後台比對只能靠人工 Excel 處理。 | 無法在 PDM 做跨後台比對與稽核 |
| 2-5 | **門市分群與後台類型設定缺失**：PDM 未維護「哪些門市對接 EMC、哪些門市對接 NCR 中繼端」的對應關係，品項資料推送時無法自動判斷目標後台，需人工每次判斷分流。 | 品項推送無法自動分流，人工作業風險高 |
| 2-6 | **套餐（Combo）組成管理缺失**：套餐的主品 + 副品搭配關係需在 PDM 定義，套餐結構來源為 EMC，再經由 NCR 中繼端同步至對應門市。目前 PDM 無套餐結構定義，無法承接此資料流。 | 套餐無法正確經由中繼端同步至門市 |
| 2-9 | **同步排程與異常通知缺失**：PDM 與 EMC、NCR 中繼端之間目前無同步排程機制。當三邊資料因操作時序不同而產生飄移時，系統無法主動發出通知，需仰賴人工定期比對才能發現不一致。建議加入同步排程或資料異常通知機制。 | 資料飄移時無法即時發現，影響門市上架正確性 |

## ③ BOM 維護（食材類 ＋ 非食材類）

| # | 缺少的功能 | 業務影響 |
|---|---|---|
| 3-1 | **非食材 BOM 類型缺失**：包材、耗材無獨立分類欄位，與食材混用同一 BOM 結構，成本計算時無法分開彙整食材成本與包材成本。 | 單份成本結構不清，食材與包材成本無法分離 |
| 3-2 | **BOM 有效期間缺失**：無法設定 BOM 版本的生效起訖日，換季配方或新品上市無法提前設定，只能到當天手動切換。 | 換季或活動切換時需人工介入，風險高 |
