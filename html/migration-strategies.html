漢堡王 ERP 遷移：前後端並行策略

基於漸進式遷移（Phase 0–5）的前端與後端雙軌執行指南
版本：v1.0 日期：2026-05-20 參考：java-to-laravel-migration.html § 8

後端並行核心原則

Java 凍結維護期
  • 除阻斷性 bug，不接受新功能 PR
  • 繼續服務「尚未遷移」的模組路由
  • PostgreSQL 保持唯讀副本，直到階段 5
Laravel 主力開發
  • 所有新功能、所有新模組一律寫在 Laravel
  • 共用 MySQL，表前綴 erp_<module>_
  • 沿用 OA 的使用者體系、流程引擎、JWT

核心架構：Nginx Gateway 雙 Upstream

┌─────────────┐ │ Vue 前端 │ ← 前端代碼一份，不因後端分叉 └──────┬──────┘ │ HTTP / WebSocket ┌──────▼──────────────────────────────┐ │ Nginx Gateway │ │ 路由表決定每個 URL 打到哪個後端 │ └──┬──────────────────────────────┬───┘ │ │ ┌──▼──────────────┐ ┌──────────▼───────────────┐ │ Java ERP │ │ Laravel OA + ERP │ │ :48080 (PG) │ │ :8080 (MySQL) │ │ │ │ │ │ pdm/forecast/ │ │ pdm/ingredient/ ✅ P1 │ │ whs/* │ │ pdm/bom/ ✅ P1 │ │ pmm/* │ │ bhm/* ✅ P1 │ │ bhm/* │ │ whs/* ✅ P2 │ │ (待遷移) │ │ pmm/* ✅ P3 │ └─────────────────┘ │ pdm/forecast/ ✅ P4 │ └───────────────────────────┘
切換開關：新增一條 Nginx location，然後 nginx -s reload — 零停機切換。回退同理：刪掉那行重載即可。

各階段後端工作清單

Phase 0 地基（並行前提）

1.5 ~ 2 人月

目標：讓 Java 與 Laravel 能在同一個 JWT 下共存，且流程引擎 PoC 通過。

Java 側工作

  • 改為接受 Laravel OA 發的 JWT（公鑰驗簽）
  • 停止自己的 /auth/login，統一走 OA 登入
  • 凍結新功能合併：建立 java-freeze 分支政策

Laravel 側工作

  • 建立 BaseModel（id / creator / createTime / deleter / deleteTime）
  • 建立 CommonResult、PageResult Response Macro
  • 建立 ErrorCode 字典與全局 Exception Handler
  • 流程整合 helper（MenuFlowProcessInstanceHelper 對應）
  • DB 命名規約：erp_pdm_、erp_whs_、erp_pmm_
風險閘門：流程引擎 PoC 必須跑通（建立假單 → OA 起流程 → callback 回寫狀態），否則時程估算無效，禁止進入 Phase 1。

Nginx 初始狀態（所有流量打 Java）

upstream java_erp    { server 127.0.0.1:48080; keepalive 64; }
upstream laravel_erp { server 127.0.0.1:8080;  keepalive 64; }

server {
    listen 80;
    server_name erp.example.com;

    proxy_http_version 1.1;
    proxy_set_header Connection        "";
    proxy_set_header Authorization     $http_authorization;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    location /admin-api/ { proxy_pass http://java_erp; }   # ← 全部 Java
}

Phase 1 BHM + PDM 主檔

2 ~ 2.5 人月

選擇理由：純 CRUD、無流程、無跨模組寫操作依賴。失敗影響面最小，最適合熱身。

Laravel 實作範圍

  • BHM 客戶基礎資料（若解凍）
  • PDM 食材、產品、BOM、配方（不含需求預測計算）
  • 漢堡王中繼 API：TokenManager + 3 條 Guzzle 呼叫
  • PG dump → DDL 翻 MySQL（保留原始 ID）

Java 側動作

  • 對應 Controller 加 @Deprecated + 寫入 log 警告
  • 尚未遷移的模組（whs / pmm）改走 Laravel REST API 取 PDM 主檔
  • Feign Client：PdmIngredientFeignApi 改指 Laravel 端

Nginx 切換（新增 4 行）

server {
    # ─── Phase 1：已遷移到 Laravel ─────────────────────────
    location ^~ /admin-api/pdm/ingredient/ { proxy_pass http://laravel_erp; }
    location ^~ /admin-api/pdm/bom/        { proxy_pass http://laravel_erp; }
    location ^~ /admin-api/pdm/product/    { proxy_pass http://laravel_erp; }
    location ^~ /admin-api/bhm/            { proxy_pass http://laravel_erp; }

    # 其餘 PDM 子路徑（forecast）仍走 Java
    location ^~ /admin-api/pdm/            { proxy_pass http://java_erp; }

    # 預設：Java
    location /admin-api/ { proxy_pass http://java_erp; }
}
跨系統引用：Java 的 WHS / PMM 若需要讀 PDM 食材，改成 HTTP 呼叫 GET /admin-api/pdm/ingredient/{id}（Laravel），不再直接 JOIN PostgreSQL。

Phase 2 WHS 庫存模組

2 人月

選擇理由：WHS 只依賴 PDM 食材（已遷移完畢）；盤點單是第一個走 OA 流程的業務單，驗證 Phase 0 helper。

Laravel 實作範圍

  • 倉庫、儲位主檔 CRUD
  • 進出庫單、即時庫存
  • 盤點單（接 OA 流程）
  • 庫存扣減：SELECT ... FOR UPDATE 或 Cache::lock

Java 側動作

  • WHS Controller 全部標 @Deprecated
  • PMM 的驗收寫庫動作改呼叫 Laravel WHS API
  • WHS Feign Client 改指 Laravel
壓測要求：庫存併發扣減必須跑負載測試（模擬 50 並發倉操），確認 SELECT ... FOR UPDATE 不會死鎖，且 QPS 滿足業務需求後才切換。

Nginx 新增

location ^~ /admin-api/whs/ { proxy_pass http://laravel_erp; }

Phase 3 PMM 採購模組

2.5 人月

選擇理由：PMM 驗收單寫 WHS — WHS 已在 Laravel，可在同一進程呼叫 Service，效率最高。

Laravel 實作範圍

  • 廠商主檔、報價維護
  • 請購 → 詢價 → 採購 → 驗收（狀態機）
  • 用 spatie/laravel-model-states 建模
  • 多階段審批（金額分級，走 OA 流程）
  • 跨單關聯：請購拆採購、採購分批驗收

Java 側動作

  • PMM Controller 全部標 @Deprecated
  • BPM biz 層的 PMM callback listener 刪除
  • 若有其他 Java 模組引用 PMM，改走 Laravel REST API

Nginx 新增

location ^~ /admin-api/pmm/ { proxy_pass http://laravel_erp; }

Phase 4 PDM 計算層 / 需求預測

2 ~ 3 人月

選擇理由：計算依賴 PDM 主檔（P1）、PMM（P3）、WHS（P2）全部就位，計算輸入完整。

Laravel 實作範圍

  • 拉漢堡王 API 銷售資料（P1 已有 TokenManager）
  • 需求預測計算（建議寫單元測試覆蓋邊界）
  • BOM 樹狀展開 → 原物料需求（遞迴）
  • 預測單審批 → 觸發 PMM 請購
  • 批次計算包成 Queue Job（Horizon），必要時上 Octane

Java 側動作

  • PDM forecast Controller 標 @Deprecated
  • BPM 的預測單 listener 刪除
  • Java 已幾乎只剩基礎設施，無業務邏輯

Nginx 新增

location ^~ /admin-api/pdm/forecast/ { proxy_pass http://laravel_erp; }

Phase 5 拆除 Java 後端

0.5 人月

前提：Phase 1–4 全部驗收、UAT 跑過至少一個完整業務週期（月結 / 採購週期）。

  • Nginx 路由表清空 java_erp 相關 location，所有 /admin-api/* 指向 Laravel
  • Java 服務停機；保留 Docker Image + PG snapshot 作為 90 天回退保障
  • PostgreSQL 退役；歷史資料 read-only 副本保留 6 個月
  • 刪除 Java upstream 區塊，Nginx config 清潔完成
不可逆動作：停機前先確認 PG snapshot 完整可還原，並在低峰時段執行，預留 4 小時回退窗口。

Java ↔ Laravel 跨模組呼叫規則

情境做法說明
Java 讀已遷移的 Laravel 主檔 Java Feign Client → Laravel REST API 不再 JOIN PostgreSQL；保持 ID 不變，呼叫 GET /admin-api/pdm/ingredient/{id}
Java 寫已遷移的 Laravel 表 禁止直接寫 MySQL；改呼叫 Laravel API 避免雙寫導致資料不一致
Laravel 讀尚未遷移的 Java 資料 Laravel Guzzle / Saloon → Java REST API 僅在 Phase 過渡期使用；遷移完成後刪除
同一 Phase 內跨服務（已全遷 Laravel） 直接呼叫 Laravel Service 層 無需 HTTP；同進程內調用效率最高

回退機制

每個 Phase 切換的標準回退流程（30 秒內完成）：
  1. 回退 PR：移除對應 location ^~ 行
  2. 執行 nginx -t && nginx -s reload
  3. 流量立即回到 Java；資料寫入自動走回 PostgreSQL
  4. Laravel 側資料（若有）人工 rollback migration，或保留等待再次切換

切換 PR 與回退 PR 應同時準備好，並在 Code Review 時一起審查。

前端並行核心原則

鐵律 Vue 代碼只有一份，不分叉

前端（erp-kingmaker）不因後端切換而建立分支版本。所有 API base URL 切換由 Nginx Gateway 在伺服器端完成，前端完全感知不到後端語言。

前端需要做的事
  • 維持 CommonResult<T> / PageResult<T> 回應格式
  • 調整因後端重構而變動的欄位名稱
  • 移除對已廢棄 Java 端 API 的特殊處理
  • WebSocket 通道切換（Reverb 取代 Java WS）
前端不應該做的事
  • 建立 if (env === 'java') ... else ... 的 API 分支
  • 為 Java / Laravel 各自維護一套 API service 檔
  • 硬編碼後端 IP / Port
  • 自行做流量切換（應由 Nginx 負責）

API 契約：前端的唯一關注點

前端只需關心 API 路徑 與 回應結構 是否一致，不管後端是 Java 還是 Laravel。

合約項目Java 原版Laravel 目標前端需改動？
成功回應結構 {"code":0,"data":{...},"msg":"ok"} 完全相同（Response Macro） 不需
分頁結構 {"list":[...],"total":N} 完全相同 不需
錯誤碼結構 {"code":1001002,"msg":"..."} 完全相同（ErrorCode 字典對齊） 不需
日期格式 timestamp（ms） 或 ISO 8601 需統一 — 若格式有差，前端要加 adapter 需確認
欄位命名 camelCase（Java 慣例） Laravel 預設 snake_case → 需設定 JsonResource::withoutWrapping + transform 需確認
JWT Token 位置 Authorization: Bearer <token> 完全相同（Nginx 轉發） 不需
WebSocket 通道 Java WebSocket Starter ws://erp.example.com/ws Laravel Reverb ws://erp.example.com/app/KEY 需改（Phase 2 起）
檔案上傳路徑 /admin-api/infra/file/upload 沿用 OA 既有路徑或建立相容路徑 需確認
最常踩的雷：camelCase vs snake_case。建議在 Laravel 全域設定 JsonResource 的 key 輸出為 camelCase，或在 Phase 0 時一次性在前端加 Axios response interceptor 轉換，避免每個模組個別處理。

環境設定：API Base URL 管理

前端 API base URL 只有一個入口，不因後端語言而分叉：

# .env.development
VITE_API_BASE_URL=http://localhost:80   # 打 Nginx dev proxy

# .env.staging
VITE_API_BASE_URL=https://erp-staging.example.com

# .env.production
VITE_API_BASE_URL=https://erp.example.com
// src/api/request.ts  — Axios 全域設定
import axios from 'axios'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 30000,
})

// Response interceptor：統一處理 CommonResult
request.interceptors.response.use(res => {
  const { code, data, msg } = res.data
  if (code !== 0) throw new ApiError(code, msg)
  return data
})

export default request
開發環境 Vite Proxy：本機不需要 Nginx，Vite server.proxy 也只設一個 target（打本機 Nginx 或直接打 Laravel dev server 視開發階段而定）。

各階段前端工作清單

Phase 0
地基
準備期 前端配合事項
  • 確認 VITE_API_BASE_URL 已統一，移除任何硬編碼後端位址
  • 加入 Axios response interceptor（code / data / msg 解構）
  • 驗證 camelCase / snake_case 策略，決定是否加 key transformer
  • 與後端確認 JWT 格式（payload 欄位名稱）一致
  • PoC：使用假帳號呼叫 OA 登入 → 取得 JWT → 打 Java ERP API，確認 token 可穿透
Phase 1
PDM 主檔
低風險切換 前端驗收清單
  • Nginx 切換後，食材 / BOM / 產品頁面功能測試（CRUD 完整 e2e）
  • 分頁、搜尋、Excel 匯出確認正常
  • 確認 PDM 主檔 API 回應格式與 Java 版完全一致（可用 Postman collection 對比）
  • 若有欄位名稱差異 → 前端加 adapter 或後端對齊，不寫 if/else 分支
Phase 2
WHS 庫存
含 WebSocket 前端改動
  • 庫存異動通知：從 Java WebSocket → Laravel Reverb
  • 前端 Laravel Echo 設定（echo.channel('inventory').listen(...)）
  • 盤點單頁面：確認流程狀態欄位（status）與 OA 流程回調格式一致
  • 庫存並發警告：壓測若發現前端輪詢太頻繁，改為 WebSocket push
WebSocket 切換是本次遷移前端工作量最大的單點，建議單獨安排 1~2 天 spike。
Phase 3
PMM 採購
含多步驟審批 UI
  • 審批流程 UI：確認 OA 流程引擎回傳的審批節點資料格式
  • 請購 → 詢價 → 採購 → 驗收 四個頁面狀態機顯示測試
  • 驗收單觸發的庫存寫入：確認 WHS 庫存頁面即時更新（WebSocket）
  • 如需在審批頁嵌入 OA 流程圖 — 確認 OA 提供對應的 iframe 或 API
Phase 4
PDM 計算
含非同步計算 UI
  • 需求預測頁面：計算為非同步（Queue Job），前端需實作「計算中 / 完成 / 失敗」狀態輪詢或 WebSocket push
  • BOM 展開結果：樹狀表格 / 折疊展開 UX 確認
  • 預測單觸發請購：確認跳轉到 PMM 頁面帶入預填資料的流程正確
Phase 5
收尾
清理期
  • 移除所有對 Java API 的特殊 workaround 或 adapter
  • 移除開發工具中 proxy 指向 Java 的設定
  • 更新 API 文件（Swagger / Postman collection）
  • 確認 Vue Router 所有路由都對應到 Laravel 端有實作的 API

WebSocket 切換詳解（Phase 2）

Java 側（舊）

// 舊：直連 Java WebSocket
const ws = new WebSocket(
  'ws://erp.example.com/ws?token=...'
)
ws.onmessage = (e) => {
  const { type, data } = JSON.parse(e.data)
  // ...
}

Laravel Reverb（新）

// 新：Laravel Echo + Reverb
import Echo from 'laravel-echo'
import Pusher from 'pusher-js'
window.Pusher = Pusher

const echo = new Echo({
  broadcaster: 'reverb',
  key: import.meta.env.VITE_REVERB_KEY,
  wsHost: import.meta.env.VITE_REVERB_HOST,
  wsPort: import.meta.env.VITE_REVERB_PORT,
  forceTLS: false,
  auth: { headers: {
    Authorization: `Bearer ${token}`
  }}
})

echo.channel('inventory')
  .listen('StockUpdated', (e) => {
    store.updateStock(e.warehouseId, e.qty)
  })
Phase 2 切換期間，兩套 WebSocket 可能同時存在（Java 舊通知 + Laravel 新通知）。建議用 feature flag（env var 或 Cookie）讓前端在切換期間只連一套，避免重複通知。

前端 QA 策略

QA 層次工具時機
API 契約比對 Postman Collection Runner（Java vs Laravel 同路徑回應對比） 每個 Phase 切換前
功能迴歸 Playwright E2E（關鍵頁面流程：CRUD + 流程審批） 每個 Phase 切換後
灰度測試（QA 帳號） Cookie erp_canary=new → Nginx map 導到 Laravel 切換前 1 週，讓內部 QA 先用 Laravel 版本
UAT 真實業務場景跑一個完整週期（週採購、月盤點） Phase 4 完成後，Phase 5 前

灰度切換：讓 QA 先體驗 Laravel（指定使用者）

# Nginx：map cookie 決定後端
map $cookie_erp_canary $whs_backend {
    default  java_erp;
    "new"    laravel_erp;
}
server {
    location ^~ /admin-api/whs/ {
        proxy_pass http://$whs_backend;
    }
}

QA 在 DevTools 手動設定：

document.cookie = "erp_canary=new; path=/"

同一個 URL，QA 帳號體驗 Laravel 版本，其他使用者繼續用 Java — 前端代碼完全不需要改動。
