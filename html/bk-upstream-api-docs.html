🔐 JWT 認證機制 — BurgerKingTokenManager

⚙️ Feign 設定 — BurgerKingFeignConfig

所有通過 BurgerKingStoreClient 發出的請求，都會被 RequestInterceptor 自動附加 Authorization: Bearer ... Header，不需在每個方法上手動傳入 Token。

@FeignClient(
    name = "burgerkingStoreClient",
    url  = "http://61.218.209.215:80/api",    // 注意：url 已含 /api 前綴
    configuration = BurgerKingFeignConfig.class
)
public interface BurgerKingStoreClient { ... }

// Interceptor 邏輯
@Override
public void apply(RequestTemplate template) {
    // 僅當 Header 尚未含 Authorization 時才注入
    if (!headers.containsKey("Authorization")) {
        String auth = tokenManager.getAuthorization(null); // 強制走遠端 BK 登入
        if (auth != null) template.header("Authorization", auth);
    }
    // 不透傳 tenant-id（目前不考慮多租戶）
}
注意： BurgerKingFeignConfig 中 tokenManager.getAuthorization(null) 傳入 null，意即完全忽略當前使用者請求頭，只使用系統配置的帳密登入。這確保所有呼叫都使用同一組 BK 管理員憑證。

ERP 業務邏輯中的使用

// 1. 有使用者區域/門店權限時，自動過濾資料
public List<GroupWithStoresInnerVO> getRemoteGroupWithStoresInner() {
    CommonResult<List<GroupWithStoresInnerVO>> remote =
        burgerKingStoreClient.getGroupWithStoresInner();

    Integer loginUserAreaId  = SecurityFrameworkUtils.getLoginUserAreaId();
    Integer loginUserStoreId = SecurityFrameworkUtils.getLoginUserStoreId();

    // 無任何限制 → 全量返回
    if (loginUserAreaId == null && loginUserStoreId == null) return remote.getData();

    // 有區域限制 → 過濾區域
    // 有區域+門店限制 → 再過濾門店
    return remote.getData().stream()
        .filter(group -> loginUserAreaId == null || loginUserAreaId.equals(group.getId()))
        .map(group -> {
            if (loginUserAreaId != null && loginUserStoreId != null) {
                group.setStores(group.getStores().stream()
                    .filter(s -> loginUserStoreId.equals(s.getId()))
                    .collect(Collectors.toList()));
            }
            return group;
        }).collect(Collectors.toList());
}
關鍵設計：ERP 實作了「依登入使用者的 areaId / storeId 自動過濾」機制，確保區域主管只看到自己負責區域的門店，門店人員只看到自己門店的資料。資料來源仍是 BurgerKing 全量 API，過濾在 ERP 層完成。

🧮 需求預測核心流程

ERP 的需求預測主流程 getProductRecipeAnalysis() 整合了 BurgerKing 上游 API 與本地資料庫，計算每個門店、每個商品對應食材的需求量預測。

呼叫 BK API 取得銷量統計
呼叫 getCompletedOrdersFilterCached(groupAreaId, startTime, endTime, storeId)，取得指定區域、時間範圍內各門店各商品的銷量（已含平日/假日拆分）。
提取商品 ID，批量查詢食材配方
從銷量資料中提取所有 productId（去重），一次性查詢本地 DB（pdmProductRecipeRelMapper）取得各商品對應的食材配方，建立 productId → List<配方> 的 Map。
LONG 儲存類型食材：預載安全存量與門市庫存
篩選儲存類型為「LONG + 食材小類」的食材，批量查詢安全存量（whs_stock_safe）和門市現有庫存（whs_stock），避免後續逐筆查詢。
查詢廠商報價（有指定門店時）
若 storeId != null，查詢各食材對應的最新廠商報價（pmm_vendor_quote），建立 prodCode → 報價 的 Map，用於計算預測成本。
獲取區域名稱
呼叫 getGroupSummaries()（內部再次呼叫 BK API group-with-stores/inner）取得區域名稱，填入預測結果中。
逐門店、逐商品建構需求預測明細
對每個門店的每個商品，結合銷量統計（平日/假日均量）× 食材配方標準用量 × 預測加成倍數，計算每種食材的預測需求量（projected），輸出 StoreDemandForecastVO 結構。

📅 平日 / 假日計算規則

BurgerKing 後端已根據以下規則計算好平日/假日拆分，ERP 直接使用回傳的統計欄位。

📅 平日（Weekday）
週一 ～ 週四
一般客流量較平穩
🎉 假日（Holiday）
週五 ～ 週日
客流量通常高於平日
設計說明：OrderCompletedFilterVO.productsList 原本是 JSON 字串（[{"productId":2,"ProductTasteId":3},...]），需要在 ERP 端解析後結合 combinationQuantity 計算商品實際數量，再依 completedTime 的星期幾分入平日或假日桶。目前此邏輯已由 BurgerKing 後端直接處理並回傳 ProductSalesStatisticsVO 格式，ERP 不需再重複計算。
