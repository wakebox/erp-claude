什麼是 HMR？

Hot Module Replacement (HMR) is a mechanism that allows modules to be updated at runtime without a full page refresh. It dramatically improves the development experience by preserving application state during updates. — Vite 官方文件

HMR 解決的核心問題：

🔄
無需整頁重載
修改一個 Button 元件，只更新那個元件，其他頁面狀態（表單輸入、捲軸位置、store 資料）完全保留。
狀態保留
⚡
毫秒級回饋
不需重新打包整個 bundle，只針對變更模組做 ESM 動態 import，更新時間通常 < 100ms。
效能
🧩
模組粒度更新
以單一 ES Module 為單位更新，透過 Module Graph 精確計算最小影響範圍。
精確性
🎨
CSS 即時注入
CSS/SCSS 修改直接替換 <style> tag，不產生任何 JS 更新，無 FOUC（樣式閃爍）。
UX

HMR Boundary — 邊界概念

HMR Boundary 是 HMR 最關鍵也最容易被誤解的概念。它決定了「更新要傳播到哪裡為止」。

什麼是 Boundary？

一個模組如果呼叫了 import.meta.hot.accept()，它就是一個 HMR Boundary（邊界）。當這個模組或其子依賴變更時，更新會在這裡「被吸收」，不再往上傳播。

graph BT A["App.vue\n(無 accept)"] --> B["Layout.vue\n(無 accept)"] B --> C["Sidebar.vue\n(有 accept ✓)"] C --> D["NavItem.vue\n(有 accept ✓)"] D --> E["Icon.vue\n修改這個！"] style E fill:#f85149,stroke:#f85149,color:#fff style D fill:#3fb950,stroke:#3fb950,color:#000 style C fill:#58a6ff,stroke:#58a6ff,color:#000 style B fill:#21262d,stroke:#30363d,color:#8b949e style A fill:#21262d,stroke:#30363d,color:#8b949e
🔍
傳播邏輯
修改 Icon.vue → 往上找 boundary → NavItem.vue 有 accept() → 在這裡停止，只更新 NavItem.vue 及其子樹。
如果 NavItem.vue 也沒有 accept，則繼續往上找 Sidebar.vue，它有 accept，在此停止。
如果一直找到根都沒有 accept → full page reload。

Vue Plugin 自動注入 accept()

Vue 的 Vite plugin (@vitejs/plugin-vue) 會自動為每個 .vue 檔案末尾注入 HMR 代碼，所以開發者通常不需要手動寫：

編譯後的 Button.vue（簡化） JavaScript 
// 你寫的 SFC 內容（編譯後）
const __default__ = {
  setup() { /* ... */ }
}

// @vitejs/plugin-vue 自動注入的 HMR 代碼 ↓
if (import.meta.hot) {
  __default__.__hmrId = "7ba5bd90"  // 元件唯一 ID

  // 在全域 HMR 系統註冊這個元件
  __VUE_HMR_RUNTIME__.createRecord(__default__.__hmrId, __default__)

  // 宣告這個模組接受自身更新（self-accepting）
  import.meta.hot.accept((mod) => {
    if (!mod) return  // 模組被刪除

    const { default: updated } = mod

    // 通知 Vue runtime 更新這個元件的定義
    // Vue 會保留所有 reactive state，只替換 render/setup
    __VUE_HMR_RUNTIME__.rerender(updated.__hmrId, updated.render)
    __VUE_HMR_RUNTIME__.reload(updated.__hmrId, updated)
  })
}

手動宣告 accept — 純 JS/TS 模組

對於非 Vue SFC 的模組（如 Pinia store、utils），需要手動宣告：

src/store/user.ts TypeScript 
export const useUserStore = defineStore('user', {
  state: () => ({ name: 'admin', role: 'manager' }),
  // ...
})

// 手動宣告這個 store 接受 HMR
if (import.meta.hot) {
  // accept() 不傳 deps 參數 = self-accepting（接受自身更新）
  import.meta.hot.accept()

  // dispose() = 在舊模組被替換前做清理
  import.meta.hot.dispose(() => {
    // 清除 Pinia store，讓新模組重新初始化
    useUserStore().$dispose()
  })
}

accept() 的三種形式

寫法意義適用場景
hot.accept(cb) Self-accepting，模組接受自己的更新 Vue SFC（自動注入）、Pinia store
hot.accept('./dep', cb) 接受特定依賴的更新 需要對特定子模組更新作出反應
hot.accept(['./a','./b'], cb) 接受多個依賴的更新 聚合多個子模組的模組

WebSocket 通訊協議

Vite 的 HMR 完全靠 WebSocket 推送通知，不是傳模組內容，只傳「要去重新 import 哪個 URL」。

訊息格式

HMR WebSocket 訊息結構（TypeScript 型別） TypeScript 
// Server → Client 的訊息類型
type HMRPayload =
  | ConnectedPayload       // 連線建立
  | UpdatePayload          // 模組更新（最常見）
  | FullReloadPayload      // 整頁重載
  | CustomPayload          // plugin 自定義事件
  | ErrorPayload           // 編譯錯誤覆蓋層
  | PrunePayload           // 模組被刪除

interface UpdatePayload {
  type: 'update'
  updates: Update[]
}

interface Update {
  type: 'js-update' | 'css-update'
  path: string            // 模組路徑，如 /src/components/Button.vue
  acceptedPath: string    // 接受 HMR 的 boundary 路徑
  timestamp: number       // 用於 cache-busting，加在 import URL 後面
  explicitImportRequired?: boolean
}

interface FullReloadPayload {
  type: 'full-reload'
  path?: string           // 觸發 reload 的檔案路徑（debug 用）
}

interface ErrorPayload {
  type: 'error'
  err: {
    message: string
    stack: string
    id?: string
    frame?: string        // 錯誤的代碼框架（紅色高亮顯示在瀏覽器上）
    plugin?: string
    loc?: { line: number; column: number; file?: string }
  }
}

@vite/client 收到訊息後的處理邏輯

@vite/client 核心邏輯（簡化） JavaScript 
// WebSocket 訊息處理器（@vite/client 內部邏輯）
socket.addEventListener('message', async ({ data }) => {
  const payload = JSON.parse(data)

  switch (payload.type) {
    case 'connected':
      console.debug('[vite] connected.')
      // 啟動心跳 ping
      setInterval(() => socket.send('{"type":"ping"}'), __HMR_TIMEOUT__)
      break

    case 'update':
      // 通知所有監聽器（如 vite-plugin-vue-devtools）
      notifyListeners('vite:beforeUpdate', payload)

      await Promise.all(
        payload.updates.map(async (update) => {
          if (update.type === 'js-update') {
            return await fetchUpdate(update)  // 動態 import 新模組
          } else {
            // CSS update：直接操作 DOM 的  tag
            const el = document.querySelector(`link[href="${update.path}"]`)
            if (el) {
              el.href = `${update.path}?t=${update.timestamp}`
            }
          }
        })
      )

      notifyListeners('vite:afterUpdate', payload)
      break

    case 'full-reload':
      notifyListeners('vite:beforeFullReload', payload)
      location.reload()
      break

    case 'error':
      // 在頁面上顯示紅色錯誤覆蓋層
      createErrorOverlay(payload.err)
      break
  }
})

// 動態 import 更新後的模組
async function fetchUpdate({ path, acceptedPath, timestamp }) {
  const mod = hotModulesMap.get(path)  // 找到這個模組的 HMR 資訊
  if (!mod) return  // 尚未 accept，會觸發 full-reload

  const moduleMap = new Map()
  const isSelfUpdate = path === acceptedPath

  // 加上時間戳 → 繞過 HTTP cache
  const [acceptedPathWithoutQuery] = acceptedPath.split('?')
  const newMod = await import(
    `${acceptedPathWithoutQuery}?t=${timestamp}`
  )

  // 呼叫所有已登記的 accept callback
  for (const { deps, fn } of mod.callbacks) {
    if (deps.includes(acceptedPath) || isSelfUpdate) {
      fn([newMod])  // 傳入新模組，讓 callback 更新 UI
    }
  }
}

Plugin 系統與 HMR Hook

Vite Plugin 可以透過 handleHotUpdate hook 介入 HMR 流程，自訂哪些模組需要更新。

Plugin HMR Hook

Plugin handleHotUpdate Hook 介面 TypeScript 
interface Plugin {
  name: string

  // 當檔案變更時呼叫，可以：
  // 1. 回傳 undefined → 讓 Vite 按預設處理
  // 2. 回傳空陣列 [] → 阻止任何 HMR 更新
  // 3. 回傳 ModuleNode[] → 指定要更新的模組
  handleHotUpdate?(ctx: HmrContext): Array | void | Promise | void>
}

interface HmrContext {
  file: string            // 變更的檔案絕對路徑
  timestamp: number       // 變更時間戳
  modules: Array  // 受影響的模組
  read: () => string | Promise  // 讀取變更後的內容
  server: ViteDevServer
}

大型專案 規模不影響 HMR 速度 專案越大越慢

首次請求 需要時間（多個 HTTP 請求） 需要時間（bundle 很大）
Production Build 使用 Rollup 打包（非 Vite dev server） 使用 Webpack 打包
The main difference is that in Webpack, HMR propagates up through the module tree and may require a full reload if the update bubble can't be handled. In Vite, HMR is performed over native ESM, so Vite only needs to precisely invalidate the chain between the edited module and its closest HMR boundary. — Vite 官方文件

HMR 常見問題排障

症狀 原因 解法
改了檔案但頁面沒更新 WebSocket 斷線 開 DevTools Network → WS tab，確認 ws://5173 有連線
每次都 full reload 而非 HMR 找不到 HMR boundary 確認 .vue 有被 @vitejs/plugin-vue 正確處理；檢查 console 的 [vite] invalidate 訊息
console 有 [vite] connecting... WS 連線中斷後重試 通常是伺服器重啟，@vite/client 會自動重連，屬於正常
CSS 改了沒反應 SCSS 全域注入問題 檢查 additionalData SCSS 設定是否造成 transform 錯誤
透過 Nginx 代理後 HMR 失效 Nginx 沒有代理 WebSocket 在 location 加上：
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
Pinia store 更新後狀態消失 Store 沒有 HMR dispose 處理 在 store 末尾加 import.meta.hot?.accept(acceptHMRUpdate(useStore, import.meta.hot))
大量 vite: invalidate 造成慢 index file (barrel) 重新匯出太多模組 減少 index.ts 中的 re-export；或用 optimizeDeps.include 預先打包
