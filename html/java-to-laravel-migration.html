3. 可行性結論

綠燈 在新情境下：技術可行、工程效益合理
  • 不需要重造 Flowable（OA 流程引擎接手）→ 原本最大的紅燈消失
  • 不需要重造芋道 14 個 Starter（OA 已涵蓋通用基建）→ 第二大紅燈消失
  • 實際需翻譯的 Java 檔從 1,777 縮到 ~565，工作量降到三分之一以下
  • PostgreSQL 用得淺（搜遍全專案幾乎無 PG-only 語法），DB 翻譯成本極低
  • ERP 尚未上線，沒有 production data migration 包袱
注意 成敗關鍵：Laravel OA 流程引擎的能力是否對齊 Flowable

這是整個專案在「動工前」就必須驗證的事。若 OA 流程引擎缺關鍵能力（動態 approver、會簽、回退、業務 callback），可能導致策略需要重來。建議第一週做 PoC 驗證。

5. 風險分析

紅燈 已消除

原本獨立重寫情境下的兩個紅燈（Flowable 無替代、需重造 14 個 Starter），在「整合進 OA」情境下都消失。

黃燈 1 Laravel OA 流程引擎能力對齊（最關鍵）

必須在動工前驗證以下能力 OA 引擎是否具備：

能力是否必須用在哪
動態 approver（資料驅動決定下一關）✅ 必須採購單按金額分級審批
會簽（多人並行，全簽/任一簽）⚠️ 需確認多部門會簽
抄送 CC（不影響流程）⚠️ 需確認bpm_process_instance_copy
加簽 / 回退⚠️ 需確認退回上一關修改
業務 callback（流程結束更新業務單）✅ 必須所有 ERP 單據共用
流程綁定選單機制❓ 看需求同單據綁不同流程
子流程 / 跳轉 / 超時提醒❓ 看需求視實際使用

黃燈 2 使用者體系與資料權限對齊

  • ERP 業務資料的 creator_id、approver_id、dept_id 全部要對應到 Laravel OA 的 user 主檔
  • 芋道 RBAC、部門、職位 ↔ OA 既有角色定義 — 需 mapping 表
  • @DataPermission（按部門過濾資料）OA 若無對應機制，需自寫 Global Scope

黃燈 3 PHP 性能模型

  • PHP-FPM 為 per-request 短生命週期，不利常駐運算（需求預測批次、BOM 展開）
  • 對策：批次計算包成 Queue Job（Horizon），必要時上 Laravel Octane（Swoole / RoadRunner）
  • WHS 庫存併發扣減：用 Cache::lock 或 SELECT ... FOR UPDATE，務必壓測

綠燈 低風險項目

  • PostgreSQL → MySQL：用得淺，dump → 改 DDL → 重灌即可
  • REST API 契約：Vue 前端可保留，只要維持 CommonResult 結構
  • 漢堡王中繼 API：FeignClient → Guzzle，2~3 天工作
  • JWT 認證：直接吃 Laravel OA 既有 auth guard
  • 排程、Excel、檔案、Redis：Laravel 生態現成

6. 關鍵整合決策

決策建議說明
資料庫策略 OA 與 ERP 共用同一個 MySQL，ERP 表前綴 erp_pdm_、erp_whs_、erp_pmm_ 避免跨 DB JOIN 複雜度，前提是命名不衝突
流程引擎使用方式 業務單在 ERP 模組，發起審批呼叫 OA 引擎 API，結束 callback 回 ERP 更新狀態 OA 為單一流程事實來源
使用者 / 組織 ERP 不再有自己的 user/dept/role，全走 OA 主檔 避免雙系統使用者不同步
ID 策略 遷移時保留原 ID（不重編號） 過渡期跨模組引用（Java 端引用 PDM 食材）才不需 mapping
新功能寫哪 從遷移啟動日起，所有新功能一律寫在 Laravel Java 進入凍結維護，否則永遠追不上
BHM 是否轉 視業務是否解凍決定，可放到後期 純 CRUD，3~5 人天可補

8. 漸進式遷移策略（6 階段）

核心架構

┌─────────────┐ │ Vue 前端 │ └──────┬──────┘ │ ┌──────▼──────┐ │ Nginx Gw │ ← 路由表決定每個模組打到哪邊 └──┬───────┬──┘ │ │ ┌──▼───┐ ┌─▼────────┐ │ Java │ │ Laravel │ │ ERP │ │ OA + ERP │ │ (PG) │ │ (MySQL) │ └──────┘ └──────────┘

原則

  1. Vue 不分叉 — 前端代碼一份，靠 Nginx 切流量
  2. 每個業務模組一次性切換，不在單一模組內做雙寫
  3. 新功能一律寫在 Laravel，Java 進入凍結維護
  4. 跨模組引用走 ID 一致策略 — 遷移時保留原始 ID

階段 0：地基

1.5 ~ 2 人月

目的：架好「平台」，後面每個模組才能用同一套規範開發。

  • Nginx Gateway 路由表（初始全打 Java）
  • 用戶體系打通：ERP Java 改接受 Laravel OA 發的 JWT
  • 基礎建設：ErrorCode、CommonResult、BaseModel、PageResult、Response Macro
  • 業務單據 ↔ OA 流程綁定 helper（對應 MenuFlowProcessInstanceHelper）
  • DB 命名規約：erp_<module>_<entity>
  • 流程引擎能力 PoC：用假單據走完「建立 → OA 流程 → callback 回寫狀態」
風險閘門：PoC 通過才能進階段 1。若 OA 流程引擎缺關鍵能力，整個專案的時程基礎就錯了。

階段 1：BHM + PDM 主檔

2 ~ 2.5 人月

為什麼先做：純 CRUD、無流程、無跨模組寫操作依賴、變動最少。

  • BHM 客戶基礎資料（若解凍）
  • PDM 主檔：食材、產品、BOM、配方（不含需求預測計算）
  • 漢堡王中繼 API：Token Manager + 三條 API 翻譯成 PHP

遷移做法：

  1. PG dump → DDL 翻譯 → MySQL（保留原始 ID）
  2. Laravel 完成 CRUD 與 API
  3. Nginx 切換：/api/pdm/ingredient/*、/api/pdm/bom/* → Laravel
  4. Java 端對應 controller 標 @Deprecated，不再接受寫請求
  5. 未遷移的 Java 模組改用 Laravel REST API 取主檔

階段 2：WHS 庫存模組

2 人月

為什麼第二：庫存只依賴 PDM 食材主檔（已遷移），自身不涉複雜跨模組寫操作。

  • 倉庫、儲位主檔 CRUD
  • 進出庫單、即時庫存查詢
  • 盤點單（接 OA 流程）— 第一次驗證階段 0 的 helper
風險：庫存併發扣減。Java 端用 Redisson 鎖，PHP 改 Cache::lock 或 SELECT ... FOR UPDATE，務必壓測。

階段 3：PMM 採購模組

2.5 人月

為什麼這時做：PMM 驗收單會寫 WHS 庫存，先把 WHS 搞定，PMM 就能同進程呼叫 Service。

  • 廠商主檔、報價維護（金額類審批）
  • 請購 → 詢價 → 採購 → 驗收 四階段狀態機
  • 用 spatie/laravel-model-states 建模狀態機
  • 跨單據關聯（請購拆採購、採購分批驗收）

階段 4：PDM 計算層 / 需求預測

2 ~ 3 人月

為什麼最後做：計算密集，且依賴的 PDM 主檔、PMM 採購、WHS 庫存都已就位。

  • 漢堡王 API 拉銷售資料（階段 1 已完成基礎）
  • 需求預測計算（複雜數值邏輯，建議寫測試）
  • BOM 展開 → 原物料需求（樹狀遞迴）
  • 需求預測單審批 → 觸發 PMM 請購
  • 計算包成 Queue Job（Horizon），必要時上 Octane

階段 5：拆除 Java 後端

0.5 人月

前提：階段 1~4 全部驗收通過、UAT 至少跑過一個完整業務週期。

  • Nginx 路由表清空，所有 /api/* 指向 Laravel
  • Java 服務停機，保留映像檔 + DB snapshot 作為災難回退
  • PostgreSQL 退役，歷史資料 read-only 副本保留 6 個月
  • 更新 CLAUDE.md、CONTEXT.md 等文件

9. Nginx Gateway 設定

9.1 基礎：路徑分流（推薦從這開始）

# /etc/nginx/conf.d/erp-gateway.conf

upstream java_erp {
    server 127.0.0.1:48080;
    keepalive 64;
}

upstream laravel_erp {
    server 127.0.0.1:8080;
    keepalive 64;
}

server {
    listen       80;
    server_name  erp.example.com;

    client_max_body_size  50M;     # Excel 匯入用
    proxy_http_version    1.1;
    proxy_set_header      Host              $host;
    proxy_set_header      X-Real-IP         $remote_addr;
    proxy_set_header      X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header      X-Forwarded-Proto $scheme;
    proxy_set_header      Authorization     $http_authorization;  # JWT 必傳
    proxy_set_header      Connection        "";
    proxy_read_timeout    120s;

    # Vue 前端靜態資源
    location / {
        root   /var/www/erp-kingmaker/dist;
        try_files $uri $uri/ /index.html;
    }

    # 已遷移到 Laravel 的模組（階段 1：PDM 主檔）
    location ^~ /admin-api/pdm/ingredient/   { proxy_pass http://laravel_erp; }
    location ^~ /admin-api/pdm/bom/          { proxy_pass http://laravel_erp; }
    location ^~ /admin-api/bhm/              { proxy_pass http://laravel_erp; }

    # 還沒遷移的 PDM 子路徑仍走 Java
    location ^~ /admin-api/pdm/forecast/     { proxy_pass http://java_erp; }

    # 預設：所有 API 走 Java ERP
    location /admin-api/ {
        proxy_pass http://java_erp;
    }
}
指令作用
^~前綴匹配且優先於 regex，避免被後面 /admin-api/ 攔截
proxy_set_header Authorization必須，否則 JWT 會掉
proxy_http_version 1.1 + Connection ""啟用 upstream keepalive

切換做法：增加一行 location 後 nginx -s reload 即可零停機切換。

9.2 進階：百分比灰度

# 在 http {} 區塊裡定義
split_clients "${remote_addr}${http_user_agent}" $whs_backend {
    10%     laravel_erp;     # 10% 打 Laravel
    *       java_erp;        # 其餘打 Java
}

server {
    location ^~ /admin-api/whs/ {
        proxy_pass http://$whs_backend;
    }
}
注意：寫操作模組（建單、改單）不建議百分比灰度，同一筆資料可能被兩邊各寫一半。比較適合唯讀查詢。

9.3 進階：指定使用者灰度（內部測試）

# 從 Cookie 控制
map $cookie_erp_canary $whs_backend {
    default   java_erp;
    "new"     laravel_erp;
}

server {
    location ^~ /admin-api/whs/ {
        proxy_pass http://$whs_backend;
    }
}

QA 在瀏覽器手動設 Cookie: erp_canary=new，就能用同一個 URL 體驗新版。

9.4 推薦：map 集中管理路由

# 路由表
map $uri $erp_backend {
    default                              java_erp;
    ~^/admin-api/pdm/ingredient/         laravel_erp;
    ~^/admin-api/pdm/bom/                laravel_erp;
    ~^/admin-api/bhm/                    laravel_erp;
    ~^/admin-api/whs/                    laravel_erp;   # 階段 2 開放
    # ~^/admin-api/pmm/                  laravel_erp;   # 階段 3 再放開
    # ~^/admin-api/pdm/forecast/         laravel_erp;   # 階段 4 再放開
}

server {
    location /admin-api/ {
        proxy_pass http://$erp_backend;
    }
}

好處：所有遷移狀態一目了然，PR diff 也清楚（只動 map 那段）。

9.6 踩雷清單

問題原因對策
切換後 401 / 403Authorization 沒傳顯式加 proxy_set_header Authorization $http_authorization;
大檔上傳失敗預設 client_max_body_size 1M調大（範例給 50M）
Laravel 拿到內網 IP沒設轉發 headerX-Real-IP + X-Forwarded-For，TrustProxies 加 Nginx IP
location 命中錯誤沒加 ^~，被 regex 蓋掉一律加 ^~
WebSocket 502沒升級協議加 Upgrade / Connection header
cookie / CSRF 衝突兩邊都用 cookieAPI-only 用 Bearer Token；或分 cookie path
切換瞬間少數請求失敗upstream 還在處理reload 是安全的，壓力大時排低峰；或用 proxy_next_upstream
CORS header 重複兩個 upstream 都吐CORS 集中在 Nginx 處理，後端關掉

9.7 工作流建議

  1. 環境變數化 upstream：dev / staging / prod 同一份 config，靠 include 不同 upstream.conf
  2. 路由表進版控：erp-gateway.conf 進 Git，每次遷移就是一個 PR
  3. 觀測性：nginx-prometheus-exporter + Grafana 觀察每個 location 的 RT 與錯誤率
  4. 演練回退：切換 PR 同時準備好回退 PR，30 秒內能 revert + reload

10. 馬上可以做的事

  1. 架 Nginx Gateway（半天工）並把所有流量都先指到 Java — 這是後續切換的開關
  2. 跑階段 0 的流程引擎 PoC — 用一張假單據做完整 round-trip：
    • 業務單建立 → 呼叫 OA 流程 API 起單
    • OA 引擎走完流程
    • callback 回業務單 → 更新狀態
  3. 盤點 Flowable 實際用到的能力：閱讀 kingmaker-module-bpm-biz/listener 19 個 listener、MenuFlowProcessInstanceHelper、所有 runtimeService.startProcessInstanceBy* 的呼叫點，列出真正在用的能力清單
  4. 對 OA 流程引擎做能力盤點，把第 3 點的清單交叉比對，識別缺口
  5. 凍結 Java 新功能：除非阻斷性 bug，否則所有新需求都進 Laravel 待辦
