🔍 核心結論

  • 所有 BurgerKing 外部資料抓取均為 On-Demand（使用者觸發），無任何排程任務或背景自動同步。
  • 唯一有效快取是 BurgerKingTokenManager 的 JWT Token（ConcurrentHashMap，55 分鐘 TTL）。所有業務資料均為每次請求即時呼叫 BK API。
  • getRemoteGroupWithStoresInner() 上的 @Cacheable(cacheNames = "groupWithStoresInnerList#32h") 已被註解掉，實際上沒有任何資料快取。
  • getCompletedOrdersFilterCached() 方法名稱具有誤導性——實際上沒有 @Cacheable，每次呼叫都直接打 BK API。
  • 唯一持久化的操作是 GET /system/store/one-areas-with-stores-sync，會將 BK 的區域/門市資料同步寫入本地 ERP StoreDO 表。

// BurgerKingTokenManager.java private final ConcurrentHashMap<String, TokenHolder> tokenCache = new ConcurrentHashMap<>(); public String getAuthorization(HttpServletRequest request) { // 1. 先嘗試從請求 Header 取得（若有傳入則直接使用） // 2. 再檢查 Cache 中是否有有效 Token TokenHolder holder = tokenCache.get(KEY); if (holder != null && holder.expireAt > System.currentTimeMillis() + 60_000) { return holder.token; // 60秒緩衝，避免 token 在傳輸中過期 } // 3. 重新 Login 取得新 Token String token = loginAndGetToken(); long expireMillis = parseJwtExpireMillis(token); // Base64 解碼 JWT payload，取 exp 欄位 if (expireMillis <= 0) expireMillis = System.currentTimeMillis() + 55 * 60 * 1000; // 預設 55分鐘 tokenCache.put(KEY, new TokenHolder(token, expireMillis)); return token; }
⚠
注意：TokenManager 使用null呼叫（tokenManager.getAuthorization(null)）， 始終使用系統管理員帳號，不會使用當前登入使用者的憑證。BurgerKingFeignConfig 中的 RequestInterceptor 對所有 FeignClient 呼叫自動注入此 Token。

📅 排程任務確認

❌ 不存在的機制

  • 沒有 @Scheduled 任務抓取 BK 資料
  • 沒有 Cron Job 定時同步
  • 沒有 Spring Cache 有效快取業務資料（@Cacheable 已停用）
  • 沒有背景線程定期預取資料
  • BK 資料不會在應用啟動時預載

✔ 確認存在的機制

  • JWT Token in-memory 快取（55min TTL）— BurgerKingTokenManager
  • 所有業務資料為 On-Demand 即時抓取
  • 唯一持久化：/system/store/one-areas-with-stores-sync（手動觸發）
  • BK FeignConfig RequestInterceptor 自動注入 Bearer Token
  • ERP 在應用層做權限過濾（不依賴 BK 的 RBAC）

⚡ 效能影響分析

⚠
效能風險：由於所有業務資料均為即時抓取且無快取，以下情境可能導致較高延遲：
  • 需求預測分析（/product-recipe-analysis）：同時抓取 BK 銷售資料 + 大量本地 DB 查詢，且可能需要多次 BK 呼叫
  • 廠商報價匯入：每個廠商的 Excel 匯入都會呼叫一次 getAllAreasWithStores（建議：已有 comment 說明先取一次）
  • 門市清單頁面：group-with-stores-inner、group-summaries、stores-by-group 三個端點都各自呼叫 BK API，如果同時被前端觸發則產生 3 次重複呼叫
  • BK API 不可用時：無 fallback 機制，所有依賴 BK 資料的功能將直接失敗
💡
建議（恢復被停用的快取）： 在 DemandForecastServiceImpl.getRemoteGroupWithStoresInner() 上恢復 @Cacheable(cacheNames = "groupWithStoresInnerList#32h")， 可大幅減少 BK API 呼叫頻率（店鋪資料不頻繁變動）。 同時建議將 getCompletedOrdersFilterCached() 方法名更名為 getCompletedOrdersFilter() 以消除誤導。
